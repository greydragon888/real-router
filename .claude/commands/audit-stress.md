Проведи мутационно-валидированный аудит дисциминирующей силы стресс-тестов пакета (или одного файла) и приведи недискриминирующие тесты в порядок.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Путь к пакету (`packages/<name>`) — аудит всех `tests/stress/*.stress.ts`
- ИЛИ путь к конкретному `*.stress.ts` файлу
- Опционально `--report-only` — только отчёт, без правок (по умолчанию: аудит + правки)

Пример:
```
/audit-stress packages/sources
/audit-stress packages/core/tests/stress/dependencies-store.stress.ts --report-only
```

---

## Принцип

Покрытие НЕ измеряет дисциминирующую силу стресс-теста. Тест-«театр» проходит, даже когда защищаемое поведение полностью сломано. Прочитай в корневом `CLAUDE.md` буллет «Heap-threshold stress tests MUST have proven discriminating power» — это база. Цель аудита: для КАЖДОГО ассерта доказать (мутацией), что он падает, когда ломается то, что он клеймит охранять; иначе — починить или удалить.

НЕ трогай: уже-дискриминирующие тесты (healthy ≪ порог ≪ leak с запасом ≥3× обе стороны) и **детерминированные fake-timer тесты** (`vi.useFakeTimers`/`advanceTimersByTime` — виртуальное время не флакает).

## Классификация ассертов (определи тип КАЖДОГО)

1. **HEAP-THRESHOLD** — `expect(delta).toBeLessThan(N байт)` на `takeHeapSnapshot`/`heapUsed`.
2. **COUNT-LEAK** — счётчик listener/entry/subscription/cache после churn.
3. **CONCURRENCY/RACE** — инвариант после гонки.
4. **NO-CRASH THEATRE** — проверяет лишь «не бросило»/«нет console.error»/«нет unhandled»/«промис разрешился», без инварианта на РЕЗУЛЬТАТ. 🚩
5. **TAUTOLOGY** — `completed === N` (N = счётчик цикла), `name.length > 0`, и т.п. 🚩
6. **TIMING wall-clock** — реальный `Date.now`/`performance.now` + `< Xms`.
7. **TIMING fake-timer** — виртуальное время. Детерминирован → ОСТАВИТЬ.

## Протокол на каждый ассерт (кроме fake-timer)

1. **Измерь healthy:** временно замени порог на `throw new Error("PROBE_<uniq> " + <величина>)`, прогони тест в изоляции 3× (`pnpm vitest run --config vitest.config.stress.mts <file> -t "<name>"`), запиши min/med/max. Откати пробу СРАЗУ.
2. **Определи точную утечку/баг, который тест клеймит охранять.** Проверь structural-ловушки, которые делают тест невалидируемым:
   - **GC-masking:** цикл create→drop (новый router/компонент за итерацию, ссылка отброшена) → per-cycle утечка недостижима на снимке → heap её НЕ видит независимо от cleanup. Heap-проверка здесь принципиально не дискриминирует.
   - **Hard-cap:** структура ограничена (EventEmitter ≤10k listeners/event, deps-store ≤100, Map last-write-wins → одно поколение) → max-возможная утечка может быть НИЖЕ порога → театр by construction.
   - **N-too-low:** сигнал утечки остаётся в KB, ниже межтестового шумового пола (~100–300KB) → не отличить от шума. Поднять N (runtime <~2s).
3. **Мутация:** инъекцируй ИМЕННО тот failure-mode в src (удали `unsubscribe()`/`teardown()`/short-circuit, или удержи ссылку на поколение/состояние за итерацию). Прогони только этот тест → ассерт ДОЛЖЕН упасть. Откати src. Проверь `grep`-ом, что мутация/проба не осталась.
   - Если ассерт НЕ падает на мутации → он недискриминирующий.
   - Если ассерт падает на ТЕКУЩЕМ (корректном) коде → это, скорее всего, **реальный баг** — СТОП, сообщи, НЕ ослабляй ассерт, чтобы спрятать.

## Починка по типу (если не --report-only)

- **HEAP genuine-linear** (живой объект, утечка достижима): порог СТРОГО между измеренным healthy и смоделированной утечкой, запас ≥3× обе стороны. Инлайн-комментарий: healthy/leak/порог.
- **HEAP GC-masked / throughput:** heap не поймает per-cycle утечку — не притворяйся. Либо переведи на **count-based** ассерт по счётному прокси (listener count / cache size — он РЕАЛЬНО дискриминирует), либо порог = round(8–10× измеренного healthy) как честный throughput-гвард с комментарием «throughput guard, per-cycle cleanup проверяется функциональными тестами».
- **HEAP hard-cap-bounded:** убери heap-ассерт, оставь/добавь count/`has`-инвариант (cap/cleanup), который дискриминирует.
- **NO-CRASH / TAUTOLOGY:** добавь инвариант на результат (final state / parity URL↔state / точный счётчик вызовов / membership), не «отработало».
- **COUNT слишком свободный** (`≥N`, `>0` без верхней границы): ужесточи до `===` детерминированного значения ИЛИ добавь верхнюю границу (ловит дублирование).
- **TIMING wall-clock:**
  - запас ≥~100× (catastrophe-guard, не флакает) → можно оставить, опционально добавить op-count.
  - запас тугой (флакает под нагрузкой) → **конвертируй в детерминированный op-count** (заспай дорогую операцию, ассерть O(1)/вызов), ИЛИ удали, если корректность покрыта другими тестами.

## Жёсткие правила

- Порог ВСЕГДА ≥3× от макс. наблюдённого healthy (иначе вводишь новый флак). Никогда ниже healthy.
- Для concurrency с легитимно-плавающими счётчиками — инвариант / тугая безопасная граница, не хрупкий точный счётчик.
- НЕ ослабляй уже-дискриминирующий тест: если healthy≪порог≪leak с ≥3×, оставь, отметь «OK».
- Заголовки тестов держи правдивыми (обнови `× N cycles`, убери ложные «heap»/«leak» клеймы).

## Валидация и отчёт

1. Прогони ПОЛНЫЙ stress-suite пакета 3× зелёным (`pnpm vitest run --config vitest.config.stress.mts`). Heap-тесты стохастичны — прогоняй серийно, без параллельного контеншна (он зашумляет замер). Учитывай: property-тесты пакета могут флакать под turbo-контеншном — это не связано.
2. `grep` на остатки проб/мутаций (`PROBE_`, `MUTANT`, и т.п.) в `packages/**/*.ts` — ноль.
3. `git diff` затрагивает ТОЛЬКО `*.stress.ts` (никаких `src/`).
4. **Отчёт-таблица:** file | test | тип | старый | healthy (min/med/max) | мутация (падает?) | новый | вердикт (OK/CHANGED/REPORTED-BUG). Веди с любым найденным реальным багом.

## Замечание

Изменения только в тестах → changeset не нужен. Push не делай без явной просьбы.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Валидация п.4) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного теста/пакета (разовую) → в отчёт/память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ прогону на ДРУГОМ пакете.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
