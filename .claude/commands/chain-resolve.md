## Принцип — копать до КОРЕННОЙ ПОРОДЫ, не до первого «правдоподобного»

Вердикт `/debt-map` (STRONG/WEAK) и гипотеза корня в issue — **СТАРТ, не ответ**. Самый дорогой провал — остановиться на ПЕРВОМ корне, который «объясняет» членов: он часто сам симптом более глубокого. Цель — не «закрыть issue», а **дойти до владельца концерна, который гасит весь класс**.

**Признаки, что ты ещё НЕ на дне (не останавливайся):**
- предлагаемый фикс **ДОБАВЛЯЕТ слой/абстракцию** (Variant-A-запах) → глубже лежит «у концерна нет авторитетного владельца», а не «нет абстракции»;
- тело члена само ссылается «**same class as #X / same seam / parallel to #Y**» в ДРУГОМ слое → это сквозная **ОСЬ**, копай туда;
- **каноническое issue корня дисклеймит члена** («separate root cause / should not be folded in») → спин-офф, ОТДЕЛЬНЫЙ корень;
- «фикс» оставляет хоть один band-aid / не удаляет ни костыля → ты нашёл симптом, не владельца.

Канон-предостережение: cancellation-арка выглядела «done», но оказалась лишь ГРАНЬЮ оси A1 (тело #807 это и выдало). **«STRONG + унифицировано» ≠ «дно».**

## Фазы

**0. Загрузи цепочку как ГИПОТЕЗУ.** Прочитай issue + `project_architectural_debt_map` (память): члены, off-root, missed, оси, вердикт. Всё это подлежит ПЕРЕпроверке, не принятию.

**1. Читай ФИКСЫ и КОД, не заголовки.** Для каждого члена: тело (`gh issue view <n> --comments`) + фикс (`git log --all --grep "#<n>"`, `git show <sha>`) + текущий код. Re-derive корень из кода фиксов, НЕ из root-claim'а самого issue. Keystone часто НЕ `label:bug` (трекается как `[test]`/epic — ищи шире).

**2. Копай до дна (СУТЬ скила — глубина).** На каждый кандидат-корень спрашивай «это дно, или симптом глубже?» — code-grounded «5 почему». Не останавливайся, пока:
   - не найден РЕАЛЬНЫЙ владелец концерна (существующий `FSM`/`RoutesStore`/emitter/`onListenerError`/codec — **Тип 1**) ИЛИ доказано, что владельца нет (**Тип 2**);
   - не проверены сквозные само-ссылки тел (ось?) и дисклеймеры канона (спин-офф?);
   - не ясно, что именно УДАЛЯЕТСЯ (нет удаляемых band-aid'ов → это не дно, а симптом).
   **Эмпирика > рассуждения:** пиши одноразовые пробы и НАБЛЮДАЙ поведение (как coalesce/discrimination-пробы той сессии), не доказывай корень «в уме». Перед сносом любого band-aid — **sweep**: докажи, что он мёртв по ВСЕМ путям (урок replace→SUCCESS: «мёртвый» RecursionDepthError оказался живым).

**3. Адверсариальный flip-тест.** Прогони STRONG/WEAK на НАЙДЕННОМ глубоком корне (не на исходном). STRONG-доказательство = **convergence в коде** (фиксы разных issue правят ОДНО место) + существующий владелец. Параллельные скептик-агенты пытаются ОПРОВЕРГНУТЬ корень (читают тела+код, default-refute, цитаты `file:line`+commit). **Вердикт МОЖЕТ перевернуться (STRONG↔WEAK) — это успех скила, не провал.** На несущем не доверяй вердикту субагента — открой helper и протрассируй convergence сам.

**4. Реши артефакт по ГЛУБОКОМУ корню (не по метке issue):**
   - **STRONG + существующий владелец** → **RFC** (Тип-1 collapse: усилить владельца авторитетом, удалить ad-hoc-координацию + band-aid'ы; шаблон-эталон `navigation-cancellation-unification.md`) → `/brainstorm` (если есть развилка дизайна, как Variant A/B) → `/audit-rfc` → `/implement-rfc`.
   - **WEAK / независимые точечные** → `/bugfix` (или `/bugfix-batch`), без RFC-overhead.
   - **Спин-офф под-корни** → отдельные цепочки (новый issue / `/debt-map --verify`).
   - **Найдена более глубокая ОСЬ** → эскалируй: текущая цепочка — её грань; полное разрешение = ось (шире RFC или явная пометка оси в карте), не только эта цепочка.

**5. Исполни** артефакт через профильные скилы (`/implement-rfc` / `/bugfix`). При `--dry` остановись на решении (корень + вердикт + план артефакта).

**6. Замкни контур.** Обнови `project_architectural_debt_map` (вердикт подтверждён/перевёрнут, новые оси/спин-оффы, закрытая грань). Подготовь комментарий в issue (корень, артефакт, что осталось осью). Кросс-референс соседних issue. Push/commit/`gh issue create`/комментарий — только по явной просьбе.

## Anti-stop (не останавливайся на простом выводе — это про что весь скил)
- Принял вердикт/гипотезу issue за факт → стоп, это старт.
- Первый корень «объясняет» членов → спроси «а он сам не симптом?» (Variant-A-запах / ось-ссылка в теле / дисклеймер канона).
- Предложение ДОБАВЛЯЕТ слой → ты не нашёл владельца (копай: кто ДОЛЖЕН этим владеть и уже существует?).
- «Done» без проверки, что унификация — не грань более глубокой оси.
- Один правдоподобный агент-вердикт → перепроверь сам в коде (convergence).

## Жёсткие правила
- Вердикт и корень из issue — ГИПОТЕЗА; re-derive из КОДА фиксов, не из заголовка/тела-claim'а.
- Корень доказывается **convergence в код**, не темой; на flip-тесте default = refute.
- **Тип 1 (усилить СУЩЕСТВУЮЩЕГО владельца, ядро ↓) > Тип 2 (новый примитив, fallback).** «Добавляю слой вместо удаления координации» = не дно (Variant-A).
- Эмпирика (пробы/sweep) > рассуждения на несущем.
- Не тащи WEAK в RFC; не плоди issue под недоказанный корень; не дублируй существующие.
- Push/commit/issue/комментарий — по явной просьбе.

## Оговорки (НЕ ошибка)
- **Перевёрнутый вердикт** (STRONG→WEAK или наоборот по итогам раскопки) — успех скила, не провал.
- **WEAK→bugfix** — валидный, частый исход; не натягивай унификацию ради унификации.
- **Найденная ось** важнее «закрытия» одной цепочки (плечо × пакеты).
- Раскопка может **расщепить** цепочку на под-корни — это норма, не неудача.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце каждого прогона)

Этот файл живой. После артефакта вынеси секцию **«Самокоррекция»** — критику не своей работы, а САМОГО СКИЛА: где его текст промолчал/соврал/был двусмыслен и это стоило времени или дало неглубокий/ложный корень.

1. **Привязка к инциденту этого прогона**, не общие советы: «скил сказал X / промолчал про X → реальность Y → остановился рано / чуть не принял симптом за корень на Z». Нет инцидента — «правок нет».
2. **Высокая планка:** правка, только если изменила бы исход (предотвратила бы остановку на симптоме, перебор, ложный артефакт).
3. **Разовое (специфика цепочки) → в карту долга/память, НЕ в скил.** В скил — повторяющийся класс, полезный СЛЕДУЮЩЕЙ раскопке.
4. **Сначала заточи существующий абзац, потом дописывай** (анти-раздувание).
5. **Предлагай таблицей (раздел | инцидент | правка), спроси разрешения.** Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
