Улучши mutation score пакета на основе логов прогона: классифицируй КАЖДОГО выжившего мутанта, убей killable точечными тестами, докажи и точечно подави эквиваленты, не трогая src-поведение.

Входные данные:
$ARGUMENTS

Формат аргументов (цель — любое из):
- **Путь к пакету** (`packages/<dir>`) — берёт его `reports/mutation-report.json`
- **Имя пакета** — npm-имя (`@real-router/logger`, `search-params`) ИЛИ короткое имя директории (`logger`, `path-matcher`)
- **Путь к конкретному** `reports/mutation-report.json`
- Опционально:
  - `--report-only` — только классификация + отчёт, без правок (по умолчанию: классификация + правки)
  - `--file <src/...>` — сфокусироваться на одном мутируемом файле
  - `--no-disable` — не добавлять `// Stryker disable` (только убивать killable)

**Резолв имени → директории** (имя ≠ путь и не `.json`):
1. Сними scope если есть (`@real-router/logger` → `logger`), попробуй `packages/<это>`.
2. Если такой директории нет — **матчи по полю `name`** в `packages/*/package.json` (директория может отличаться от npm-имени: напр. `@real-router/types` лежит в `packages/core-types`). Однозначного совпадения нет → СТОП, спроси.
3. Дальше работаешь с `packages/<резолв>/reports/mutation-report.json` (как в Шаге 0; нет файла → СТОП).

Пример:
```
/mutation-score @real-router/logger
/mutation-score search-params --report-only
/mutation-score packages/core --file src/api/getRoutesApi.ts
/mutation-score packages/path-matcher/reports/mutation-report.json --report-only
```

## Принцип

Mutation score измеряет, **ассертят** ли тесты поведение, а не **исполняется** ли код. Выживший мутант = строка исполнена, но ни один ассерт не падает при её порче. 100% строкового покрытия сосуществует с дырами: `?.`-short-circuit, не-проверенные сообщения, не-наблюдаемые ветки.

Цель — **не гнаться за 100%**. Потолок честного score ≈ 88–92%: значимая доля выживших — **эквиваленты** (мутант не меняет наблюдаемого поведения, убить корректным тестом НЕЛЬЗЯ). Цель: для КАЖДОГО выжившего вынести вердикт (killable / equivalent / suspected-bug / intentional) и действовать по классу. «Закрыть» эквивалент — это `// Stryker disable` с обоснованием ИЛИ документирование, **а не тест-театр** (тест, пиннящий поведение, который пройдёт и со сломанным кодом).

**Скептицизм в ОБЕ стороны.** Чтение кода систематически переоценивает критичность (defense-in-depth даёт эквиваленты) И недооценивает (рутина прячет реальные дыры). Единственный арбитр — **эмпирическая мутация** (ручная инъекция + полный сьют), не рассуждение. Перед объявлением «баг» — проверь design-intent (рядом лежащие тесты, комментарии, issue): сильное «воспроизводимое» подозрение бывает намеренным поведением.

**Третий класс системной ошибки: вердикт «недостижимо публично».** Read-based «эту ветку публично не достать» так же ненадёжен, как read-based «критично/некритично» (урок сессии 2026-06: из 7 вердиктов «недостижимо» **устояло 0** — все либо достижимы через пропущенный механизм, либо мёртвый код, где вход не доходит до ветки). Та же дисциплина: эмпирика (перебор ВСЕХ механизмов + проба покрытия конкретной строки), не аналогия. Даже одна «проверка» обманчива — пробуешь 2 механизма, пропускаешь 3-й; планка — перебор всех вызывающих и всех публичных точек входа.

## Шаг 0 — данные (источник правды — JSON, не incremental)

1. **Источник — `reports/mutation-report.json`. Файла нет → СТОП.** Не парси HTML-отчёт (Stryker встраивает данные в HTML как JS-литерал с конкатенацией строк, `"path.length <"+"= 1"` — это НЕ валидный JSON, и обходные экстракторы — пустая трата времени). Если `reports/mutation-report.json` отсутствует — **прекрати работу и сообщи** пользователю: добавить `json`-reporter в `stryker.config` и прогнать `test:mutation`. Конфиг:
   ```js
   reporters: ["progress", "clear-text", "html", "json"],
   jsonReporter: { fileName: "reports/mutation-report.json" },
   ```
   Для **верификации** после правок (Шаг «Валидация») цифры бери из **clear-text stdout** прогона (он всегда свежий: killed/survived/Ignored) — не из файла, который частичный `--mutate`-прогон может не перезаписать. Для пер-мутантной детали остатка нужен свежий JSON (полный прогон).
2. **НЕ доверяй статусам существующего отчёта — он мог быть incremental.** Incremental-кэш отдаёт **устаревшие** статусы: мутант помечен `Survived`, хотя существующие тесты его УЖЕ убивают (просто кэш не переоценил). Если базовый `reports/mutation-report.json` сделан с `incremental: true` (см. конфиг), его пер-мутантные `Survived` **ненадёжны** — часть «выживших» уже мертва. Последствие, если поверить: пишешь лишние kill-тесты под уже-убитых мутантов. **Перед классификацией — получи СВЕЖИЙ non-incremental отчёт.**
   - **Форсирование свежести: УДАЛИ кэш-файл** `rm -f .stryker-tmp/incremental.json` (путь из конфига, поле `incrementalFile`). Один лишь `--incrementalFile /tmp/throwaway.json` НЕ гарантирует свежесть (Stryker может всё равно подхватить кэш из конфигового пути) — проверено, не сработало; удаление файла — надёжно. `--incremental false` НЕ используй: Stryker трактует `false` как glob-паттерн `--mutate` → пустой прогон.
   - Для финальной цифры — полный `stryker run` без `--mutate` (кэш удалён → свежий).
   - **Запускай из директории ПАКЕТА** (`cd packages/<name>`), не из корня монорепо: из корня Stryker копирует в sandbox посторонние корневые артефакты (сокеты вроде `.codegraph` и пр.) и падает с `ENOTSUP`/copyfile.
   - **Дешёвая проба «а не убит ли уже?»** на подозрительном `Survived` из stale-базы: инъецируй мутацию, прогони сьют. Red → уже покрыт существующими тестами (не пиши новый тест — это был stale-артефакт). Green → действительно выжил, действуй по классу.
3. **Stryker видит только то, что в его vitest-include** (обычно `*.test.ts`). `*.properties.ts` / `*.stress.ts` ему НЕвидимы — инварианты, проверяемые только там, для Stryker = дыры. Целевые kills клади в `*.test.ts` (или зеркаль ключевые примеры из property).
4. **`coverageAnalysis: "perTest"` не атрибутирует module-level код.** Литералы на уровне модуля (regex-/массив-`const`, вычисляемые при import) не привязываются ни к одному тесту → их мутанты выходят как `cov=0` **Survived**, даже когда поведение реально покрыто (тест гоняет функцию, использующую этот regex). Не спутай с настоящей NoCoverage-дырой: проверь инъекцией (red на полном сьюте = покрыто, просто не атрибутировано). Что даёт `coverageAnalysis: "all"`: гоняет ВСЕ тесты против КАЖДОГО мутанта (без per-test фильтра) → такие module-const мутанты честно убиваются и пробел закрывается; цена — дольше прогон (нет per-test отсечения). Применяй точечно/для финального прогона, если такие `cov=0`-артефакты мешают классификации. **НЕ путай с `ignoreStatic: true` — это ДРУГОЙ механизм.** При `ignoreStatic:true` (частый дефолт, напр. route-tree без `STRYKER_FULL=true`) static-мутанты (regex/const-литералы, default-значения, всё вычисляемое при загрузке модуля) выходят как **`Ignored`**, а НЕ `Survived` — они исключены из знаменателя, классифицировать нечего. Сценарий «module-const regex даёт `cov=0` **Survived**» возникает только при `ignoreStatic:false`. Перед классификацией const/regex/default-выжившего сверь режим в `stryker.config`: при `ignoreStatic:true` его в Survived и не будет; если такой ВСЁ ЖЕ в Survived — значит static включены, и это настоящий перебор покрытия.
5. **Агрегация — `pnpm mutation:analyze <pkg|packages/dir|path-to-report.json>`** (`scripts/mutation-analyze.mjs`). Считает статусы (Killed/Survived/NoCoverage/Timeout/CompileError/Ignored), `Score=Detected/Valid` (`Detected=Killed+Timeout`, `Valid=Detected+Survived+NoCoverage`; CompileError + Ignored вне знаменателя), **per-file score**, и группирует каждого Survived/NoCoverage по файл+строка+**КОЛОНКА**+мутатор с дословным `replacement`, помечая **DISABLE-SAFE** (нет killed-собрата этого мутатора на строке — `disable` не уронит kill, выдаёт готовую строку) / **ENTANGLED-KEEP** (killed-собрат есть → подавлять нельзя, документируй). `--lines <file>:n,n` — дамп ВСЕХ мутантов строк (killed+survived+ignored) для ручной сверки. ⚠ **DISABLE-SAFE — СТРУКТУРНЫЙ вердикт (что МОЖНО подавить не теряя kill), НЕ «эквивалент»**: равенство поведения доказывай инъекцией (Эмпирический протокол) и только потом disable — иначе disable-театр. **Почему колонка обязательна** (это и считает тул): на одной строке несколько sub-expression мутантов одного мутатора в разных колонках с РАЗНЫМИ статусами — `a && b && c` даёт `ConditionalExpression→true` для всего условия (col X, killed) И для операнда `c` (col Y, survived); без колонки сольёшь killed и survived и будешь час гоняться за «противоречием Stryker». (Тул недоступен → тот же расчёт node-скриптом по JSON; формула и группировка выше.)

## Классификация выживших (вынеси класс КАЖДОМУ)

1. **KILLABLE** — тест не проверяет наблюдаемый эффект мутации. Чаще всего:
   - StringLiteral в сообщении ошибки / `caller`/`methodName` аргументе → тест есть, но `toThrow(TypeError)` без текста. 🚩
   - structural input-guard → невалидные входы (null/массив/примитив/битое поле) не подаются.
   - алгоритм (segments/ids/diff) → проверяется через `navigate()`, но не через точные массивы.
   - cleanup/one-shot logic → side-effect не считается (spy/membership).
   - validator call-site (`ctx.validator?.ns.fn(arg,"m")`) → в core-сьюте `validator===null`, ветка short-circuit'ится.
2. **EQUIVALENT** — мутант не меняет наблюдаемого поведения; убить корректным тестом НЕЛЬЗЯ. Подклассы:
   - **defensive-redundancy** — инвариант держат ≥2 механизма (concurrency: navigationId + signal.aborted + isActive; cleanup-guard, чей эффект дублирован downstream). 🚩 частый ложный «критический баг».
   - **entangled object-guard** — `→false`-операнд цепочки `a || b || c`, перекрытый соседним операндом/downstream-проверкой; его `→true`-сосед на той же строке **killed**.
   - **fast-path ↔ fallback** — ветка-оптимизация, general-путь даёт идентичный результат (`nameToIDs` fast-paths, `rest.length===0` vs базовый case).
   - **boundary-on-min** (`i<maxI`→`<=`: индекс на min даёт `undefined`, сравнение возвращает тот же индекс), **cache short-circuit** (`if(cached) return` → пересчёт того же), **idempotent merge** (`{...d,...{...d,...p}}`≡`{...d,...p}`), **`length>0`→`>=0`** где 0-случай no-op (emit на 0 слушателей), **`++`→`--`** для уникального id.
   - **equivalent-by-visibility** — мутант **функционально** ненаблюдаем (никакой `*.test.ts` его не убьёт), но ловится `*.stress.ts`/`*.properties.ts` вне Stryker-include (heap-утечка cleanup-строки, инвариант проверяемый только генеративно). Строго это не «убить нельзя», а «убить нечем из видимого Stryker'у». Действие: `// Stryker disable` с reason, **цитирующим КОНКРЕТНЫЙ** stress/property-тест, что его держит (напр. «leak guarded by stress S2») — честное «covered elsewhere», не театр. Перед подавлением: (а) докажи функц.-ненаблюдаемость инъекцией (полный `*.test.ts`-сьют зелёный); (б) убедись, что мутацию реально ловит названный тест (его инвариант/порог дискриминирует — иначе это NoCoverage-дыра, а не covered). Если дёшево зеркалится в быстрый `*.test.ts` (без heap/introspection) — лучше УБЕЙ там, чем подавляй.
3. **LOW-IMPACT** — ObjectLiteral error-metadata (`{routeName}`→`{}`), StringLiteral логов/сообщений, не влияющие на поведение. Killable (assert текст), но ROI низкий.
4. **INTENTIONAL NoCoverage** — `/* v8 ignore @preserve */` defensive; validator-path (валидация opt-in в отдельном пакете); default-параметр (Stryker-артефакт атрибуции `cov=0`). НЕ дыра.
5. **SUSPECTED BUG** 🚩 — мутант указывает на реальную дыру в ЛОГИКЕ (не в тесте). СТОП — см. протокол.

## Эмпирический протокол (арбитр killable vs equivalent)

Для любого неочевидного выжившего (особенно concurrency / cleanup / defensive-guard / «похоже на баг»):

1. **Инъекцируй ровно эту мутацию в src** (по `replacement` из лога: `if(false)`, `===`→`!==`, удали тело блока, `→true`).
2. **Прогони ПОЛНЫЙ сьют пакета. «Зелёный/красный» — по числу упавших ТЕСТОВ, НЕ по exit-коду.** В пакетах с enforced 100% покрытия (тут — большинство) инъекция, отключающая ветку/fast-path/dead-branch, ВСЕГДА роняет coverage-гейт (`ELIFECYCLE: threshold not met`) — даже когда все ассерты прошли (ветка стала мёртвой → покрытие просело). Читай строку `Tests X passed (X)` / `0 failed`, а не код возврата `pnpm test`. Coverage-ELIFECYCLE на пробе эквивалента — ОЖИДАЕМ, не сигнал killable.
   - **Зелёный** (0 тестов упало) → эффект не наблюдается → **EQUIVALENT** (или непокрытый edge — проверь, есть ли наблюдаемый эффект В ПРИНЦИПЕ; если нет — эквивалент).
   - **Красный** (упал ≥1 тест) → эффект наблюдаем; если сьют уже ловит, а лог говорит Survived → это **incremental-артефакт**, перепрогони non-incremental. Иначе — KILLABLE, пиши тест.
3. **Откати мутацию НЕМЕДЛЕННО.** `grep` по `src/**` на остатки (`if (false)`, `PROBE_`, `MUTANT`, инвертированные операторы) — ноль. Сверь `git diff src` — пусто.
4. **Перед вердиктом «BUG»:** найди design-intent — рядом лежащий тест (`*-cleanup.test.ts`), комментарий, ссылку на issue, stress-коммент. Если поведение задокументировано → это фича, не баг (а survivors — узкий edge задокументированной фичи). Если intent НЕ найден и эффект вредный → **REPORTED-BUG**: СТОП, сообщи, не пиннь поведение и не ослабляй (фикс — отдельной TDD-задачей через `/bugfix`).

**Батч-проверка (ускорение).** Несколько **независимых** подозреваемых-эквивалентов инъецируй РАЗОМ и прогони полный сьют ОДИН раз: зелёный подтверждает ВСЕ сразу; красный → изолируй по одному. Условие — непересекающиеся пути (иначе один маскирует другой). Сильный приём: подтвердить целую оптимизацию одной инъекцией её отключения — напр. `if (false)` на ОБЕИХ гард-строках static-кэша; сьют зелёный (coverage падает, поведение идентично) ⇒ весь кэш — чистая оптимизация ⇒ все его мутанты эквивалентны.

  **Два капкана ложного RED (батч соврёт «killable», хотя каждый эквивалент):**
  - **Defensive-пары.** Два гарда держат ОДИН инвариант, каждый эквивалентен в одиночку, но инъекция ОБОИХ сразу ломает (`writable`+`configurable` у `defineProperty`; `length===0`+`start===length`; `?? 0`-fallback + bounds-check). Stryker мутирует по ОДНОМУ → каждый — настоящий эквивалент. Подозреваешь пару (общий инвариант) → верифицируй ЧЛЕНОВ по отдельности, не в батче.
  - **Сдвиг строк.** Несколько УДАЛЕНИЙ строк в ОДНОМ файле сбивают нумерацию (удалил L48 → бывшая L56 теперь L55). Делай все удаления одним проходом по ИСХОДНЫМ номерам (`perl -ne 'print unless ($.==48 || $.==56)'`) или снизу-вверх; не цепочкой отдельных правок. После инъекции — `sed -n` по целевым строкам, убедись, что испортил именно их.

## Действия по классу (если не --report-only)

- **KILLABLE** → точечный тест (паттерны ниже). Цель — ассерт на НАБЛЮДАЕМЫЙ эффект, не на исполнение. Для kill'а, зависящего от **runtime-семантики** (frozen-ness, case-sensitivity, порядок, тайминг — где твоё рассуждение о поведении мутанта может быть НЕВЕРНЫМ), сделай дешёвую **пробу kill'а ДО дорогого Stryker**: инъецируй ровно эту мутацию, прогони сьют, убедись что падает **РОВНО твой новый тест** (и никакой другой — иначе тест бьёт мимо или ловит соседний мутант). Падает он один → kill дискриминирует, откатывай инъекцию. Для детерминированных killable (точный текст ошибки, `toThrow("...")`) проба не нужна — string-сравнение убьёт гарантированно.
- **EQUIVALENT** → точечный `// Stryker disable next-line <mutators>: equivalent — <причина>` (правила ниже). Если entangled — НЕ подавляй, документируй.
- **LOW-IMPACT** → assert текст/метаданные, если дёшево; иначе оставь, отметь.
- **INTENTIONAL** → оставь; опционально `// Stryker disable` чтобы убрать из знаменателя (для v8-ignored / default-param).
- **SUSPECTED BUG** → REPORTED-BUG, СТОП (см. протокол).

## Стратегия при большом объёме выживших (100+)

Поштучная upfront-классификация всех выживших не масштабируется в один контекст. Работай **kills-first эмпирическим циклом** — пусть Stryker сам сортирует killable от equivalent:

1. **Триаж по файлам.** Бери файлы по очереди (маленькие/худший-score сначала — быстрый импульс и валидация подхода). Веди задачу на файл.
2. **Сначала kills, потом классификация ОСТАТКА.** Для файла набросай точечные тесты на ЯВНЫЕ killable (наблюдаемый эффект через публичную поверхность) → прогони targeted Stryker (`--mutate "<file>"`) → он переведёт killable в **Killed**, оставив в Survived ровно equivalent + неочевидное. Классифицируй и подавляй/документируй уже этот **сократившийся** остаток. Это дешевле и эмпирически честнее, чем рассуждать о каждом мутанте заранее.
3. **Батч-инъекция эквивалентов** (см. Эмпирический протокол) — подтверждай группу независимых эквивалентов одним прогоном.
4. **Ограничь объём.** После kills + disable доказанных эквивалентов — СТОП по файлу: остаток (entangled / intentional / low-impact / сложно-killable, требующий экзотических входов) **документируй** в аудите как явный follow-up, не закрывай ценой театра или дня на один мутант. Лучше частичный честный результат, чем замаскированный.
5. **Финальная верификация — пакетом** (targeted прогон затронутых файлов разом), цифры из clear-text (это per-file score по `--mutate`-файлам, НЕ package-score — для общего числа см. Валидация п.3).

## Точечный disable (не слепой, не теряющий покрытие)

- **НЕ** `mutator.excludedMutations` в конфиге — отключает мутатор глобально, теряет десятки честно-убитых того же класса. Только inline.
- Синтаксис: `// Stryker disable next-line <Mutator1,Mutator2>: equivalent — <доказанная причина>`. Причина — обязательна и конкретна.
- **Перечисляй ТОЛЬКО мутаторы, у которых на этой строке НЕТ killed-варианта** (сверь по логу статусы всех мутантов строки). Пример: на строке `if (a !== b)` `EqualityOperator ===` killed, а `ConditionalExpression →false` + `BlockStatement` survived-эквиваленты → подавляй `ConditionalExpression,BlockStatement`, НЕ `EqualityOperator`.
- **Entangled**: один мутатор даёт И killed, И survived-эквивалент на одной строке — подавить нельзя (потеряешь kill). Оставь survived, задокументируй в аудите. Два под-случая:
  - **по значению мутации** на одной позиции: `ConditionalExpression →true` killed + `→false` survived.
  - **по КОЛОНКЕ**: `disable next-line <Mutator>` глушит ВСЕ мутанты этого мутатора на строке, **независимо от колонки**. Если тот же мутатор имеет killed-вариант в ДРУГОЙ колонке той же строки (sub-expression) — подавлять нельзя. Пример: `const r = type === "object" && v === null ? "null" : type;` — `ConditionalExpression→true` целого тернарника (col тернарника) killed, а `type === "object"→true` (col под-выражения) survived-эквивалент; общий мутатор `ConditionalExpression` → НЕ подавляй, документируй.
- **Конфликт с `eslint-disable-next-line`**: Stryker `disable next-line` пропускает строки-комментарии и целит следующую КОДОВУЮ строку — ставь Stryker-коммент ВЫШЕ eslint-коммента. Проверь прогоном, что мутант ушёл в `Ignored`, а eslint не сломался.
- **Prettier переносит коммент посреди выражения.** Disable, поставленный внутри тернарника/цепочки (напр. перед `: foo.toLowerCase()`), prettier утащит за `:` на новую строку — `next-line` начнёт целить не туда. Ставь disable перед ЦЕЛЫМ стейтментом, и **после `lint --fix` перепроверь** позицию (мутант → `Ignored` на верификации).

## Паттерны kill

Примеры ниже сформулированы на API роутера (`navigate`/`state.transition`/`validator`) — это **иллюстрации механики**, а не требование, что пакет такой. Общий принцип везде один: **ассерт НАБЛЮДАЕМОГО выхода через настоящую поверхность пакета** (для чистой функции — её возвращаемое значение; для фасада — публичный вызов), а не факт исполнения. Переноси паттерн на поверхность своего пакета.

- **validator call-site** → spy-валидатор через `getInternals(router).validator = <spy>` (Proxy, лениво создающий `vi.fn()` per-метод). НЕ ставь реальный validation-plugin (цикл зависимостей). Ассертируй `toHaveBeenCalledWith(arg, "caller")` — убивает И context-строку, И логику вокруг вызова. _(Обоснованный white-box — нет чисто-публичной поверхности; см. «Публичный контракт».)_
- **StringLiteral сообщения/caller** → ассерт ТЕКСТА: `toThrow("точный текст")`, `toHaveBeenCalledWith(..., "method")`. `toThrow(TypeError)` без текста НЕ убивает `""`-мутант.
- **structural guard** → через ПУБЛИЧНУЮ границу, где guard живёт: `createRouter([], {}, <badDeps>)` / `getRoutesApi(r).add([<bad>])` / `createRouter([], { logger: <bad> })` — по-одному невалидным входом (null / примитив / массив / битое поле) + текст сообщения. (Guard достижим публично — НЕ импортируй его из `src`; см. «Публичный контракт».)
- **алгоритм (segments/ids)** → ассерт ТОЧНЫХ массивов через ПУБЛИЧНУЮ поверхность: `state.transition.segments.{activated,deactivated,intersection}` после `navigate()` по иерархии 4-5+ уровней. `toStrictEqual` ловит off-by-one / отсутствие reverse / битый slice. Прямой вызов чистой функции из `src` — крайне редкое исключение; СНАЧАЛА исчерпай публичные двери к «внутренним» веткам: `start()` (initial nav) бьёт `fromState===undefined` fast-path; `router.shouldUpdateNode(router.makeState(name, params))` БЕЗ meta-аргумента подаёт meta-less состояния в `getTransitionPath` (его meta-less fast-path + `reverseArray`); `forwardTo` с роута на `/` даёт corner-матчи. В сессии 2026-06 ВСЕ подозреваемые «недостижимые pure-fn ветки» (вкл. `getTransitionPath` FAST PATH 1/3) оказались публично достижимы — прямой вызов не понадобился ни разу.
- **one-shot / cleanup / splice** → ассерт side-effect: spy-счётчик вызова (`toHaveBeenCalledTimes(1)`), membership (`has(name)===false`), длина реестра до/после. Двойной вызов проверяет идемпотентность.
- **reactivation / edge-условие** → сконструируй вход, где условие именно различает (route в `toDeactivate` И `toActivate` одновременно — смена `:param` у предка и т.п.).

## Публичный контракт, НЕ white-box (+ поиск мёртвого кода)

**Что есть «контракт» — зависит от пакета** (примеры ниже — иллюстрации, не привязка к конкретному пакету). Определи поверхность ОДИН раз перед killing:
- **Пакет с богатым публичным API** (роутер, плагин с фасадом): контракт — публичные точки входа (`navigate`/`buildPath`/`get*Api`/события). Прямой импорт внутренней функции, `getInternals`, namespace-классы, чтение приватных полей — 🚩 white-box (ниже — почему).
- **Внутренний пакет / пакет чистых функций** (matcher, кодеки, утилиты): для **чистой функции её вход→выход ЕСТЬ полный контракт** — прямой импорт из `src` и ассерт результата это **контракт-тест, НЕ тот white-box**, о котором предупреждает правило (нет bookkeeping, нет wiring, нечего пиннить кроме I/O). Это санкционированная конвенция, если она уже принята в пакете (напр. соседние unit-тесты импортят из `../../src`). Предпочитай всё же убивать через самую внешнюю достижимую поверхность (если внутренний хелпер достижим через публичный вызов пакета — бей оттуда, это доказывает и интеграцию).

Дальше — про первый случай (богатый API), где white-box опасен.

**Убивай через публичную точку входа, не прямым вызовом внутренней функции.** White-box-kill даёт ложную защищённость: (а) пиннит реализацию, не контракт; (б) ассертит **bookkeeping** (внутренняя map/массив/счётчик), а не то, что исполняет система — десинхронизация двух структур оставляет тест зелёным при сломанном поведении; (в) обходит wiring — изолированный прогон внутренностей не доказывает, что публичный путь их зовёт; (г) **подпирает 100%-coverage-гейт над мёртвым/недостижимым кодом.**

**Перед kill'ом — спроси: достижима ли ветка через публичный API?**
- **ДА** → убей через публичную точку входа. Доказывает контракт И убивает мутанта.
- **НЕТ** (никакая публичная точка не достигает/не ассертит эффект) → это **СИГНАЛ МЁРТВОГО / OVER-DEFENSIVE КОДА**, а не повод тестировать напрямую.

**«НЕТ» — высокая планка, доказывается перебором, не аналогией.** Read-based «эту ветку не достать» крайне ненадёжен. Прежде чем объявить недостижимой:
1. `grep` ВСЕ вызывающие функцию в `src/**`; для КАЖДОГО — какой публичный API его драйвит и каким входом.
2. Перебери ВСЕ триггер-механизмы пакета, не 1–2 (разные публичные методы, краевые/малформ-входы, sync-побочки, init-пути). Пробуй КАЖДЫЙ эмпирически: проба + покрытие конкретной строки.
3. Для «защитной» ветки проверь, доходит ли вход ДО неё: upstream-gate (внешний `if` отсёк falsy раньше) или TS-narrowing вызывающего (`x ? fn(x)`) фильтруют триггер до этой строки. «Функционально нужная защита» при отсечённом входе = мёртвый код, а не KEEP.

**Достижимость ≠ наблюдаемость.** Ветка может быть публично ДОСТИЖИМА, но конкретная деталь её выхода — нет; это НЕ повод для white-box. Типичный случай: внутренний transform (сортировка/реверс/нормализация), чей результат потребитель читает по членству, а не по порядку → порядок ненаблюдаем. Вывод: покрой НАБЛЮДАЕМУЮ часть публично, а ненаблюдаемую деталь классифицируй EQUIVALENT — ИЛИ удали ненужный transform из src (то, что никто не читает). Одна ветка через одну дверь даёт И killable, И equivalent.

**Поиск мёртвого кода (побочный продукт аудита).** Выживший в публично-недостижимой ветке часто = мёртвый код, который white-box-тест держит «covered». Протокол:
1. `grep` вызывающих функцию/параметр/поле в `src/**` по ПУБЛИЧНЫМ путям.
2. Только тесты / нет конструирующего кода → кандидат на удаление (мёртвый параметр без вызывающего; публичное поле, которое нигде не конструируется; ветка, достижимая лишь из теста).
3. Перед удалением — проверь design-intent (issue/комментарий/история; «зачем добавляли?»). Подтверждено мёртвым → **удали ветку**. **НЕ маскируй white-box-тестом** и предпочитай удаление `v8 ignore`-у. Toolkit упрощения (в порядке предпочтения):
   - **built-in уже обрабатывает кейс** → удали guard целиком (напр. `Object.freeze(null) === null` не бросает ⇒ `if (!x) return x` перед freeze избыточен).
   - **TS требует narrowing, но вход gated-out** → non-null assertion, документирующая инвариант вызывающего (`config!`, `arr.at(-1)!`) + `eslint-disable no-non-null-assertion -- <почему gate гарантирует>`. Чище `v8 ignore`.
   - **только при реальном TS-defensive без чистого упрощения** → `v8 ignore` (последний выбор).
   Публичное API → changeset + migration note. Перепись white-box→public ОГОЛИТ мёртвый код (coverage/branches упадут) — это нужный сигнал, не регресс; глобально score растёт (мёртвые ветки уходят из знаменателя).

**Обоснованный white-box (редкое исключение, KEEP) держится на СТРУКТУРНОМ барьере, не на «не нашёл публичный путь»:** напр. seam между пакетами через санкционированный subpath, или spy на точку, тестирование которой иначе тянет цикл зависимостей. Барьер тут — архитектурный (dependency cycle / sanctioned seam), доказуемый, а не «перебрал и не достал». Прежде чем заявить такой KEEP — пройди перебор выше; «недостижимо по перебору» почти всегда значит «мёртвый код».

## Жёсткие правила

- **Тесты ищут баги, не пиннят.** Если мутант эквивалент — `// Stryker disable`, НЕ тест, который «проходит и так».
- **Kill через публичный контракт, НЕ white-box** (в пакетах с богатым API). Прямой импорт внутренней функции/`getInternals`/namespace-класса ради kill'а — 🚩 запрещён. **Исключение:** чистая функция во внутреннем пакете — её I/O и есть контракт, прямой импорт из `src` легитимен (см. «Публичный контракт» — определение поверхности по типу пакета). «Недостижимо публично» доказывай перебором всех вызывающих + механизмов, не аналогией (см. «НЕТ — высокая планка»); подтверждено → мёртвый/over-defensive код (упрости по toolkit; `v8 ignore` — последний выбор), а не повод тестировать напрямую.
- **НЕ ослабляй и не удаляй src ради score** без доказательства мёртвого кода. Мёртвый код (напр. неиспользуемый возврат функции) — удаляй (это валидный kill), но докажи отсутствие потребителей (`grep`).
- **Откатывай ручные мутации немедленно**, `grep` остатки, сверяй `git diff src`.
- **Не гонись к 100%.** После закрытия killable + disable доказанных эквивалентов — СТОП. Остаток (defensive/entangled/low-impact) документируй, не выдумывай театр.
- **non-incremental** для любой финальной/сравнительной цифры.

## Валидация и отчёт

1. Полный сьют пакета зелёный. Если enforced 100% покрытия упало после перевода white-box→public или удаления кода — это ОГОЛЁННЫЙ мёртвый/недостижимый код: упрости его по toolkit (built-in / gated non-null assertion / удаление ненаблюдаемого transform), `v8 ignore` — последний выбор; НЕ возвращай white-box-тест ради покрытия.
2. Lint чистый (disable-комментарии длинные — проверь max-len; eslint-disable не сломан).
3. non-incremental прогон затронутых файлов (`rm -f .stryker-tmp/incremental.json` для свежести, затем `--mutate "<files>"`) → подтверди по **clear-text stdout** (он свежий; файл-репорт частичный прогон может не перезаписать): killable → Killed, эквиваленты → Ignored, **ни один killed-собрат не пропал**, число killed не упало. Если нужна пер-строчная детализация остатка (по колонкам!) — гони полный прогон ради свежего JSON и читай его через `pnpm mutation:analyze <pkg>` (даст score + остаток с DISABLE-SAFE/ENTANGLED вердиктами; `--lines <file>:n,n` сверит, что killed-собратья на затронутых строках целы). Сверь, что КАЖДЫЙ disable реально ушёл в `Ignored` (как в path-matcher: disable перед `} else if`-блоком/посреди тернарника может не приклеиться — см. prettier-капкан).
   - **ID и номера строк в свежем отчёте СМЕЩЕНЫ относительно базового** — добавленные disable-комментарии сдвигают все строки ниже (а Stryker перенумеровывает мутанты при каждом прогоне). Сопоставляй остаток по кортежу **`(file, mutator, replacement, status)`**, НЕ по id или ИСХОДНОМУ номеру строки — иначе спутаешь мутанты (напр. бывшая L157 после вставки коммента на L136 станет L158, и старый «[44] L157» укажет не туда).
   - **clear-text targeted-прогона даёт score ТОЛЬКО по `--mutate`-файлам** (свой знаменатель из этих файлов) — это НЕ package-score. Для headline before/after: либо полный прогон без `--mutate`, либо пересчёт от baseline (выжившие в нетронутых файлах не меняются: `новый Killed = baseline Killed + убитые`, `новый Valid = baseline Valid − removed-dead − disabled − перешедшие-в-Killed-как-было-Survived`, плюс заявленные `Survived = только остаток`). НЕ выдавай per-file targeted-процент за общий по пакету.
4. `grep` остатки проб/мутаций в `src/**` — ноль. `git diff src` — только намеренные (dead-code/disable, все no-op для runtime).
5. **Отчёт-таблица:** file | survived было→стало | killed | Ignored | класс остатка | вердикт (CLOSED / EQUIVALENTS-DISABLED / REPORTED-BUG). Веди с любым найденным реальным багом и его доказательством.
6. Обнови `packages/<pkg>/.claude/mutation-audit-*.md` (если есть): новый score, per-file расклад, классификация остатка, любые REPORTED-BUG. Не удаляй историю — добавляй веху.

## Замечание

- Изменения только в тестах → changeset не нужен. Изменения в `src/` (dead-code → `void`, `// Stryker disable`-комментарии) затрагивают публичный пакет → при коммите нужен changeset (даже если no-op для runtime). `stryker.config.mjs` / vitest-конфиги — infra, в master напрямую, без changeset.
- Push/commit не делай без явной просьбы. Финальный полный прогон — дорогой; предложи запустить, не запускай молча в долгом цикле без нужды.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Валидация п.5–6) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику пакета (разовую) → в `mutation-audit-*.md` / память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ прогону на ДРУГОМ пакете.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй. Скил уже длинный.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
