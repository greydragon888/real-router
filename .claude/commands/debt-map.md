Найди архитектурные корни роутера по заведённым багам: кластеризуй баг-историю в цепочки долга (по КОРНЮ, не по заголовку), адверсариально верифицируй ТЕЛАМИ issue + КОДОМ фиксов, вынеси вердикт STRONG/WEAK + сквозные оси, оформи research-артефакты (RFC если STRONG → /brainstorm→/audit-rfc→/implement-rfc; bugfix если WEAK). Цель — НЕ «починить N багов», а доказать, что они схлопываются в M owned-инвариантов, и обновить карту долга.

Входные данные:
- Без аргумента — полный анализ баг-истории → карта долга (`project_architectural_debt_map` в памяти). **Если карта уже существует — это дельта-сверка, НЕ пересборка:** новые баги с прошлого прогона + сироты прошлой кластеризации (систематически некластеризованные группы корпуса). Верифицированные цепочки не пере-открывать — они проверены `/chain-resolve` глубже, чем свежий прогон сможет. Пересборка с нуля — только по явному запросу пользователя.
- `<package>` / `<label>` / `<issue-range>` — сузить корпус (напр. `path-matcher`, `label:bug`, `#700-900`).
- `--verify <chain>` — прицельная адверсариальная ПЕРЕпроверка одной цепочки (перевернуть STRONG↔WEAK по коду).
- `--axes` — искать только сквозные оси (паттерны поверх цепочек), пропустив пер-цепочную классификацию.
- `--update` — обновить существующую карту новыми багами с прошлого прогона (не строить с нуля).

Пример:
```
/debt-map
/debt-map path-matcher
/debt-map --verify "Route-CRUD commit"
/debt-map --axes
```

## Принцип

Баги в зрелом роутере **не независимы** — они кластеризуются в **архитектурные корни** (отсутствующий инвариант / отсутствующий единый владелец концерна). На реальном прогоне 145 bug-issue свелись к ~10 цепочкам → ~6 owned-инвариантам + 5 сквозным осям. Ценность не в количестве фиксов, а в **схлопывании**: один владелец гасит класс багов и позволяет УДАЛИТЬ накопленные костыли.

Это **обратная археология** к `/audit-rfc` и `/brainstorm`: там валидируешь дизайн ВПЕРЁД (claim → первоисточник); здесь раскапываешь корни НАЗАД из симптомов (баг → общий код-корень). И тот же мутационный дух (`audit-pbt`/`audit-stress`): истинность «это корень» доказывается походом в **тела issue + код фиксов**, а не доверием к заголовку или к резюме субагента.

**Самый опасный режим — title/area-кластеризация.** Заголовки и метки систематически путают «пакет» или «волну аудита» (эпик, area-sweep) с единым корнем. На реальном прогоне часть «цепочек» оказалась тематическими area-группами, а не корнями. **Корень доказывается телами + кодом + сигналом «фиксы РАЗНЫХ багов сошлись в ОДИН helper».** Без этого — это группа по теме, не корень.

## КОРЕНЬ vs AREA-GROUP — сигналы (ядро метода)

**Корень (STRONG) — хотя бы один сильный код-сигнал:**
- **Convergence (сильнейший):** фиксы РАЗНЫХ багов приземлились в ОДИН helper/файл/функцию. (Пример с прогона: #1036 Route-CRUD — четыре commit-пути `add/replace/remove/clear/update` сошлись в `assertAddable`/`adoptRouteArtifacts`/`throwIfReentrantTreeMutation` → доказано КОДОМ.)
- **Whack-a-mole:** один и тот же класс бага рецидивит в РАЗНЫХ пакетах (фикс в одном — всплывает в соседнем). Сигнал отсутствующего общего примитива (пример: 3 реимпла isolation вместо переиспользования `onListenerError` — #1039).
- **Band-aid-накопление:** вокруг одного концерна налипли спец-кейсы/предохранители (признак отсутствующего single-owner). (Пример: `maxEventDepth`/`RecursionDepthError`/#308/#945/#935 вокруг отмены — все band-aid'ы одного отсутствующего владельца.)
- **Доказуемо нет инварианта:** нет enforced-свойства (`parse(build(x))==x` — #1037), нет единой точки эмиссии/коммита.

**Area-group (WEAK, НЕ корень) — объединяет только тема:**
- баги в одном пакете / одной волне аудита (эпик, dom-utils-sweep) / под одним заголовком, но механически разнородны;
- «фиксы» рассыпаны без общего helper (нет convergence);
- корень на поверку — 1–2 issue в одном сайте (пример: logger-singleton #1043 — реально только #724+#790, один сайт `Router.ts`).

**Правило-фильтр:** прежде чем назвать цепочку — спроси «что общего у фиксов в КОДЕ?». Общий helper/инвариант → корень. Общая тема/пакет → area-group. На реальном прогоне это перевернуло несколько кандидатов.

## STRONG-сигнал НЕОБХОДИМ, но НЕ ДОСТАТОЧЕН — 3 фильтра перед эмиссией STRONG

**Эмпирика сессии 2026-06-29: ВСЕ 5 STRONG-вердиктов карты — #1036 / #1037 / #1038 / #1039 / #1040 — перевернулись в WEAK на прицельном `/chain-resolve`.** Код-сигнал (convergence / whack-a-mole / band-aid / нет-инварианта) ловит «тут что-то есть», но НЕ отличает «живой корень под RFC» от «уже решено / Variant-A / расщепляется». Все ТРИ примера-сигнала выше (#1036 convergence, #1039 whack-a-mole, #1037 нет-инварианта) — ровно те, что флипнули: сигнал был верен, корня под RFC не было. Прежде чем эмитить STRONG, прогони три дешёвых фильтра — любой проваленный = WEAK:

1. **realized?** (ловил 3/5: #1036/#1037/#1040) — общий helper / инвариант УЖЕ существует и потребляется? Фиксы «сошлись в `assertAddable`» — но он УЖЕ извлечён; «нет `parse(build(x))==x`» — а он УЖЕ enforced property-тестом; «асимметричные дефолты» — а они УЖЕ single-sourced. **Реализованная convergence = ЗАКРЫТАЯ цепочка, RFC-payload ПУСТ → WEAK** (остаток = gap в существующем шве → bugfix, не новый примитив). Convergence-сигнал НЕОБХОДИМ, не достаточен: спроси «это уже сделано?».
2. **существующий владелец? Тип 1 vs Variant-A** (ловил #1039) — для whack-a-mole / missing-primitive: есть ли УЖЕ переиспользуемый владелец, НА которого схлопнуть? Нет → новый примитив НАРАЩИВАЕТ ядро = Variant-A = WEAK+bugfix. (`onListenerError` — emitter-internal private, не переиспользуем не-эмиттерными сайтами → «extract invokeSafely» = новый слой, НЕ collapse-onto-owner.) «3 реимпла» доказывают дублирование, НЕ существование владельца.
3. **примитив enforce'ит adoption?** (ловил #1039) — предотвратит ли НОВЫЙ примитив рецидив «забыли вызвать / изолировать»? Обычно НЕТ (helper существует ≠ сайты его зовут: `throwIfReentrantTreeMutation` существует, но hand-wired в 5 CRUD-ops — 6-я забудет). Структурный превентер «забыли» = lint / типы, **а для perf/hot-path осей — бенчмарк-гейт** (регрессия ловится метрикой, не статикой: cross-router `link-build`/`active-links` ловит ось «adapter reactive-bridge fan-out» #1099/#1103/#1247), — (tooling), НЕ примитив → WEAK+lint. «Рецидив класса» сам по себе ⇏ RFC.

**И проверь РАСЩЕПЛЕНИЕ до STRONG:** STRONG-цепочка часто = «owned-realized половина + несводимая-hand-coded половина (заблокированная дизайн-инвариантом) + cross-cutting члены ЧУЖИХ нитей». (#1038: `finalize`/`BaseSource` владеют scaffolding [realized]; error-ветки несводимы под non-terminal-error #775; #767 → isolation-нить #1039; #771 → ось A1.) Дели — не выдавай гибрид за одну STRONG.

## Протокол (фазы)

**0. Собрать корпус — тела, не заголовки.**
- `gh issue list --label bug --state all --limit 500` (+ **closed** обязательно — закрытые баги несут историю фиксов) → для несущих `gh issue view <n> --comments`.
- **Keystone цепочки часто НЕ `label:bug`.** Отсутствующий инвариант сам по себе трекается как `[test]`/`[docs]`/epic — невидимо для bug-корпуса, но это и есть КОРЕНЬ. Ищи шире `label:bug` (по теме цепочки: «property-oracle», «INVARIANTS», «atomicity», эпики аудита). На прогоне keystone симметрии search-params — #746 (`[test]`: «oracle зеркалит реализацию → все баги ушли зелёными») — был невидим bug-кластеру, хотя именно он объясняет ВЕСЬ класс (#387/#742/#743/#744).
- Для КАЖДОГО релевантного бага вытащи **фикс**: связанный PR/коммит (`gh pr view`, `git log --all --grep "#<n>"`, `git show <sha>`) — **diff фикса — это и есть код-сигнал корня**.
  - **Дельта СРАЗУ после аудит-волны — члены OPEN, фиксов ещё НЕТ.** Fix-diff-метод неприменим. Convergence тогда = ОБЩИЙ код-локус, который **тела сами цитируют** (`file:line`), + само-декларация семьи в теле («the #NNN family», «Nth facet of the same …», «same seam as …»), **верифицированные ЛИЧНЫМ чтением этого локуса** (открой файл — убедись, что фиксы разных issue правили бы ОДНО место). Это валидный code-signal взамен fix-diff — но именно ЧТЕНИЕ локуса, не доверие цитате в теле. Пример дельты №3 (2026-07-04): #1170/#1201/#1236 — три OPEN-бага, все цитируют `getRoutesApi.ts:477-496`; чтение блока (`nextState = {...revalidated, transition}` → `setState` без гвардов) подтвердило единый локус → «revalidation rebuilds from matchPath, dropping X».
- Fan-out `Explore`/`general-purpose`-агентов по кластерам багов (по пакету/подсистеме для ШИРОТЫ сбора, не для классификации). Агенты читают **рабочее дерево текущей ветки** (read-only), цитируют `file:line` + commit, возвращают тела+фиксы, НЕ заголовки.

**1. Кандидат-кластеры по подозреваемому КОРНЮ** (не по пакету/теме). На каждый — однофразовая гипотеза корня («нет единого владельца X», «нет инварианта Y»). **Инфра-баги (ci/cd, release, examples) — полноправный кандидат-кластер:** «роутерная» рамка системно их отбрасывает (прогон 2026-06-29 оставил весь CI-корпус вне кластеризации, хотя #730–735/#810–814 уже существовали; ось A6 «CI verification-model drift» нашлась только дельта-прогоном 2026-07-03).

**2. Адверсариальная верификация (fan-out, скептик на каждый кандидат-в-STRONG).** Каждый субагент-скептик: читает ТЕЛА issue + КОД фиксов и пытается **ОПРОВЕРГНУТЬ** «единый корень» — доказать, что это area-group. **Default = area-group, пока корень не доказан convergence.** Требуй цитат `file:line` + commit + вердикт STRONG/WEAK/PARTIAL. (На реальном прогоне так работали 5 адверсариальных субагентов.) **Скептик обязателен для кандидата в STRONG; для кластера, который сам классифицируешь WEAK/closed по трём фильтрам + личной трассе кода, — опционален:** роль скептика — понижать, а систематическая ошибка метода — over-call STRONG, не занижение (занижение позже ловится `--verify`).

**3. Сам перепроверь несущее — не доверяй вердикту субагента.** Открой заявленный общий helper и **протрассируй convergence**: реально ли фиксы разных issue правят ОДНО место? Нет convergence в коде → downgrade к WEAK/area-group, как бы убедительно ни звучал заголовок. (Это прямой аналог «несущий claim перепроверь компиляцией/прогоном» из `/audit-rfc`.)
   - **Overlap-баг (симптом ДВУХ корней) — назови PRIMARY по fix-locus** (куда падает однострочный фикс), второй корень — лишь тематический кросс-реф, не членство. (Пример: #787/#903 числились и в isolation, и в hardening; фикс — `Object.hasOwn`/typeof в диагност-хелпере → PRIMARY = hardening, isolation отпадает.)
   - **Reachability/severity-квалификатор проверяй ПОЧЛЕННО.** Если корень несёт оговорку «reachable via history.state/URL», «production-silent» — провер каждого члена против неё. Член, не дотягивающийся по квалификатору, — мисс-ассайн или сосед. (Пример: в «untrusted input» квалификатор держался лишь для ~3 из 10; Symbol #934 — developer-only, structured-clone его не пропустит → не корень-член.)

**4. Сквозные оси — поверх цепочек.** Найди паттерны, пересекающие НЕСКОЛЬКО цепочек/пакетов: **одна правка гасит баги в N пакетах** (для infra-оси единица — слой CI: turbo-hash / path-фильтр / планировщик / needs-агрегация / required-ruleset, не пакет). Оси **стратегически важнее слабых цепочек** (плечо). (Пример с прогона: A1 sync-commit-vs-deferred-microtask seam; A2 emission-discipline; A3 shared-state-не-SSR-safe; A4 prototype-key hasOwn; A5 core-не-гейтит-вход.) Ось может быть реальной даже когда соответствующая «цепочка» WEAK.
   - **Как искать (сильнейший приём): добывай само-кросс-ссылки из ТЕЛ.** Баги часто сами называют своих ось-собратьев — «same class as #X», «parallel to #Y», «sibling of #Z», «same seam as …». На прогоне вся ось A1 нашлась из ОДНОЙ фразы в теле #807 («same class as core `lastSync*`, rx `state$`») — без вычитки этих фраз ось пропускается целиком. `gh issue view <n>` + grep тел на «same class/seam/parallel/sibling».

**5. Шаблон схлопывания «следующий #1030» — два рода унификации, по умолчанию ПЕРВЫЙ.** Целевой исход цепочки — НЕ «починить N багов», а дать концерну авторитетного владельца и **удалить band-aid'ы + ad-hoc-координацию** вокруг него.
   - **Тип 1 — collapse-onto-СУЩЕСТВУЮЩЕГО-владельца (дефолт; стоимость ОТРИЦАТЕЛЬНАЯ — ядро ↓, ноль новых слоёв):** владелец УЖЕ есть, баги — от кода, что координирует вокруг него ad-hoc. Делаем существующего владельца авторитетным + удаляем координацию. *Канон: cancellation-арка #1030–1034 НЕ вводила контроллер-слой (это отвергнутый Variant A) — сделала авторитетным УЖЕ существующий `FSM` (abort = эффект действия `CANCEL`), удалив `maxEventDepth`/`RecursionDepthError`/#308/#945/#935 + ручную координацию.*
   - **Тип 2 — новый примитив (FALLBACK, только если владельца НЕТ):** добавляет слой; оправдан, лишь если убранное дублирование > слоя. Не дефолт.
   - **Владелец concern-specific** (НЕ тащи всё в FSM → god-object): FSM — lifecycle/transition; `RoutesStore` — дерево/CRUD; emitter/`onListenerError` — события/isolation (часто уже существует — переиспользовать, не плодить); codec — grammar/roundtrip.
   Для STRONG-цепочки сформулируй: владелец (существует → Тип 1) + что удаляется.

**6. Артефакт + чейнинг.**
- На STRONG-цепочку — **research-RFC** (`.claude/<topic>-unification.md`, формат под `/audit-rfc`→`/implement-rfc`; шаблон-эталон: `navigation-cancellation-unification.md`): гипотеза корня · код-доказательство convergence (`file:line`+commit) · предлагаемый single-owner + удаляемые band-aid'ы · затронутые пакеты + breaking-рябь · членство в оси.
- На WEAK-цепочку — `/bugfix` (точечно), без RFC-overhead. **НО различай форму WEAK:** WEAK-**независимые** корни в РАЗНЫХ местах → N точечных `/bugfix`; **WEAK-но-≥2-фасета-в-ОДНОМ-код-локусе → `/bugfix-batch`** (один PR, один RED per facet), НЕ N точечных — иначе whack-a-mole (фикс одного фасета оставит остальные живыми в том же блоке, а Phase-0.8-гейт bugfix при заходе на любой из них всё равно эскалирует обратно сюда). Пример дельты №3: #950-семья (#1170 meta / #1201 guards / #1236 context) — все в `getRoutesApi.ts:477-496` → ОДИН batch, не три /bugfix.
- Сквозные оси — отдельные стратегические заметки (плечо × пакеты).
- **Обнови `project_architectural_debt_map`** (память): новые/перевёрнутые вердикты, новые оси, закрытые цепочки.
- **Проверь существующие issues** перед заведением (не дублируй — `feedback_check_existing_issues_before_creating`).

## Классификация вердикта

- **STRONG** — корень доказан КОДОМ (convergence в общий helper / доказанное отсутствие инварианта). → RFC.
- **WEAK** — area-group или корень из 1–2 сайтов; нет convergence. → точечный bugfix, не RFC.
- **PARTIAL / over-broad** — корень есть, но цепочка склеила несколько под-корней (пример: #1040 path-matcher — STRONG-ядро, но over-broad; #1041 input-hardening — 3 под-корня). → расщепить на под-цепочки.
- **Вердикт — ГИПОТЕЗА, и STRONG — СЛАБАЯ гипотеза:** на сессии 2026-06-29 5/5 STRONG-вердиктов карты перевернулись в WEAK на `/chain-resolve` (см. «3 фильтра» выше). Эмитя STRONG — сначала прогони фильтры (realized? / владелец? / enforce?) + проверь расщепление, и помечай «STRONG-suspect, вероятно флипнет на `--verify`». Не выдавай за факт.

## Жёсткие правила

- Доказывай корень **ТЕЛОМ issue + КОДОМ фикса**, не заголовком. Title/area-кластеризация — главный систематический обман этого анализа.
- **Convergence в общий helper — сильнейший код-сигнал.** Нет общего кода у фиксов → не корень (area-group).
- Адверсариально: скептик **опровергает** единый корень (default area-group). На несущем НЕ доверяй вердикту субагента — открой helper и протрассируй сам.
- **Оси > слабые цепочки** по плечу — не утони в WEAK-точечных; сначала то, что гасит N пакетов.
- Целевой исход STRONG — **схлопывание**: дать АВТОРИТЕТ существующему владельцу (FSM/store/emitter/codec) + удалить band-aid'ы и ad-hoc-координацию, а не латать симптомы. **Предпочитай усиление СУЩЕСТВУЮЩЕГО владельца (Тип 1, ядро ↓) новому примитиву (Тип 2, fallback).** Предложение, которое ДОБАВЛЯЕТ слой/абстракцию вместо того, чтобы УДАЛИТЬ координацию (или не удаляет ни костыля), — Variant-A-запах, не унификация.
- STRONG/WEAK — гипотеза. Не плоди issue под недоказанный корень. Не дублируй существующие.
- Push/commit/`gh issue create` — только по явной просьбе.

## Оговорки (НЕ метить как находку)

- **Area-group честно назван группой** (не выдан за корень) — это правильная дисциплина, не ошибка. WEAK-вердикт — валидный результат, а не неудача.
- **Корень, расщепляющийся на под-корни** — норма (фикс корня вскрывает корень глубже: #1036 спин-оффнул guard-storage как ОТДЕЛЬНЫЙ корень). Не натягивай всё в одну цепочку. **Сильнейший спин-офф-сигнал: КАНОНИЧЕСКОЕ issue корня само дисклеймит члена** как «separate root cause / should not be folded in» — верь ему (на прогоне #698 написал ровно это про #952; #952 — отдельный корень в `RouteLifecycleNamespace`, не tree-commit).
- **Ось, чья «цепочка» WEAK** — ось всё равно реальна (#1045/A3): мети ось, не понижай из-за слабости цепочки.
- **Уже-унифицированное** (cancellation-арка) — не пере-открывай; отметь как закрытую грань (но проверь: закрыта ли ВСЯ ось или только одна грань — A1 закрыл nav-грань, memory/rx/Safari открыты). **STRONG-унификация сама может быть лишь ГРАНЬЮ более глубокой, ещё-не-найденной оси:** после схлопывания перечитай тела/код самой унификации на «same class as …» — cancellation-арка (STRONG, done) оказалась гранью A1, и выдала это тело #807. **«STRONG + унифицировано» ≠ «долг закрыт».**

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце каждого прогона)

Этот файл живой. После артефактов вынеси секцию **«Самокоррекция»** — критику не своей работы, а САМОГО СКИЛА: где его текст промолчал/соврал/был двусмыслен и это стоило времени или едва не дало ложный корень.

1. **Привязка к инциденту этого прогона**, не общие советы: «скил сказал X / промолчал про X → реальность Y → потерял/чуть не ошибся на Z». Нет инцидента — «правок нет» (честнее придуманной).
2. **Высокая планка:** правка, только если изменила бы исход (предотвратила ложный корень, сократила перебор, сняла двусмысленность).
3. **Разовое (специфика этого репо/цепочки) → в карту долга/память, НЕ в скил.** В скил — повторяющийся класс, полезный СЛЕДУЮЩЕМУ прогону.
4. **Сначала заточи существующий абзац, потом дописывай** (анти-раздувание).
5. **Предлагай таблицей (раздел | инцидент | правка), спроси разрешения.** Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
