Преврати сырую идею в утверждённый дизайн через структурированный сократовский диалог — ДО написания кода: исследуй контекст, провалидируй реальную потребность, задавай по одному вопросу за раз, предложи 2–3 подхода с trade-off'ами, презентуй дизайн по секциям с апрувом, запиши design-doc в `.claude/` и передай его в `/implement-rfc` (или `/bugfix`). НЕ пиши код и НЕ вызывай implementation-скиллы, пока пользователь не одобрил дизайн.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Идея/проблема в свободной форме (напр. «хочу typesafe-роуты», «нужен DevTools-плагин», «scroll restoration для адаптеров»).
- Опционально номер/ссылка на issue (`#699`) — если идея уже заведена.
- Опционально `--quick` — компактный режим для мелкой правки (меньше секций дизайна, но процесс тот же).

Пример:
```
/brainstorm typesafe route params для React-адаптера
/brainstorm #699
/brainstorm заменить ручной changeset на авто-детект --quick
```

---

## Принцип

Брейнсторминг — это **исследование пространства решений до фиксации одного из них**, а не презентация заранее готового плана. Самый дорогой провал — молча начать кодить под первую же интерпретацию запроса: реальная потребность, ограничения и критерии успеха выясняются в диалоге, а не угадываются. Дисциплина двойная: **(1)** ни строки кода и ни одного implementation-скилла (`/implement-rfc`, `/bugfix`, `/changeset`, `/create-pr`) до явного апрува дизайна; **(2)** каждый проект проходит процесс — но **глубина масштабируется по сложности**: однострочная правка получает один абзац и один вопрос, новый пакет — полный цикл с design-doc и 2–3 подходами.

Это брейнсторм-аналог `audit-rfc`/`implement-rfc`: там claim'ы документа проверяются против первоисточника и живой системы; здесь — **до того как зафиксировать дизайн, проверь, что задача вообще существует и стоит решения** (ср. memory `feedback_no_user_request_no_problem`).

## Фаза 0 — Контекст

1. **Исследуй проект перед вопросами.** Прочитай релевантные `CLAUDE.md`/`ARCHITECTURE.md`/`IMPLEMENTATION_NOTES.md`, затронутые пакеты, недавние коммиты и открытые issue. Не задавай вопрос, ответ на который уже есть в коде или доках.
2. Если передан номер issue — `gh issue view <number> --comments` (комментарии читай всегда: там уточнения скоупа и контр-примеры).
3. Свериться с memory-правилами проекта, релевантными идее (границы слоёв, dogfooding, decomposition, canonical-pattern-first — см. ниже).

## Фаза 1 — Валидация потребности (gate)

Прежде чем проектировать решение — докажи, что проблема реальна. Это триаж-гейт, который может закоротить работу в «делать не нужно».

1. **Есть ли user-driven запрос?** issue, реальный пользователь, конкретный pain — или это «codifies convention»/DRY-уборка без потребителя? Нет запроса + нет аналога у standalone-конкурентов = нет gap (memory `feedback_no_user_request_no_problem`). Останови и сообщи.
2. **Если идея — аналог конкурентской фичи** (TanStack/React Router и т.п.): проверь на fake-аналогию — одинаковое имя ≠ одинаковая семантика; прикинь ROI (memory `feedback_competitor_feature_analogies`, `feedback_competitive_categories`). Framework-bundled роутинг (Next/SvelteKit) — не конкурент.
3. **Применимость гейтов проекта:** не нарушает ли идея границы слоёв core/plugin/dom-utility (memory `feedback_layer_consistency`); проходит ли design-review-тест «хорошая архитектура позволяет писать плохой код, не ломая систему» (memory `feedback_failure_isolation`).

Если потребность не подтверждена — НЕ переходи к дизайну: изложи находку и спроси, продолжать ли.

## Фаза 2 — Сократовские вопросы (по одному за раз)

Выясни **назначение, ограничения, критерии успеха** — не реализацию.

- **Один вопрос на сообщение.** Если тема большая — разбей на несколько последовательных вопросов, не вали всё разом.
- **Предпочитай multiple-choice через `AskUserQuestion`** — варианты с короткими описаниями trade-off'ов; рекомендованный — первым с пометкой «(Рекомендую)». Пользователь всегда может выбрать «Other».
- Спрашивай только то, что меняет дизайн. Очевидное с разумным дефолтом — выбери сам и упомяни, не переспрашивай.
- Типовые оси: какие пакеты/адаптеры в скоупе (один или все 8); SSR/CSR/оба; breaking change допустим (pre-1.0 → `minor`); публичный API или внутренняя механика; нужен ли новый пакет или расширение существующего.

## Фаза 3 — Подходы (2–3 варианта)

Предложи **2–3 принципиально разных подхода**, для каждого: суть, trade-off'ы, влияние на API/слои/размер бандла/совместимость, и явная **рекомендация с обоснованием**.

- Учти canonical-pattern-first: helper, который попадёт в >10 точек, проектируется как Stage 0 (pre-feature), иначе двойной mechanical refactor (memory `feedback_canonical_pattern_first`).
- Учти dogfooding: каждый плагин/фича должны быть использованы в example до выпуска; ≥2 повторения оправдывают helper, иначе YAGNI (memory `feedback_dogfooding`, `feedback_dogfooding_one_consumer_insufficient`).

## Фаза 4 — Презентация дизайна (апрув по секциям)

Изложи дизайн, **масштабируя глубину по сложности**, и получай апрув после каждой содержательной секции (не вали весь документ сразу):
- Проблема и цель · публичный API/сигнатуры · затронутые пакеты и границы слоёв · модель данных/состояния · влияние на SSR/гидрацию (если применимо) · стратегия тестов (functional 100% + нужны ли property/stress) · breaking changes и миграция · декомпозиция.
- **Большую multi-package задачу декомпозируй на 4 stage** (ядро+React → остальные адаптеры → dogfooding → документация) (memory `feedback_large_task_decomposition`).

## Фаза 5 — Запись design-doc

После апрува запиши документ в **`.claude/<topic>-design.md`** (или `.claude/rfc-<topic>.md`, если это полноценный RFC) — то же место и формат, что потребляют `/audit-rfc` и `/implement-rfc`. Это и есть точка передачи между скиллами.

Документ содержит: контекст и потребность (с доказательством из Фазы 1) · принятые решения (О-N) · открытые вопросы · публичный API · план шагов (изменение → верификация → откат) · карту гейтов (локально-автоматизируемое vs handoff) · стратегию тестов и доков.

## Фаза 6 — Self-review дизайна

Перед передачей прогони документ по себе (ср. memory `feedback_plan_self_critique`):
- Нет ли плейсхолдеров, внутренних противоречий, неоднозначностей, расползания скоупа.
- Backward-compat / recovery: есть ли у каждого шага откат; продумана ли миграция для breaking changes.
- Encoding/priming-ловушки; не размывает ли дизайн границы слоёв; не требует ли private-API (только публичный — memory `feedback_no_private_api`, `feedback_core_as_library`).
- Каждое несущее утверждение дизайна — это гипотеза, помеченная для эмпирической проверки на этапе реализации (не «зелёный дизайн = истина»).

## Фаза 7 — Переход к реализации

Брейнсторминг НЕ реализует. Единственное, что делаешь дальше — **передаёшь design-doc**:
- Полноценный RFC/дизайн → сначала `/audit-rfc <path>` (провалидировать claim'ы), затем `/implement-rfc <path>`.
- Точечная правка под issue → `/bugfix #<n>` или прямая реализация по апрувленному плану.
- Спроси пользователя, запускать ли переход сейчас, или он сам вызовет нужный скилл.

## Жёсткие правила

- НЕ писать код и НЕ вызывать implementation-скиллы до явного апрува дизайна. Терминальное действие брейнсторма — записанный design-doc + (по запросу) передача в `/implement-rfc`.
- Сначала исследовать контекст (Фаза 0), потом спрашивать — не задавать вопрос, ответ на который есть в коде/доках.
- Валидация потребности — gate: нет user-driven запроса и нет аналога у standalone-конкурента → стоп, не проектировать.
- Один вопрос на сообщение; multiple-choice через `AskUserQuestion`; рекомендованный вариант первым.
- Глубина процесса масштабируется по сложности, но процесс проходят все задачи, включая мелкие.
- 2–3 подхода с trade-off'ами и явной рекомендацией до фиксации дизайна.
- Дизайн писать в `.claude/` в формате, потребляемом `/audit-rfc` и `/implement-rfc`.
- Дизайн — план, а не ground-truth: несущие утверждения помечать для эмпирической проверки при реализации.
