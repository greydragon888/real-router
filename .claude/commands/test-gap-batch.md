$ARGUMENTS

Формат аргументов:
- **Явный список** `[test]`-issue одной PR-группы (`#1178 #1181 #1179`) — обычно одна тема (reentrancy / start / stress-hygiene) одного пакета.
- **Без аргументов (или `all`) → discovery-режим:** сам находит все открытые test-gap issue (Шаг 0.0), группирует по пакету, подтверждает скоуп ПЕРЕД правкой. Сузить: `all <фильтр>` — напр. `all core`, `all priority:medium`.

Этот скилл — **ОРКЕСТРАТОР**. Оркестрацию ГРУППЫ (worktree/ветка `gh issue develop`, `EnterWorktree`, `pnpm install` без dist, возобновление, точечная валидация, один PR с `Closes`, rebase-before-push, гейты, worktree-жизненный-цикл) он **НЕ дублирует — берёт ВЕРБАТИМ как `/bugfix-batch` Шаги 1+3**. Здесь — только тест-геп-специфика: классификация гепа + **дисциплина дискриминации** + маршрут «тест vs всплывший баг». Механику property берёт из `/audit-pbt` / `/pbt-scenarios`, stress — из `/stress-scenarios` / `/audit-stress`, метод мутации — из `/mutation-score`, всплывший баг — в `/bugfix`.

## Шаг 0 — Discovery (если нет списка) + fetch + классификация КАЖДОГО гепа

**0.0 — Discovery (только без явного списка).** `gh issue list --state open --label "area: tests" --json number,title,labels` → отбери test-gap по заголовку (`[test]` / `[<pkg>][test]`, «untested»/«unpinned»/«coverage gap»/«never exercised»/«property gap»/«stress gap»/«vacuous»/«survives mutation»/«not asserted»). **ОТСЕЙ не-test с меткой `area: tests`** (чистая tooling/CI-задача без нового теста). Сгруппируй по пакету/области, выведи таблицу `#N · пакет · приоритет · класс гепа`, **подтверди скоуп у пользователя ПЕРЕД правкой** (набор бывает большой — юзер может срезать до `all core` или до темы). Один пакет с 10+ гепами разбей на тематические под-группы (reentrancy / start / stress) — одна под-группа = одна ветка/PR.

`gh issue view <N>` на каждый. Классифицируй КАЖДЫЙ пункт — класс определяет метод:

- **MISSING-coverage** (непокрытая ячейка/ветка/путь): новый тест, ПИННящий КОРРЕКТНОЕ поведение существующего кода.
- **TEST-INTEGRITY** (существующий тест зелен на СЛОМАННОМ коде — green-pin, self-catching try/catch, sleeping/`await sleep` assertion, tautology на ложной посылке, title≠body, duplicate describe): **ПОЧИНИ лживый тест**, чтобы он дискриминировал. Это НЕ «добавить покрытие» — это ремонт теста, который уже «зелёный», но ничего не ловит (#1189/#1243/#1209/#1183 класс).
- **PROPERTY-gap** (инвариант без генеративного локка): property-тест — метод из `/audit-pbt` / `/pbt-scenarios`; сверь/допиши строку в `INVARIANTS.md`.
- **STRESS-gap** (concurrency / memory / throughput без нагрузочного локка): stress-тест — метод из `/stress-scenarios` / `/audit-stress`; **порог обязан иметь доказанную дискриминирующую силу** (healthy < threshold < leak, ≥3× margin; см. корневой CLAUDE.md про heap-thresholds — round-MB guess = theatre).
- **BUG-revealing** (геп-филл тест ПАДАЕТ на текущем коде): это **БАГ**, не тест-геп. Эскалируй в `/bugfix` (отдельная ветка + changeset на src-фикс). НЕ подгоняй тест под green — issue остаётся про баг, тест-пин коммить вместе с фиксом.

## Шаг 1 — Один worktree/ветка на группу — **как `/bugfix-batch` Шаг 1**

Ведущая задача = первый номер группы. `gh issue develop <lead> --name <lead>-test-gaps` (slug — общий идентификатор группы, `core-reentrancy-tests`, не имя одного гепа) → worktree в `.claude/worktrees/<lead>-test-gaps` → `EnterWorktree path:` → **`pnpm install` в worktree ПЕРЕД любой пробой/прогоном** (свежий worktree не делит `node_modules`; dist НЕ собирай). Возобновление: `pwd` + `git branch --show-current` + `git log master..HEAD` перед пере-созданием. Детали (флаги, resume-ловушки) — в `/bugfix-batch` Шаг 1, не дублирую.

## Шаг 2 — Per-gap цикл (ЯДРО этого скила)

Для КАЖДОГО гепа по очереди:

1. **Заземли на РЕАЛЬНОМ src+тесте, НЕ на issue-черновике.** Issue — гипотеза на момент заведения: описанная непокрытая ячейка могла быть покрыта позже, «vacuous» тест — уже починен, «survives mutation» мутант — уже killed, счётчик/строка — уплыть. Прочитай ЦЕЛЕВОЙ тест-файл + целевой src ПЕРЕД правкой; если геп уже закрыт — триаж-non-gap (Гейты). (Симметрично doc-batch verify-against-source.)
2. **Напиши/почини тест — и ДОКАЖИ дискриминацию МУТАЦИОННО (несущее правило).** Тест, который не краснеет на сломанном поведении, — мусор: это РОВНО анти-паттерн, что эти issue репортят. Для КАЖДОГО добавленного/починенного теста: **сломай ТОЧНУЮ src-ветку/поведение, что он целит** (или удали фикс/инвертируй условие) → тест ОБЯЗАН покраснеть; верни → зелёный.
   - **Вердикт — по падению ТЕСТА, эмпирически** (`feedback_mutation_audit_empirical`): не чтением, не exit-кодом Stryker, не «оно должно ловить». Green-до-мутации тест — недискриминирующий пин, находка сама по себе (в TEST-INTEGRITY именно это и чинишь).
   - **Тесты ОХОТЯТСЯ за багами, не пиннят** (`feedback_tests_hunt_bugs_not_pin`): пиши как скептичную пробу на реальном src. Геп-филл ПАДАЕТ на текущем коде → это БАГ (Шаг 0 BUG-revealing → эскалация в `/bugfix`), а не «сделать зелёным».
   - **Через ПУБЛИЧНЫЙ API** (`feedback_kill_mutants_via_public_api`, `/whitebox-audit`): закрывай геп через публичную поверхность, не белым ящиком по внутренним символам — white-box-тест прячет мёртвый/недостижимый код от coverage. Внутренний доступ — только с явным обоснованием (memory/perf-инвариант).
   - **TEST-INTEGRITY-ремонт:** сними маскирующий try/catch (глотает throw), убери sleeping/`allSettled`-глушилку, разверни tautology, приведи title к ФАКТИЧЕСКОМУ assert. Докажи: ПОЧИНЕННЫЙ краснеет на мутации, СТАРЫЙ — нет (в этом весь ремонт).
   - **Timing-правка ядра** → прогони `test:properties` + `test:stress` затронутого пакета (`feedback_sweep_properties_stress_on_timing_change`).
   - **Coverage:** 100% пакета; **per-FILE** для phantom-порога адаптеров (svelte/solid/vue/angular — aggregate `Tests N passed` НЕ доказывает покрытие ТВОЕГО файла; смотри строку файла в coverage-таблице).
3. **Отдельный коммит на КАЖДЫЙ геп** с его `#<N>` (`/commit-msg`). **Changeset НЕ нужен** (правится `tests/`, не `src/`). Коммить `--no-verify` + точечный `pnpm -F <pkg> test -- --run` (+ `test:properties`/`test:stress` по классу) ПЕРЕД коммитом (per-package, не полный turbo — он делегируется пользователю). **Всплыл баг на Шаге 2** → src-фикс отдельным коммитом **+ changeset** (via `/bugfix`), RED-тест-пин — своим коммитом; issue тогда про баг, `Closes` вешается на баг-фикс. Гранулярность и `git add <файл>`-дисциплина — как `/bugfix-batch` Шаг 2.

## Шаг 3 — Финал на группу — **как `/bugfix-batch` Шаг 3**

Точечная валидация объединённого набора пакетов (`type-check`/`lint`/`test -- --run` + `test:properties`/`test:stress`; подтверждай `Tests N passed`, не только `exit 0`; foundation — bare-имена). Один PR через `/create-pr` с **`Closes #<N>` на КАЖДЫЙ геп-issue** (иначе merge закроет лишь ведущую). Push — гейт, `git fetch && git rebase origin/master` ПЕРЕД push, `--no-verify` при полной-сборке-в-pre-push. Wiki обычно НЕ при чём (тест-геп доки не трогает — если тронул поведенческую доку, отдельный wiki-коммит как `/bugfix-batch` 3.2). Полный `pnpm build` НЕ гоняй. Детали — в `/bugfix-batch` Шаг 3.

## Гейты

- **НЕ push / НЕ PR / НЕ issue-comment без явной просьбы** — готовь, показывай, жди «go». Локальные per-gap коммиты — воркфлоу, не гейт.
- **НЕ помечай геп закрытым, пока тест не ДОКАЗАН дискриминирующим** (мутация красит эмпирически). Зелёный-но-недискриминирующий = геп НЕ закрыт (иначе воспроизведёшь ровно тот дефект, что issue репортит).
- **BUG-revealing геп ≠ тест-геп.** Геп-филл падает на текущем коде → НЕ подгоняй под green; эскалируй src-фикс в `/bugfix` (ветка + changeset), issue остаётся про баг. Пометь в отчёте.
- **Триаж-non-gap** (issue-claim не репродуцируется — ячейка уже покрыта / тест уже дискриминирует / мутант уже killed): корректный исход «проверено, гепа нет» — коммит не нужен, issue закрывается с объяснением. НЕ частичный провал, группой не блокируй.
- Не помечай группу выполненной при падающих тестах или недоказанной дискриминации хотя бы одного гепа.

## Итоговый отчёт

- Workspace/ветка: `.claude/worktrees/<lead>-test-gaps`.
- Таблица по гепам: `#<N>` → **класс** (missing / integrity / property / stress) → тест(ы) + **доказательство мутации** (что сломал → покраснело, что вернул → зелёно) → коммит → статус (**закрыт** / **not-reproduced** триаж / **BUG → эскалирован в `/bugfix` #NN**).
- Точечная валидация объединённого набора (полный `pnpm build` делегирован пользователю; отметь поведенчески-затронутых потребителей, если баг всплыл в широком пакете).
- PR: ссылка/черновик, `Closes #<N>` на все геп-issue. Всплывшие баги — отдельные ветки/PR + changeset.
- Черновики issue-комментариев.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта — секция **«Самокоррекция»**: где ТЕКСТ ЭТОГО скила промолчал/соврал/был двусмыслен и это стоило времени или риска.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона. Особый кандидат: недоказанная дискриминация (закрыл геп green-pin'ом), пропущенный BUG-revealing (подогнал под green вместо эскалации), ошибка классификации (integrity-ремонт спутал с missing-coverage), phantom-coverage адаптера.
2. **Высокая планка: изменило бы исход?** Только если правка реально предотвратила бы ошибку, на которой споткнулся.
3. **Граница sub-скиллов ↔ test-gap-batch.** Метод одного property-теста → `/audit-pbt`; stress-порога → `/stress-scenarios`; мутации → `/mutation-score`; оркестрация группы → `/bugfix-batch`. Сюда — только батч-специфика тест-гепов (классификация, дискриминация-как-гейт, тест-vs-баг маршрут). Не дублируй.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Усиливай существующий абзац; не плоди секции.
5. **Предлагай, не применяй молча.** Кандидаты таблицей (раздел | инцидент | текст), спроси. Применил по согласию → `.claude/commands/` infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон чистый → «правок нет».
