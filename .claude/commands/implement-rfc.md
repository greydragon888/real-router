Реализуй RFC/дизайн-документ/план-миграции по этапам: декомпозируй на атомарные шаги (изменение → верификация → откат), ВЕРИФИЦИРУЙ каждое несущее утверждение документа против ЖИВОЙ системы (а не доверяй «зелёному» RFC), выполни всё локально-автоматизируемое, а негейтируемое-локально (publish/OIDC/CI-матрица/merge/секреты) передай пользователю с точными командами и рисками.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Путь к RFC/дизайн-документу/плану (напр. `.claude/pnpm-11-migration-analysis.md`, `.claude/rfc-file-routes.md`, `.claude/oxc-migration-analysis.md`).
- Опционально `--dry-run` — только декомпозиция + pre-flight (план шагов, открытые вопросы, гейт-карта), без правок.
- Опционально `--no-commit` — не создавать коммитов (по умолчанию: следовать модели доставки репо; high-risk → ветка, коммит по явному go-ahead; НИКОГДА не push без явной просьбы).

Пример:
```
/implement-rfc .claude/pnpm-11-migration-analysis.md
/implement-rfc .claude/rfc-isomorphic-fetch-cache.md --dry-run
```

---

## Принцип

RFC, помеченный «готов к реализации» — и даже «факты перепроверены эмпирически» — это **план, а не ground-truth**. Реализация — это не транскрипция документа в файлы, а его *исполнение против реальной системы*, где КАЖДОЕ несущее допущение проверяется в момент, когда на него опираешься. Самый дорогой режим провала — **plausible-but-wrong допущение в RFC**: документ уверенно утверждает «эта группа конфига мертва после миграции / этот вывод инструмента имеет формат X / этот флаг безопасен», ты доверяешь и правишь код — а гейт молча краснеет (или, хуже, краснеет только на CI/релизе).

Это implementation-аналог `audit-rfc`: там истинность claim'а доказывается походом в первоисточник; здесь — **перед тем как опереться на claim в правке кода, докажи его прогоном на живом репо**. «Документ так сказал» — не основание; основание — `git diff`, exit-код гейта, реальный вывод инструмента.

Вторая половина дисциплины — **честная граница автоматизации**. Часть RFC физически не валидируется локально (OIDC-публикация, реальная CI-матрица, наблюдение за первым релизом, что-то за секретами/правами). Сделай ВСЁ, что можно сделать и проверить локально; остальное передай пользователю с **точной командой и точным риском** — и не выдавай суррогат за проверку (`--dry-run`/verdaccio НЕ проходят OIDC-обмен; «собралось локально» ≠ «прошло на CI-матрице»).

## Перед стартом (pre-flight) — обязательно

1. **Прочитай RFC ЦЕЛИКОМ.** Не начинай правки по первому разделу. Внутренние ревизии («Ред. N», «✅ СНЯТО», «решено») меняют смысл ранних разделов — поздний раздел может отменить ранний.
2. **Извлеки/построй план шагов.** Если в RFC есть пошаговый план (§«план», «по шагам») — возьми его шаги, верификации и откаты как есть. Если нет — декомпозируй сам на **атомарные, независимо-верифицируемые** шаги; у каждого: (а) изменение, (б) как проверю успех (конкретный прогон/exit-код), (в) откат.
3. **Раздели РЕШЕНИЯ и ОТКРЫТЫЕ вопросы.** Принятые в RFC решения (О-N, «Принято») — НЕ перерешивай, исполняй. Дизайн-вопросы, оставшиеся открытыми и гейтящие код, — вынеси пользователю (AskUserQuestion) ДО правок, не выбирай молча.
4. **Построй карту гейтов: automatable-now vs gated.** Для каждого шага реши: можно ли выполнить И проверить локально сейчас? Что требует пользователя / CI / внешнего сервиса / секрета / публикации / merge? Это разделение определяет, где заканчивается твоя работа и начинается handoff.
5. **Сверься с конвенциями и памятью репо.** Модель доставки (этот репо: infra → master напрямую, НО high-blast-radius → ветка+PR как осознанное исключение — см. правило в CLAUDE.md и §«модель доставки» в RFC). Запреты (memory `feedback_no_full_build`: полную `pnpm build`/turbo делегировать пользователю). Версии тулинга (стек bleeding-edge — память лагает). Чужой незакоммиченный WIP в дереве — не трогать.
6. **Сними baseline.** Зелёное стартовое состояние (нужный гейт проходит), хэши/счётчики, по которым потом докажешь «изменилось только то, что должно» (напр. sha lockfile, число resolution'ов, diff-stat).

При `--dry-run` остановись здесь и выдай: план шагов + карту гейтов + открытые вопросы.

## Цикл исполнения (на каждый шаг)

**изменение → верификация → (откат наготове)**, атомарно, по одному шагу.

- **Предпочитай инструмент, предписанный RFC (кодмод/скрипт), но ВЕРИФИЦИРУЙ его вывод, а не его рапорт.** Сводка инструмента врёт: кодмод `pnpm-v10-to-v11` рапортовал «Modified 0 files», фактически изменив три. Истина — `git diff`/`git status`, а не строка «0 files».
- **Перед опорой на claim — проверь claim.** Если RFC говорит «X безопасно удалить / Y теперь мёртв / формат вывода Z / флаг W — no-op», и на этом висит правка — **проверь эмпирически перед правкой**. Реальный урок: RFC утверждал, что группа `pnpmOverrides` в `syncpack.config.mjs` станет мёртвой после переезда overrides → пробное удаление **уронило `lint:deps`** (syncpack 15.x читает overrides и из `pnpm-workspace.yaml`). Группу вернули + задокументировали. Если б доверились RFC — красный гейт.
- **Claim о стороннем инструменте → проверь на самом инструменте.** Поведение/формат вывода/флаги (turbo, pnpm, syncpack, eslint, browser API) — прогон/`--help`/context7-доки/исходник, НЕ память. Пример: `npm view <p> version` → `pnpm view <p> version` безопасно только потому, что вывод эмпирически идентичен (annotation «published X ago» есть в полной сводке, но не в field-запросе) — это проверено, а не предположено.
- **Не доверяй codemod-резолюции версий/значений из сети.** Кодмоды молча падают в pinned-fallback при недоступном реестре (напр. `pnpm@11.0.1` ниже нужного минимума). Несущие значения (пин версии, integrity-хэш) — пиши руками поверх вывода кодмода и проверяй строгой ассерцией.
- **Изолируй чужой WIP.** Коммить ТОЛЬКО файлы RFC; незакоммиченные правки пользователя оставь нетронутыми (селективный `git add`, не `git add -A`).
- **Каждый «зелёный» — это exit-код, а не отсутствие вывода.** `outputLogs: errors-only`/`loglevel=error` дают пустой вывод при успехе; читай реальный код возврата (в zsh — `$pipestatus`), не «вывод пустой → ок».

## Дисциплина верификации (ядро)

- **«Зелёный RFC» ≠ «корректно в этом репо».** Пере-выведи каждый load-bearing факт против живой системы: переустанови/пересобери затронутое, прогони затронутые гейты, пересчитай счётчики.
- **Гони КАЖДЫЙ гейт, который RFC задевает,** и подтверди зелёным — не предполагай. Для миграции пакет-менеджера/тулинга это минимум: чистая установка из локфайла (ловит ошибки конфига/build-скриптов под строгими флагами), формат локфайла, dep-tooling (lint:deps/lint:dedupe/lint:audit). Полную `build`/тест-матрицу — по правилам репо (здесь: делегировать).
- **Измеренное vs предположенное.** Для количественных claim'ов (тайминг, порог, «уложится в ≤Xs») первоисточник — замер/симуляция, не число из RFC; добудь, если дёшево, иначе явно пометь ASSUMPTION+RISK в отчёте.
- **Докажи «изменилось только нужное».** Сравни с baseline: diff-stat по ожидаемому набору файлов; неожиданная мутация (локфайл «поплыл», лишние файлы) — стоп и разбор.
- **Нашёл, что RFC НЕ ПРАВ — чини репо ПРАВИЛЬНО (не как в RFC), задокументируй расхождение** (комментарий в коде у места + запись в infra-notes + явный флаг в отчёте). По желанию — поправь сам RFC (если это твой локальный план-док). Не «исполняй ошибку, потому что так в документе».

## Что НЕЛЬЗЯ валидировать локально — передать пользователю (handoff)

- Определи заранее (pre-flight шаг 4) и НЕ имитируй проверку: `--dry-run`/локальный реестр НЕ доказывают OIDC/provenance; «собралось у меня» не заменяет CI-матрицу; первый реальный релиз — отдельный наблюдаемый гейт.
- Для каждого gated-пункта дай: **точную команду**, что именно проверить, и **точный риск/blast-radius** при поломке.
- **НИКОГДА** не push/publish/merge/issue-create без явной просьбы пользователя (CLAUDE.md). Коммит — по модели доставки/просьбе; outward-facing шаги (PR, релиз) — только по go-ahead.
- Если локальный тулчейн пользователя расходится с целевым (напр. глобальный pnpm 10 при миграции на 11) — дай команду-мост (`corepack pnpm …` / обновление) и предупреди, что обычный вызов даст неверный результат.

## Документация (конвенция репо — обязательна для infra)

- Обнови то, что предписывает RFC и CLAUDE.md: таблицу тулчейна, Release/New-Package разделы, `IMPLEMENTATION_NOTES.md` в формате **Problem → Solution → Why** (с эмпирически-проверенными фактами и найденными gotcha'ми; «never delete historical decisions» — добавляй запись, а не переписывай старую).
- Зафиксируй найденные расхождения с RFC прямо в коде у места (короткий комментарий «почему так, а не как казалось»).

## Финальный отчёт + само-критика (перед «готово»)

Отчёт строго делит: **СДЕЛАНО и проверено** (с результатами гейтов) / **НЕ автоматизируемо → на пользователе** (с командами и рисками) / **найденные расхождения с RFC**.

Само-критика (прогони по себе, ср. memory `feedback_plan_self_critique`):
- Где я ПРЕДПОЛОЖИЛ вместо того, чтобы проверить? (особенно claim'ы RFC, на которых висели правки)
- Backward-compat / recovery: есть ли у каждого шага рабочий откат; оставил ли систему в собираемом состоянии на каждом коммите?
- Encoding/формат-ловушки (хэши, кодировки, кавычки в YAML/JSON, разные оболочки)?
- Не закоммитил ли чужой WIP; изменилось ли ровно то, что планировал (diff-stat vs baseline)?
- Не выдал ли суррогат за проверку gated-пункта?

## Жёсткие правила

- Прочитать RFC целиком до первой правки; уважать внутренние ревизии (поздняя отменяет раннюю).
- Принятые решения — исполнять, не перерешивать; открытые гейтящие вопросы — пользователю ДО кода.
- Атомарный шаг → верификация exit-кодом → откат наготове. По одному шагу.
- Перед опорой на claim RFC, висящий на правке, — проверить claim прогоном; вывод инструмента читать из артефакта (`git diff`), не из его сводки.
- Несущие значения из сети (версии, хэши) — руками + строгая ассерция, не доверять codemod-fallback.
- Чужой WIP не трогать; `git add` селективно.
- Полную сборку/тест-матрицу — по правилам репо (здесь делегировать пользователю).
- Не push/publish/merge без явной просьбы; gated-пункты — handoff с командой и риском, без имитации проверки.
- Нашёл ошибку RFC — чинить правильно + задокументировать, не транскрибировать ошибку.
