Проведи аудит RFC/дизайн-документа: провалидируй КАЖДОЕ утверждение против первоисточника (код, компилятор, прогон, смежные доки, доки/CLI/исходники сторонних продуктов) — НЕ против самого документа — найди ошибки, противоречия и непропагированные решения, вынеси вердикт о готовности к реализации.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Путь к RFC/дизайн-документу (напр. `.claude/rfc-file-routes.md`, `.claude/rfc-type-safe-routing-generator.md`, `.claude/ci-acceleration-poc-ru.md`).
- Опционально `--fix` — применить ТОЛЬКО механические правки (фактические исправления, починка цитат, пропагация уже-ПРИНЯТОГО решения по всем таблицам/псевдокоду/DoD, снятие противоречий с однозначной резолюцией). Дизайн-вопросы НИКОГДА не правятся автоматически — выносятся автору. По умолчанию: report-only (доказанный порядок — сперва отчёт, правки по явной просьбе).

Пример:
```
/audit-rfc .claude/rfc-file-routes.md
/audit-rfc .claude/rfc-type-safe-routing-generator.md --fix
```

---

## Принцип

Статус «готов к работе» в шапке НЕ значит, что документ корректен. RFC-«театр» проходит, когда: утверждение ссылается на код, который говорит не то, что заявлено; решение принято в прозе, но не пропагировано в таблицы/псевдокод/DoD; в алгоритме ordering-баг, который никто не протрассировал; цитата на смежный док не резолвится.

Цель аудита: для КАЖДОГО утверждения и вывода доказать его **первоисточником** — кодом, компилятором, реальным прогоном, содержимым смежного документа — а НЕ доверием к само-утверждению RFC или к резюме субагента. Это RFC-аналог мутационной валидации (см. `audit-pbt`/`audit-stress`): истинность claim'а доказывается походом в ground-truth, а не зелёным цветом документа.

Ключевая интуиция: самый опасный режим — **plausible-but-wrong**. Claim звучит убедительно, субагент его «подтверждает», но компилятор/код говорит обратное. Несущие и неожиданные claim'ы проверяй САМ, эмпирически (см. Жёсткие правила).

**Ground-truth включает данные, которые нужно ПОЛУЧИТЬ, а не только код, который нужно прочитать.** Для количественных claim'ов (тайминги, пороги, «уложится в ≤Xs», «доминирует пакет Y») первоисточник — это **замер**, а не число из документа. Не принимай оценку на веру и не понижай находку до «вероятно ок», когда вопрос закрывается 5-минутным замером / симуляцией / калибровкой по реальной истории. Эталон дисциплины — `.claude/ci-acceleration-poc-ru.md`: он сам ведёт ревизии «факты перепроверены эмпирически» (Ред. 16: `turbo --dry=json` на репо нашёл **3 бага** центрального скрипта) и явно метит **измеренное** (`32m 29s` на real CI run) против **оценочного** («arm64-дельта неизвестна → замер, не гипотеза»; «`K=5` — стартовый, подобрать по замерам»). Аудит обязан: (1) для каждой load-bearing-цифры определить — измерена она или предположена; (2) если предположена и гейтит решение — добыть данные или явно пометить ASSUMPTION+RISK.

**Claim'ы о СТОРОННИХ продуктах — отдельный первоисточник: сам продукт, не репо и не память.** Если документ ссылается на чужой API / поведение / лимит / опыт («turbo делает X», «GitHub Actions arm64 free для public-repo», «Nx Cloud умеет Y», «Vite-плагин поддерживает Z», «browser API ведёт себя так») — проверь в первоисточнике ПРОДУКТА (его CLI/`--help`/прогон, context7-доки, исходники, офиц. доки, web), а НЕ по training-памяти: стек bleeding-edge (TS 6 / turbo 2.9 / ESLint 10 / Vitest 4…), и память систематически отстаёт от актуальных версий. «Я знаю, как работает X» без захода в источник — это plausible-but-wrong в чистом виде (см. memory `feedback_no_hallucinated_spec_facts`).

## Классификация утверждений (определи тип КАЖДОГО claim'а)

1. **CODE-FACT** — ссылка на файл/функцию/строку/regex/сигнатуру API/поведение. Проверяется чтением реального кода.
2. **TYPE/COMPILER** 🔬 — утверждение о поведении TypeScript (`keyof`, `Omit`, inference, module augmentation, index-signature). Проверяется ТОЛЬКО компиляцией — никогда рассуждением и никогда доверием агенту.
3. **CROSS-REF (внутренний)** — цитата на ВНУТРЕННИЙ док/RFC/секцию/issue/номер-решения. Должна резолвиться в существующий таргет с заявленным содержимым.
4. **EXTERNAL/THIRD-PARTY** 🌐 — утверждение о СТОРОННЕМ продукте: его API / поведение / лимит / прайсинг / «как делает инструмент X» (turbo, Nx, GitHub Actions, Vite, Vitest, browser API…). Первоисточник — САМ продукт: установленный CLI (`--help`/прогон), context7-доки, его исходники/репозиторий, офиц. доки (WebFetch), web (WebSearch). НЕ из памяти (training лагает на bleeding-edge) и НЕ из утверждения RFC.
5. **COUNT/INVENTORY** — счётчики пакетов, «N адаптеров», «9 инвариантов». Пересчитай сам.
6. **ALGORITHM/PSEUDOCODE** — читается как исполнение: ordering-баги, недостижимые ветки, необработанные edge-кейсы, расхождение сигнатуры вызова и определения.
7. **DESIGN-DECISION** 🔁 — выбор, принятый в прозе («Принято», О-N). Должен быть пропагирован в КАЖДЫЙ downstream-артефакт: convention-таблицы, псевдокод, DoD этапов, Risks, сводную таблицу.
8. **INVARIANT/TEST-PLAN** — предлагаемые инварианты/тесты. Проверяется на корректность (свойство реально держится с учётом ОСТАЛЬНЫХ решений?) и неизбыточность (не дубль другого инварианта).
9. **PERF/EMPIRICAL-ESTIMATE** 📊 — количественный claim: тайминг, порог (`K=5`), «уложится в ≤Xs», «пакет Y доминирует wall-time», счётчик, варьирующийся по входу. Первоисточник — **замер / симуляция / калибровка по реальной истории**, не число из документа. Проверь дополнительно: помечено ли явно как измеренное vs оценочное.
10. **PROSE/MOTIVATION** — мотивация/рамка, не привязанная к коду. Низкий приоритет; проверяется лишь на внутреннее противоречие.

## Протокол на каждый claim

1. **Классифицируй** (тип выше) и выдели первоисточник, который его докажет/опровергнет.
2. **Собери ground-truth.** Для широты — fan-out `Explore`-агентов, сгруппированных по кластерам (типы; конкретный пакет; смежные доки; инфра/тесты); каждому — точные вопросы + требование цитировать `file:line` и выносить вердикт CONFIRMED/REFUTED/PARTIAL/NOT-FOUND. Агенты должны читать **рабочее дерево текущей ветки** (read-only, без worktree-изоляции — иначе ветвление от origin/master даст аудит устаревшего кода).
3. **Несущие и неожиданные claim'ы перепроверь САМ, эмпирически** — НЕ доверяй вердикту агента на load-bearing-утверждении:
   - **TYPE/COMPILER:** напиши scratch-`.ts`, прогони `npx tsc --noEmit --strict --ignoreConfig <file>` (флаг обязателен — иначе TS5112 на найденном tsconfig); читай, какие именно строки дали ошибку. (Пример из сессии-родителя: агент «опроверг» claim про `Omit` выдуманным результатом компиляции; реальный `tsc` доказал, что claim ВЕРЕН — `keyof` с индекс-сигнатурой коллапсирует в `string|number`, и `Omit` ничего не убирает.)
   - **regex/parsing:** прогони паттерн на реальных строках-нарушителях (`node -e`), не «на глаз».
   - **code-path:** открой функцию и протрассируй конкретный вход до точки, где решается поведение (порядок веток!). (Так найден silent-баг optional-splat: `segment.endsWith("?")` перехватывает `*slug?` в optional-param-fork раньше splat-ветки.) **Если RFC включает собственный проверочный сниппет/гейт** (`node -e`-страж версии, валидатор, regex-чек, выданный за доказательство свойства) — он сам claim, а не инструмент: прогони его поперёк ВСЕЙ граничной полосы, которую он берётся гейтить (чуть-ниже / на / чуть-выше порога), не только на target/happy-значении. Гейт, пропускающий значение, которое заявляет отбраковывать, — ложно-зелёный BLOCKER. (Сессия-родитель: страж `(min===0 && pat<=1)` для «≥ 11.1.3» пропускал 11.0.2–11.1.2 — поймано только прогоном по диапазону, а на 11.8.0 он зелёный.)
   - **CROSS-REF (внутренний):** `grep`-ни цитируемый файл на цитируемый заголовок/секцию — резолвится ли.
   - **EXTERNAL/THIRD-PARTY:** проверь в первоисточнике ПРОДУКТА, по убыванию силы: (1) **установленный инструмент** → запусти его (`<tool> --help`, реальный прогон/вывод — сильнейшее); для **существования/версии npm-пакета — `npm view <pkg> version`** (registry-факт; `npmjs.com` часто 403'ит WebFetch, а `E404` = пакета нет — валидное доказательство ОТСУТСТВИЯ, сильнее «я не нашёл»); (2) **context7** (`resolve-library-id` → `query-docs`) для актуальных доков библиотеки/CLI/сервиса; (3) **офиц. доки** через `WebFetch`; (4) `WebSearch`. Для load-bearing — cross-check ≥2 источника. Память НЕ источник. Не проверяемо (платное/закрытое/нерелизнутое API) → пометь **ASSUMPTION**, не утверждай. (Примеры сессии-родителя: «turbo не шардит matrix OOTB / нет DTE» — CLI `--help` + context7 Turborepo; shape `turbo query affected` — прогон CLI; «arm64-runners free для public-repo» — внешний GitHub-факт → офиц. доки/web.) **Когда RFC даёт shell-команду, чтобы ДОБЫТЬ значение для конфиг-поля/токена** (хэш/id для вставки в `packageManager`, lockfile, `.npmrc`) — прогони её и сверь **формат вывода** (кодировку/алгоритм/длину) с форматом ЦЕЛЕВОГО поля, не только корректность значения: правильное значение в неправильной кодировке тихо ломает шаг и проваливает его же цель. (Сессия-родитель: `npm view … dist.integrity` отдаёт `sha512-<base64>`, а поле `packageManager` требует `+sha512.<hex>`; `dist.shasum` вовсе sha1 — наивная вставка = малформед.)
   - **COUNT:** `ls`/`grep` и пересчитай (в сессии-родителе агент насчитал 33 пакета, реально — 32).
   - **PERF/EMPIRICAL:** прогони реальную команду и засеки (`time`, turbo `--dry=json`/`--summarize`, `.turbo/runs/*.json`); порог — откалибруй по реальной истории (`gh pr list` + `git diff`/turbo affected на последних N PR — распределение, не одна точка); скрипт-эскиз — прогони на реальных сценариях (его вывод = функциональный unit-тест без обвязки). Если замер ПРИНЦИПИАЛЬНО недоступен (нужен CI-hardware / прод-трафик) — пометь claim как **ASSUMPTION** и вынеси, что решение висит на неизмеренном числе; НЕ выдавай оценку за факт.
   - **Чистота рабочего дерева для git-diff/affected-проб (скил read-only → одноразовый worktree).** Инструменты, считающие affected по git (`turbo affected`/`turbo query affected`/`nx affected`/`--filter='...[ref]'`/`lint-staged`), **сворачивают незакоммиченные правки рабочей копии в результат** — грязное дерево даёт ложный affected (твои правки читаются как «затронуто»). Гоняй такую пробу в **detached-worktree на нужном ref** (`git worktree add --detach <scratch> <ref>` → прогон → `git worktree remove`; turbo-бинарь из основного репо через `--cwd`/`--skip-infer`), удаляемом в конце — скил ничего не пишет в репо, worktree безопасно сносится. Сначала `git status`; дерево грязное и проба про affected → worktree обязателен. (Сессия-родитель: A5-проба `turbo query affected` на грязном `packages/core` вернула `core` в affected → выглядело как REFUTED линчпина; чистый worktree дал `[rx]` без core.)
   - **Hardware-bound vs hardware-invariant — ключевое различие, не путать.** *Логика/структура* (классификация, routing-решение, граф affected, счётчики) — hardware-**инвариантна**: локальный прогон = истина и для CI, переносится 1:1. *Тайминги* — hardware-**bound**: локальная абсолютная цифра на CI-цель НЕ переносится, и сравнивать её с CI-оценкой дока **нельзя** (apples/oranges → не пиши «REFUTED» из локального замера CI-числа). Переносится лишь **отношение**, и только когда обе величины упираются в ОДИН ресурс (CPU/CPU — `angular` vs `preact` bundle: да; CPU/IO — dry-run-операция vs checkout+install: нет, разные узкие места). Абсолютный CI-таргет (`≤Xs`) меряется ТОЛЬКО на CI-железе — иначе ASSUMPTION. Прежде чем мерить тайминг локально, спроси: «это про скорость (hardware-bound → нужен CI или same-bottleneck ratio) или про корректность (hardware-invariant → локально достаточно)?».
4. **Раздели «claim технически верен как сформулирован» и «вывод держится».** Посылка может быть верна при неверном выводе — и наоборот (верный вывод на слегка кривой цитате).
5. **Для DESIGN-DECISION — охота за непропагированными копиями.** `grep`-ни сигнатурные строки старого решения по всему документу; каждое вхождение, противоречащее принятому решению — находка. Решение в прозе ≠ решение в алгоритме: если псевдокод не enforce-ит решение, реализация воспроизведёт баг, против которого решение и принято.

## Классификация находок (severity)

- **BLOCKER** — реализация по документу даст неверный код / ложно-зелёный тест / воспроизведёт описанный баг (непропагированное решение в алгоритме; ordering-баг в псевдокоде; инвариант, противоречащий другому решению; внутреннее противоречие в контракте/представлении данных).
- **SHOULD-FIX** — не ломает реализацию, но недоспецифицировано/вводит в заблуждение (over-promise механизма; недоопределённая область инварианта; неозвученный trade-off).
- **MINOR** — цитата/факт/счётчик мимо при верном по сути выводе; редундантный инвариант; формулировка.

Каждая находка: тип claim'а + `file:line` доказательства из ground-truth + **конкретная однострочная правка**.

## Починка (только при `--fix`)

Правится МЕХАНИЧЕСКОЕ и однозначное:
- фактические исправления (неверная сигнатура/счётчик/цитата → верная, подтверждённая ground-truth);
- пропагация уже-ПРИНЯТОГО решения во все таблицы/псевдокод/DoD/сводку (+ `grep`-доказательство нулевого остатка старой формулировки);
- снятие противоречия, когда корректная резолюция однозначна (одна сторона уже канонизирована решением);
- ordering-фиксы в псевдокоде, восстанавливающие заявленную семантику.

При правке псевдокода **читай его как исполнение** — наивная вставка может оказаться мёртвым кодом (в сессии-родителе throw для `[[...slug]]`, вставленный после dot-check, был бы недостижим, т.к. `[[...slug]]` содержит `.`; пришлось переупорядочить проверки и сузить dot-check до имени токена).

НЕ правится автоматически: выбор IN/OUT фичи, смена архитектуры, переименование API — любой вопрос, требующий решения автора → выносится в отчёт как открытый.

Подними ревизию в шапке + добавь запись в журнал ревизий (что и почему).

## Жёсткие правила

- Доказывай **первоисточником**, не документом. RFC — не свидетель собственной корректности.
- На load-bearing/неожиданном claim'е НЕ доверяй вердикту субагента без собственной перепроверки (компиляция/прогон/трассировка). Subagent-резюме — гипотеза, не факт.
- TYPE/COMPILER/regex/algorithm — только эмпирика (`tsc`/`node`/трассировка), никогда «логически очевидно».
- Claim о стороннем продукте (API/поведение/лимит/опыт) проверяется в **первоисточнике продукта** (CLI/context7/доки/исходники/web), НЕ из памяти и НЕ из утверждения RFC. Стек bleeding-edge — training систематически отстаёт; «я знаю, как работает X» без захода в источник = находка-в-зеркале. Для установленного инструмента прогон его CLI бьёт любые доки.
- Решение «исправлено» ТОЛЬКО когда пропагировано везде; докажи `grep`-ом нулевой остаток старой формулировки.
- Верный вывод на кривой цитате → чини цитату, вывод сохрани. Не выбрасывай хороший вывод из-за плохого адреса.
- Дизайн-выбор автора — не «ошибка», пока не порождает противоречие. Не навязывай вкусовщину под видом находки.
- Раздели **измеренное и оценочное**. Оценка, поданная как факт — находка. Load-bearing-оценка, гейтящая решение, должна быть либо измерена, либо явно помечена ASSUMPTION+RISK с планом замера. 5-минутный замер бьёт хедж-вердикт «вероятно ок» — стремись получить данные, а не обойти их.
- Push/commit не делай без явной просьбы.

## Оговорки (НЕ метить как находку)

- **PROSE/MOTIVATION** не обязана цитировать код — мети только противоречия в ней.
- **CROSS-REF в правильную сущность, но слегка не тот файл/секция** — MINOR (суть важнее адреса), не BLOCKER.
- **Субъективное дизайн-предпочтение** без противоречия — не ошибка.
- **Не каждое число нужно мерить** — только load-bearing, гейтящее решение. Цифра, честно помеченная «оценочно, замер в фазе X» с планом — это правильная дисциплина (награждай, не мети как находку). Estimate ≠ ошибка; estimate, выданный за факт — ошибка.
- **Сторонний продукт, чьё поведение нельзя проверить** (закрытое/платное/нерелизнутое API) → пометь **ASSUMPTION** с указанием, что нужен доступ/релиз; не выдавай память за факт. Если RFC сам честно назвал это гипотезой / «надо измерить» (как arm64-дельта) — это дисциплина, не находка.
- **Re-audit гигиена:** после слов автора «исправил» сперва сверь, что читаешь ТЕКУЩИЙ файл — fingerprint (`stat` mtime/size + `md5`); если твой read выглядит неизменным, а автор настаивает — пере-`stat`, кэш инструментов/ФС может отставать от свежего сохранения. Не выноси «изменений нет», не сверив fingerprint. (RFC под `.claude/` — gitignored: `git diff`/`git status` для него всегда пусты — верь содержимому/хэшу, не git.)

## Валидация и отчёт

1. **Отчёт-таблица:** claim | тип | ground-truth (`file:line` / замер / context7·CLI·доки продукта) | вердикт (CONFIRMED / REFUTED / PARTIAL / STALE-CITATION / NOT-FOUND / MEASURED / ASSUMPTION) | severity | правка.
2. Раздели находки на **BLOCKER / SHOULD-FIX / MINOR**; для каждой — однострочный конкретный фикс с `file:line`.
3. **Вердикт о готовности:** оцени архитектуру (зелёная/нет) ОТДЕЛЬНО от исполнения; перечисли, что закрыть до «готов к реализации»; явно раздели «ревизия документа» и «пересмотр архитектуры».
4. При `--fix`: после правок — `grep`-доказательство нулевого остатка старых формулировок + новый fingerprint файла; перечитай наиболее переструктурированные зоны, убедись, что псевдокод/таблицы не порвались.

## Замечание

RFC/дизайн-доки под `.claude/` обычно gitignored (не трекаются) — изменения не видны в `git`, верифицируй содержимым и fingerprint'ом. Правки документа changeset не требуют. Push не делай без явной просьбы.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Валидация п.1–3) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного RFC/стороннего продукта (разовую) → в отчёт/память по этому RFC, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ аудиту ДРУГОГО документа.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
