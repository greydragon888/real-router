Проведи аудит RFC/дизайн-документа: провалидируй КАЖДОЕ утверждение против первоисточника (код, компилятор, прогон, смежные доки, доки/CLI/исходники сторонних продуктов) — НЕ против самого документа — найди ошибки, противоречия и непропагированные решения, вынеси вердикт о готовности к реализации.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Путь к RFC/дизайн-документу (напр. `.claude/rfc-file-routes.md`, `.claude/rfc-type-safe-routing-generator.md`, `.claude/ci-acceleration-poc-ru.md`).
- Опционально `--fix` — применить ТОЛЬКО механические правки (фактические исправления, починка цитат, пропагация уже-ПРИНЯТОГО решения по всем таблицам/псевдокоду/DoD, снятие противоречий с однозначной резолюцией). Дизайн-вопросы НИКОГДА не правятся автоматически — выносятся автору. По умолчанию: report-only (доказанный порядок — сперва отчёт, правки по явной просьбе).

Пример:
```
/audit-rfc .claude/rfc-file-routes.md
/audit-rfc .claude/rfc-type-safe-routing-generator.md --fix
```

---

## Принцип

Статус «готов к работе» в шапке НЕ значит, что документ корректен. RFC-«театр» проходит, когда: утверждение ссылается на код, который говорит не то, что заявлено; решение принято в прозе, но не пропагировано в таблицы/псевдокод/DoD; в алгоритме ordering-баг, который никто не протрассировал; цитата на смежный док не резолвится.

Цель аудита: для КАЖДОГО утверждения и вывода доказать его **первоисточником** — кодом, компилятором, реальным прогоном, содержимым смежного документа — а НЕ доверием к само-утверждению RFC или к резюме субагента. Это RFC-аналог мутационной валидации (см. `audit-pbt`/`audit-stress`): истинность claim'а доказывается походом в ground-truth, а не зелёным цветом документа.

Ключевая интуиция: самый опасный режим — **plausible-but-wrong**. Claim звучит убедительно, субагент его «подтверждает», но компилятор/код говорит обратное. Несущие и неожиданные claim'ы проверяй САМ, эмпирически (см. Жёсткие правила).

**Ground-truth включает данные, которые нужно ПОЛУЧИТЬ, а не только код, который нужно прочитать.** Для количественных claim'ов (тайминги, пороги, «уложится в ≤Xs», «доминирует пакет Y») первоисточник — это **замер**, а не число из документа. Не принимай оценку на веру и не понижай находку до «вероятно ок», когда вопрос закрывается 5-минутным замером / симуляцией / калибровкой по реальной истории. Эталон дисциплины — `.claude/ci-acceleration-poc-ru.md`: он сам ведёт ревизии «факты перепроверены эмпирически» (Ред. 16: `turbo --dry=json` на репо нашёл **3 бага** центрального скрипта) и явно метит **измеренное** (`32m 29s` на real CI run) против **оценочного** («arm64-дельта неизвестна → замер, не гипотеза»; «`K=5` — стартовый, подобрать по замерам»). Аудит обязан: (1) для каждой load-bearing-цифры определить — измерена она или предположена; (2) если предположена и гейтит решение — добыть данные или явно пометить ASSUMPTION+RISK.

**Claim'ы о СТОРОННИХ продуктах — отдельный первоисточник: сам продукт, не репо и не память.** Если документ ссылается на чужой API / поведение / лимит / опыт («turbo делает X», «GitHub Actions arm64 free для public-repo», «Nx Cloud умеет Y», «Vite-плагин поддерживает Z», «browser API ведёт себя так») — проверь в первоисточнике ПРОДУКТА (его CLI/`--help`/прогон, context7-доки, исходники, офиц. доки, web), а НЕ по training-памяти: стек bleeding-edge (TS 6 / turbo 2.9 / ESLint 10 / Vitest 4…), и память систематически отстаёт от актуальных версий. «Я знаю, как работает X» без захода в источник — это plausible-but-wrong в чистом виде (см. memory `feedback_no_hallucinated_spec_facts`).

## Классификация утверждений (определи тип КАЖДОГО claim'а)

1. **CODE-FACT** — ссылка на файл/функцию/строку/regex/сигнатуру API/поведение. Проверяется чтением реального кода.
2. **TYPE/COMPILER** 🔬 — утверждение о поведении TypeScript (`keyof`, `Omit`, inference, module augmentation, index-signature). Проверяется ТОЛЬКО компиляцией — никогда рассуждением и никогда доверием агенту.
3. **CROSS-REF (внутренний)** — цитата на ВНУТРЕННИЙ док/RFC/секцию/issue/номер-решения. Должна резолвиться в существующий таргет с заявленным содержимым.
4. **EXTERNAL/THIRD-PARTY** 🌐 — утверждение о СТОРОННЕМ продукте: его API / поведение / лимит / прайсинг / «как делает инструмент X» (turbo, Nx, GitHub Actions, Vite, Vitest, browser API…). Первоисточник — САМ продукт: установленный CLI (`--help`/прогон), context7-доки, его исходники/репозиторий, офиц. доки (WebFetch), web (WebSearch). НЕ из памяти (training лагает на bleeding-edge) и НЕ из утверждения RFC.
5. **COUNT/INVENTORY** — счётчики пакетов, «N адаптеров», «9 инвариантов», **а также usage/frequency-счётчики, которыми RFC оправдывает УДАЛЕНИЕ фичи («used once», «barely used», «dead code», «нигде не вызывается»)**. Пересчитай сам: инвентарь — `ls`/скан; usage-частоту — эмпирически (`gh pr list --search`, `git log -S<маркер>`, code-скан по вызовам), НЕ со слов автора / по памяти. «Remove-because-unused» — проверяемый claim, а не мотивация (в сессии-родителе «`#trivial` использован однажды» → `gh pr list` показал 2 PR-title). **Family-claim — свойство, заявленное разом для ВСЕХ N копий/вариантов/файлов** («все 5 копий byte-identical», «эти 4 варианта читают `?n=`») — проверяй на КАЖДОМ члене семьи детерминированным сканом (`md5` всех копий, скрипт-инвентарь по каждому файлу), никогда образец+аналогией: «проверил один, обобщил на пять» — plausible-but-wrong, переживающий даже само-аудит дока. (Сессия bench-app-linter: Rev-2 сам заявлял «byte-identical, verified empirically» — md5 ВСЕХ копий дал расхождение в 4/5 файлов; из «4 вариантов на `?n=`» реально читали его 2/4.) Инвентарь «где живут копии X» собирай и по **смежным слоям-потребителям** (драйверы/раннеры/тест-фикстуры/CI-скрипты), не только по слою, который док называет: потребитель часто держит собственную hardcoded-копию контракта. (Та же сессия: RFC инвентаризовал spec-копии в `_shared/` + angular-инлайны, а сценарные драйверы несли ТРЕТЬЮ копию свип-констант — целая дрейф-поверхность мимо инвентаря и мимо проектируемого паритет-чека.) **Инвентарь ссылок при переименовании/удалении пакета обязан отдельным пунктом покрывать КОРЕНЬ репо** — root-конфиги (codecov/sonar/size-limit/knip/CI-guard-скрипты) держат пакетные списки, а walk по подкаталогам корень не захватывает; плюс читай guard-скрипты (`check-*-scope` и т.п.): их заголовки называют файлы-потребители списков, отсутствующие в любом import-графе. (Сессия engine-merge: `codecov.yml` с 3 компонентами и двусторонним pre-commit-гейтом нашёлся только через header `check-coverage-scope.mjs` — главная S1-находка аудита лежала вне скана подкаталогов.)
6. **ALGORITHM/PSEUDOCODE** — читается как исполнение: ordering-баги, недостижимые ветки, необработанные edge-кейсы, расхождение сигнатуры вызова и определения. **Если док заменяет правило парсинга/классификации — прогони НОВОЕ правило по полному словарю форм СТАРОЙ грамматики** (и валидных, и отвергаемых, включая реверс- и двойные-глифы): каждая старая форма обязана получить названный исход (валидна / та же ошибка / новый рецепт); форма, тихо меняющая интерпретацию или тихо легализующая бывший reject, — находка. (Сессия m1-drop: правило среза `?` тихо превращало сегодня-валидную `/:id??tab` в мёртвую query-декларацию, а сегодня-отвергаемые `?<x>`/`a<b`/`a>b` — в тихо-принятые; вскрыто только перебором словаря форм прогоном на dist.)
7. **DESIGN-DECISION** 🔁 — выбор, принятый в прозе («Принято», О-N). Должен быть пропагирован в КАЖДЫЙ downstream-артефакт: convention-таблицы, псевдокод, DoD этапов, Risks, сводную таблицу. **Если решение меняет форму/сигнатуру осевого поля или API, а док даёт список затронутых/незатронутых потребителей — построй инвентарь читателей этого поля (и вызовов меняемых сигнатур) НЕЗАВИСИМО**, сканом по всем слоям — включая symlink-shared источники и каналы мимо названного механизма (док говорит «sweep интерцепторов» — а читатели бывают не-интерцепторными), — и сравни со списком дока: потребитель вне списка — находка; «не затронут» доказывается инвентарём читателей, не доверием списку. (Сессия url-model-v2: независимый sweep читателей `state.params` вскрыл sources / shared/browser-env / preload / logger вне списка «не затронуты» §7.5 и вне плана Этапа 3 — три дыры скоупа, одна с silent-обеднением пользовательских колбэков.)
8. **INVARIANT/TEST-PLAN** — предлагаемые инварианты/тесты. Проверяется на корректность (свойство реально держится с учётом ОСТАЛЬНЫХ решений?) и неизбыточность (не дубль другого инварианта). **Особый под-класс — критерий «эквивалентность текущему поведению» (fuzz-паритет, golden-сравнение): если тот же док обещает чинить известные баги, критерий обязан явно исключать баг-грани (на них эталон — корректный вывод, расхождение ожидаемо), иначе он самопротиворечив — первая же генерация на баг-грани уронит прогон.** (Сессия invertible-grammar: спайк-критерий «выходы ≡ текущим codec'ам» на срезе, намеренно включающем optional-leading-баг #1147, — BLOCKER: половина omit-генераций попадает в грань.)
9. **PERF/EMPIRICAL-ESTIMATE** 📊 — количественный claim: тайминг, порог (`K=5`), «уложится в ≤Xs», «пакет Y доминирует wall-time», счётчик, варьирующийся по входу. Первоисточник — **замер / симуляция / калибровка по реальной истории**, не число из документа. Проверь дополнительно: помечено ли явно как измеренное vs оценочное.
10. **PROSE/MOTIVATION** — мотивация/рамка, не привязанная к коду. Низкий приоритет; проверяется лишь на внутреннее противоречие.

## Протокол на каждый claim

1. **Классифицируй** (тип выше) и выдели первоисточник, который его докажет/опровергнет.
2. **Собери ground-truth.** Для широты — fan-out `Explore`-агентов, сгруппированных по кластерам (типы; конкретный пакет; смежные доки; инфра/тесты); каждому — точные вопросы + требование цитировать `file:line` и выносить вердикт CONFIRMED/REFUTED/PARTIAL/NOT-FOUND. Агенты должны читать **рабочее дерево текущей ветки** (read-only, без worktree-изоляции — иначе ветвление от origin/master даст аудит устаревшего кода).
3. **Несущие и неожиданные claim'ы перепроверь САМ, эмпирически** — НЕ доверяй вердикту агента на load-bearing-утверждении:
   - **TYPE/COMPILER:** напиши scratch-`.ts`, прогони `npx tsc --noEmit --strict --ignoreConfig <file>` (флаг обязателен — иначе TS5112 на найденном tsconfig); читай, какие именно строки дали ошибку. (Пример из сессии-родителя: агент «опроверг» claim про `Omit` выдуманным результатом компиляции; реальный `tsc` доказал, что claim ВЕРЕН — `keyof` с индекс-сигнатурой коллапсирует в `string|number`, и `Omit` ничего не убирает.)
   - **regex/parsing:** прогони паттерн на реальных строках-нарушителях (`node -e`), не «на глаз». **Инвентарь regex-литералов в коде НЕ собирай грепом:** метасимволы искомой строки (`[^>]`, скобки, кванторы) + вариативность grep-бинаря (BSD/ugrep) дают молчаливую пустоту на реально существующих строках, а пустота читается как «отсутствует» (класс feedback_tool_error_not_absence). Первичный инструмент инвентаризации — детерминированный скан (node-скрипт: `fs` + `String.includes`) или `Read` файла; grep — лишь вторичное подтверждение НАЛИЧИЯ, никогда доказательство отсутствия. (Сессия-родитель: два грепа подряд «не нашли» `<[^>]+>` в buildParamMeta.ts — ложное «desync устранён» успело попасть в RFC ещё до аудита; вскрыто только Read + node-сканом.) Это касается и **glob-расширений**: grep-verify цитаты/комментария обязан покрывать ВСЕ реальные расширения цели — `--include='*.ts'` молча пропускает `.tsx`/`.svelte.ts`/`.mts` — и вердикт «отсутствует» выносится только после `Read` конкретного файла. (Сессия rfc-1435: grep с `*.ts` «не нашёл» parity-литерал в solid `useRouteEnter.tsx:105` — ложный REFUTED предотвращён только Read'ом.) Node-скан сам тоже валидируй `Read`'ом — минимум один позитив и один негатив на каждое извлекаемое поле; классические артефакты: numeric separators (`30_000` → regex-обрезок «30») и совпадение токена в **комментарии** (`includes` не отличает код от комментария, объясняющего, почему фичи НЕТ — так конфиг type-guards с текстом «No --expose-gc» посчитался как имеющий флаг).
   - **code-path:** открой функцию и протрассируй конкретный вход до точки, где решается поведение (порядок веток!). (Так найден silent-баг optional-splat: `segment.endsWith("?")` перехватывает `*slug?` в optional-param-fork раньше splat-ветки.) **Если пакет собран — прогони заявленное поведение напрямую вместо/поверх трассировки:** сверь свежесть dist (mtime dist новее последнего src-коммита: `stat` + `git log -1 -- <pkg>/src`), затем `node -e` + `require('<pkg>/dist/…')` на заявленном входе. Исполнение бьёт трассировку и попутно ловит соседние контракт-claims. (Сессия invertible-grammar: три бага §1.3 доказаны прогоном за минуты — `buildPath('home',{})` → `"//home"` → `match` → undefined — и тот же прогон попутно REFUTED соседний claim «parse тотальна»: URIError на `%E0%41`.) **Если RFC включает собственный проверочный сниппет/гейт** (`node -e`-страж версии, валидатор, regex-чек, выданный за доказательство свойства) — он сам claim, а не инструмент: прогони его поперёк ВСЕЙ граничной полосы, которую он берётся гейтить (чуть-ниже / на / чуть-выше порога), не только на target/happy-значении. Гейт, пропускающий значение, которое заявляет отбраковывать, — ложно-зелёный BLOCKER. (Сессия-родитель: страж `(min===0 && pat<=1)` для «≥ 11.1.3» пропускал 11.0.2–11.1.2 — поймано только прогоном по диапазону, а на 11.8.0 он зелёный.) **Claim о `production`-поведении, зависящий от ОПЦИОНАЛЬНОГО слоя (плагин / флаг / dev-guard), проверяй на ДЕФОЛТНОМ/минимальном конфиге, не максимальном** — он может держаться лишь с opt-in-компонентом, которого в дефолте нет. (Сессия parse-segment: «production gate-first» держался лишь С `validation-plugin`; дефолтный bare core — валидация opt-in — ПРИНИМАЛ `/:y*`, что и вскрыло BLOCKER, замаскированный максимально-конфигурным чтением.) **Проба, читающая ПОТОК СОБЫТИЙ/логов, обязана зафиксировать снимок ДО своих же пост-вызовов** (диагностических navigate/cleanup-хелперов) — иначе события артефактов читаются как события испытуемого сценария; неожиданную event-находку ре-исполняй изолированной пробой без хелперов. (Сессия commit-gate: «spurious TRANSITION_ERROR на guard-пути» оказался SAME_STATES-эмитом post()-хелпера самого пробника — ложная грань дошла до RFC, вскрыта только чистым ре-прогоном.)
   - **Расхождение оракулов (IDE/tsserver/линтер vs сборка/прогон):** прав тот инструмент, который ИСПОЛНЯЕТ заявленный контракт — «builds green» доказывается сборкой (+ артефакт, дошедший до bundle/dist), не отсутствием IDE-подчёркиваний, и наоборот; второй оракул фиксируй в отчёте как residual-шум, не как вердикт — особенно если репо УЖЕ документирует свой IDE-шум как ложный. (Сессия bench-app-linter: Q6-проба «angular может импортировать top-level `_shared`» — сборка зелёная + маркер в бандле, а IDE тем же моментом выдал TS6059 «not under rootDir»; доверие IDE-диагностике инвертировало бы ответ на открытый вопрос RFC.)
   - **Claim о сложности/инкрементальности — проверяй на ДВУХ уровнях: примитив И системный вызывающий путь.** Инкрементальный/O(1) примитив пакета не доказывает инкрементальность фичи — интеграция может сводить его к полному ребилду; трассируй от публичного API вниз до примитива. (Сессия invertible-grammar: «addRoute O(1)» — примитив `insertIntoTrie` действительно per-route и не зависит от числа роутов, но `core.addRoute` гоняет полный `rebuildTree` — новый матчер + `registerTree` всего дерева; claim REFUTED на системном уровне, а RFC опирался на него в 4 секциях, включая аргумент против отклонённой альтернативы.)
   - **CROSS-REF (внутренний):** `grep`-ни цитируемый файл на цитируемый заголовок/секцию — резолвится ли. Не резолвится, а репо недавно пережил реорганизацию (merge/rename/split пакетов)? Прежде чем выносить NOT-FOUND, проверь предка реорганизации (`git show <merge-commit>^:<старый-путь>`): ссылка может быть валидной на файл, потерянный при реорганизации — это ДВОЙНАЯ находка (STALE-CITATION у дока + потерянный артефакт у самой реорганизации, отдельный репорт владельцу), а не битая цитата. (Сессия url-model-v2: «INVARIANTS Matching #25» не резолвился в engine/INVARIANTS.md — `git show e50042ac^` показал его в пре-мерж `path-matcher/INVARIANTS.md:45`; при engine-merge оба слойных INVARIANTS-файла не переехали.)
   - **EXTERNAL/THIRD-PARTY:** проверь в первоисточнике ПРОДУКТА, по убыванию силы: (1) **установленный инструмент** → запусти его (`<tool> --help`, реальный прогон/вывод — сильнейшее); для **существования/версии npm-пакета — `npm view <pkg> version`** (registry-факт; `npmjs.com` часто 403'ит WebFetch, а `E404` = пакета нет — валидное доказательство ОТСУТСТВИЯ, сильнее «я не нашёл»); **для статуса github-issue `/issues/N` (через API) НЕДОСТАТОЧЕН** — `state_reason: "completed"` бывает административным закрытием БЕЗ code-fix; cross-check `/issues/N/timeline` (есть ли merged-PR / release-tag у close-события) + `npm view <pkg> version` (вышел ли релиз с фиксом). `closed` ≠ resolved-in-code; один эндпоинт ведёт в обе стороны (closed-as-fixed ↔ still-broken). (Сессия-родитель: #822 `closed/completed` 6 дней назад, но `/timeline` без merged-PR + latest changesets без новой опции → workaround всё ещё за пользователем; вывод «фикс есть, peer-правка не нужна» был бы ложным.) (2) **context7** (`resolve-library-id` → `query-docs`) для актуальных доков библиотеки/CLI/сервиса; (3) **офиц. доки** через `WebFetch`; (4) `WebSearch`. Для load-bearing — cross-check ≥2 источника. Память НЕ источник. Не проверяемо (платное/закрытое/нерелизнутое API) → пометь **ASSUMPTION**, не утверждай. (Примеры сессии-родителя: «turbo не шардит matrix OOTB / нет DTE» — CLI `--help` + context7 Turborepo; shape `turbo query affected` — прогон CLI; «arm64-runners free для public-repo» — внешний GitHub-факт → офиц. доки/web.) **Когда RFC даёт shell-команду, чтобы ДОБЫТЬ значение для конфиг-поля/токена** (хэш/id для вставки в `packageManager`, lockfile, `.npmrc`) — прогони её и сверь **формат вывода** (кодировку/алгоритм/длину) с форматом ЦЕЛЕВОГО поля, не только корректность значения: правильное значение в неправильной кодировке тихо ломает шаг и проваливает его же цель. (Сессия-родитель: `npm view … dist.integrity` отдаёт `sha512-<base64>`, а поле `packageManager` требует `+sha512.<hex>`; `dist.shasum` вовсе sha1 — наивная вставка = малформед.)
   - **COUNT:** `ls`/`grep` и пересчитай (в сессии-родителе агент насчитал 33 пакета, реально — 32).
   - **PERF/EMPIRICAL:** прогони реальную команду и засеки (`time`, turbo `--dry=json`/`--summarize`, `.turbo/runs/*.json`); порог — откалибруй по реальной истории (`gh pr list` + `git diff`/turbo affected на последних N PR — распределение, не одна точка); скрипт-эскиз — прогони на реальных сценариях (его вывод = функциональный unit-тест без обвязки). Если репо ведёт трекаемый perf-ledger — бенч-клеймы сверяй с ним, ища сценарий по синонимам: имя в RFC может не совпадать с именем в ledger, и пустой grep по RFC-имени ≠ отсутствие данных (сессия url-model-v2: «query-scaling» в RFC = `search-param-scaling` в ledger, где посылка «лидерство 4×» оказалась «4/5 когорт + code-traced lag в пятой»). Если замер ПРИНЦИПИАЛЬНО недоступен (нужен CI-hardware / прод-трафик) — пометь claim как **ASSUMPTION** и вынеси, что решение висит на неизмеренном числе; НЕ выдавай оценку за факт.
   - **Exit-код пробы снимай с самой команды, не через пайп** (`cmd; echo $?` или `PIPESTATUS`) — пайп возвращает статус ПОСЛЕДНЕЙ стадии, и зелёный хвост (парсер/`jq`/`head`) маскирует красную пробу. Вердикт — по наблюдаемому выводу пробы; exit-код лишь сверка. (Сессия R3: `validator --json | node-парсер; echo $?` дал `exit=0` от парсера — на несущем факте «валидатор отвергает пустые changeset'ы» это читалось бы как ПРОТИВОПОЛОЖНЫЙ вывод; спасло только то, что JSON-ошибки были в выводе перед глазами.) Для **async-пробника** (node async-IIFE) `exit=0` вдобавок ≠ «дошёл до конца»: невыполненный `await` при опустевшем event loop даёт молчаливый exit 0 и срезает хвостовые кейсы — печатай маркер-строку на КАЖДЫЙ кейс и суди по напечатанным маркерам, не по коду выхода. (Сессия rfc-1435: вечно-pending leave-промис завесил ВТОРУЮ навигацию того же пробника — CASE2/3 молча не выполнились при exit=0.)
   - **Чистота рабочего дерева для git-diff/affected-проб (скил read-only → одноразовый worktree).** Инструменты, считающие affected по git (`turbo affected`/`turbo query affected`/`nx affected`/`--filter='...[ref]'`/`lint-staged`), **сворачивают незакоммиченные правки рабочей копии в результат** — грязное дерево даёт ложный affected (твои правки читаются как «затронуто»). Гоняй такую пробу в **detached-worktree на нужном ref** (`git worktree add --detach <scratch> <ref>` → прогон → `git worktree remove`; turbo-бинарь из основного репо через `--cwd`/`--skip-infer`), удаляемом в конце — скил ничего не пишет в репо, worktree безопасно сносится. Сначала `git status`; дерево грязное и проба про affected → worktree обязателен. (Сессия-родитель: A5-проба `turbo query affected` на грязном `packages/core` вернула `core` в affected → выглядело как REFUTED линчпина; чистый worktree дал `[rx]` без core.)
   - **Hardware-bound vs hardware-invariant — ключевое различие, не путать.** *Логика/структура* (классификация, routing-решение, граф affected, счётчики) — hardware-**инвариантна**: локальный прогон = истина и для CI, переносится 1:1. *Тайминги* — hardware-**bound**: локальная абсолютная цифра на CI-цель НЕ переносится, и сравнивать её с CI-оценкой дока **нельзя** (apples/oranges → не пиши «REFUTED» из локального замера CI-числа). Переносится лишь **отношение**, и только когда обе величины упираются в ОДИН ресурс (CPU/CPU — `angular` vs `preact` bundle: да; CPU/IO — dry-run-операция vs checkout+install: нет, разные узкие места). Абсолютный CI-таргет (`≤Xs`) меряется ТОЛЬКО на CI-железе — иначе ASSUMPTION. Прежде чем мерить тайминг локально, спроси: «это про скорость (hardware-bound → нужен CI или same-bottleneck ratio) или про корректность (hardware-invariant → локально достаточно)?».
4. **Раздели «claim технически верен как сформулирован» и «вывод держится».** Посылка может быть верна при неверном выводе — и наоборот (верный вывод на слегка кривой цитате).
5. **Для DESIGN-DECISION — охота за непропагированными копиями.** `grep`-ни сигнатурные строки старого решения по всему документу; каждое вхождение, противоречащее принятому решению — находка. Решение в прозе ≠ решение в алгоритме: если псевдокод не enforce-ит решение, реализация воспроизведёт баг, против которого решение и принято.

## Классификация находок (severity)

- **BLOCKER** — реализация по документу даст неверный код / ложно-зелёный тест / воспроизведёт описанный баг (непропагированное решение в алгоритме; ordering-баг в псевдокоде; инвариант, противоречащий другому решению; внутреннее противоречие в контракте/представлении данных).
- **SHOULD-FIX** — не ломает реализацию, но недоспецифицировано/вводит в заблуждение (over-promise механизма; недоопределённая область инварианта; неозвученный trade-off).
- **MINOR** — цитата/факт/счётчик мимо при верном по сути выводе; редундантный инвариант; формулировка.

Каждая находка: тип claim'а + `file:line` доказательства из ground-truth + **конкретная однострочная правка**.

## Починка (только при `--fix`)

Правится МЕХАНИЧЕСКОЕ и однозначное:
- фактические исправления (неверная сигнатура/счётчик/цитата → верная, подтверждённая ground-truth);
- пропагация уже-ПРИНЯТОГО решения во все таблицы/псевдокод/DoD/сводку (+ `grep`-доказательство нулевого остатка старой формулировки);
- снятие противоречия, когда корректная резолюция однозначна (одна сторона уже канонизирована решением);
- ordering-фиксы в псевдокоде, восстанавливающие заявленную семантику.

При правке псевдокода **читай его как исполнение** — наивная вставка может оказаться мёртвым кодом (в сессии-родителе throw для `[[...slug]]`, вставленный после dot-check, был бы недостижим, т.к. `[[...slug]]` содержит `.`; пришлось переупорядочить проверки и сузить dot-check до имени токена).

НЕ правится автоматически: выбор IN/OUT фичи, смена архитектуры, переименование API — любой вопрос, требующий решения автора → выносится в отчёт как открытый.

Подними ревизию в шапке + добавь запись в журнал ревизий (что и почему).

## Жёсткие правила

- Доказывай **первоисточником**, не документом. RFC — не свидетель собственной корректности. Это включает код, который RFC **цитирует** (комментарий, заголовок скрипта, docstring): проверяй против *первичного* источника (config / поведение / прогон), а не против цитаты — комментарий устаревает, RFC наследует устаревание, а claim, звучащий и в RFC, и в цитируемом им коде, ощущается дважды-подтверждённым, оставаясь **одним эхом**. (Сессия-родитель: и RFC, и `cap-major.mjs:15` хором «no config to prevent this» — а реальный `config.json` уже содержал `onlyUpdatePeerDependentsWhenOutOfRange`; claim REFUTED только походом в config, не в цитату.) **Тот же статус — у audit-log / ревизионной записи в шапке САМОГО документа, особенно оставленной прошлым прогоном `/audit-rfc`:** её вердикты (CONFIRMED / No BLOCKER / «ready to implement» / выбранный bump) — subagent-grade claims, не установленный факт; несущую посылку под выводом «ready/patch/GREEN» перепроверь исполнением с нуля — прошлый прогон мог рубер-стемпнуть plausible-but-wrong. (Сессия parse-segment: Rev-1 audit-log заявлял «production gate-first → patch → No BLOCKER → ready»; bare-core прогон `createRouter([{path:'/:y*'}])` опроверг посылку — BLOCKER, а не patch.) Это касается и документа, написанного ЭТОЙ ЖЕ сессией: свежесть прогона защищает от устаревания, но не от артефактов интерпретации — неожиданные грани, не подтверждённые независимым источником, ре-исполняй изолированной пробой прежде чем принять их как ground-truth.
- На load-bearing/неожиданном claim'е НЕ доверяй вердикту субагента без собственной перепроверки (компиляция/прогон/трассировка). Subagent-резюме — гипотеза, не факт.
- TYPE/COMPILER/regex/algorithm — только эмпирика (`tsc`/`node`/трассировка), никогда «логически очевидно».
- Claim о стороннем продукте (API/поведение/лимит/опыт) проверяется в **первоисточнике продукта** (CLI/context7/доки/исходники/web), НЕ из памяти и НЕ из утверждения RFC. Стек bleeding-edge — training систематически отстаёт; «я знаю, как работает X» без захода в источник = находка-в-зеркале. Для установленного инструмента прогон его CLI бьёт любые доки.
- Решение «исправлено» ТОЛЬКО когда пропагировано везде; докажи `grep`-ом нулевой остаток старой формулировки. Резидью-паттерн включай с уникальным якорем строки (ключ строки / первая ячейка), не только заменённым хвостом — одинаковые хвосты соседних строк таблицы дают ложный остаток на строке, которую менять и не требовалось. (Прогон rfc-1439: хвост «есть ×4 + AOT-only» изменённой M4 легитимно жил и в неизменяемой M3 — потерян верификационный раунд на разборе ложного остатка.)
- Верный вывод на кривой цитате → чини цитату, вывод сохрани. Не выбрасывай хороший вывод из-за плохого адреса.
- Дизайн-выбор автора — не «ошибка», пока не порождает противоречие. Не навязывай вкусовщину под видом находки.
- Раздели **измеренное и оценочное**. Оценка, поданная как факт — находка. Load-bearing-оценка, гейтящая решение, должна быть либо измерена, либо явно помечена ASSUMPTION+RISK с планом замера. 5-минутный замер бьёт хедж-вердикт «вероятно ок» — стремись получить данные, а не обойти их.
- Push/commit не делай без явной просьбы.

## Оговорки (НЕ метить как находку)

- **PROSE/MOTIVATION** не обязана цитировать код — мети только противоречия в ней.
- **CROSS-REF в правильную сущность, но слегка не тот файл/секция** — MINOR (суть важнее адреса), не BLOCKER.
- **Субъективное дизайн-предпочтение** без противоречия — не ошибка.
- **Не каждое число нужно мерить** — только load-bearing, гейтящее решение. Цифра, честно помеченная «оценочно, замер в фазе X» с планом — это правильная дисциплина (награждай, не мети как находку). Estimate ≠ ошибка; estimate, выданный за факт — ошибка.
- **Сторонний продукт, чьё поведение нельзя проверить** (закрытое/платное/нерелизнутое API) → пометь **ASSUMPTION** с указанием, что нужен доступ/релиз; не выдавай память за факт. Если RFC сам честно назвал это гипотезой / «надо измерить» (как arm64-дельта) — это дисциплина, не находка.
- **Re-audit гигиена:** после слов автора «исправил» сперва сверь, что читаешь ТЕКУЩИЙ файл — fingerprint (`stat` mtime/size + `md5`); если твой read выглядит неизменным, а автор настаивает — пере-`stat`, кэш инструментов/ФС может отставать от свежего сохранения. Не выноси «изменений нет», не сверив fingerprint. (RFC под `.claude/` — gitignored: `git diff`/`git status` для него всегда пусты — верь содержимому/хэшу, не git.) То же — для **ground-truth файлов, изменившихся ПО ХОДУ прогона** (нотификация системы / линтер / правка пользователя): line-цитаты по такому файлу пере-сними `grep -n` ПОСЛЕ изменения — вердикт «line-цитата верна/сдвинута», вынесенный по до-изменения read'у, невалиден. (Прогон rfc-1439: react/CLAUDE.md изменился внешней правкой посреди аудита — цитируемый диапазон :169-174 подтверждён только повторной съёмкой.)

## Валидация и отчёт

1. **Отчёт-таблица:** claim | тип | ground-truth (`file:line` / замер / context7·CLI·доки продукта) | вердикт (CONFIRMED / REFUTED / PARTIAL / STALE-CITATION / NOT-FOUND / MEASURED / ASSUMPTION) | severity | правка.
2. Раздели находки на **BLOCKER / SHOULD-FIX / MINOR**; для каждой — однострочный конкретный фикс с `file:line`.
3. **Вердикт о готовности:** оцени архитектуру (зелёная/нет) ОТДЕЛЬНО от исполнения; перечисли, что закрыть до «готов к реализации»; явно раздели «ревизия документа» и «пересмотр архитектуры».
4. При `--fix`: после правок — `grep`-доказательство нулевого остатка старых формулировок + новый fingerprint файла; перечитай наиболее переструктурированные зоны, убедись, что псевдокод/таблицы не порвались.

## Замечание

RFC/дизайн-доки под `.claude/` обычно gitignored (не трекаются) — изменения не видны в `git`, верифицируй содержимым и fingerprint'ом. Правки документа changeset не требуют. Push не делай без явной просьбы.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Валидация п.1–3) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного RFC/стороннего продукта (разовую) → в отчёт/память по этому RFC, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ аудиту ДРУГОГО документа.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset; коммить строго `git commit -o .claude/commands/<skill>.md -m …` (only-путь): staged-зона общая, в репо может работать параллельная сессия — голый `git commit` после неудачной первой попытки (напр. commitlint-реджект scope) утащит твой файл в её коммит (инцидент url-model-v2: правка скилла уехала в чужой feat(ci)-коммит).
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
