Проведи white-box аудит unit-тестов пакета: заставь их ходить через ПУБЛИЧНОЕ API, чем структурно вскрой недостижимый/мёртвый код, и резолвь КАЖДУЮ находку (simplify / v8-ignore / KEEP-narrow exception), доказывая эмпирически — замером покрытия и прогоном, не рассуждением.

Входные данные:
$ARGUMENTS

Формат аргументов:
- **Модуль (ОБЯЗАТЕЛЬНО)** — какой пакет аудировать. Любое из:
  - путь к пакету: `packages/path-matcher`, `packages/route-tree`
  - npm-имя: `@real-router/logger`, `search-params`
  - короткое имя директории: `path-matcher`, `fsm`, `event-emitter`
- Опционально `--report-only` — только Фазы 0–3 (guardrail + замер + карта кандидатов), без правок. По умолчанию: аудит + переписывание + резолюция находок под гейтом подтверждения на каждой src-правке.

> **⛔ Без аргумента НЕ запускать.** Если `$ARGUMENTS` пуст или модуль не резолвится в `packages/<dir>` — остановись, попроси указать модуль и приведи пример (`/whitebox-audit path-matcher`). НЕ выбирай модуль сам, НЕ гони по всем пакетам, НЕ угадывай.

> **Диспетчер.** Это НЕ `/mutation-score` (тот убивает выживших мутантов точечными тестами — ось «сила ассертов») и НЕ `/deep-audit` (тот аудирует конкретный метод/namespace `@real-router/core`). `/whitebox-audit` — про **границу** теста: unit-тесты обязаны ходить через public API, тогда недостижимый код всплывает как непокрытая ветка. Пересекается с `/mutation-score` (оба про качество тестов), но дополняет его: whitebox = ЧЕСТНОСТЬ границы, mutation = сила ассертов внутри границы.

**Прецеденты (изучи перед прогоном — это готовая эталонная реализация):**
- `packages/core/eslint.config.mjs` — первый white-box guardrail + аудит-доктрина в комментарии.
- `packages/path-matcher/eslint.config.mjs` + `tests/unit/*` — ПОЛНАЯ миграция всеми тремя исходами (мигрирован / v8-ignore / KEEP-exception); `createSegmentNode.test.ts` и `percentEncoding.test.ts` — образцы KEEP-исключений с заголовком-обоснованием.
- `.claude/whitebox-test-audit-2026-06-23.md` — исходный аудит core (если есть локально).

## Философия (зачем правило вообще)

White-box-тест может убить мутанта, **не усилив ПУБЛИЧНЫЙ контракт** — тогда зелёный тест не доказывает то, на что полагается консьюмер, и **прячет публично-недостижимый (мёртвый) код** от 100%-гейта покрытия. Инвариант правила: **зелёный тест ⟺ реальная гарантия для консьюмера**. Форсирование публичной поверхности делает каждый ассерт настоящей гарантией И вскрывает мёртвый код как непокрытую ветку. Выигрыш — в честности, а не только в детекции мёртвого кода (он есть даже при нулевом мёртвом коде).

## Фаза 0 — Локализация + гейт аргумента

1. Резолвь модуль → `packages/<dir>` (путь / npm-имя / короткое имя, как `/mutation-score`). Не резолвится или `$ARGUMENTS` пуст → **СТОП** (см. гейт выше).
2. Найди тестовую директорию для «unit-подобных» тестов пакета: обычно `tests/unit/`, у некоторых пакетов — `tests/functional/` (core). Определи по факту (`ls`). Property (`tests/property/`), stress (`tests/stress/`), helpers, benchmarks — **вне области** (у них легитимно другое отношение к внутренностям, см. Фазу 5).
3. Сверься с публичной поверхностью: прочитай `src/index.ts` пакета — что РЕАЛЬНО экспортится. (Часть «внутренних» на деле публичны — не суди по имени модуля.)
4. **Fast-path «пакет уже пред-мигрирован» (`grep` импортов ПЕРВЫМ делом).** Прежде чем строить карту кандидатов, `grep -rn "from \"" tests/` по ВСЕМ сьютам. Если в unit/functional-тестах **ноль** `src/*`-импортов (все идут из имени пакета) **И** unit-гейт-прогон (`vitest run`, functional-alone) даёт **100% stmt/branch/func/line** по каждому src-модулю (в т.ч. `internal/*`) — пакет уже полностью честен: публично-недостижимого кода нет, Фазы 2–5 сворачиваются в ноль (нечего классифицировать/мерить/резолвить), а весь дедаврабл — **добавить guardrail с ПУСТЫМ allowlist**, фиксируя уже-честное состояние (не чинить регресс). Это НЕ «пропустить аудит»: пустой allowlist здесь — доказанный результат (100%-functional-coverage = каждая ветка достижима публично), а не необследованность. Прецедент: `type-guards` (2026-07-17, весь набор functional+property+stress уже на public API). Дисциминирующую силу guardrail'а всё равно докажи мутацией (инъекция `src/*`-импорта в functional-тест → `no-restricted-imports` краснеет).

## Фаза 1 — Guardrail (eslint white-box правило)

1. Если у пакета НЕТ локального `eslint.config.mjs` — создай `[...eslintConfig, {блок}]` (`import eslintConfig from "../../eslint.config.mjs"`), как у `path-matcher`/`core`. Если есть — добавь блок.
2. Блок: `files: ["tests/<unit-dir>/**/*.test.ts"]`, правило `no-restricted-imports` с `patterns: [{ group: ["**/src", "**/src/**"], message: "…" }]`. В message: публичная поверхность ЭТОГО пакета (импорт из имени пакета, не `src/*`) + подсказка «public-символ через src-путь → сменить путь; чистая внутренняя функция → в tests/property; недостижимо публично → KEEP-narrow ignores-исключение здесь». Скопируй тон из `path-matcher`/`core`.
3. Прогони eslint по тест-директории → собери СПИСОК `src/*`-импортов (нарушения). Это карта кандидатов. (Правило пока «краснит» — норм; резолвим по ходу.)

## Фаза 2 — Классификация каждого нарушения

Для каждого импорта из `src/*` реши по `src/index.ts`:
- **Class-1 — публичный символ через src-путь** (символ ЕСТЬ в индексе, импортится как `../../src/X`): тривиально репойнтнуть на имя пакета (`from "<package>"`). Смешанный импорт (часть public, часть internal) — расщепить на два. Чинить сразу. **Гоча:** self-import `from "<package-name>"` в тестах пакета резолвится (Node self-ref + `exports` + tsc `customConditions` + vitest alias) — но без прецедента в пакете **проверь vitest-прогоном И type-check'ом**, что резолвится, до массовой правки.
- **Class-2 — истинно внутренний** (символа НЕТ в индексе: энкодеры, node-фабрики, токенайзер-примитивы, char-сканы): в Фазу 3.

## Фаза 3 — Замер: что всплывёт непокрытым (эмпирика, не рассуждение)

Для каждого Class-2-файла замерь покрытие ИМПОРТИРУЕМОГО src-модуля от полного набора **МИНУС этот тест-файл**:

```
pnpm -F <pkg> exec vitest run tests/ \
  --coverage --coverage.include='src/<module>.ts' \
  --exclude='**/<file>.test.ts'
```

Непокрытые строки = что ТОЛЬКО этот white-box-файл покрывает напрямую. Затем для каждой такой строки установи, покрыта ли она:
- непрямыми public-тестами других файлов (тогда прямой тест избыточен для покрытия);
- **exempt-property-тестом** (`*.properties.ts`) — **ГОЧА:** property-тесты НЕ гоняются в `vitest run tests/` (default config), их покрытие в unit-гейт НЕ входит; строка, покрытая ТОЛЬКО exempt-property, всплывёт «непокрытой» в unit-гейте — это НЕ мёртвый код, это кандидат в v8-ignore (Фаза 5);
- кросс-пакетно (напр. gate `route-tree` зовёт публичную функцию `path-matcher`) — проверь консьюмеров grep'ом.

## Фаза 4 — Honest-переписывание через public API

Перепиши Class-2-тесты на публичную поверхность:
- ошибки/реджекты → публичная validation-функция (`findSegmentGrammarError`) ИЛИ throw на `registerTree`/фабрике — точно, лоссли;
- извлечённые данные (имя/констрейнт/kind) → `buildParamMeta`/аналог;
- **поведение** (`optional`, декод, матч) → через `match`/`buildPath` — обе формы, реальные входы.

**Ключевой принцип:** ассертить **наблюдаемое консьюмером поведение**, не внутренний кортеж/массив/структуру. Ленивый рерайт слабее: `optional` проверяй `match`'ем обеих форм (take/omit), не отсутствующим полем meta — иначе мутант выживёт. Прогони покрытие src-модуля от ПЕРЕПИСАННОГО файла: **мёртвый код всплывёт непокрытой веткой** (branch < 100%).

## Фаза 5 — Резолюция КАЖДОЙ всплывшей находки (дерево решений)

| Ситуация | Исход | Как |
|---|---|---|
| **Truly dead** — ветка недостижима НИ через public, НИ через property, НИ через другой тест | **SIMPLIFY** (удалить из `src`) | Доказать недостижимость (трассировка + мутация); удалить; прогнать equivalence-property/parity — зелёные. changeset. |
| **Dead-via-public, но покрыто** exempt-property/другим слоем | **v8-ignore + коммент** | `/* v8 ignore next -- unreachable via <public path>; covered by <exempt property/др.> */`. Паттерн core «validator-opt-in: dead in core, covered in plugin». |
| **Reachable, но вердикт публично НЕРАЗЛИЧИМ** (fast-reject, backstopped downstream — напр. `validatePercentEncoding` ← decode try/catch) | **KEEP-narrow exception** | Оставить прямой тест; вынести (при необходимости) в отдельный файл; в `ignores` allowlist + заголовок-обоснование в файле. |
| **Внутренний memory/perf/hidden-class инвариант**, консьюмером ненаблюдаемый (напр. `createSegmentNode` sentinel #1009/#1379) | **KEEP-narrow exception** | Так же; в заголовке — какой инвариант и почему только прямым тестом. |
| Покрытие полностью **избыточно** (public + property уже дают 100%), mutation-ценности нет | **DELETE** | Удалить файл/блок. |

**Не путать критерии:** truly-dead-нигде → SIMPLIFY; dead-via-public-но-покрыто-property → v8-ignore; reachable-но-неразличимо ИЛИ внутренний-инвариант → KEEP-exception. Каждое SIMPLIFY/v8-ignore — доказать эмпирически (мутация + прогон), не рассуждением. KEEP-exception при необходимости выноси в ОТДЕЛЬНЫЙ exempted-файл (не exemptить весь большой mostly-public файл ради одного блока).

## Фаза 6 — Верификация

- **Unit-гейт 100%** (`pnpm -F <pkg> test`) — все src-строки покрыты/ignored.
- **Property** (`test:properties`) + **stress** (если есть) — зелёные (после сигнатурных правок src).
- **Downstream-консьюмеры** — при правке движка прогнать прямых потребителей (`route-tree`, `core`).
- **type-check** (`tsc --noEmit`) + **eslint** (`tests/**` + тронутый `src/**`) чисты; allowlist держит только KEEP-исключения.
- **Мутационно** валидируй ЛЮБУЮ тронутую src-ветку/новый лок-тест: сломать ветку → тест/property краснеет (иначе тест не дискриминирует).
- **Форматирование:** после рерайта `eslint --fix` по тест-директории (prettier/padding).

## Фаза 7 — Оформление

- **changeset** на затронутые public-пакеты. Приватный src (path-matcher/route-tree) → `@real-router/core` **patch** (behavior-preserving cleanup, `#NN`-референс на код-линейку находки). См. `.changeset/README.md`.
- Обнови гочи в `<pkg>/CLAUDE.md`, если менялись тест-конвенции («tests import directly from src/*» → «unit через public API; чистые внутренние — в exempt property; memory/perf-инварианты — KEEP-exception»).
- **Коммиты (каждый зелёный):** миграция + src-находки одного файла-кластера — вместе (покрытие связывает тест и src); правило-guardrail — отдельным коммитом ПОСЛЕ миграций (иначе коммит с правилом до миграций краснит линт). Инкрементально: правило можно приземлять с TODO-`ignores` на ещё-не-мигрированные, освобождая список по мере готовности.
- **push/PR — `--no-verify`** (pre-commit/pre-push = полный turbo, таймаутит; таргетная валидация локально, полная в CI). Не пушить без явной просьбы.

Финальный `eslint.config.mjs` держит **только задокументированные KEEP-исключения** (каждое — с обоснованием в заголовке файла И в конфиге). Все остальные unit-тесты — public-API. Отчёт: карта {файл → исход}, список удалённого/ignored мёртвого кода со ссылкой на почему-недостижимо, список KEEP-исключений с критерием.

## Заточка скилла

После прогона — если всплыл новый класс находки, критерий резолюции или гоча (напр. новый способ, которым замер обманывает; новый вид backstopped-неразличимости) — **допиши этот файл** (таблицу Фазы 5 / гочи Фазы 3) и, при необходимости, комментарий-доктрину в `<pkg>/eslint.config.mjs`, чтобы следующий прогон стартовал с накопленного знания.
