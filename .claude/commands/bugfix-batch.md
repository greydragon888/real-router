Оркеструй батч багфиксов одной PR-группы (несколько issues одного пакета / dependency-chain): один worktree/ветка → per-bug фиксы по методологии `/bugfix` → один общий PR. Один CI-прогон вместо N.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Список номеров issue одной PR-группы (`#736 #738 #749`) — обычно строка PR-очереди из эпика #756 (один пакет или одна dependency-chain). Группируй по **непересекающимся файлам** между группами; перекрытие багов по одному файлу ВНУТРИ группы — норма (один PR их и покрывает).

Этот скилл — **ОРКЕСТРАТОР**. Он НЕ дублирует TDD-методологию: per-bug ядро берётся из скилла `/bugfix` (Фазы 0.4→3 + коммит+changeset). Здесь — только то, что специфично для группы: общая ветка, порядок, один PR. Меняется методология фикса → правится `/bugfix`, этот скилл наследует.

## Шаг 1 — Один worktree/ветка на всю группу

1. **Ведущая задача = первый номер группы** (по dependency-chain — корневой баг). Заведи ветку и worktree ОДИН раз, под ведущую задачу:
   - `gh issue develop <lead> --name <lead>-<slug>` — ветка `<lead>-<slug>`, привязана к ведущей задаче (секция Development). `<slug>` — **общий идентификатор группы** (`path-matcher-fixes`, а НЕ имя одного бага), всё имя ≤ 64 символов. Базируется на `origin/master`; для другой базы — `--base <branch>`.
   - `git fetch origin` → `git worktree add .claude/worktrees/<lead>-<slug> <lead>-<slug>` → **`EnterWorktree` с `path: .claude/worktrees/<lead>-<slug>`** (именно `path:`, не `name:`).
2. Весь батч — внутри этого ОДНОГО worktree, в ветке `<lead>-<slug>`. Несовпадение «номер ветки (ведущая) ≠ номера остальных issues» здесь ожидаемо.

## Шаг 2 — Per-bug цикл (по dependency-chain)

Упорядочи баги по dependency-chain (#756 «Order = recommended execution order»): корневой → зависящие. Для КАЖДОГО бага по очереди:

1. Выполни per-bug ядро **по скиллу `/bugfix`, Фазы 0.4→3** (контекст из issue → RED → GREEN → REFACTOR), но **без** worktree-setup (Фаза 0.3 — сделан в Шаге 1) и **без** финальных фаз (валидация/PR/comment — они в Шаге 3). Все гейты `/bugfix` применяются: эмпирическое воспроизведение (0.6), критика предложенного решения (0.7), флапающий-RED guard (1.3).
   - Если фикс корневого уже закрыл зависимый баг — перепроверь эмпирически (0.6), НЕ пиши лишний фикс, отметь это в отчёте.
2. **Отдельный коммит + changeset на КАЖДЫЙ баг** с его `#<N>` (`/commit-msg` → `/changeset`). Гранулярность критична: review читается по одному дефекту, история привязана к issue, спорный баг выдёргивается без развала всего PR. Перекрытие файлов между багами разделяй `git add -p` по hunk'ам.

## Шаг 3 — Финал на всю группу (один раз)

1. **Точечная валидация** по ОБЪЕДИНЁННОМУ набору затронутых пакетов: на каждый — `pnpm -F <pkg> type-check`, `lint`, `test -- --run` (+ `test:properties` / `test:stress`, если фиксы касались инвариантов / конкурентности / памяти). 100% coverage обязательно. Полный `pnpm build` **не запускай** — финальная сборка делегируется пользователю (сообщи в отчёте).
2. **Docs/wiki sync** одним проходом (`/bugfix` Фаза 5) по всем затронутым пакетам/API — баг-фиксы часто означают расхождение доков с реальным поведением.
3. **Один PR** через `/create-pr` из N коммитов:
   - В теле — **`Closes #736`, `Closes #738`, `Closes #749`: по отдельной строке на КАЖДЫЙ issue группы.** Без этого при merge GitHub закроет только ведущую задачу, остальные останутся открытыми.
   - Заголовок отражает группу/пакет, не один баг.
4. **Push** (только по явной просьбе пользователя): `git push -u origin <lead>-<slug>`. Pre-push `lint:audit` (osv-scanner) в git-worktree известно даёт спурьозные падения (обход дерева от `/`). Если pre-push спотыкается **только** на нём — верифицируй `pnpm lint:audit` в основном чекауте и повтори с `--no-verify`. Прочие падения pre-push — настоящие, не обходи.
5. **Комментарии в issue** (черновики; публикуй только по явной просьбе): по комментарию на каждый баг ИЛИ один сводный со ссылками на коммиты — по ситуации. Содержание — как `/bugfix` Фаза 7 (корень, фикс, тесты, скоуп-решения).

## Гейты (наследуются от `/bugfix`)

- **НЕ коммить / НЕ пуш / НЕ публикуй комментарии в issue без явной просьбы пользователя** — только готовь и показывай.
- Не помечай группу выполненной при падающих тестах или частичном фиксе хотя бы одного бага.

## Итоговый отчёт

- Workspace/ветка: `.claude/worktrees/<lead>-<slug>` на ветке `<lead>-<slug>`.
- Таблица по багам: `#<N>` → корень → тест(ы) → коммит → статус (исправлен / не нужен, если корневой фикс уже покрыл).
- Точечная валидация объединённого набора пакетов (полный `pnpm build` делегирован пользователю).
- Docs/wiki: что проверено/обновлено.
- PR: ссылка (или черновик), `Closes #<N>` для всех issues группы.
- Черновики issue-комментариев.

## Жизненный цикл worktree

Держи worktree и ветку до мёржа PR. После мёржа почисти вручную: `git worktree remove .claude/worktrees/<lead>-<slug>` (worktree, в который вошли через `EnterWorktree path:`, харнесс автоматически не удаляет). `git worktree list` — обзор всех активных батчей.

## Параллельные батчи

Несколько групп можно вести **параллельно** — каждый `/bugfix-batch` живёт в своём worktree (изолированы). Условие чистоты: группы **не пересекаются по файлам** (в #756 PR-очередь разбита по пакетам — пересечений почти нет, PR'ы мержатся независимо). Запускай параллельные группы в **отдельных сессиях** — один Claude в одной сессии работает в одном worktree за раз; параллельность даёт число сессий, не один Claude.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После итогового отчёта вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки. Особый кандидат для батча: проблемы координации (порядок багов, перекрытие файлов между коммитами, `Closes` не закрыл задачу, конфликт ветки).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку или сняла двусмысленность, на которой ты споткнулся. Косметику отбрасывай.
3. **Граница bugfix ↔ bugfix-batch.** Методология одного фикса (RED/GREEN/критерии) → правь `/bugfix`, НЕ здесь. Сюда идёт только то, что про оркестрацию группы (ветка/порядок/PR/параллельность). Не дублируй per-bug инструкции.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Усиливай существующий абзац; не плоди секции.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто → так и скажи: «правок нет».
