Оркеструй батч багфиксов одной PR-группы (несколько issues одного пакета / dependency-chain): один worktree/ветка → per-bug фиксы по методологии `/bugfix` → один общий PR. Один CI-прогон вместо N.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Список номеров issue одной PR-группы (`#736 #738 #749`) — обычно строка PR-очереди из эпика #756 (один пакет или одна dependency-chain). Группируй по **непересекающимся файлам** между группами; перекрытие багов по одному файлу ВНУТРИ группы — норма (один PR их и покрывает).

Этот скилл — **ОРКЕСТРАТОР**. Он НЕ дублирует TDD-методологию: per-bug ядро берётся из скилла `/bugfix` (Фазы 0→3 + коммит+changeset). Здесь — только то, что специфично для группы: общая ветка, порядок, один PR. Меняется методология фикса → правится `/bugfix`, этот скилл наследует.

## Шаг 1 — Один worktree/ветка на всю группу

1. **Ведущая задача = первый номер группы** (по dependency-chain — корневой баг). Заведи ветку и worktree ОДИН раз, под ведущую задачу:
   - `gh issue develop <lead> --name <lead>-<slug>` — ветка `<lead>-<slug>`, привязана к ведущей задаче (секция Development). `<slug>` — **общий идентификатор группы** (`path-matcher-fixes`, а НЕ имя одного бага), всё имя ≤ 64 символов. Базируется на `origin/master`; для другой базы — `--base <branch>`.
   - `git fetch origin` → `git worktree add .claude/worktrees/<lead>-<slug> <lead>-<slug>` → **`EnterWorktree` с `path: .claude/worktrees/<lead>-<slug>`** (именно `path:`, не `name:`).
   - **После входа — `pnpm install` в этом worktree ПЕРЕД любой валидацией/пробой.** Свежий git-worktree не делит `node_modules` с основным чекаутом (он gitignored), поэтому `vitest`/`tsx` там отсутствуют, пока не установишь; pnpm store общий → это в основном линковка (секунды), но без неё первый же `pnpm -F <pkg> test` / запуск пробы упадёт «command not found».
2. Весь батч — внутри этого ОДНОГО worktree, в ветке `<lead>-<slug>`. Несовпадение «номер ветки (ведущая) ≠ номера остальных issues» здесь ожидаемо.

**Возобновление (worktree уже существует).** Не пересоздавай. Сначала `pwd` + `git branch --show-current`: прерванная сессия могла восстановиться **уже внутри** worktree (тогда `EnterWorktree` не нужен — повторный вызов падает «is the current working directory», а `path:`-форма — ENOENT с удвоением пути). Затем `git log master..HEAD` + `git status`: что уже закоммичено / лежит в рабочем дереве — чтобы не делать заново.

## Шаг 2 — Per-bug цикл (по dependency-chain)

Упорядочи баги по dependency-chain (#756 «Order = recommended execution order»): корневой → зависящие. Для КАЖДОГО бага по очереди:

1. Выполни per-bug ядро **по скиллу `/bugfix`, Фазы 0→3** (контекст из issue → RED → GREEN → REFACTOR), **без** финальных фаз (валидация/PR/comment — они в Шаге 3). Worktree/ветку на группу уже создал Шаг 1; `/bugfix` инфраструктуру не заводит — лишь проверяет, что ветка не `master` (в батче это уже выполнено), поэтому отдельных оговорок про worktree-setup не нужно. Все гейты `/bugfix` применяются: эмпирическое воспроизведение (0.6), критика предложенного решения (0.7), флапающий-RED guard (1.3).
   - Если фикс корневого уже закрыл зависимый баг — перепроверь эмпирически (0.6), НЕ пиши лишний фикс, отметь это в отчёте.
   - **Design-call баг** (audit/issue помечает «Design call» — несколько валидных разрешений с разной архитектурной ценой: слой core/plugin/doc, breaking surface): задай `AskUserQuestion` с рекомендацией **ДО** RED. Выбор подхода — решение пользователя, не деталь реализации; реализуешь по доверию — рискуешь полной переделкой фикса. Сам вопрос формулируй по разделу «Как формулировать вопрос пользователю» в `/bugfix` — просто, для человека без накопленного тобой контекста.
2. **Отдельный коммит + changeset на КАЖДЫЙ баг** с его `#<N>` (`/commit-msg` → `/changeset`). Гранулярность критична: review читается по одному дефекту, история привязана к issue, спорный баг выдёргивается без развала всего PR. Когда баги группы трогают РАЗНЫЕ участки одного файла, доводи каждый баг до коммита ПЕРЕД правкой следующего в том же файле — рабочее дерево на момент коммита содержит изменения только одного бага, и хватает обычного `git add <файл>` (предпочтительный путь, без риска зацепить чужой hunk). `git add -p` по hunk'ам нужен, только если перекрывающиеся правки уже сделаны до коммита. **Коммить с `--no-verify` + точечная per-package валидация (type-check/lint/test) перед каждым коммитом**: pre-commit hook гонит `turbo run test` по всему графу (минуты, легко >2-мин лимита Bash), а за батч это N прогонов одного и того же. Полнографовая валидация — та самая делегируемая пользователю финальная сборка (Шаг 3.1), не гоняй её на каждый коммит. **Два бага — одна неразделимая реализация?** Если фиксы двух issues сходятся в ОДНУ atomic-правку (общий helper/функция, которую `git add -p` не режет начисто на два коммита — **или общий тест-артефакт: один генератор/property/RED-фикстура, зеленеющий сразу для двух багов, плюс переслоённый describe**), допусти **один коммит + отдельный changeset на каждый issue** (`Closes #A` + `Closes #B` в теле PR) — гранулярность сохраняется в CHANGELOG, а искусственное разделение дало бы неполный intermediate-коммит (RED упал бы без ещё-не-влитого src-фикса второго бага). Перекрытие по РАЗНЫМ участкам файла (выше) дели на коммиты; перекрытие в ОДНОЙ функции / ОДНОЙ тест-фикстуре — нет.

## Шаг 3 — Финал на всю группу (один раз)

1. **Точечная валидация** по ОБЪЕДИНЁННОМУ набору затронутых пакетов: на каждый — `pnpm -F <pkg> type-check`, `lint`, `test -- --run` (+ `test:properties` / `test:stress`, если фиксы касались инвариантов / конкурентности / памяти). 100% coverage обязательно. Полный `pnpm build` **не запускай** — финальная сборка делегируется пользователю (сообщи в отчёте). **«Затронутый набор» = пакеты, чей `src/` правил батч, НЕ их транзитивные потребители:** если фикс в широко-консумируемом пакете (`sources` / `core` / `types`) меняет ПОВЕДЕНИЕ, потребительские сьюты (адаптеры и пр.) валидирует та самая делегированная полная сборка — не гоняй их per-package, это не входит в «затронутый набор». В отчёте отметь, какие потребители затронуты поведенчески, чтобы пользователь знал, за чем следить в полной сборке. **⚠️ `pnpm -F <name>` на НЕсовпавшем фильтре выходит `exit 0` с `No projects matched the filters` — молчаливый no-op, неотличимый от успеха по коду возврата (легко отчитаться о валидации пакета, который не прогонялся).** Подтверждай, что в выводе есть строка `Tests N passed` / `Test Files N passed`, а не только `exit=0`. Foundation/private-пакеты фильтруй **bare-именем** (`-F route-tree`, `-F path-matcher`, `-F search-params`, `-F event-emitter`), НЕ `@real-router/*` (это имена публичных; `fsm` и `logger` — публичные, но остальные foundation-пакеты приватны с bare-именами).
2. **Docs/wiki sync** одним проходом (`/bugfix` Фаза 5) по всем затронутым пакетам/API — баг-фиксы часто означают расхождение доков с реальным поведением. **Вики — отдельный git-repo (`../real-router.wiki/`), нередко с посторонними незакоммиченными правками от прошлой/чужой работы: коммить ТОЛЬКО свои файлы явным `git add <файл>`, НИКОГДА `-a`/`commit -am`** — иначе утащишь чужое в свой wiki-коммит. Один отдельный wiki-коммит на всю группу; push wiki вместе с push ветки (Шаг 3.4), тот же гейт явной просьбы.
3. **Один PR** через `/create-pr` из N коммитов:
   - В теле — **`Closes #736`, `Closes #738`, `Closes #749`: по отдельной строке на КАЖДЫЙ issue группы.** Без этого при merge GitHub закроет только ведущую задачу, остальные останутся открытыми.
   - Заголовок отражает группу/пакет, не один баг.
4. **Push** (только по явной просьбе пользователя). **Перед push ОБЯЗАТЕЛЬНО синхронизируй ветку с `master`: `git fetch origin && git rebase origin/master`** — worktree базировался от `origin/master` на момент Шага 1, за время батча master уходит вперёд; rebase (НЕ merge — master защищён, «Merge commits are not allowed», история линейна) гонит CI и мёрж PR на актуальной базе и снимает конфликты заранее. Конфликты разрешай до push. Затем `git push -u origin <lead>-<slug>`. Pre-push `lint:audit` (osv-scanner) в git-worktree известно даёт спурьозные падения (обход дерева от `/`). Если pre-push спотыкается **только** на нём — верифицируй `pnpm lint:audit` в основном чекауте и повтори с `--no-verify`. Отдельный случай: если pre-push репозитория запускает **полную сборку** (turbo по всему графу), которую проект делегирует пользователю (см. Шаг 3.1) или которая таймаутит в свежем worktree с холодным кэшем, — `--no-verify` допустим **при условии**, что точечная валидация затронутых пакетов (type-check/lint/test, Шаг 3.1) плюс `lint:audit` в **основном** чекауте (валидно, только когда ветка не меняла зависимости) покрывают то, что проверил бы pre-push. Это не «обход настоящего падения», а перенос делегированной полной сборки на финал пользователя. Прочие падения pre-push — настоящие, не обходи.
5. **Комментарии в issue** (черновики; публикуй только по явной просьбе): по комментарию на каждый баг ИЛИ один сводный со ссылками на коммиты — по ситуации. Содержание — как `/bugfix` Фаза 7 (корень, фикс, тесты, скоуп-решения).

## Гейты (наследуются от `/bugfix`)

- **НЕ пуш / НЕ создавай PR / НЕ публикуй комментарии в issue без явной просьбы пользователя** — эти исходящие действия готовь и показывай, жди явного «go». Локальные per-bug коммиты (Шаг 2) — это сам воркфлоу, НЕ гейт: коммить по ходу, не переспрашивай.
- Не помечай группу выполненной при падающих тестах или частичном фиксе хотя бы одного бага. **Исключение — триаж-короткое замыкание (`/bugfix` 0.6):** issue группы оказался не дефектом (observation / wontfix) → корректный исход «задокументировано, без правок src» (коммит-доки, БЕЗ changeset, issue закрывается через `Closes`). Это НЕ частичный фикс — зафиксируй такой статус в отчёте, группу им не блокируй.

## Итоговый отчёт

- Workspace/ветка: `.claude/worktrees/<lead>-<slug>` на ветке `<lead>-<slug>`.
- Таблица по багам: `#<N>` → корень → тест(ы) → коммит → статус (исправлен / не нужен, если корневой фикс уже покрыл / **документирован, без правок src** — триаж-non-defect 0.6, коммит-доки без changeset).
- Точечная валидация объединённого набора пакетов (полный `pnpm build` делегирован пользователю).
- Docs/wiki: что проверено/обновлено.
- PR: ссылка (или черновик), `Closes #<N>` для всех issues группы.
- Черновики issue-комментариев.

## Жизненный цикл worktree

Держи worktree и ветку до мёржа PR. После мёржа почисти вручную: `git worktree remove .claude/worktrees/<lead>-<slug>` (worktree, в который вошли через `EnterWorktree path:`, харнесс автоматически не удаляет). `git worktree list` — обзор всех активных батчей.

## Параллельные батчи

Несколько групп можно вести **параллельно** — каждый `/bugfix-batch` живёт в своём worktree (изолированы). Условие чистоты: группы **не пересекаются по файлам** (в #756 PR-очередь разбита по пакетам — пересечений почти нет, PR'ы мержатся независимо). Запускай параллельные группы в **отдельных сессиях** — один Claude в одной сессии работает в одном worktree за раз; параллельность даёт число сессий, не один Claude.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После итогового отчёта вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки. Особый кандидат для батча: проблемы координации (порядок багов, перекрытие файлов между коммитами, `Closes` не закрыл задачу, конфликт ветки).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку или сняла двусмысленность, на которой ты споткнулся. Косметику отбрасывай.
3. **Граница bugfix ↔ bugfix-batch.** Методология одного фикса (RED/GREEN/критерии) → правь `/bugfix`, НЕ здесь. Сюда идёт только то, что про оркестрацию группы (ветка/порядок/PR/параллельность). Не дублируй per-bug инструкции.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Усиливай существующий абзац; не плоди секции.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто → так и скажи: «правок нет».
