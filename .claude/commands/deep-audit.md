Выполни глубокий аудит метода или namespace'а @real-router/core по соответствующему промпту из `packages/core/.claude/prompts/`, верифицируя КАЖДУЮ находку эмпирически (mitata-проба / `npx tsx -e` / временный vitest с прогоном), и в конце заточи как сам целевой промпт, так и этот скилл.

Входные данные:
$ARGUMENTS

Формат аргументов:
- **Цель** — что аудировать. Принимается любое из:
  - slug метода/namespace'а: `claim-context-namespace`, `navigate`, `routes`, `event-bus`
  - имя файла или путь: `method-deep-audit-navigate.md`, `packages/core/.claude/prompts/namespace-deep-audit-routes.md`
  - имя метода/подсистемы прозой: `cloneRouter`, `RouteLifecycleNamespace`
- Опционально `all` — прогнать по всем целевым промптам последовательно (для повторных полных проходов).
- Опционально `--report-only` — не применять правки самокоррекции (только вывести таблицы кандидатов). По умолчанию: аудит + предложение правок с гейтом подтверждения.

Пример:
```
/deep-audit claim-context-namespace
/deep-audit navigate
/deep-audit RouteLifecycleNamespace
/deep-audit all
```

## Что это и зачем

Скилл — **дирижёр**, промпты в `packages/core/.claude/prompts/` — **партитура**. Он НЕ дублирует правила из партитуры (Дискриминация Bug vs Test-gap, протокол трёх артефактов, Probe-протокол, severity recalibration) — он их **исполняет с дисциплиной** и добавляет то, чего в отдельном промпте нет: резолв цели, гигиену scope при параллельных агентах, эмпирический гейт, связку с остальными audit-скиллами и **двухуровневую самокоррекцию** (целевой промпт + сам скилл).

Целевые промпты (исполняемые этим скиллом):
- `method-deep-audit-<X>.md` — add-route, can-navigate-to, claim-context-namespace, clone-router, create-request-scope, dispose, navigate, replace, serialize-router-state, start, subscribe, subscribe-leave, update-route, use-plugin.
- `namespace-deep-audit-<X>.md` — event-bus, plugins, route-lifecycle, routes.

**Базовые шаблоны** `method-deep-audit.md` и `namespace-deep-audit.md` — НЕ цели: это партитура общих правил, которую конкретный промпт обязывает прочитать первой. Если цель резолвится в них — это template, не запускай как аудит (см. Фаза 0).

**Вне scope скилла:** `architecture-review-core-fable.md` — у него отдельный workflow и свои выводы (`architecture-review-core-fable-*.md`). На запрос по нему — откажись и укажи, что это другой процесс.

## Фаза 0 — Резолв цели и гигиена scope

1. Резолвни аргумент в **один** файл `packages/core/.claude/prompts/<...>.md`:
   - slug → `method-deep-audit-<slug>.md` или `namespace-deep-audit-<slug>.md` (проверь оба; `ls packages/core/.claude/prompts/`).
   - имя метода прозой (`cloneRouter` → `clone-router`, `RouteLifecycleNamespace` → `route-lifecycle`) — нормализуй в kebab-case и сопоставь.
   - Неоднозначность или нет совпадения → **выведи список доступных целевых промптов и спроси**, не угадывай.
   - Резолв в `architecture-review-core-fable.md` → откажись (вне scope). Резолв в голый `method-deep-audit.md` / `namespace-deep-audit.md` → это шаблон: перечисли конкретные цели соответствующего семейства и спроси, какую (или предложи `all`).
2. **Гигиена scope при параллельных агентах.** Каталоги `packages/core/.claude/audit/` и `benchmarks/core/audit-probes/` общие — туда параллельно пишут другие агенты/прогоны.
   - **Не трогай untracked-файлы, которые создал не ты.** Твой отчёт и probe-директория стамплены датой + slug'ом — коллизия маловероятна; всё равно проверь `ls` перед записью и **никогда не перезаписывай** чужой файл.
   - Чужой WIP-аудит **сегодняшней** даты — не источник правды. Для «пометить legacy как archived» опирайся на **дату в имени файла < сегодня** (плюс mtime при сомнении); git-статус критерием не является — `packages/*/.claude/` целиком gitignored, отчёты **никогда** не закоммичены (коммитятся только probe-директории в `benchmarks/`).
   - **Отчёт по ЭТОЙ ЖЕ цели с СЕГОДНЯШНЕЙ датой уже существует** → раздели «завершённый» и «WIP» по хвостам, не по mtime: probe-директория закоммичена в `benchmarks/` И хвосты маршрутизации исполнены (wiki-фикс / filed issues / заточенный промпт — проверь их git-лог/gh). Завершённый → это **re-run**: регрессионный режим (Фаза 1), и дельта-секцию «Прогон №N — регрессионная сверка» **допиши append'ом в существующий отчёт** (имя-коллизия — не повод ни для нового файла, ни для перезаписи); статусы находок сверь с исполненными хвостами (fixed/filed/открыта). Незавершённый (проб в git нет, таблица оборвана) → параллельный WIP: не трогай, сообщи пользователю и спроси (инцидент 2026-07-03: subscribe re-run через 70 мин после завершённого прогона №2 — распознавание и место дельта-записи решались ad-hoc).
3. Зафиксируй фактическую дату прогона: `date +%F` (для имён `*-deep-YYYY-MM-DD.md` и `audit-probes/<slug>-YYYY-MM-DD/`).

## Фаза 1 — Чтение цепочки зависимостей (порядок обязателен)

Сначала **партитура**, потом **код**. Прочитай ЦЕЛИКОМ, в этом порядке:
1. **Базовый шаблон** соответствующего семейства (`method-deep-audit.md` для method-аудитов, `namespace-deep-audit.md` для namespace-аудитов) — универсальные правила, формат отчёта, Probe-протокол, severity-шкала и её recalibration. Это контракт, по которому ты работаешь.
2. **Целевой промпт** — его Target / Scope / разделы / известные риски.
3. **Cross-references**, которые промпт называет в шапке (другие `*-deep-audit-*.md`, разделы CLAUDE.md, wiki) — но только для понимания границ, не для дублирования.
4. **Исходники, тесты и доки из Scope** целевого промпта — целиком, с привязкой `файл:строка`.

**Сверь scope с git-дельтой от baseline.** Прежде чем читать по промптовым спискам,
сними дельту с даты последнего отчёта: `git log --since=<дата-baseline> --oneline --
packages/<pkg>/src packages/<pkg>/tests`. Scope-списки промпта отстают от свежих
мержей — коммиты, не видевшие ни одного аудита (и добавленные ими property/stress
файлы), это **главный фокус** прогона; их отсутствие в промпте не значит «вне scope».
Найденные новые файлы добавь в scope-списки промпта в Фазе 6.

**Нулевая дельта по scope + недавний исчерпывающий baseline → РЕГРЕССИОННЫЙ режим**
(инцидент 2026-07-03: serialize-router-state — дельта нулевая, «исполни разделы
промпта» тянул к полному проходу, «не дублируй legacy» его запрещал): перегон
baseline-probe-набора (src+dist), сверка контракт-дискриминаторов (wiki/доки — они
меняются НЕЗАВИСИМО от src, проверяй их git-лог отдельно), сверка статусов открытых
находок baseline; дельта-отчёт вместо полного 10-секционного прохода. Полный проход —
только если дельта непуста или baseline неполон/устарел по формату.

Чтение объёмное → можно распараллелить read-only разведку поверхности sub-агентами (`Explore`/`general-purpose`): построение surface-map, grep по консьюмерам, инвентаризация тестов. Но **суждение** (Bug vs Test-gap, severity, вердикт пробы) и **эмпирика** остаются в основном потоке — их не делегируем. **Если git-дельта показала, что промпт-описание архитектуры устарело** — субагентам передавай СВОЮ выжимку текущей архитектуры из прочитанного src (5-10 строк: структуры состояния, ключевые семантики, номера строк), а не отсылку к промпту: иначе они аудируют фантомные структуры (инцидент 2026-07-03: route-lifecycle промпт описывал pre-#676 Sets, которых нет в коде). Это касается и **собственных проб**: перед дёрганьем метода в probe сверь его публичность grep'ом по фасаду/api-экспортам — промптовая API-map может врать (инцидент 2026-07-03: routes-промпт назвал `matchPath` фасадным методом; проба упала на `router.matchPath is not a function`, реальный публичный путь — `getPluginApi(router).matchPath`).

**Грабли consumer-grep'а:** вывод об **отсутствии** использования метода (для «не dogfood'ится» / «нет consumer'ов») делай только по **полному** счёту матчей. Не обрезай `| head` и не доверяй порядку аргументов `grep -r … packages examples` — `packages` обрабатывается первым и `head -N` может срезать раздел `examples` целиком, дав **ложную absence**. Считай `grep -rc` / `wc -l` или grep'ай per-dir прежде чем писать находку (созвучно lint-survey blind spots — источник правды шире одного glob'а).

## Фаза 2 — Исполнение разделов промпта

Иди по структуре целевого промпта (10-секционный шаблон, namespace-формат с API Surface Map, или фокусный мини-аудит — что задаёт промпт). На каждое утверждение — **обязательная** привязка `файл:строка`. Непроверенное в коде — `[ГИПОТЕЗА]` + конкретный способ верификации.

Жёстко держи дисциплину из базового шаблона (не переписывай её здесь — применяй):
- Категория строки — только `Bug` (с прогнанным failing-тестом + ссылкой на нарушаемый контракт) или `Test-gap`. `Bug-risk` / «risk» / «may break» — **запрещены**.
- Каждая `Bug`-строка = **три артефакта**: (а) контракт `файл:строка`; (б) физический reproduction-артефакт в репозитории; (в) **прогон выполнен**, его stdout/assertion процитирован в отчёте. Не прогнал — `[FAILING-TEST НЕ ВЕРИФИЦИРОВАН]` + downgrade в `Test-gap (LOW)`.
- **Probe-verified ≠ автоматический CRITICAL.** После репродукции пересчитай severity по downstream-impact (какой production-path? typical или niche? magnitude?), а не по чёткости репродукции.
- Репро **через публичный контракт** (`createRouter`, `getPluginApi`, `getRoutesApi`, …), а не прямым дёрганьем приватных функций — white-box репро прячет мёртвый код от coverage-гейта и завышает severity.

Тесты должны **искать баг**, а не пиннить текущее поведение: скептически проверь гипотезу промпта на реальном src — промпт мог устареть, и «известный риск» уже закрыт. Закрытую находку фиксируй как закрытую, не воскрешай по тексту промпта.

**Перед записью Bug-строки как НОВОЙ** сверь её с issues параллельных волн (`gh issue list --state all --search "created:$(date +%F)" --limit 30`) и со свежими закоммиченными probe-директориями соседних прогонов — параллельная сессия могла уже зафайлить твою находку. Совпало → ре-верифицируй пробой и помечай «(filed #NNN)» в отчёте, НЕ пере-открывай (инцидент 2026-07-03: #1171/#1174 были зафайлены clone-router/review-волнами за часы до route-lifecycle прогона).

## Фаза 3 — Эмпирический гейт (mitata / tsx / vitest — ПРОГНАТЬ)

Это сердце аудита: ни одна находка и ни одна предложенная оптимизация не идёт в severity-таблицу без прогона.

- **Probe-файл в `benchmarks/core/audit-probes/<slug>-YYYY-MM-DD/probe-<N>-<short>.ts`** (коммитится вместе с отчётом — становится регрессионным набором, **не** в `/tmp`; запуск `npx tsx <path>`; образцы стиля: contract-behavior пробы — закоммиченные `benchmarks/core/audit-probes/<prev>/probe-*.ts` (прямой импорт `@real-router/core` / `@real-router/core/api`); micro-latency — `benchmarks/core/isolated-anomalies.ts` / `isolated-navigate.ts` (mitata `measure()` + локальный `./helpers`)). ⚠️ **Подводные камни запуска проб:** (1) каталог `audit-probes/` — **CJS** (нет `"type":"module"`): top-level await падает с `TLA not supported with cjs` — оборачивай async-тело в `void (async () => { … })()`. (2) **Плавный `npx tsx` резолвит `@real-router/core/*` в `dist/esm`, НЕ в src** (условие `@real-router/internal-source` активно только для `tsc`/vitest-alias, не для рантайма tsx) — проба против устаревшего dist даёт **ложный** вывод «правка не сработала». Чтобы верифицировать **src-правку**, гоняй `NODE_OPTIONS='--conditions=@real-router/internal-source' npx tsx <path>` (для heap — добавь `--expose-gc`), **либо** верифицируй временным vitest (он алиасит на src). ⚠️ **Sourcemap-иллюзия:** при прогоне против dist стек-трейс всё равно показывает `src/...` пути (sourcemap бандла), но имена классов **мангленные** (`n.dispose`, `e.disposeAll`, `a`) — НЕ принимай src-путь в стеке за доказательство прогона src. Доказательство — **идентичность вывода** плавного прогона и forced-conditions прогона (если совпали — dist в синхроне с src, находки валидны независимо от резолва; **разошлись** — сначала `diff`, не только md5: расхождение, сводящееся к мангленным идентификаторам внутри ТЕКСТОВ ошибок (`e.startsWith` vs `name.startsWith`), — не расхождение поведения). ⚠️ **Все импорты в пробах — только статические:** динамический `await import("@real-router/core/api")` рядом со статическими под dist-резолвом загружает ВТОРОЙ инстанс пакета (dual-package CJS/ESM split) с собственным internals-WeakMap — `getXxxApi(router)` падает ложным «Invalid router instance» ТОЛЬКО в dist-прогоне (инцидент 2026-07-03: probe add-route ложно «упал» на dist, потерян diff-цикл, пока dynamic-import не заменили статическим). ⚠️ **Импорты в пробах — только из deps `benchmarks/package.json`:** workspace-резолв покрывает лишь перечисленные там пакеты; `@real-router/logger` в deps НЕТ — проба, проверяющая логгер-категории, ловит их перехватом `console.error`, не импортом логгера (инцидент 2026-07-03: цикл потерян на `Cannot find module '@real-router/logger'`). ⚠️ **Перегон legacy-проб предыдущих дат:** их stdout — не истина. Захардкоженные копии конфигов/таблиц и комментарии в пробах стареют вместе с кодом (инцидент 2026-07-03: event-bus probe-04 печатал ложное «missing DISPOSE transitions» из до-#669 копии FSM-таблицы; probe-09 нёс до-#670 комментарий) — вердикт legacy-пробы сверяй с текущим src, прежде чем заносить в отчёт или заводить баг. В **свежем worktree** dist отсутствует вовсе (плавный прогон падает на резолве) — для dist-контроля собери точечно `npx turbo bundle --filter=@real-router/<pkg>` (десятки секунд; это НЕ «полная pnpm build» и не нарушает запрет ниже), либо ограничься src-режимом и пометь в отчёте, что dist-контроль не выполнялся. Это слот для **обоих** классов проб:
  - **Контракт-поведение** (фокусный мини-аудит из N вопросов; верификация «by-design / not-a-bug») — публичный API, печатай построчно `наблюдение + вердикт` на каждый вопрос. Один файл `probe-01-<slug>.ts` на весь набор вопросов — нормально.
  - **Микро-латентность / hot-path** — mitata-проба по шаблону `method-deep-audit.md` (раздел 10d). Правила приёмки: RME > 5% → `[NOISE]`; Δ < 2σ → `[НЕ ПОДТВЕРЖДЕНО]`; от батареи → `[SKIPPED: battery]`; **свежий `router`-инстанс на каждый вариант** (IC megamorphism).
- **Тривиальная one-shot проверка** (форма экспорта, структура константы, один контракт) → `npx tsx -e '<…>'`. ⚠️ `tsx -e` — тоже CJS-контекст: top-level await падает с «TLA not supported with cjs» — оборачивай в `void (async () => { … })()` как в probe-файлах (инцидент 2026-07-03: потерян цикл на голом await). Цитируй полную команду + stdout. Для многовопросной behavioral-верификации предпочитай коммитящийся probe-файл выше, а не ephemeral `-e`. **Резолв src vs dist:** package-имена `@real-router/core/*` под tsx резолвятся через `import`-condition в **`dist/`** (возможно устаревший) — src-condition `@real-router/internal-source` активна только в `tsc`. Чтобы аудировать **исходник**, импортируй по относительному пути (`../../../../packages/core/src/<...>`) в probe-файле, не по package-имени; голый `node -e` — тоже только `dist/`. **Control/non-printable символы** (NUL, U+2028, U+0000–U+001F) — только probe-файл с `String.fromCharCode`, НЕ `tsx -e`: хост отклоняет команды с control-байтами в строке.
- **Нужны фикстуры / mocks / beforeEach** → временный vitest в `packages/<pkg>/tests/functional/<area>/audit-bug-<N>.test.ts`, цитируй assertion-строку. Временные пробы, не ставшие регрессионным артефактом, удали до финала (НЕ оставляй красный тест в дереве — pre-commit гоняет весь сьют). ⚠️ **Запуск одного файла:** `pnpm -F @real-router/<pkg> test -- <path>` часто НЕ фильтрует — package-скрипт `test` бывает захардкожен на `vitest run functional/` и игнорирует `-- <path>` (прогонит весь сьют, ты не увидишь свой red). Для одного файла: `cd packages/<pkg> && npx vitest run <relative-path> --coverage.enabled=false`. ⚠️ **Parity-репро read-only-предиката против мутирующего сиблинга** (canNavigateTo↔navigate, isActiveRoute↔navigate): используй ДВА инстанса (twin) ЛИБО вызывай предикат ПЕРВЫМ. Мутатор, вызванный первым на общем инстансе, коммитит state → второй вызов предиката становится same-state no-op → тест **ложно ЗЕЛЕНЕЕТ** и прячет баг (опаснее ложно-красного — инцидент 2026-06-25: HIGH-баг #970 чуть не уехал зелёным; пробы спаслись twin-роутерами, первый vitest — нет).

Эмпирика бьёт чтение кода: критичность мутанта/находки подтверждается прогоном, а не рассуждением о defense-in-depth. **Не запускай полную `pnpm build`** — аудит работает точечными прогонами; финальную полную валидацию (если из аудита родится src-фикс) делегируй пользователю.

## Фаза 4 — Связка с остальными audit-скиллами

Глубокий аудит **находит** пробелы; их закрытие — работа специализированных скиллов. Маршрутизируй находки (предлагай, не запускай молча длинные цепочки):
- **Пропущенный инвариант / property-gap** → `/pbt-scenarios <pkg>` (сгенерирует дискриминирующий PBT) либо точечно `/audit-pbt <file>` для валидации силы существующего property-теста.
- **Стресс-чувствительная поверхность без теста** → `/stress-scenarios <pkg>`; силу существующего стресс-теста проверяет `/audit-stress <file>`.
- **Подозрение на слабые тесты в зоне аудита** → отметь и передай мутационную валидацию в `/mutation-score <pkg> --file <src>`.
- **Подтверждённый `Bug` (3 артефакта на руках)** → не чини здесь. Заведи задачу `/create-issue` (он дедуплицирует против существующих — частая ловушка дублей) или, если чинишь сразу, `/bugfix <issue>` (TDD-цикл в своём worktree). Затронут `packages/*/src/` → фикс несёт `/changeset` + `/commit-msg`. Сам этот скилл **src не правит** — он report-only по отношению к коду.

## Фаза 5 — Отчёт

Сохрани в `packages/core/.claude/audit/<slug>-deep-YYYY-MM-DD.md` (фактическая дата) — на русском, таблицы, обязательные `файл:строка`, без воды и драматизации. ⚠️ **Прогон в worktree:** `.claude/` (промпты И аудиты) — gitignored → в worktree каталога НЕТ; отчёт пиши сразу в `.claude/audit/` **основного чекаута** (абсолютный путь), а probe-директорию из worktree скопируй в `benchmarks/core/audit-probes/` основного чекаута до завершения (перед копированием `ls` — не перезаписывай чужие датированные директории параллельных прогонов). Структура — как задаёт целевой промпт; обязательны:
- **Сводная таблица проблем** в конце: severity (CRITICAL/HIGH/MEDIUM/LOW) · категория (`Bug`/`Test-gap`) · `файл:строка` · описание · предложенный фикс. Это смотрят глазами первым.
- Для namespace-аудита — **API Surface Map** в начале.
- Для прошедших probe оптимизаций — колонки Probe file · Δ(before→after) · RME b/a · Verdict.
- Legacy-аудиты с датой < сегодня — пометить archived, не дублировать их содержимое.

## Фаза 6 — Заточка ЦЕЛЕВОГО промпта (Level-1, явный запрос)

После отчёта вынеси секцию **«Правки целевого промпта»** — критику не своей работы и не находок, а **текста `packages/core/.claude/prompts/<целевой>.md`**: где он при ЭТОМ прогоне промолчал, соврал, был двусмыслен, указал на устаревший `файл:строка` или избыточен — и это стоило времени, лишнего перебора или риска ошибки. 
Цель — чтобы повторный прогон был быстрее и точнее. Дисциплина — та же 6-правиловая, что и для самого скилла ниже (привязка к инциденту, высокая планка «изменило бы исход», разовое≠паттерн, анти-раздувание, предлагай-не-применяй-молча, честность про «нечего править»). Применяй правки в файл промпта только по согласию (или сразу, если был дан `--fix`-режим прогона); `prompts/` — infra, в master без changeset. `--report-only` → только таблица кандидатов.
Отнесись серьезно! Это многократные промпты для контроля качества

## Итоговый отчёт

- Цель и резолв: какой промпт исполнен (`файл`), фактическая дата.
- Путь к отчёту + к probe-директории; сколько проб прогнано, сколько `confirmed` / `[НЕ ПОДТВЕРЖДЕНО]` / `[NOISE]`.
- Сводка находок: сколько `Bug` (по severity) и `Test-gap`; каждый `Bug` — с тремя артефактами и процитированным прогоном.
- Маршрутизация в другие скиллы (что предложено: `/pbt-scenarios` / `/stress-scenarios` / `/mutation-score` / `/create-issue` / `/bugfix`).
- Таблица «Правки целевого промпта» (Level-1) — применены/предложены или «правок нет».

Важно:
- Не помечай аудит выполненным, если хоть одна `Bug`-строка не верифицирована прогоном.
- Не правь `packages/*/src/` из этого скилла — только отчёт + probe/repro-артефакты + (по согласию) тексты промпта/скилла.
- Не перезаписывай чужие untracked-файлы в общих каталогах.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После итогового отчёта вынеси короткую секцию **«Самокоррекция»** — критику не своей работы и не целевого промпта (это Фаза 6), а **САМОГО ЭТОГО СКИЛА**: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного метода/namespace'а (разовую) → в отчёт/память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ прогону по ДРУГОЙ цели. Специфику конкретного промпта → в Фазу 6 (заточка целевого промпта), не сюда.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй. Помни: скил — дирижёр; правила, живущие в базовых шаблонах, сюда не копируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset. `--report-only` → только таблица, без правок.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
