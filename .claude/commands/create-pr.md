Создай и опубликуй pull request для текущей ветки на основе решённых задач, истории коммитов и changeset-файлов.

Входные данные (опционально):
$ARGUMENTS

Формат аргументов (всё необязательно):
- Явные номера задач для переопределения автодетекта (например `#807 #808`) — если автоопределение ошибается.
- `--draft` — создать черновик PR.

## Фаза 0 — Проверка состояния

Базовая ветка PR — **всегда `master`**.

1. Текущая ветка: `git branch --show-current`.
2. **Проверь базовую ветку репозитория**: `gh repo view --json defaultBranchRef -q .defaultBranchRef.name`.
   - **Если это не `master` — ПРЕКРАТИ** и сообщи пользователю: скилл рассчитан на базу `master`, продолжать нельзя.
3. Убедись, что в ветке есть что оформлять: `git log master..HEAD --oneline`.
   - **Если пусто — ПРЕКРАТИ**: ветка не опережает `master`, PR оформлять не из чего. Сообщи пользователю.
4. Проверь, нет ли уже открытого PR для этой ветки: `gh pr view --json number,url,state 2>/dev/null`.
   - Если PR уже существует — **не создавай дубль**. Сообщи URL и предложи обновить тело через `gh pr edit` (один PR на ветку — это конвенция проекта).

## Фаза 1 — Сбор номеров решённых задач (из трёх источников)

В ветке может быть решено **несколько** задач — номер каждой записан в commit message и/или changeset-файле. Собери кандидатов из всех источников, объедини и верифицируй.

1. **Источник A — имя ветки** (ведущий числовой сегмент, формат `<number>-<slug>`):
   `git branch --show-current | grep -oE '^[0-9]+'`
2. **Источник B — commit messages ветки** (все `#N`):
   `git log master..HEAD --format='%s%n%b' | grep -oE '#[0-9]+' | sort -u`
3. **Источник C — changeset-файлы, добавленные/изменённые в ветке** (все `#N`, в т.ч. `([#N](…/issues/N))`):
   `git diff master...HEAD --name-only -- .changeset | grep -v README` → прочитай каждый → извлеки `#N`.
4. Если в `$ARGUMENTS` переданы явные номера — используй их как авторитетный список (источники A–C только для перекрёстной проверки).
5. **Объедини кандидатов и верифицируй каждый** — это должен быть открытый issue, а не PR и не несуществующий номер:
   `gh issue view <N> --json number,title,state,url`
   - Команда падает (номер — это PR или его нет) → исключи кандидата, сообщи об этом.
   - `state == CLOSED` → предупреди (возможно, задача уже закрыта другим PR) и **спроси**, включать ли `Closes` для неё.
6. **Если источники расходятся — НЕ угадывай.** Источник правды — что реально изменено в коде (коммиты + changeset); имя ветки вторично (ветку могли переиспользовать). Пример расхождения: имя ветки `757-…`, а коммиты/changeset ссылаются на `#807`. Перечисли расхождение и **спроси**, какие задачи реально решены в этой ветке, прежде чем продолжать.
7. Для каждой подтверждённой задачи прочитай её целиком **вместе с комментариями**: `gh issue view <N> --comments`. Комментарии учитывай ВСЕГДА — там уточнения скоупа, корректировки и контр-примеры.

## Фаза 2 — Изучение изменений ветки

1. **Коммиты**: `git log master..HEAD --format='%H %s%n%b'`. В этом проекте действует «один PR — несколько независимых коммитов»: каждый коммит обычно решает свою задачу. Сопоставь коммиты с задачами из Фазы 1.
2. **Diff-обзор**: `git diff master...HEAD --stat`; точечно `git diff master...HEAD -- <ключевые файлы>`.
3. **Добавленные тесты** (доказательство фикса) — найди их в diff (`tests/functional`, `tests/property`, `tests/stress`). Они идут в раздел **Test plan**.
4. **Changeset-файлы**: тип бампа (`patch`/`minor`) и тексты — основа для Summary и для типа PR (`fix` vs `feat`). Пакет из frontmatter changeset (`@real-router/<pkg>`) даёт `<scope>` заголовка.

## Фаза 3 — Сборка PR по шаблону проекта

### Заголовок (Conventional Commits, как squash-commit)

- **Одна задача**: `<type>(<scope>): <description> (#N)`
- **Несколько задач**: каждый клауз со своим `(#N)`, через ` + ` —
  `fix(memory-plugin): same-tick navigate-after-back() (#807) + short-circuit no-emission contract (#808)`
- `<type>`: `fix` для баг-фиксов, `feat` для фич (определи по changeset-бампу и характеру задач). `<scope>` — из changeset-пакета / скоупов коммитов.

### Тело (на английском, по `.github/PULL_REQUEST_TEMPLATE.md`)

```markdown
## Summary

<!-- одна задача: 2–4 bullet'а про то, что получает пользователь, не детали реализации -->

### #807 — <короткий заголовок задачи>      <!-- по подсекции на задачу, если задач несколько -->
- Что починено/добавлено (поведение, а не реализация).
- **Root cause:** корень проблемы и как фикс его устраняет.

### #808 — <короткий заголовок задачи>
- …

### Maintainability                          <!-- опционально: побочные чистки (lint/refactor) -->
- …

## Test plan

- [x] <добавленные тесты: путь + что проверяют — fail before / pass after>
- [x] `pnpm build` passes
- [x] 100% coverage maintained

Closes #807
Closes #808
```

Правила тела:
- **Английский**, безличный тон (как в существующих PR и changeset'ах).
- **Summary** — польза для пользователя, не детали реализации (буквально как просит шаблон).
- Для каждого бага укажи **Root cause** (так оформлены реальные PR проекта).
- Раскомментируй секции шаблона **`## New API`** (короткий пример нового публичного API) или **`## Breaking changes`** (before/after), если применимо.
- **Test plan**: ставь `[x]` только для реально проверенного. Если не уверен, что `pnpm build` зелёный, — либо запусти его, либо оставь `[ ]` и явно предупреди пользователя.
- **`Closes #N` — отдельной строкой для КАЖДОЙ решённой задачи**, в самом низу тела. Это обязательно: каждая задача должна автоматически закрыться при merge. Перепроверь, что число строк `Closes` равно числу подтверждённых задач.

## Фаза 4 — Публикация

1. Покажи пользователю финальные заголовок и тело.
2. Запушь ветку (вызов `/create-pr` = явная авторизация на push **именно этой** ветки и создание PR):
   `git push -u origin HEAD`
3. Создай PR в базовую ветку:
   ```bash
   gh pr create --base master --head "$(git branch --show-current)" --title "<title>" --body "<body>"
   ```
   Добавь `--draft`, если он был передан в аргументах.
4. Выведи URL созданного PR.

## Итоговый отчёт

- Подтверждённые задачи и их источники (имя ветки / коммиты / changeset).
- Заголовок PR.
- Список `Closes #N` (по строке на задачу) — подтверди, что покрыты все решённые задачи.
- URL опубликованного PR.

Важно:
- Один PR на ветку — не создавай дубль, если PR уже открыт.
- Не угадывай номера задач при расхождении источников — спрашивай.
- `Closes` обязателен для каждой решённой задачи.

Примеры:
```
/create-pr
/create-pr --draft
/create-pr #807 #808
```
