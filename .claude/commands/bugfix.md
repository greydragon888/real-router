Исправь баг по методологии TDD (Red → Green → Refactor), затем синхронизируй локальную документацию и вики.

Входные данные:
$ARGUMENTS

Формат аргументов:
- **Первый (обязательный)** — номер issue (`#812`, `812`) или ссылка на него (`https://github.com/greydragon888/real-router/issues/812`).
- **Второй (опциональный)** — имя workspace (worktree). Если задан — фикс идёт в отдельном worktree `.claude/worktrees/<workspace>` (создаётся, если его ещё нет). Если НЕ задан — фикс делается в **текущей ветке**, без отдельного worktree.

## Фаза 0 — Контекст из issue

1. Извлеки номер issue из **первого** аргумента. Чаще всего передаётся номер (`#812` или `812`) — возьми его как есть. Если передана ссылка — номер это последний числовой сегмент URL. **Второй** аргумент (если есть) — имя workspace, оно понадобится в шаге 3.
2. Получи описание бага **вместе с комментариями**: `gh issue view <number> --comments`. Комментарии читай и учитывай ВСЕГДА — в обсуждении почти всегда есть уточнения, корректировки скоупа или контр-примеры, без которых фикс будет неполным.
   - **Если issue не найден или закрыт как duplicate/invalid — ПРЕКРАТИ** и сообщи об этом.
3. **Где чинить — определяет workspace-аргумент.** Сориентируйся: `git rev-parse --show-toplevel` (корень) + `git branch --show-current` (ветка).
   - **Workspace-аргумент задан** → фикс в отдельном worktree `.claude/worktrees/<workspace>`:
     - Если такой worktree уже существует — войди в него: `EnterWorktree` с `path: .claude/worktrees/<workspace>`, и продолжай в нём.
     - Если ещё нет — заведи ветку и worktree четырьмя шагами:
       1. `gh issue develop <number> --name <workspace>` — создаёт ветку `<workspace>` на `origin` и привязывает её к issue (секция Development). Имя ≤ 64 символов (латиница, цифры, дефисы). Базируется на HEAD default-ветки (`origin/master`); для другой базы — `--base <branch>`.
       2. `git fetch origin` — подтянуть ветку локально.
       3. `git worktree add .claude/worktrees/<workspace> <workspace>` — worktree с checkout этой ветки (локальная ветка начинает трекать `origin/<workspace>`).
       4. **`EnterWorktree` с `path: .claude/worktrees/<workspace>`** — переключить сессию в этот worktree. Именно `path:` (вход в уже созданный worktree), **не** `name:` — `name:` создал бы свою новую ветку и конфликтовал бы с веткой от `gh issue develop`.
     Дальше **весь** цикл (RED → GREEN → REFACTOR → валидация → доки) — внутри этого worktree, в ветке `<workspace>`.
   - **Workspace-аргумент НЕ задан** → фикс в **текущей ветке**, без отдельного worktree:
     - Ты **не на `master`** → продолжай прямо в текущей ветке. Если она явно про **другую** задачу — предупреди (правки лягут поверх чужой работы), но не блокируй: пользователь сам выбрал «в той же ветке».
     - Ты **на `master`** → код-фикс (правка `packages/*/src/`) нельзя слать в master напрямую, нужна ветка под задачу для PR + changeset. Заведи её **в текущем же чекауте, без worktree**: `gh issue develop <number> --name <number>-<slug>` → `git fetch origin` → `git switch <number>-<slug>` (`<slug>` — короткий идентификатор из заголовка issue). Если фикс заведомо инфраструктурный (не трогает `packages/*/src/`) — master допустим, продолжай на нём.
   - В любом случае дублируй `#<number>` в коммите и changeset.
4. Из issue выдели: симптом, шаги воспроизведения, ожидаемое поведение, затронутый пакет/файл (если указан).
5. Если в issue нет данных для написания падающего теста (нет воспроизведения или ожидаемого результата) — остановись и спроси, не угадывай.
6. **Сначала воспроизведи баг эмпирически** — временной пробой через публичный API или собранный пакет, ПРЕЖДЕ чем писать формальный тест. Это триаж-гейт, который может закоротить работу в «только отчёт».
   - Проверь статус связанных/блокирующих issue (`blocked by #N`, эпики, ссылки в теле): если блокер закрыт — перепроверь, не починил ли его фикс уже этот баг.
   - Если репродукция из issue **не воспроизводится** на текущем HEAD — премиса устарела (баг уже исправлен или описание неверно). НЕ чини «по тексту»: останови RED, сообщи находку, при необходимости предложи пересмотр скоупа и спроси, продолжать ли.
   - Если баг воспроизводится, но корень — иной класс входов, чем в issue, — фиксируй фактический корень, а не формулировку issue.
   - Временные пробы удаляй до конца фазы — никаких `__probe*.test.ts` или отладочных файлов в коммите.
7. **Если в issue предложен способ решения бага — относись к нему максимально критически.** Это гипотеза автора, а не истина. НЕ применяй его на доверии. Сначала самостоятельно найди корень бага, затем проверь предложенное решение: действительно ли оно устраняет причину (а не симптом), не ломает ли инварианты/другие сценарии, не противоречит ли границам слоёв и архитектуре проекта. Применяй его, только если есть **реальные доказательства** (тест, анализ кода), что фикс корректен **и** оптимален. Если твоё независимое решение лучше — используй его и объясни, почему предложенное в issue не подошло.

## Фаза 1 — RED (тест, доказывающий баг)

1. Локализуй причину: найди модуль и функцию, ответственные за поведение. Сначала пойми корень, не патч симптома.
2. Напиши **падающий** тест, воспроизводящий баг, в `tests/functional/` соответствующего пакета (или ближайший подходящий уровень — property/stress, если баг про инвариант/память/конкурентность).
   - Тест проверяет **поведение**, а не реализацию.
   - Один логический assert на причину провала; имя теста описывает ожидаемое поведение.
3. Запусти тест и **убедись, что он падает именно по причине бага** (а не по опечатке/компиляции). Падающий тест — доказательство дискриминирующей силы.

## Фаза 2 — GREEN (минимальный фикс)

1. Напиши **минимальный** код, чтобы тест прошёл. Никакой спекулятивной функциональности (YAGNI) — только то, что требует тест.
2. Запусти новый тест + весь набор тестов затронутого пакета: `pnpm -F <пакет> test -- --run`. Все зелёные.
3. Проверь регрессии в зависимых пакетах (особенно при правке `shared/dom-utils`, `shared/browser-env`, `shared/ssr` — изменение пропагируется во всех консьюмеров через симлинки; для `shared/dom-utils` синхронизируй git-tracked копию в `packages/angular/src/dom-utils`).

## Фаза 3 — REFACTOR

1. Улучши структуру фикса и теста, не меняя поведения. Тесты остаются зелёными.
2. Проверь, что не осталось дубля логики и фикс лежит на правильном слое (core / plugin / dom-utility — границы симметричны, не размывать ради одной правки).
3. **Пост-фикс анализ (когда фикс подтверждён тестами и проверками).** Вернись к фиксу свежим взглядом и подумай: можно ли его упростить, оптимизировать, улучшить?
   - Нет ли более короткого/прямого решения, которое устраняет ту же причину.
   - Нет ли лишних аллокаций, обходов, копий, повторных вычислений на горячем пути.
   - Не дублирует ли фикс существующий хелпер/утилиту, которые можно переиспользовать.
   - Читается ли он в стиле окружающего кода (именование, идиомы, плотность комментариев).
   - Любое упрощение применяй, только если все тесты остаются зелёными и `pnpm build` проходит. Если упрощение ухудшает читаемость или ломает границы слоёв — оставь как есть и зафиксируй, почему.

## Фаза 4 — Валидация

1. Запусти полную сборку: `pnpm build` (type-check → lint → test → bundle, 100% coverage обязательно).
2. **Changeset** — если правились `packages/*/src/` публичного пакета, вызови скилл `/changeset` (он сам определит затронутые публичные пакеты и тип бампа по issue + diff). Для баг-фикса тип обычно `patch`; в pre-1.0 фазе breaking changes идут как `minor`. Инфраструктурные правки (не трогают `packages/*/src/`) идут в master без changeset — `/changeset` сам это распознает.

## Фаза 5 — Синхронизация документации и вики (обязательно после фикса)

Баг-фикс часто означает, что задокументированное поведение расходилось с реальным. Пройди по документации и проверь, не описывает ли она старое (баговое) поведение, нет ли устаревших примеров или сигнатур.

1. **Локальная документация** — проверь и при расхождении обнови:
   - Корневые: `README.md`, `ARCHITECTURE.md`, `IMPLEMENTATION_NOTES.md`, `CLAUDE.md` (формат записи решений — Problem → Solution → Why, исторические решения не удалять).
   - Затронутого пакета: `packages/<pkg>/README.md`, `ARCHITECTURE.md`, `CLAUDE.md`, при наличии — `INVARIANTS.md` (если фикс касался инварианта).
   - **Контракт INVARIANTS ↔ property-тесты:** если фикс ввёл или изменил инвариант — добавь/обнови соответствующий кейс в `tests/property/*` И синхронизируй таблицу маппинга «инвариант → тест-файл» в `INVARIANTS.md` (в этом репозитории строки инвариантов маппятся 1:1 на property-тесты; прозаическая строка без теста ломает контракт).
   - Точечно ищи устаревшие упоминания: `grep -rn "<имя метода/опции>" packages/<pkg>/*.md README.md ARCHITECTURE.md`.
2. **Вики** (отдельный репозиторий `../real-router.wiki/`):
   - Найди страницы, описывающие затронутый API/метод/компонент (например `Route.md`, `RouterOptions.md`, `<Plugin>.md`, `*-Integration.md`).
   - Обнови сигнатуры, примеры и описания поведения, если они расходятся с фиксом.
   - Поиск: `grep -rn "<имя метода/опции>" ../real-router.wiki/`.
3. Сообщи список проверенных файлов и что именно обновил (или явно укажи, что расхождений в документации не найдено).

## Фаза 6 — Commit message

1. Когда фикс, валидация, changeset и документация готовы, вызови скилл `/commit-msg` — он сгенерирует сообщение в формате Conventional Commits по текущему diff. Для баг-фикса ожидаемый тип — `fix(<scope>): ...`.
2. Выведи готовое сообщение пользователю. **НЕ коммить и НЕ пуш без явной просьбы** — только предложи сообщение.
3. **Когда пользователь явно попросит запушить** — пуш в ветку задачи: `git push -u origin <branch>` (`<branch>` — текущая ветка: `<workspace>`, если работал в worktree, иначе ветка текущего чекаута). Pre-push hook гоняет полную валидацию. **Только если работал в git-worktree:** шаг `lint:audit` (osv-scanner) известно даёт там спурьозные падения (обход дерева от `/`) — если pre-push спотыкается **только** на нём, верифицируй чисто в основном чекауте (`pnpm lint:audit` из корня основного репозитория), затем повтори пуш с `--no-verify`. Все прочие падения pre-push — настоящие, не обходи.
4. **Жизненный цикл worktree** (только если работал в worktree — workspace был задан). Держи worktree и ветку до мёржа PR. После мёржа почисти вручную: `git worktree remove .claude/worktrees/<workspace>` (worktree, в который вошли через `EnterWorktree path:`, харнесс автоматически не удаляет — в отличие от созданного через `name:`). Если worktree не заводился (фикс в текущей ветке) — этого шага нет.

## Фаза 7 — Комментарий в issue (как решён баг)

Подготовь комментарий, кратко и по делу объясняющий **как решён баг** — это остаётся в публичной истории issue и закрывает контур обратной связи с автором.

1. Содержание (на **английском** — issues ведутся на английском):
   - **Корень бага** — подтверждённый, в одну-две фразы.
   - **Фикс** — суть подхода + ключевой фрагмент кода/диффа; плюс почему поведение в остальном сохранено (какие инварианты/проверки остались на месте).
   - **Тест(ы)**, доказывающие фикс — имена.
   - **Скоуп-решения** — если в issue предлагалось решение или несколько пунктов: что принято, что задокументировано вместо рефакторинга, что осознанно отложено как follow-up — каждое с краткой причиной. Это самая ценная часть для автора issue.
   - Не переобещай статус: описывай реализованный подход, **не** заявляй «merged/released», если фикс ещё на ветке.
2. Публикация: `gh issue comment <number> --body-file <file>` — используй `--body-file` (не `--body`), чтобы сохранить markdown и код-блоки. **Публикуй только по явной просьбе пользователя** — тот же гейт, что коммит/пуш: комментарий в публичный issue — внешнее действие (обычно идёт в паре с просьбой закоммитить/запушить). Без просьбы — только покажи черновик в итоговом отчёте.

## Итоговый отчёт

- Workspace и ветка: worktree `.claude/worktrees/<workspace>` на ветке `<workspace>` (если workspace задан), либо текущая ветка `<branch>` без отдельного worktree (если не задан).
- Корень бага и где исправлен.
- Если в issue было предложено решение — принято оно или нет и почему.
- Тест, доказывающий фикс (путь + имя).
- Пост-фикс анализ: что упрощено/оптимизировано (или почему оставлено как есть).
- Результат `pnpm build`.
- Changeset: создан через `/changeset` (какие пакеты/бамп) или не нужен — почему.
- Документация и вики: что проверено и что обновлено.
- Предложенный commit message из `/commit-msg`.
- Черновик комментария в issue (или подтверждение, что опубликован, если была явная просьба).

Важно:
- НЕ коммить и НЕ пуш без явной просьбы пользователя.
- НЕ публикуй комментарий в issue без явной просьбы пользователя — только подготовь черновик.
- Не помечай задачу выполненной при падающих тестах или частичном фиксе.

Примеры:
```
/bugfix 812                       # фикс в текущей ветке (без worktree)
/bugfix 812 812-config-guard      # фикс в отдельном worktree .claude/worktrees/812-config-guard
/bugfix https://github.com/greydragon888/real-router/issues/812
```

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Red→Green→Refactor + синхронизация локальных доков/вики) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного бага/пакета (разовую) → в черновик issue-комментария/память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ фиксу ДРУГОГО бага.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
