Исправь баг по методологии TDD (Red → Green → Refactor), затем синхронизируй локальную документацию и вики.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Номер issue (`#812`) или ссылка на него (`https://github.com/greydragon888/real-router/issues/812`).

> **Несколько багов одной PR-группы в один PR?** Это отдельный скилл — `/bugfix-batch` (оркеструет группу в одну ветку/PR, **переиспользуя per-bug методологию этого скилла**). Здесь — один баг, линейно Фазы 0→7.
>
> **Один КОРЕНЬ в N call sites / пакетах ≠ батч.** Кросс-адаптерный / кросс-пакетный фикс одного дефекта (напр. одна и та же ошибка в `<Link>` всех 6 адаптеров) — это ЭТОТ скил, линейно: один RED→GREEN per call site, один changeset per затронутый публичный пакет, один PR. `/bugfix-batch` — для N **различных** багов (разных корней) одной PR-группы. Симметричное размножение одной правки по адаптерам — НЕ батч. Попутно всплывший «7-й call site» или директива-близнец с тем же дефектом (тот же общий хелпер) — дотягивается в этом же прогоне ради симметрии (feedback_layer_consistency), отдельным RED-тестом.

## Как формулировать вопрос пользователю

Если по ходу багфикса нужно спросить пользователя (нет данных для RED — 0.5; премиса issue устарела — 0.6; design-call с несколькими валидными разрешениями), формулируй вопрос **просто и понятно — для человека без накопленного тобой контекста**. Не вываливай плотную сводку из внутренних имён, аббревиатур и ссылок на спецификации.

Структура:
1. **Что это за фича/область** — одна-две фразы простым языком: что она делает для пользователя, на бытовом примере.
2. **При чём тут затронутый механизм/опция** — простыми словами, зачем он и когда применяется.
3. **Сам баг и его видимое последствие** — что идёт не так и чем это бьёт по пользователю.
4. **Сам выбор** — один чёткий вопрос с конкретными вариантами (напр. «чинить поведение по-настоящему или оставить как есть и честно описать ограничение в доках»).

Плохо: «scroll-spy: IntersectionObserver root + MutationObserver target резолвятся ОДИН РАЗ при создании, а опция обещает resolved on every event…» — плотно, на жаргоне, нечитаемо без контекста. Хорошо: то же по структуре выше — сначала «что такое scroll-spy и зачем `scrollContainer`», затем «в чём баг и последствие», затем один вопрос с двумя вариантами.

## Фаза 0 — Контекст из issue

1. Извлеки номер issue из аргумента. Чаще всего передаётся номер (`#812` или `812`) — возьми его как есть. Если передана ссылка — номер это последний числовой сегмент URL.
2. Получи описание бага **вместе с комментариями**: `gh issue view <number> --comments`. Комментарии читай и учитывай ВСЕГДА — в обсуждении почти всегда есть уточнения, корректировки скоупа или контр-примеры, без которых фикс будет неполным.
   - **Если issue не найден или закрыт как duplicate/invalid — ПРЕКРАТИ** и сообщи об этом.
3. **Не заводи инфраструктуру — работай в той ветке, в которой запущен.** Этот скил НЕ создаёт ветку/worktree и не решает, где работать: для одиночного запуска отдельную ветку под задачу готовишь заранее ты сам, для группового — её готовит `/bugfix-batch` на всю группу. Единственная проверка — `git branch --show-current`:
   - **Текущая ветка `master` → немедленно ПРЕКРАТИ и сообщи**, что багфикс на `master` недопустим: нужна отдельная ветка под задачу (заведи её сам — напр. `gh issue develop <number> --name <number>-<slug>` + checkout/worktree — либо запусти этот баг через `/bugfix-batch`). Дальше не продолжай.
   - **Любая другая ветка → продолжай в ней.** `#<number>` дублируй в коммите и changeset.
4. Из issue выдели: симптом, шаги воспроизведения, ожидаемое поведение, затронутый пакет/файл (если указан).
5. Если в issue нет данных для написания падающего теста (нет воспроизведения или ожидаемого результата) — остановись и спроси, не угадывай.
6. **Сначала воспроизведи баг эмпирически** — временной пробой через публичный API или собранный пакет, ПРЕЖДЕ чем писать формальный тест. Это триаж-гейт, который может закоротить работу в «только отчёт».
   - Проверь статус связанных/блокирующих issue (`blocked by #N`, эпики, ссылки в теле): если блокер закрыт — перепроверь, не починил ли его фикс уже этот баг.
   - Если репродукция из issue **не воспроизводится** на текущем HEAD — премиса устарела (баг уже исправлен или описание неверно). НЕ чини «по тексту»: останови RED, сообщи находку, при необходимости предложи пересмотр скоупа и спроси, продолжать ли.
   - Если баг воспроизводится, но корень — иной класс входов, чем в issue, — фиксируй фактический корень, а не формулировку issue.
   - **Issue-на-УДАЛЕНИЕ (dead code / unused / «недостижимо из X»): «0 покрытия / достижимости из публичного API X» ≠ «удаляемо».** Метрика достижимости аудита (0 проб через core, 0 внешних импортов) говорит лишь «этот потребитель не зовёт» — а НЕ «мёртвый код». Пакет держит СВОИ тесты + property + **INVARIANTS** на этот код → это живая, тестированная возможность, не мусор. Перепроверь КАЖДЫЙ пункт на HEAD: существование + потребители + **выделенные тесты/property/INVARIANTS + `git grep` на использование внутри самого пакета/тест-хелперов**. Есть выделенное покрытие → удаление это **product-решение** (сознательный API-shrink с потерей тестированной возможности), НЕ cleanup: вынеси как design-call (`AskUserQuestion`), не сноси на доверии к слову «dead» ([[feedback_legacy_skepticism]]). И даже под «да, сокращаем» — верифицируй границу КАЖДОГО пункта отдельно: аудит систематически врёт про «impl vs отдельный», «мёртвая ветка vs test-only», «фантом vs используется тест-хелпером» (прогон #1302: 7 «мёртвых» пунктов = тестированный API с INVARIANTS; `createRouteTree` оказался ПОСТРОЕН на «мёртвом» билдере — удаление стало рефактором-инлайном, не сносом).
   - **Воспроизводишь через консьюмер/пример?** Примеры и внешние приложения без `customConditions` в своём `tsconfig` резолвят `@real-router/*` из `dist` (через `exports`), а не из `src`. Значит баг, завязанный на опубликованные типы/бандл, проявляется только против собранного `dist` — **пересобери пакет** (`pnpm -F <pkg> bundle`) перед type-check/прогоном консьюмера, иначе и RED, и GREEN считаются против устаревшего `dist` (ложный результат).
   - Временные пробы удаляй до конца фазы — никаких `__probe*.test.ts` или отладочных файлов в коммите.
7. **Если в issue предложен способ решения бага — относись к нему максимально критически.** Это гипотеза автора, а не истина. НЕ применяй его на доверии. Сначала самостоятельно найди корень бага, затем проверь предложенное решение: действительно ли оно устраняет причину (а не симптом), не ломает ли инварианты/другие сценарии, не противоречит ли границам слоёв и архитектуре проекта. Применяй его, только если есть **реальные доказательства** (тест, анализ кода), что фикс корректен **и** оптимален. Если твоё независимое решение лучше — используй его и объясни, почему предложенное в issue не подошло.
   - **Своё собственное предложение — ТОЖЕ гипотеза, не повышай к нему доверие из-за авторства.** Если fix-direction в теле issue написал ТЫ САМ (в прошлой сессии, при заведении задачи), на старте фикса его тянет принять как данность — не надо: пере-выведи минимум из кода на момент фикса. Особенно перед **добавлением API-поверхности** (новый метод/поле/экспорт на публичном типе) — сначала проверь, не достаётся ли нужное через СУЩЕСТВУЮЩИЙ surface (getter'ы, уже-прокинутые коллабораторы, реэкспорт). «Добавить метод» рефлекторно раздувает API и дифф; часто хватает 0 новых точек. (Прогон #1301: я сам авторил «add 2 getters to RouterInternals»; пользователь дважды поправил — реальность 0 новых методов: дерево уже даёт `getTree()`, сегменты/существование — матчер, а реэкспорт понадобился ровно для ОДНОЙ функции без эквивалента на матчере.)
   - **Особый случай — REJECT малформед-конфига (throw там, где раньше молча проходило): он легко ПЕРЕ-ОХВАТЫВАЕТ.** Issue задаёт заявленный скоуп находки, но истинная граница дефекта — то, что принимают СУЩЕСТВУЮЩИЕ тесты. Перед коммитом reject'а прогони полный сьют пакета И downstream-потребителя (Фаза 2/4): падающий существующий тест, ассертящий отвергаемую форму как VALID, — сигнал **сузить** reject (у формы есть рабочий под-кейс — прогон #1242 §5.4: детект ловил ВСЕ param-родители slash-child, а работал только required, дефект лишь у optional+splat) или **вынести как design-call** (форма толерируется by-design — §5.2 `?name=value`: buildParamMeta даёт мусорное имя, но существующие «valid query» тесты его принимают), а НЕ форсить reject + переписывать тест под него. Это не «подгонка фикса под тест» (её мы избегаем на баг-пиннящих тестах) — это признание, что заявленный скоуп reject'а шире реального дефекта.
8. **Гейт корня-цепочки: не заплатка ли фикс поверх более глубокого владельца концерна.** Пункт 7 сомневается в *предложенном решении*; этот пункт сомневается в *самой рамке issue* — а вправе ли этот баг вообще чиниться локально, или он лишь **симптом-член цепочки долга**, где локальный RED→GREEN станет очередным band-aid'ом поверх ненайденного владельца, а класс багов продолжит рецидивировать. Триггеры (хоть один → сначала мини-проверка ниже, НЕ патч):
   - **Issue ссылается на другие / связан:** «follow-up of #N», «parallel to #N», «residual of #N», «same class as #N», «same seam», «sibling of #N», часть эпика, `blocked by #N`; или тело/комментарии называют баги-собратья. (Такие само-кросс-ссылки в телах — сильнейший сигнал сквозной **оси**; ровно из них `/debt-map` восстанавливает оси.)
   - **Whack-a-mole:** тот же класс симптома уже чинили в другом пакете/сайте (фикс там — всплыл здесь) → сигнал отсутствующего общего примитива.
   - **Band-aid-запах:** очевидный фикс ДОБАВЛЯЕТ спец-кейс / предохранитель / флаг вокруг концерна, а не удаляет координацию (Variant-A-запах из `/chain-resolve`).
   - **Уже в карте:** issue значится членом цепочки в `project_architectural_debt_map` (память).

   **Мини-проверка — convergence по КОДУ, не рассуждением** (дешёвая версия `/debt-map`-теста): вытяни тела+фиксы названных собратьев (`gh issue view <N> --comments`; фиксы — `git log --all --grep "#<N>"` / `git show <sha>`) и протрассируй: сходятся ли фиксы РАЗНЫХ issue в ОДИН helper/владельца? Есть ли у концерна авторитетный **существующий** владелец (`FSM` / `RoutesStore` / emitter / `onListenerError` / codec), на который класс должен схлопнуться? Три исхода:
   - **Сходятся к более глубокому корню + владелец существует + вокруг налипли костыли** → это **цепочка долга, не одиночный баг**; локальный фикс — следующая заплатка. **Не эскалируй в отдельный скил** — `/chain-resolve` / `/debt-map` на практике почти не срабатывают; проведи раскопку **здесь, инлайн** (дёшево, по коду, БЕЗ RFC-крестового-похода):
     - **`git blame` багованных строк → residual-vs-original:** строки предшествуют родителю → это НЕ residual (родитель ортогонален — пиши «related to #N», не «follow-up»); введены/оставлены родителем → настоящий residual (#1206: blame дал оригинал #474, фикс #767 тронул другой шов → изолированный).
     - **Follow-up? Проверь «ложную полноту» родительского ФИКСА:** прочитай его коммент/диф и сверь ЗАЯВЛЕННОЕ покрытие с кодом — родительские фиксы часто заявляют, что покрыли ныне-падающий кейс («also falls under the catch», «wraps all four»), а шов/порядок-вычисления это отрицают (#1222: #798 заявил «compile-throw falls under the catch», но `runHook(compileHook())` вычисляет compile аргументом ДО try — покрытия не было). Это сильнейший тэг residual'а.
     - **Sweep сиблингов** (см. Фазу 2.5): смапь ВСЕ call-site/каналы/координаты концерна и пробь КАЖДЫЙ — фикс покроет их все или назовёт полный радиус.
     - Вынеси находку (корень · собратья · владелец · какие band-aid'ы удалятся · **рекомендованный глубокий фикс + class-guard**) и **вынеси развилку пользователю как design-call**: глубокий структурный фикс сейчас (контракт/умбрелла для single-surface корня; конвенция+sweep+инвариант для кросс-поверхностного рекуррентного класса) **или** осознанная **временная** заплатка (interim, TODO-ссылка на корневой issue). Не блокируй; не форсируй umbrella молча.
   - **Realized-владелец (gap в существующем шве)** — общий владелец/helper УЖЕ существует и потребляется, а остаток = дыра в шве (фильтр «realized?» из `/debt-map`: реализованная convergence = закрытая цепочка, RFC-payload пуст): тогда локальный фикс — ПРАВИЛЬНЫЙ ход (bugfix, не RFC). Продолжай линейно.
   - **Нет convergence, корень самодостаточен** — обычный одиночный баг. Продолжай, отметив в отчёте, что член цепочки исключён.

   **Не спутай с симметричным размножением одной правки** (шапка скила): один и тот же корень в N call sites / адаптерах — ЭТО ЭТОТ скил (чини все сайты линейно, один RED per site), НЕ цепочка. Цепочка — когда корень глубже РАЗНЫХ багов и его владелец не тронут.
   **Калибровка (жёсткий урок `/debt-map`):** «более глубокий корень» — ГИПОТЕЗА, и её легко переоценить (на реальном прогоне 5/5 STRONG-корней флипнули в WEAK при раскопке). Твоя роль в этом гейте — **триаж + ОГРАНИЧЕННАЯ инлайн-раскопка** (blame · ложная-полнота · sweep — дёшево и по коду), НЕ полная RFC-раскопка: ссылка на другой issue сама по себе ≠ глубокий корень, она лишь повод прогнать мини-convergence + инлайн-дig и решить «одиночный / член цепочки / class-guard». Не превращай каждую ссылку в RFC-крестовый-поход и не блокируй фикс, если корень не доказан кодом.

## Фаза 1 — RED (тест, доказывающий баг)

1. Локализуй причину: найди модуль и функцию, ответственные за поведение. Сначала пойми корень, не патч симптома. (Это корень *внутри* бага; вопрос «не симптом ли сам баг более глубокого корня-цепочки» уже прошёл гейтом Фазы 0.8 — если тот выявил цепочку, ты уже провёл инлайн-раскопку и согласовал с пользователем скоуп: глубокий структурный фикс или осознанная interim-заплатка — с ним сюда и заходишь.)
2. Напиши **падающий** тест, воспроизводящий баг, в `tests/functional/` соответствующего пакета (или ближайший подходящий уровень — property/stress, если баг про инвариант/память/конкурентность).
   - Тест проверяет **поведение**, а не реализацию.
   - **Если фикс по замыслу НЕ меняет видимый output** (дедуп / экономия ресурса / перф — результат до и после идентичен): чистый output-тест невозможен, и RED легитимно проверяет **ресурсное следствие** — один источник/подписка/кэш-вход вместо N, отсутствие лишнего пересчёта. Это и есть тестируемое поведение, не реализация. Дискриминатор — наблюдаемый побочный эффект кэш-промаха vs шеринга (напр. повторный вызов коллаборатора при miss), а не чтение приватного поля. Не трать время на поиск output-разницы, которой по построению нет.
   - Один логический assert на причину провала; имя теста описывает ожидаемое поведение.
3. Запусти тест и **убедись, что он падает именно по причине бага** (а не по опечатке/компиляции). Падающий тест — доказательство дискриминирующей силы.
   - **Недетерминированный баг** (конкурентность / тайминг / память — RED обычно на property/stress-уровне): одно падение силу НЕ доказывает — тест мог упасть случайно, а на исходном коде иногда проходить. Прогони RED **несколько раз** (или зафиксируй `seed` у `fast-check`) и убедись, что он падает **стабильно** до фикса. Иначе после фикса «зелёный» не отличить от случайной удачи — фикс не доказан. Если стабильного падения добиться не удаётся, воспроизведение неполное: вернись к Фазе 0.6, не закрепляй флапающий тест.
   - **Код пакета корректен, а дефект — в опубликованных артефактах или консьюмере.** Если src пакета ведёт себя правильно, но баг живёт в том, что доезжает до потребителя (`dist/*.d.ts`/бандл — например, не доходящая до консьюмера module augmentation, отсутствующий ре-эмит `.d.ts`) или в самом примере/приложении-консьюмере — package-level unit-теста для RED может **не существовать**. Ставь RED на **интеграционной границе**: собери пакет (`pnpm -F <pkg> bundle`), затем type-check/прогон консьюмера (например `tsc` примера). RED = консьюмер падает на собранном `dist`; GREEN = после фикса доставки. Дополни постоянным контракт-пином (type-level `@ts-expect-error` или conditional-type assert) против регресса, проверив его дискриминирующую силу (временно сломай тип — пин должен упасть).

## Фаза 2 — GREEN (минимальный фикс)

1. Напиши **минимальный** код, чтобы тест прошёл. Никакой спекулятивной функциональности (YAGNI) — только то, что требует тест.
   - **Переносишь/сплитишь существующий код с \u-escape (\\uXXXX), HTML-entity (&amp;), regex-литералами через Write/Edit?** Тул ИНТЕРПРЕТИРУЕТ escape-последовательности при записи (unicode-escape → символ, `&amp;` → `&`) и молча искажает исходник — **критично для security-escape / сериализации** (напр. split-модуля, где escape-таблица переезжает в новый файл; property-тест поймает, но лучше не ломать). НЕ ретайпи такой блок руками: вырежи byte-identical node-сплайсом из оригинала (`readFileSync` → `slice` по маркерам → `writeFileSync`), верифицируй байты (`node -e 'JSON.stringify(...)'`), затем прогони его property/security-тест.
2. Запусти новый тест + весь набор тестов затронутого пакета: `pnpm -F <пакет> test -- --run`. Все зелёные.
3. Проверь регрессии в зависимых пакетах (особенно при правке `shared/dom-utils`, `shared/browser-env`, `shared/ssr` — изменение пропагируется во всех консьюмеров через симлинки; для `shared/dom-utils` синхронизируй git-tracked копию в `packages/angular/src/dom-utils`).

## Фаза 2.5 — Sweep сиблингов + class-guard (перед тем как считать фикс завершённым)

Точечный GREEN — ещё не финал: без этой фазы фикс порождает следующий follow-up (весь класс #647→#1211 / #757→#1210 / #798→#1222 — это узкие фиксы с непробованным сиблингом). Дёшево, по коду, в скоупе бага.

1. **Cross-check с `/create-issue`.** Если тело несёт готовый глубокий анализ (корень · родословную · радиус сиблингов · class-guard из `/create-issue` Шаг 2.5) — это ПЕРВАЯ из двух прошивок, а ты ВТОРАЯ: **не доверяй ей слепо.** Пере-выведи корень из кода на HEAD, пере-`blame`, пере-проверь sweep — две независимые проверки и есть ценность (одна перепроверка могла ошибиться уверенно). Твоё собственное авторство анализа доверия НЕ повышает.
2. **Sweep сиблингов — пробь КАЖДЫЙ канал, не аналогизируй иммунитет** ([[feedback_probe_each_sibling_no_analogy]]). Смапь ВСЕ сиблинги концерна (все call-site / каналы / координаты / писатели / поколения / счётчики) и пробь или протрассируй **КАЖДЫЙ** — не считай соседа иммунным «потому что похож» или «один уже проверил» (#1210: 3 координаты записи, живая лишь hash; #1212: 4 писателя кэша; #1222: 4 hook-call-site). Три исхода на сиблинг:
   - **Gap того же корня, непокрыт** → **покрой в ЭТОМ прогоне** (симметрия — отдельный RED per site, [[feedback_layer_consistency]]).
   - **Gap, но вне скоупа задачи** → заведи через `/create-issue` (с полным Шагом 2.5) и назови в отчёте.
   - **УЖЕ покрыт другим guard'ом/backstop'ом** (проба-контроль зелёная 🟢, не gap) → отметь в отчёте и **НЕ добавляй избыточный guard** — это over-reach (ср. Фаза 0.7), особенно если issue предлагает «add all N» скопом. Проба-контроль отличает уже-покрытый сиблинг от реального gap'а (#1351: конструктор молчал на dup-name **и** @@ → покрыл оба, но dup-path уже ловил path-matcher-backstop #1153 → добавил 2 ассерта из 3, а не слепо все три из тела issue).

   НЕ оставляй сиблинг непробованным «на потом».
3. **Class-guard, а не только точечный фикс** («рекомендовал более глубокий фикс»). Сформулируй **инвариант/property**, ловящий ЭТОТ дефект И его сиблингов, а не один сайт (#1224/#1225: property «plugin ⊇ core: если bare core принимает — плагин не бросает»; #1165: pin release-в-`finally` против move-мутанта; on()-atomicity: validate-then-mutate property). Добавь его тестом, если дёшево и в скоупе пакета; иначе — рекомендуй в отчёте + в комменте issue (Фаза 7). Это то, что превращает «залатал симптом» в «закрыл класс».
4. **Re-верифицируй ПОСЛЕ фикса, не только до.** Фикс мог сдвинуть шов — прогони sweep-пробы ещё раз на GREEN-коде (тот же урок: re-верификация после фикса выявила, что фикс #798 сместил изоляцию на body, оставив compile-шов → #1222). Пробы удали до коммита.

## Фаза 3 — REFACTOR

1. Улучши структуру фикса и теста, не меняя поведения. Тесты остаются зелёными.
2. Проверь, что не осталось дубля логики и фикс лежит на правильном слое (core / plugin / dom-utility — границы симметричны, не размывать ради одной правки).
3. **Пост-фикс анализ (когда фикс подтверждён тестами и проверками).** Вернись к фиксу свежим взглядом и подумай: можно ли его упростить, оптимизировать, улучшить?
   - Нет ли более короткого/прямого решения, которое устраняет ту же причину.
   - Нет ли лишних аллокаций, обходов, копий, повторных вычислений на горячем пути.
   - Не дублирует ли фикс существующий хелпер/утилиту, которые можно переиспользовать.
   - Читается ли он в стиле окружающего кода (именование, идиомы, плотность комментариев).
   - Любое упрощение применяй, только если все тесты затронутого пакета остаются зелёными (`pnpm -F <pkg> test -- --run`). Если упрощение ухудшает читаемость или ломает границы слоёв — оставь как есть и зафиксируй, почему.

## Фаза 4 — Валидация

1. **Точечная валидация затронутых пакетов** — НЕ полный `pnpm build` (он тяжёлый: turbo гонит весь граф с property/stress; его финальный прогон делегируется пользователю). На каждый изменённый пакет: `pnpm -F <pkg> type-check`, `pnpm -F <pkg> lint`, `pnpm -F <pkg> test -- --run` (+ `test:properties` / `test:stress`, если фикс касался инварианта / конкурентности / памяти). 100% coverage обязательно. **Для адаптеров с phantom-порогом (vue/solid/svelte/angular — branch/func <100% из-за compiler-кода) aggregate suite-pass НЕ доказывает покрытие ТВОЕГО файла: phantom-allowance поглощает новые непокрытые ветки/функции в изменённом `src`. Проверяй per-file (строка файла в coverage-таблице = 100% Stmts/Branch/Funcs), не только `Tests N passed` — иначе мёртвый код (напр. никогда не вызываемый `destroy`/`noop`) или новая ветка уедет в коммит под порогом.** **Финальный lint-гейт большого пакета (`core` и т.п.) гони СВЕЖИМ (`rm <pkg>/.eslintcache`) и в фоне (`run_in_background`), не с тесным таймаутом:** его скрипт — `eslint --cache --fix` (~2 мин на `core`), и прогон, оборванный SIGTERM по короткому Bash-таймауту, либо `--cache`-прогон со старым вердиктом — ложно рапортует «чисто». **Не цепляй `; echo`/иную команду ПОСЛЕ lint — она затирает `$?`; а background-нотификация `exit 0` — это код ОБЁРТКИ, не линтера. Верифицируй собственный код возврата инструмента: чтением лога (строка `✖ N problems` от eslint) или `pnpm lint; RC=$?` без хвоста (`--max-warnings 0` роняет и на warning'ах).** Вдобавок `--fix` молча СНИМАЕТ неправильно расположенный `eslint-disable` как unused, оставляя реальную ошибку непокрытой — держи `disable`-директиву и её цель (`!`, и т.п.) на СМЕЖНЫХ строках; иначе твой «зелёный» lint скроет ровно то, что поймает полный build пользователя (и уедет в коммит). Полный `pnpm build` **не запускай** — в итоговом отчёте сообщи, что финальная сборка за пользователем.
2. **Changeset** — если правились `packages/*/src/` публичного пакета, вызови скилл `/changeset` (он сам определит затронутые публичные пакеты и тип бампа по issue + diff). Для баг-фикса тип обычно `patch`; в pre-1.0 фазе breaking changes идут как `minor`. Инфраструктурные правки (не трогают `packages/*/src/`) идут в master без changeset — `/changeset` сам это распознает.

## Фаза 5 — Синхронизация документации и вики (обязательно после фикса)

Баг-фикс часто означает, что задокументированное поведение расходилось с реальным. Пройди по документации и проверь, не описывает ли она старое (баговое) поведение, нет ли устаревших примеров или сигнатур.

1. **Локальная документация** — проверь и при расхождении обнови:
   - Корневые: `README.md`, `ARCHITECTURE.md`, `IMPLEMENTATION_NOTES.md`, `CLAUDE.md` (формат записи решений — Problem → Solution → Why, исторические решения не удалять).
   - Затронутого пакета: `packages/<pkg>/README.md`, `ARCHITECTURE.md`, `CLAUDE.md`, при наличии — `INVARIANTS.md` (если фикс касался инварианта).
   - **Контракт INVARIANTS ↔ property-тесты:** если фикс ввёл или изменил инвариант — добавь/обнови соответствующий кейс в `tests/property/*` И синхронизируй таблицу маппинга «инвариант → тест-файл» в `INVARIANTS.md` (в этом репозитории строки инвариантов маппятся 1:1 на property-тесты; прозаическая строка без теста ломает контракт).
   - Точечно ищи устаревшие упоминания: `grep -rn "<имя метода/опции>" packages/<pkg>/*.md README.md ARCHITECTURE.md`.
2. **Вики** (отдельный репозиторий `real-router.wiki/`):
   - Найди страницы, описывающие затронутый API/метод/компонент (например `Route.md`, `RouterOptions.md`, `<Plugin>.md`, `*-Integration.md`).
   - Обнови сигнатуры, примеры и описания поведения, если они расходятся с фиксом.
   - Поиск: `rg "<имя метода/опции>" <wiki-path>`. **Из worktree** относительный `../real-router.wiki` НЕ достаёт до sibling-репо (worktree лежит в `.claude/worktrees/<name>/`, `../` не выходит к корню-соседу — `grep` вернёт «no matches») — бери **абсолютный** путь вики из environment (секция `Additional working directories`). И НЕ делай `cd` в вики-репо (или `cd` обратно сразу): персистентный cwd Bash уводит последующие `git`-команды (Phase 6 `git diff`/commit) в чужой репозиторий — можно собрать commit message из чужого диффа. Работай через `rg <pattern> <abs-wiki-path>` и `git -C <worktree>`; коммить в вики только СВОИ файлы (`git add <files>`) — там бывают чужие незакоммиченные правки.
3. Сообщи список проверенных файлов и что именно обновил (или явно укажи, что расхождений в документации не найдено).

## Фаза 6 — Commit message

1. Когда фикс, валидация, changeset и документация готовы, вызови скилл `/commit-msg` — он сгенерирует сообщение в формате Conventional Commits по текущему diff. Для баг-фикса ожидаемый тип — `fix(<scope>): ...`.
2. Выведи готовое сообщение пользователю. **НЕ коммить и НЕ пуш без явной просьбы** — только предложи сообщение.
3. **Когда пользователь явно попросит запушить** — пуш текущей ветки: `git push -u origin HEAD`. Pre-push hook гоняет полную валидацию; в git-worktree шаг `lint:audit` (osv-scanner) известно даёт спурьозные падения (обход дерева от `/`). Если pre-push спотыкается **только** на нём — верифицируй чисто в основном чекауте (`pnpm lint:audit` из корня основного репозитория), затем повтори пуш с `--no-verify`. Все прочие падения pre-push — настоящие, не обходи.

## Фаза 7 — Комментарий в issue (как решён баг)

Подготовь комментарий, кратко и по делу объясняющий **как решён баг** — это остаётся в публичной истории issue и закрывает контур обратной связи с автором.

1. Содержание (на **английском** — issues ведутся на английском):
   - **Корень бага** — подтверждённый, в одну-две фразы.
   - **Фикс** — суть подхода + ключевой фрагмент кода/диффа; плюс почему поведение в остальном сохранено (какие инварианты/проверки остались на месте).
   - **Тест(ы)**, доказывающие фикс — имена.
   - **Скоуп-решения** — если в issue предлагалось решение или несколько пунктов: что принято, что задокументировано вместо рефакторинга, что осознанно отложено как follow-up — каждое с краткой причиной. Это самая ценная часть для автора issue.
   - **Sweep сиблингов + class-guard** (из Фазы 2.5) — какие сиблинги концерна проверены и покрыты/заведены отдельно (полный радиус, не один сайт), и рекомендованный class-guard (инвариант/property, закрывающий класс), если он не добавлен тестом в этом прогоне. Если инлайн-раскопка (Фаза 0.8) выявила ложную полноту родительского фикса или структурный тэг (isolate-produce, false-reject, sync-seam и т.п.) — назови его: это спасает следующий follow-up.
   - Не переобещай статус: описывай реализованный подход, **не** заявляй «merged/released», если фикс ещё на ветке.
2. Публикация: `gh issue comment <number> --body-file <file>` — используй `--body-file` (не `--body`), чтобы сохранить markdown и код-блоки. **Публикуй только по явной просьбе пользователя** — тот же гейт, что коммит/пуш: комментарий в публичный issue — внешнее действие (обычно идёт в паре с просьбой закоммитить/запушить). Без просьбы — только покажи черновик в итоговом отчёте.

## Итоговый отчёт

- Ветка: текущая ветка задачи (`git branch --show-current`) — скил не заводил ветку/worktree, работал в предоставленной.
- Корень бага и где исправлен.
- Если в issue было предложено решение — принято оно или нет и почему.
- Тест, доказывающий фикс (путь + имя).
- Пост-фикс анализ: что упрощено/оптимизировано (или почему оставлено как есть).
- Sweep сиблингов (Фаза 2.5): какие каналы/сайты концерна проверены, покрыты в прогоне vs заведены отдельным issue; рекомендованный class-guard; структурный тэг/ложная-полнота, если найдены (Фаза 0.8).
- Результат точечной валидации затронутых пакетов (`pnpm -F <pkg> test/lint/type-check`); полный `pnpm build` делегирован пользователю.
- Changeset: создан через `/changeset` (какие пакеты/бамп) или не нужен — почему.
- Документация и вики: что проверено и что обновлено.
- Предложенный commit message из `/commit-msg`.
- Черновик комментария в issue (или подтверждение, что опубликован, если была явная просьба).

Важно:
- НЕ коммить и НЕ пуш без явной просьбы пользователя.
- НЕ публикуй комментарий в issue без явной просьбы пользователя — только подготовь черновик.
- Не помечай задачу выполненной при падающих тестах или частичном фиксе.

Примеры:
```
/bugfix 812
/bugfix https://github.com/greydragon888/real-router/issues/812
```

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта (Red→Green→Refactor + синхронизация локальных доков/вики) вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки (не выдумывай улучшения впрок).
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику и «было бы неплохо» — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного бага/пакета (разовую) → в черновик issue-комментария/память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ фиксу ДРУГОГО бага.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй, не дублируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто и скил нигде не подвёл → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной — пункты 1–2 это допускают.
