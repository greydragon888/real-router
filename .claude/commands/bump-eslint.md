Обнови ESLint или пакет его экосистемы (eslint, @eslint/js, eslint-plugin-*, @*/eslint-plugin, eslint-config-*, typescript-eslint, @stylistic/*, eslint-import-resolver-*): сделай diff наборов правил, замерь срабатывания реальным прогоном, разбери recommended-сдвиги и flat-config, при мажоре — путь миграции. Специализация `/bump-dep` под линтинг.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Имя пакета (обязательно), например `eslint-plugin-unicorn` или `eslint`.
- Опционально целевая версия (`eslint-plugin-unicorn@70.1.0`); по умолчанию — последняя в npm-реестре.

## Когда этот скилл, а когда `/bump-dep`

Этот скилл — **только** для eslint-экосистемы. Признаки пакета: `eslint`, `@eslint/js`, `eslint-plugin-*`, `@*/eslint-plugin` (напр. `@vitest/eslint-plugin`, `@eslint-react/eslint-plugin`), `eslint-config-*`, `typescript-eslint`, `@typescript-eslint/*`, `@stylistic/*`, `eslint-import-resolver-*`. Всё остальное (включая прочий dev-тулинг и runtime-зависимости) → обычный `/bump-dep`.

Почему отдельно: у линтера бамп влияет не через «трогает ли он наш код», а через **набор активных правил и их детекцию**. Release notes для этого ненадёжны — источник правды — diff фактических наборов правил между установленными версиями + реальный прогон eslint. Это отдельный процесс, его и кодифицирует этот скилл.

## Фаза 0 — Локализация, гейт корня, режим доставки

1. Определи имя пакета (и целевую версию, если задана) из аргумента.
2. Найди зависимость в **корневом** `package.json`:
   `node -e "const p=require('./package.json'); const d={...p.dependencies,...p.devDependencies,...p.peerDependencies}; console.log('<pkg>', d['<pkg>'])"`
   - **Нет в корневом `package.json` — ПРЕКРАТИ** и сообщи. Бамп начинается только с корня.
3. Зафиксируй текущую (точную, pinned) версию из корня (`save-exact=true`).
4. Целевая версия: из аргумента либо `npm view <pkg> version`. Если `current == target` — обновлять нечего, заверши.
5. Полный список версий в диапазоне: `npm view <pkg> versions --json`.
6. **Карта использования**: `grep -rn "\"<pkg>\"" package.json packages/*/package.json benchmarks/package.json`. syncpack держит **одну** версию во всех пакетах (`sameRange`).
7. **Режим доставки — для eslint-экосистемы почти всегда Режим A (инфраструктурный):** эти пакеты живут в `devDependencies` корня → не доставляются потребителям → changeset не нужен → правка идёт прямо в `master`, без задачи/ветки/PR. **Если пакет внезапно оказался в `dependencies` публичного пакета (`packages/*` без `"private": true`) — это аномалия для линтера: остановись и сообщи**, не делай такой бамп в `master` (это был бы Режим B — задача + ветка + changeset + PR).
8. **Кросс-совместимость версий (критично при мажоре `eslint`):** `eslint`, `@eslint/js` и `typescript-eslint` должны быть взаимно совместимы, а каждый плагин объявляет `peerDependencies.eslint`. Перед мажорным бампом `eslint` (или `typescript-eslint`) проверь peer-диапазоны всех плагинов из карты использования: `npm view <plugin> peerDependencies`. Плагин с устаревшим peer либо не загрузится, либо уронит flat-config — такие плагины придётся бампать тем же заходом (перечисли их в плане).

## Фаза 1 — Сбор релизов + diff наборов правил (ядро)

1. GitHub-репозиторий: `npm view <pkg> repository.url`.
2. Release notes по диапазону `(current, target]`: `gh release list -R <owner/repo>` + `gh release view <tag> -R <owner/repo>`.
   - **Промежуточный patch той же серии** (`8.61.2 → 8.62.0`): точную дельту бери через `gh api repos/<owner>/<repo>/compare/<current-tag>...<target-tag> --jq '.commits[].commit.message'` — auto-generated notes для `X.Y.0` задваивают изменения, уже вышедшие в твоём patch.
   - Крупный мажор — официальный migration guide (`WebFetch` на release/доки; `context7` для актуальной документации).
3. **Главный приём — diff фактических наборов правил между установленными версиями** (надёжнее notes, которые врут/отсутствуют, особенно при пакетном тегировании монорепо, напр. **SonarJS**: GitHub-релизы как `12.5.x`, а npm-пакет — `4.1.0`):
   - `Object.keys(plugin.rules)` и `Object.keys(plugin.configs.recommended.rules)` у **current** (из `node_modules` проекта) vs **target** (поставь в /tmp: `mkdir -p /tmp/x && cd /tmp/x && npm i <pkg>@<target>`).
   - ⚠️ **`eslint-config-*` пакеты правил НЕ содержат — правила в обёрнутом `eslint-plugin-*` (транзитив конфига); diff'ай именно плагин** (напр. `eslint-config-turbo` → `eslint-plugin-turbo`). Текущую версию плагина `require()`/`require.resolve` из корня часто не достанет: транзитив лежит в `.pnpm/<plugin>@<ver>_<peer-hash>/node_modules/<plugin>` (не хойстнут) — найди путь через `ls node_modules/.pnpm/<plugin>@<curver>*/node_modules/<plugin>`, либо проще поставь в /tmp **обе** версии конфига (current и target) в отдельные папки и грузи обёрнутый плагин из чистых раскладок обеих.
   - ⚠️ **Ключи `configs.*.rules` префиксованы (`unicorn/foo`), а `Object.keys(plugin.rules)` — голые (`foo`).** Нормализуй (`.replace(/^<prefix>\//,'')`) перед сравнением set'ов, иначе метки recommended/opt-in **инвертируются** и весь diff соврёт (новые recommended-правила покажутся opt-in). Считай `recommended` строго через `configs.recommended.rules` (число «recommended вырос на N» = число реально сработающих после бампа новых правил).
   - **Плагин может быть ESM-only** — унаследованный `require()` падает (`ERR_PACKAGE_PATH_NOT_EXPORTED` на `./package.json`, либо плагин под `.default` → `Object.keys(undefined)`). Грузи через dynamic import и разворачивай default:
     `node --input-type=module -e "import p0 from '<abs-path>/index.js'; const p = p0.default ?? p0; console.log(Object.keys(p.rules))"`, версию бери из `npm view <pkg> version` (не из `require('.../package.json')` — часто не в `exports`).
   - Для **`eslint` core** правила встроены: сравни `Object.keys(require('eslint').Linter ...)` неудобно — проще взять список новых/переименованных правил и сдвиги `eslint.configs.recommended`/`@eslint/js` из release notes `eslint/eslint` (там notes точные) и проверить факт через прогон (Фаза 4).
4. **Diff наборов ловит только add/remove правил и сдвиги `recommended` — он НЕ ловит изменения детекции у уже-включённого правила.** Любой `Improvement`/bugfix в notes по существующему правилу (особенно тому, что у тебя on в `src` через `recommended`) может **расширить** детекцию (не верь «improvement = меньше срабатываний» — бывает и наоборот) и заавтофиксить shipped-код молча через pre-commit `--fix`. Поэтому к diff-у наборов **обязательно** добавляй вычитку notes по каждому затронутому правилу и замер всего набора (Фаза 4).
   - **Кросс-плагинная пропагация (критично для `typescript-eslint`/`@typescript-eslint/*`):** бамп tse меняет детекцию не только своих правил. Другие плагины (`@vitest/eslint-plugin`, `eslint-plugin-import-x` и др.) переиспользуют `@typescript-eslint/utils`/type-utilities, и их правила-порты (напр. `vitest/unbound-method` = порт `@typescript-eslint/unbound-method`) наследуют изменения детекции из бампнутого tse. При этом **diff набора самого tse может быть пустым** (0 новых правил, пресеты неизменны) — бамп выглядит no-op, а сработает правило ЧУЖОГО плагина. Поэтому при бампе tse **всегда** мерь ВЕСЬ прогон (все плагины, Фаза 4), даже если rule-set diff tse пуст — не сокращай замер до «мишеней tse-правил».
5. Сгруппируй находки: **новое правило / удалённое правило / сдвиг recommended / изменение детекции существующего правила / breaking (rename, удаление опции, смена дефолта) / chore**.

## Фаза 2 — ROI по каждому новому или изменённому правилу

Для **каждого** правила из находок Фазы 1:
- **Что ловит** — суть правила.
- **ROI для этого кодбейза** — число нарушений измерь **реальным прогоном** (Фаза 4), а не grep-оценкой. Сколько срабатываний, есть ли autofix, где (исходники vs тесты).
- **Трудоёмкость** — много/мало правок, механические (autofix) или ручные; риск ложных срабатываний.
- **Рекомендация** — включить / включить-с-autofix / отложить / не включать, с обоснованием. Учитывай flat-config проекта (`eslint.config.mjs` в корне + реэкспорты в пакетах) и уже задокументированные там осознанные отключения (`// vNN: ... declined/off` — не «переоткрывай» их без причины).

## Фаза 3 — Сдвиги `recommended`-набора (иначе `pnpm lint` упадёт после бампа)

Правила, **добавленные в `recommended`** в новой версии, начнут срабатывать сразу после бампа (проект включает пресеты через `...plugin.configs.recommended.rules`). Выяви их заранее (diff из Фазы 1). По каждому реши: чинить срабатывания или осознанно отключить в `eslint.config.mjs` с комментарием-обоснованием. Это не опционально — пропущенный сдвиг роняет `pnpm lint` и pre-push.

## Фаза 4 — Замер срабатываний (реальный прогон — проверено на практике, не теория)

Изолированный временный конфиг **не работает**: flat-config игнорирует файлы вне base-path, а `--no-config-lookup` **не** изолирует от корневого `eslint.config.mjs`. Надёжный путь:

1. **Временно бампни**: `pnpm up -r <pkg>@<target>` (node_modules → target).
2. Прогоняй **родной бинарь** с родным конфигом **без `--fix`**: `node_modules/.bin/eslint <dirs> --format json --no-cache` — иначе автофиксы перепишут код до подсчёта. Родной конфиг важен — он учитывает все послабления проекта.
   - **turbo-кэш не key'ится на версию бинаря eslint** (его inputs — `eslint.config.*` + src, не node_modules). Поэтому `pnpm lint` / `turbo run lint` после бампа **вернёт старый кэш и соврёт «чисто»**. Замеряй только прямым `node_modules/.bin/eslint --no-cache`, не через turbo.
   - **Считай дельту по ВСЕМУ набору, а не только по новым правилам:** один прогон под target покажет срабатывания новых правил, но НЕ изменения существующих (их baseline неизвестен) — а там прячется расширение детекции из `Improvement`-нот. Сделай **два** прогона (current и target), сравни счётчики по правилам; как минимум проверь, не появились ли autofix-срабатывания существующих правил в `src` (файлы с `message.fix` в shipped-коде).
   - **Полный typed-lint по монорепо может не уложиться в таймаут (>2 мин)** — `parserOptions.project` медленный. Не гоняй весь репо вслепую: сначала из diff наборов + notes выдели **расширяющие детекцию** изменения (новое правило в `recommended`; фича/Improvement, добавляющие паттерн к уже-on правилу), затем замеряй **точечно только их мишени** (grep по конкретному паттерну правила или прогон одного-двух пакетов). FP-снижающие фиксы и неактивные (не включённые в конфиг) правила мерить не нужно — они не добавляют срабатываний.
   - **Сначала проверь, какие директории вообще входят в гейт** — `turbo run lint` гоняет только воркспейсы, у которых есть `scripts.lint`. В этом репо `examples/**` и `benchmarks/` его **не имеют** → они вне гейта, мерить их не нужно (а прямой `eslint` по ним ещё и падает с `exit=2` на файлах без tsconfig-проекта — это артефакт инвокации, не находки). Замеряй только `packages/*` (у них `lint` есть). Экономит большой typed-прогон.
   - **База зелёная → один целевой прогон достаточен.** Если master уже проходит lint на текущей версии (обычный случай для закоммиченного кода), baseline по всем правилам = 0, и **любое** срабатывание в прогоне под target — новое (новое правило ИЛИ расширенная детекция существующего). Тогда два прогона не нужны — хватает одного target-прогона с разбивкой по правилам и по src/test. Два прогона нужны, только когда база не гарантированно зелёная.
   - **Инвокация проекта = факт детекции:** per-package `lint` обычно `eslint --cache --ext .ts src/ tests/ --fix --max-warnings 0` — `--max-warnings 0` значит даже `warn`-правило роняет билд, а `--fix` молча переписывает автофиксируемое. Проверь реальную команду в `packages/*/package.json` перед выводами.
3. **Откатить**, если только замеряешь (не применяешь сразу): `git checkout package.json pnpm-lock.yaml && pnpm install`.

На этой dev-машине `timeout` работает напрямую (симлинк `/opt/homebrew/bin/timeout → gtimeout`) — оборачивай долгий eslint-прогон в `timeout`, или гоняй в фоне.

## Фаза 5 — План миграции (если в диапазоне breaking — мажор ИЛИ breaking в minor/patch)

> Триггер — **фактический breaking в notes, а не номер версии.** У eslint-экосистемы breaking чаще всего: переименование/удаление правила, удаление/смена дефолта опции правила, удаление пресета/конфига, изменение формата flat-config, поднятие peer `eslint`/Node.

1. Упорядоченный путь: какие файлы/конфиги и как править, в каком порядке, что проверять после каждого шага.
2. Before/after для ключевых правок (особенно `eslint.config.mjs`: переименованные правила, мигрированные опции).
3. Раздели на «механически/autofix» vs «ручное решение»; оцени риск.
4. Объёмная/рискованная миграция — **не применяй сразу**, предложи план и спроси подтверждение.

## Фаза 6 — Применение (Режим A; после согласования миграции, если она была)

1. **Предпочитай хирургический бамп:** правь версию в корневом `package.json` вручную + `pnpm install` (реюзит существующие транзитивы из lockfile) — **НЕ** `pnpm up -r <pkg>@<target>` + broad `pnpm dedupe`. Последние re-резолвят floating-диапазоны к новейшим патчам и тянут посторонние транзитивные бампы: частая ловушка на монорепо-плагинах с sibling-пакетами (`@eslint-react/*` тянет `@typescript-eslint/* ^8.x` → floating на новейший патч, при том что корневой `typescript-eslint` meta пинит сиблинги точно → **дубль версий** + раздутый lockfile-diff, хотя целевой бамп — no-op). После install свери `git diff pnpm-lock.yaml | grep -E "^[+-]  '[^']+@[0-9]"` — в focused-коммите должны меняться только ключи целевого пакета и его собственного subtree. Посторонний транзитив (особенно `@typescript-eslint` при бампе плагина) выноси в отдельный `/bump-dep`, не тащи в этот коммит. При мажоре eslint — тем же заходом бампни несовместимые по peer плагины из Фазы 0.8.
2. Консистентность и дубли: `pnpm lint:deps` (syncpack) + `pnpm lint:dedupe`. При расхождении syncpack — `pnpm lint:deps:fix`. **Фейл `lint:dedupe` (`ERR_PNPM_DEDUPE_CHECK_ISSUES`) чинится `pnpm dedupe`** — НЕ `lint:deps:fix` (то syncpack, другая проблема). Частый исход бампа meta-пакета (`typescript-eslint`): floating-транзитив (`eslint-plugin-react-x` → `@typescript-eslint/type-utils@^8`) остаётся на старой версии рядом с новой из бампнутого meta → дубль; `pnpm dedupe` схлопывает его к целевой. Это штатный шаг бампа, не посторонний транзитив — после dedupe свери, что схлопнулась только версия целевого subtree.
3. Примени согласованные адаптации: включи одобренные правила, погаси/отключи сдвинутые `recommended`-срабатывания, выполни шаги миграции. **Если фикс правит КОД (а не только конфиг) — прогони type-check на изменённых файлах** (`tsc -p <pkg>/tsconfig.json --noEmit`): eslint-гейт TS-ошибки НЕ ловит (это `tsc`, не lint), и Фаза 6.4 полный build делегирует пользователю — так что type-ошибку, внесённую твоим же фиксом, поймать больше нечем до пользователя. Особая мина: фикс, убирающий сужающую тип конструкцию (`if (guard) return` → вызов, который CFA не считает `never` — напр. overload-сигнатура с `never`-веткой), молча делает переменную `possibly undefined`.
4. Валидация линтинга: прямой `node_modules/.bin/eslint ... --no-cache` (не turbo-кэш). Полный `pnpm build` (type-check → lint → test → bundle) **делегируй пользователю** — финальную сборку не запускаем сами.

## Фаза 7 — Доставка (Режим A — инфраструктурный)

1. Changeset не нужен — `changeset-check.yml` требует его только при изменении `src/` публичного пакета.
2. Вызови `/commit-msg` (тип `ci(deps): ...` для линтер-тулинга; `chore(deps): ...` допустимо). Правка идёт прямо в `master`. **НЕ коммить и НЕ пуш без явной просьбы** — только предложи сообщение.
3. Помни (CLAUDE.md): лайнт-фейл от самого бампа плагина — это правка кода/конфига, **не** задача `resolve:dependabot`.

## Итоговый отчёт

- Пакет: `current → target` (сколько релизов в диапазоне).
- Таблица «правило → что ловит → тип (новое/recommended-сдвиг/детекция/breaking) → ROI/срабатывания → трудоёмкость → рекомендация».
- Сдвиги `recommended`: что начнёт срабатывать, решение по каждому.
- Воркэраунды, которые можно снять (файл + что удалить, напр. устаревший `eslint-disable`).
- Правила, достойные включения (где и зачем) — и явно: что включать НЕ стоит.
- При мажоре eslint: список плагинов, бампнутых ради peer-совместимости.
- План миграции (если breaking).
- Итоговая рекомендация: бампать сейчас / поэтапно / отложить — почему.
- **Режим доставки: A (инфра → master)** — с обоснованием (или флаг аномалии, если пакет в runtime).
- Предложенный commit message.

Важно:
- Гейт корневого `package.json` обязателен: нет в корне → стоп.
- Источник правды по правилам — diff наборов + реальный прогон, **не** release notes.
- Не включай правила без доказанного ROI для текущего кода.
- Не применяй объёмную миграцию без согласования.
- НЕ коммить и НЕ пуш без явной просьбы пользователя.

Примеры:
```
/bump-eslint eslint
/bump-eslint eslint-plugin-unicorn@70.1.0
/bump-eslint typescript-eslint
/bump-eslint @stylistic/eslint-plugin
```

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После отчёта вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки.
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку, сократила перебор или сняла двусмысленность, на которой ты споткнулся. Косметику — отбрасывай.
3. **Разовое ≠ паттерн.** Специфику конкретного плагина (разовую) → в задачу/память, НЕ в скил. В скил идёт только повторяющийся класс, полезный СЛЕДУЮЩЕМУ бампу ДРУГОГО eslint-пакета.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Предпочитай усиление существующего абзаца новой секции; если правка делает прежний текст избыточным — консолидируй.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто → так и скажи: «правок нет». Пустая самокоррекция честнее придуманной.
