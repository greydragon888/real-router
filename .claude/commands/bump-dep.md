Проанализируй и обнови зависимость: изучи все релизы от текущей версии в проекте до актуальной в реестре, оцени влияние каждого изменения на код и конфиги проекта, предложи внедрение полезного и (при мажоре) путь миграции.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Имя зависимости (обязательно), например `eslint-plugin-unicorn`.
- Опционально целевая версия (`eslint-plugin-unicorn@70.1.0`); по умолчанию — последняя в npm-реестре.

## Фаза 0 — Локализация и гейт «только корневой package.json»

1. Определи имя пакета (и целевую версию, если задана) из аргумента.
2. Найди зависимость в **корневом** `package.json` (`dependencies` / `devDependencies` / `peerDependencies`):
   `node -e "const p=require('./package.json'); const d={...p.dependencies,...p.devDependencies,...p.peerDependencies}; console.log('<pkg>', d['<pkg>'])"`
   - **Если зависимости нет в корневом `package.json` — ПРЕКРАТИ** и сообщи об этом. Бамп начинается только с корня; если пакет используется лишь в под-пакете/примере — это другой кейс, не для этого скилла.
3. Зафиксируй текущую (точную, pinned) версию из корня. Версии в проекте пиннятся точно (`save-exact=true`).
4. Определи целевую версию: из аргумента либо `npm view <pkg> version` (последняя в реестре). Если `current == target` — сообщи, что обновлять нечего, и заверши.
5. Полный список версий в диапазоне: `npm view <pkg> versions --json`.
6. **Карта использования в монорепо** (консистентность): где ещё объявлена зависимость —
   `grep -rn "\"<pkg>\"" package.json packages/*/package.json benchmarks/package.json`.
   syncpack держит **одну** версию во всех пакетах (`sameRange`, см. `syncpack.config.mjs`), поэтому бамп затронет всех консьюмеров — учитывай это при анализе влияния.
7. **Определи режим доставки** (это решает, как менять и куда коммитить):
   - **Режим A — инфраструктурный.** Зависимость только в `devDependencies` / корневом тулинге, либо в `dependencies` приватных пакетов (`"private": true`). Не доставляется потребителям → changeset не нужен → правка идёт прямо в `master`, без задачи/ветки/PR.
   - **Режим B — публичный runtime.** Зависимость входит в `dependencies` (runtime, не dev) хотя бы одного **публичного** пакета (`packages/*` без `"private": true`). Бамп отгружается потребителям → это пользовательское изменение: нужны **задача + отдельная ветка + changeset + доставка только через PR** (см. Фазы 5–6). Никогда не делай такой бамп в `master`.
   - Проверка публичности: `node -e "const p=require('./packages/<pkg-dir>/package.json'); console.log('private:', !!p.private, 'in deps:', !!(p.dependencies||{})['<pkg>'])"` по каждому пакету из карты использования.

## Фаза 1 — Сбор релизов и changelog (current → target)

1. Найди GitHub-репозиторий пакета: `npm view <pkg> repository.url`.
2. Собери release notes по всем релизам строго в диапазоне `(current, target]`:
   - GitHub releases: `gh release list -R <owner/repo>` + `gh release view <tag> -R <owner/repo>`.
   - Если релизов на GitHub нет или тегирование пакетное (монорепо публикует много пакетов) — возьми `CHANGELOG.md` самого пакета (на GitHub или из npm-тарбола): он версионно-точный.
   - Для крупных мажоров — официальный migration guide (`WebFetch` на страницу релиза/доки; `context7` для актуальной документации библиотеки).
3. Сгруппируй изменения по версиям; для каждого помечай тип: **bugfix / feature / deprecation / breaking / chore**.

> **Анализ всегда ПОЛНЫЙ — semver-метка не определяет глубину.** Не используй характер бампа (patch/minor/major) как фильтр: некоторые инструменты нарушают semver и кладут новые фичи (а иногда и breaking) в **patch**-релизы (например `tsdown`, `turbo`). Поэтому читай release notes **каждого** релиза в диапазоне целиком, даже если сменилась только patch-версия. Не пропускай и не «пробегай» patch-релизы, не делай вывод «это просто фиксы» по номеру версии — классифицируй по фактическому содержимому notes. Фазы 2–4 применяй к находкам независимо от того, как формально сменилась версия.

## Фаза 2 — Поэлементный анализ влияния (ядро скилла)

Пройди по **каждому** значимому изменению и сопоставь с РЕАЛЬНЫМ кодом и конфигами проекта (ищи через `grep`/`Glob` по затронутому API/опции/правилу):

- **Bugfix:** затрагивает ли он путь кода/конфиг, который мы используем?
  - Снимает ли он наш воркэраунд — можно ли удалить `eslint-disable`, `@ts-expect-error`, ручной патч, `pnpm.overrides`, хак?
  - Чинит ли латентный баг у нас, позволяет ли упростить/оптимизировать существующий код или конфиг?
  - Если изменение не касается нашего кода — отметь «нерелевантно» (без спекулятивных правок).
- **Feature:** полезна ли она здесь? **Конкретно** — в каком файле/конфиге, что улучшает (читаемость, производительность, типобезопасность, DX).
  - Если пользы нет — так и скажи. Не внедряем фичи «на всякий случай» (принцип «нет реального запроса/аналога — нет gap»).
- **Deprecation/removal:** используем ли мы устаревшее/удалённое? Где именно? Чем заменить.
- **Breaking change:** что конкретно ломается у нас → переходит в план миграции (Фаза 4).

## Фаза 3 — Особый случай: ESLint-плагины и конфиги

Если пакет — ESLint-плагин/конфиг (`eslint-plugin-*`, `@*/eslint-plugin`, `eslint-config-*`, `typescript-eslint`, `@eslint/js`, `@stylistic/*`):

1. Для **каждого нового или изменённого правила** в диапазоне версий:
   - **Что оно ловит** — суть правила.
   - **ROI внедрения для этого кодбейза** — оцени число нарушений эмпирически: временно включи правило в `eslint.config.mjs` и прогони `pnpm lint` (или точечно `pnpm -F <pkg> lint`), либо `grep` по паттерну. Сколько срабатываний, есть ли autofix.
   - **Трудоёмкость** — правок много/мало, механические (autofix) или ручные; риск ложных срабатываний.
   - **Рекомендация** — включить / включить-с-autofix / отложить / не включать, с обоснованием.
2. **Изменения `recommended`-набора:** правила, добавленные в `recommended` в новой версии, начнут срабатывать сразу после бампа — выяви их заранее, иначе `pnpm lint` упадёт после обновления. Реши по каждому: чинить срабатывания или осознанно отключить.
3. Учитывай flat-config проекта (`eslint.config.mjs` в корне и реэкспорты в пакетах).
4. Помни (CLAUDE.md): лайнт-фейл от самого бампа плагина — это правка кода/конфига, **не** задача `resolve:dependabot`.

## Фаза 4 — План миграции (если в диапазоне есть breaking changes)

> Триггер — **фактический breaking change в notes, а не номер версии.** Из-за нарушений semver (см. Фазу 1) breaking может прийти и в minor/patch — план миграции нужен в любом таком случае, не только при смене мажора.

1. Разработай **конкретный, упорядоченный** путь миграции: какие файлы/конфиги и как править, в каком порядке, что проверять после каждого шага.
2. Покажи before/after для ключевых правок.
3. Раздели объём на «механически/autofix» vs «требует ручного решения»; оцени риск.
4. Если миграция объёмная или рискованная — **не применяй сразу**: предложи план и спроси подтверждение.

## Фаза 5 — Применение (после согласования)

**Гейт ветки (по режиму доставки из Фазы 0.7) — выполни ДО любых правок:**
- **Режим A (инфра):** работаешь прямо в `master` — отдельная ветка/задача не нужны.
- **Режим B (публичный runtime):** **никогда не применяй в `master`.** Если задачи под этот бамп ещё нет — создай её: `gh issue create --title "chore(deps): bump <pkg> <current> → <target>" --body "<суть из анализа: что меняется для потребителей, миграция>"` (покажи заголовок/тело перед созданием). Затем заведи ветку и переключись: `gh issue develop <N> --checkout`. Дальше всё — только в этой ветке.

1. Обнови точную версию во всех пакетах монорепо (консистентность обязательна):
   `pnpm up -r <pkg>@<target>` → затем `pnpm dedupe`.
2. Проверь консистентность и дубли: `pnpm lint:deps` (syncpack) + `pnpm lint:dedupe`. При расхождении — `pnpm lint:deps:fix`.
3. Примени согласованные адаптации из Фаз 2–4: удали снятые воркэраунды, внедри полезные фичи, включи одобренные ESLint-правила, выполни шаги миграции.
4. Полная валидация: `pnpm build` (type-check → lint → test → bundle, 100% coverage).

## Фаза 6 — Доставка (по режиму из Фазы 0.7)

**Режим A — инфраструктурный (devDeps / приватные пакеты):**
1. Changeset не нужен — `changeset-check.yml` требует его только при изменении `src/` публичного пакета (`/changeset` сам это распознает).
2. Вызови `/commit-msg` (тип обычно `chore(deps): ...`, для тулинга — `ci(deps): ...`/`build(deps): ...`). Правка идёт прямо в `master`. **НЕ коммить и НЕ пуш без явной просьбы** — только предложи сообщение.

**Режим B — публичный runtime (изменены `dependencies` публичного пакета):**
1. **Changeset обязателен** — вызови `/changeset` (обычно `patch`; в pre-1.0 breaking-бамп → `minor`). Сошлись на номер задачи из Фазы 5.
2. Вызови `/commit-msg` (тип `chore(deps): ...`) — коммить в ветку задачи.
3. **Доставка только через PR** — вызови `/create-pr` (он соберёт тело, проставит `Closes #N` для созданной задачи и опубликует PR в `master`). Прямой коммит в `master` запрещён.

## Итоговый отчёт

- Зависимость: `current → target` (сколько релизов в диапазоне).
- Таблица по релизам: версия → ключевые изменения → тип → влияние на проект.
- Воркэраунды, которые можно снять (файл + что удалить).
- Фичи, достойные внедрения (где и зачем) — и явно: что внедрять НЕ стоит.
- ESLint (если применимо): таблица «правило → ROI → трудоёмкость → рекомендация».
- План миграции (если мажор/breaking).
- Итоговая рекомендация: бампать сейчас / поэтапно / отложить — почему.
- Результат `pnpm build` (если применяли).
- **Режим доставки: A (инфра → master) или B (публичный runtime → задача + ветка + changeset + PR)** — с обоснованием.
- Для режима B: номер созданной задачи, имя ветки, статус changeset, URL PR.
- Предложенный commit message.

Важно:
- Гейт корневого `package.json` обязателен: нет в корне → стоп.
- Режим B (изменены runtime `dependencies` публичного пакета) **никогда не идёт в master напрямую** — только задача + ветка + changeset + PR.
- Не внедряй фичи и ESLint-правила без доказанного ROI для текущего кода.
- Не применяй объёмную миграцию без согласования.
- НЕ коммить и НЕ пуш без явной просьбы пользователя.

Примеры:
```
/bump-dep eslint-plugin-unicorn
/bump-dep eslint-plugin-unicorn@70.1.0
/bump-dep typescript-eslint
```
