Оркеструй батч правок документации — по явному списку doc-issue (`#1240 #1241`) ИЛИ в discovery-режиме (сам находит все открытые doc-drift/doc-gap issue): классифицируй КАЖДЫЙ пункт doc-vs-код → чистая дока прямо в master, код-хвост отдельной веткой/PR. Никакого RED→GREEN, никакого changeset на доки.

Входные данные:
$ARGUMENTS

Формат аргументов:
- **Явный список** номеров doc-issue (`#1240 #1241`) — обычно «doc-batch»-хвосты wave-N ре-аудита. Один issue **часто смешивает doc + код** (см. Шаг 0).
- **Без аргументов (или `all`) → discovery-режим:** скилл сам находит ВСЕ открытые doc-drift/doc-gap issue (Шаг 0.0), группирует по пакету, показывает набор и подтверждает скоуп ПЕРЕД правкой. Опционально сузить: `all <фильтр>` — напр. `all core`, `all plugins`, `all priority:medium`.

Этот скилл — **ОРКЕСТРАТОР**, и его СУТЬ — не редактирование, а **классификация + маршрутизация**. В отличие от `/bugfix-batch` (весь код → одна ветка), doc-batch **расщепляет issue по природе пункта**: чистая дока идёт в master (инфра-поток CLAUDE.md, без PR/changeset), код — на ветку. Меняется механика редактирования доки → это не сюда; сюда — только как классифицировать и куда маршрутизировать.

## Шаг 0 — Discovery (если нет списка) + Fetch + классификация КАЖДОГО пункта (это и есть работа)

**0.0 — Discovery (только в режиме без явного списка).** `gh issue list --state open --label documentation --json number,title,labels` → отбери doc-drift/doc-gap по заголовку (`[docs]` / `[<pkg>][docs]`, «stale»/«drift»/«gap»/«outdated»/«JSDoc»/«CLAUDE/ARCHITECTURE/INVARIANTS/wiki … drift»). **ОТСЕЙ не-doc, помеченные `documentation`:** issue, чья СУТЬ — тест/код (`[<pkg>][test]`, enhancement, property-lock), даже если рядом стоит `documentation` — это код-задача, не текст. Сгруппируй по пакету/области (адаптер / core / плагин / util), выведи таблицу `#N · пакет · приоритет · суть` и **подтверди скоуп у пользователя ПЕРЕД правкой** — discovery-набор бывает большой (10-15 issue), «тихо все подряд» — не то; юзер может срезать до волны (`all core`) или исключить спорные. Применённый фильтр (`all core`) сужает набор здесь же.

`gh issue view <N>` на каждый issue (из явного списка или из подтверждённого discovery-набора). Для КАЖДОЙ находки внутри:

1. **Верифицируй против ТЕКУЩЕГО исходника, НЕ против issue.** Doc-issue аудита — ГИПОТЕЗА на момент заведения: цитируемая строка/символ/счётчик могли уплыть (починены, переименованы, ИЛИ добавлено больше). Грепни файл/символ/счётчик, что дока описывает, ПЕРЕД правкой. (Прогон #1240: таблица «Route Path Validation» отставала не только на строку #8, что просил issue, но и на 5 гейтов, влитых ПОЗЖЕ заведения — #1149/#1264/#1150/#1151/#1154; счётчик `VP1–VP6` сверяй с реальными describe-блоками файла, не с числом из issue.)
2. **Классифицируй → маршрут:**
   - **PURE-DOC → master:** `.md` (INVARIANTS/ARCHITECTURE/README/CLAUDE/wiki) и **comment-only** правка в `src/` (коммент — это дока, НЕ «фикс кода»; но он обязан быть ПРАВДОЙ — сверь поле/export/строку, что он описывает: аудиты находят phantom-поля и мёртвые export'ы, чей коммент врёт).
   - **CODE → ветка/PR:** всё, что трогает `src/`-логику, добавляет/меняет **ТЕСТ** (property-lock, test-lock — это код, не дока), снимает/меняет **export** или сигнатуру, добавляет `Object.freeze`/меняет поведение. Это «фикс кода» в терминах «master если не требуется фикс кода».
   Один issue часто СМЕШАННЫЙ (#1241: caveat'ы в INVARIANTS — doc; property-test-locks + снятие мёртвого export + reword reject-сообщения — код).

## Шаг 1 — Pure-doc → master

Применяй pure-doc правки батча разом — **но волнами по пакету/области при >~8 issue** (discovery-набор бывает на 10-15 issue через 10+ пакетов). Одна волна = один пакет/область → один субагент → свой `git diff`-ревью → свой коммит: мега-дифф на 15 issue глазом не отревьюишь, а completeness-drift в чужом пакете потеряешь. Малый явный список (2-3 issue) — по-прежнему разом.
- **Механическое многофайловое применение делегируй субагенту** (`general-purpose`) с точными per-пункт спеками из черновиков issue + явным списком «out of scope» (код-пункты) + «не коммить». Потом **сам РЕВЬЮ полный `git diff`** — доки требуют точности (см. гочи ниже); субагент может промахнуться мимо строки или разъехать таблицу.
  - **⚠️ Permitted-file список ОБЯЗАН покрывать КАЖДЫЙ файл, который трогают пункты issue.** Мульти-пунктовый issue часто размазан по нескольким файлам (`ARCHITECTURE.md` + `README.md` + `INVARIANTS.md` + src-коммент) — вычитай file-таргеты ВСЕХ пунктов issue ПЕРЕД делегированием и внеси их все в Touch-ONLY. Заскоупишь узко — субагент верно ПОМЕТИТ забытый пункт «out of scope», но не тронет → issue закроется наполовину, а `Closes #N` соврёт. (Прогон #1207: заскоупил на `ARCHITECTURE.md`, items 3-4 жили в README/INVARIANTS — пришлось дочинять руками перед `Closes`.)
- **Коммит на issue** (или на пакет — pure-doc обычно разбит по пакетам) через `/commit-msg`. **Changeset НЕ нужен** — `.md`+комменты не триггерят `changeset-check` (он требует changeset лишь на поведенческую правку public-`src/`).
- **`Closes #N`** — только если ВСЕ пункты issue pure-doc (закроется push'ем в master, т.к. это default-branch). **`Refs #N`** — если issue СМЕШАННЫЙ (doc-часть готова, код-хвост остаётся → issue должен остаться OPEN).
- **⚠️ Wiki-страница → коммит в ОТДЕЛЬНЫЙ репо (`real-router.wiki`), его коммиты НЕ закрывают main-repo issue.** Для wiki-issue: `Refs #N` в коммите + **ручной `gh issue close #N` ПОСЛЕ wiki-push** (в отличие от `.md`/комментов в main-repo, где `Closes #N` срабатывает push'ем master). Issue, размазанный по обоим репо (часть — main `INVARIANTS`, часть — wiki-страница), закрывай только когда ОБА запушены.

**Гочи doc-правок:**
- **Table-cell гигиена:** строка Markdown-таблицы — это ОДНА строка. Многофразовый caveat, добавляемый в ячейку, держи single-line (без сырых `\n`, экранируй литеральный `|`) — иначе разъедешь таблицу. После правки глазом сверь, что столбцы целы.
- **Коммент ≠ код-фикс, но обязан быть правдой:** comment-only в `src/` идёт в master, но верифицируй описываемое против кода (Шаг 0.1) — иначе занесёшь в master дважды-неверный коммент.
- **Verify-against-source — несущее:** первая строка каждого добавляемого caveat/row сверяется с кодом, который она описывает; не переписывай черновик issue дословно, если код с тех пор уехал.
- **`git add <конкретные файлы>`, НИКОГДА `-A`/`git commit -a`.** Целевой репо — особенно `real-router.wiki` — часто несёт ЧУЖУЮ незакоммиченную работу (в прогоне #1177/#1187/#1203: 8 посторонних wiki-страниц + stray non-prettier reformat в main `parseSegment.ts`); `-A` смёл бы их в doc-коммит. Сверь `git status` перед add; stray-изменение не по теме (авто-reformat IDE) — откати (`git checkout -- <file>`), не коммить.

## Шаг 2 — Push в master (гейт)

Pure-doc → master — инфра-поток CLAUDE.md (без PR, без changeset). Но **push — только по явной просьбе** пользователя (стандартный гейт; «сделай doc-batch» ≠ «запушь»).
- **Сперва rebase, если локальный master разошёлся:** мид-сессионный коммит в master мог лечь на устаревшую базу (за сессию origin/master ушёл вперёд) — `git fetch origin && git rebase origin/master` ПЕРЕД push. (Прогон: локальный doc-коммит сел на до-мёрж базу; харнесс ребейзнул его при клинапе worktree — проверяй `git log origin/master..master` перед push.)
- `git push origin master --no-verify` — doc-only, валидировать нечего (комменты+`.md`, поведение/тесты не тронуты; полная pre-push сборка делегируется пользователю). Push default-branch'а закрывает `Closes #N`-issue — сверь `gh issue view <N> --json state` после.

## Шаг 3 — Код-хвост → ветка/PR

CODE-пункты (test-locks, reword'ы, снятие export/сигнатуры, `Object.freeze`) прогоняй через **`/bugfix-batch`** (если их несколько issue) или **`/bugfix`** (один issue, несколько суб-правок): ветка от актуального master, per-пункт коммит + changeset (если правится public-`src/`), один PR с `Closes` на СМЕШАННЫЕ issue (закроет их остаток). Гейт на «go».
- **Design-call внутри код-хвоста** (снять мёртвый export vs оставить reserved-с-честным-комментом; move `set` vs document invalidity) → `AskUserQuestion` ДО правки — как в `/bugfix-batch`.

## Гейты

- **НЕ push (даже в master) / НЕ PR / НЕ закрытие issue без явной просьбы** — готовь, показывай, жди «go». Локальные doc-коммиты — сам воркфлоу, не гейт.
- **НЕ `Closes` смешанный issue** на doc-коммите — код-хвост не готов; `Refs`, issue остаётся OPEN.
- **Completeness-drift — не оставляй полу-обновлённую таблицу молча.** Doc-таблица/список часто отстаёт не только на находку issue, но и на работу, влитую ПОСЛЕ заведения issue (гейты/фичи/инварианты). Пометь полный зазор в отчёте + предложи дозакрыть тем же pure-doc проходом (это тоже дока → master), но НЕ раздувай молча за рамки запрошенных issue — предложи.

## Итоговый отчёт

- Таблица по issue: `#<N>` → пункты (какие doc / какие код) → куда ушли (master-коммит `<hash>` / отложены на ветку) → статус (**CLOSED** / **OPEN** с код-хвостом).
- Что запушено в master (хэши) и какие issue реально закрылись (сверено `gh issue view`).
- Код-хвост: список пунктов + предложение ветки (`/bugfix`/`/bugfix-batch`).
- Completeness-drift, если нашёл, + предложение.

## Самокоррекция скила (ОБЯЗАТЕЛЬНО, в конце КАЖДОГО прогона)

Этот файл — живой; каждый прогон обязан его затачивать. После итогового отчёта вынеси короткую секцию **«Самокоррекция»** — критику не своей работы, а САМОГО ЭТОГО СКИЛА: где его текст промолчал, соврал, был двусмыслен или избыточен — и это стоило времени, лишнего шага или риска ошибки.

1. **Привязка к ИНЦИДЕНТУ, не общие советы.** Каждая правка — из конкретного момента ЭТОГО прогона: «скил сказал X / промолчал про X → реальность Y → я потерял/чуть не ошибся на Z». Нет инцидента — нет правки. Особый кандидат: ошибка классификации (счёл код за доку или наоборот), разъехавшаяся таблица, `Closes` на смешанном issue, push-на-устаревшей-базе.
2. **Высокая планка: изменило бы исход?** Предлагай, только если правка реально предотвратила бы ошибку или сняла двусмысленность, на которой ты споткнулся. Косметику отбрасывай.
3. **Граница doc-механика ↔ doc-batch.** Как редактировать конкретный вид доки (INVARIANTS-таблица, Mermaid, wiki-страница) — НЕ сюда. Сюда — только оркестрация: классификация doc/код, маршрут master/ветка, порядок, закрытие issue.
4. **Сначала заточи, потом дописывай (анти-раздувание).** Усиливай существующий абзац; не плоди секции.
5. **Предлагай, не применяй молча.** Выведи кандидатов таблицей (раздел | инцидент | предлагаемый текст), спроси разрешения. Применил по согласию → `.claude/commands/` это infra, в master напрямую, без changeset.
6. **Честность про «нечего править».** Прогон прошёл чисто → так и скажи: «правок нет».
