Проведи глубокое code-review РЕАЛИЗАЦИИ RFC/плана: сверь diff с каноном документа пункт-за-пунктом (недовыполнение / перевыполнение / отклонение), прогони КЛАСС каждого фикса по всем параллельным путям кода, эмпирически воспроизведи каждый заявленный багфикс «до/после», проверь оптимальность решения и возможности упростить код, вынеси ранжированный вердикт.

Входные данные:
$ARGUMENTS

Формат аргументов:
- Путь к RFC/плану (напр. `packages/core/.claude/rfc-navigation-commit-gate-2026-07-08.md`) + опционально диапазон реализации (`<base>..HEAD`, номер PR или список коммитов). Диапазон не задан → выведи его сам: коммиты, ссылающиеся на issue/RFC-маркеры документа (`git log --oneline --grep`), либо текущая ветка от merge-base с master.
- Опционально `--fix` — применить находки после отчёта (correctness — с regression-тестами; simplification/doc-drift — механически). Дизайн-развилки НИКОГДА не правятся молча — выносятся автору с вариантами.
- Опционально `max` — использовать workflow-backed ревью (`Workflow({name:"code-review", args:"max — …"})`), передав в args RFC-канон, диапазон и оси §3 как инструкции; findings воркфлоу затем прогнать через §2–§7 этого скила (воркфлоу не знает RFC-контекста сам). **Нет сохранённого `code-review` workflow (`.claude/workflows/` пуст) — сразу пиши inline RFC-scoped скрипт** (finder-оси §3 как отдельные finder'ы, канон embed'ом в их промпты, адверсариальная verify-стадия на каждую находку); это штатный путь для `max`, а не фолбэк — инлайн даже точнее, т.к. finder'ы получают канон в промпте вместо «воркфлоу не знает RFC» вслепую. Не трать проверку на «а есть ли named workflow» — по умолчанию пиши inline. Калибровка: `max` — для реализаций, идущих в master, трогающих security/cancellation-швы или большие diff'ы (десятки агентов, миллионы токенов — потому и явный опт-ин; его независимые finder-углы + адверсариальная верификация ловят слепые зоны автора — прецеденты: #1331 (3 CONFIRMED-бага на clone/error-путях, пропущенные инлайн-взглядом), commit-gate-1169 (finder нашёл `onTransitionLeaveApprove`-окно, не покрытое `suspendable` — автор проверил только onTransitionStart/subscribeLeave)); без `max` — компактные правки, где хватает собственных проб. **Гигиена рабочего дерева при `max`:** finder/verify-агенты пишут пробы прямо в рабочее дерево (иногда `*.test.ts` в `tests/`, где vitest их подхватит и уронит финальный гейт); ПОСЛЕ workflow — детерминированный скан+чистка leftover'ов (`_tmp*`, `probe-*`, `zz-*`, `qa-*`) ПЕРЕД гейтами §7, и не считать их за diff. И НЕ гоняй собственные src-мутации (§2 мутационная валидация) КОНКУРЕНТНО с бегущим workflow — finder прочитает транзиентную мутацию (`if (false && …)`) как working-tree-находку; либо серии по очереди, либо своя мутация в отдельном worktree.

Пример:
```
/code-review-rfc packages/core/.claude/rfc-navigation-commit-gate-2026-07-08.md 72517c6c..HEAD
/code-review-rfc .claude/rfc-file-routes.md --fix
```

## Принцип

**RFC — канон, реализация — претендент.** Это НЕ /audit-rfc (тот валидирует сам документ до реализации) и НЕ голый /code-review (тот не знает, что код ОБЯЗАН делать). Здесь двусторонняя сверка: (а) каждый пункт/DoD/шаг RFC → реализован ли, как заявлено; (б) каждый фрагмент diff → покрыт ли каноном. Три класса расхождений, все — находки до вердикта:

- **Недовыполнение** — пункт RFC отсутствует/частичен в diff (включая doc-sync-пункты и regression-план).
- **Перевыполнение** — реализация добавила поведение/обещание, которого в RFC НЕТ. Особо опасно, когда добавка *рекламирует* побочную возможность: пин-тест и комментарий «X works» на паттерн, который RFC предписывал запретить, — это новый публичный контракт, протащенный мимо дизайн-решения. (Прецедент #1331: RFC-оговорка требовала контракт «guard-фабрики side-effect-free», реализация вместо этого зарекламировала `usePlugin`-из-фабрики комментарием + пином `usePlugin:'ok'` — ревью вскрыло дубль плагина на каждом clone и самопротиворечие CLAUDE.md.)
- **Отклонение** — реализовано иначе, чем в RFC. Не автоматически ошибка: сверь мотив, вынеси вердикт «лучше/хуже/эквивалентно» с доказательством; молчаливое отклонение без записи в RFC-журнале — находка само по себе.

**Метод — эмпирика, не чтение.** Заявленный багфикс воспроизводится на базовом коммите (`git show <base>:<file>` / прогон пробы на base) и перепроверяется на HEAD; новый тест валидируется мутационно (red без фикса — иначе он пин, а не защита). Вердикты субагентов/воркфлоу на несущих находках перепроверяй собственным прогоном.

## Шаг 0 — Скоуп и карта канона

1. Прочитай RFC целиком; выпиши таблицу обязательств: каждый шаг плана (A1…), каждый пункт DoD, regression-план, doc-sync-пункты, perf-план, заявленные метрики выгоды («−N строк», «закрывает issues X,Y»), явные НЕ-цели («что НЕ делаем»).
2. Определи diff (`git diff <base>..HEAD --stat`) и сверь границы: файлы вне карты канона → кандидаты в «перевыполнение»; пункты канона без файлов → «недовыполнение».
   - **zsh глотает мету в revision-range/grep.** `<base>^..HEAD`, `~1`, а также `|` в grep-паттерне под zsh раскрываются как glob/negation → path-limited `git diff <range> -- <path>` молча возвращает ПУСТО (не «файл не менялся», а команда не отработала как задумано). Квотируй диапазон (`git diff '<base>~1..HEAD' -- <path>`) или бери явные SHA / `git show <sha>:<path>`; пустой path-diff при непустом `--stat` того же диапазона — красный флаг тула, перепроверь другим способом, НЕ выводи «doc-sync чист» из пустоты.
   - **История могла переписаться в ходе ревью.** На активном `master` диапазон/HEAD дрейфует под тобой (squash/rebase/amend между твоими git-вызовами). Если ревью длится — перед вердиктом пере-сними HEAD и диапазон, и **якорь находки к КОНТЕНТУ рабочего дерева, не к границам коммитов**: `git diff <old-HEAD> <new-HEAD> --stat` пуст ⇒ tree идентичен (прежнее чтение файлов валидно, менялись лишь коммит-границы); непуст ⇒ перечитай изменённое.
3. RFC под `.claude/` gitignored — реализация могла разойтись с ним молча; если у RFC есть журнал ревизий, сверь, против какой ревизии писалась реализация.

## Шаг 1 — Пункт-за-пунктом

Каждая строка карты канона получает статус: **DONE** (с `file:line`-доказательством) / **PARTIAL** / **DEVIATED** (+вердикт лучше/хуже) / **OVER-DELIVERED** (+что именно добавлено сверх) / **MISSING**. Явные НЕ-цели RFC — проверь, что реализация их НЕ сделала (ползучий скоуп — находка).

## Шаг 2 — Багфиксы: «до/после» исполнением

- Для каждого заявленного фикса: воспроизведи дефект на base (проба/`git stash`-free способ: `git show`, отдельный worktree или прогон committed-проб на base) и его отсутствие на HEAD. Если RFC ссылается на committed-пробы (`benchmarks/audit-probes/…`) — прогони их: ожидания должны были «флипнуться» ровно так, как RFC предсказывал.
- Новые тесты — мутационно: временно откати фикс (или внеси эквивалентную мутацию) → тест обязан покраснеть. Зелёный-до-фикса тест — недискриминирующий пин, не регресс-защита (находка).
- Проба, читающая поток событий/логов, фиксирует снимок ДО своих же диагностических пост-вызовов — иначе артефакты хелперов читаются как события сценария.

## Шаг 3 — Класс по всем путям (главный урок)

Каждый исправленный дефект — представитель КЛАССА. «Починен на пути X» ≠ «класс закрыт»: перечисли все параллельные пути, где живёт та же механика, и прогони класс на каждом. Оси real-router (расширяй по месту):

- **constructor ↔ cloneRouter ↔ #recompileSlot** — код, исполняющий пользовательские фабрики (guard/plugin), исполняет их повторно на клоне и при рекомпиляции слота; фикс «фабрика видит готовый инстанс» обязан держаться на всех трёх. (Прецедент: план A починил конструктор — clone-путь re-исполнял фабрики на пустом config и дублировал плагины; оба бага CONFIRMED ревью.)
- **happy-path ↔ error-path нового кода** — брось исключение в середине новой процедуры: что остаётся? fail-open или fail-closed? Утёкшие наружу ссылки на полусобранный/полуразобранный объект должны быть инертны. (Прецедент: throw в середине flushPendingGuards оставлял живой router без поздних гардов — access-control bypass; фикс dispose-before-rethrow.)
- **sync ↔ async ветви**, **navigate ↔ start() ↔ navigateToState ↔ replace()-ревалидация**, **bare core ↔ с validation-plugin** (дефолтный конфиг проверяется ПЕРВЫМ), **base ↔ per-request clone (SSR)**, **dispose/teardown-симметрия** (всё, что фикс регистрирует, — снимается ли).

## Шаг 4 — Оптимальность решения (обязательный)

- **Аллокации и hot-path:** новые замыкания/объекты на per-call пути (не per-instance) — находка, если рядом уже есть shared-паттерн; сверь с Performance Notes пакета (какие пути «горячие» — задокументировано). (Прецедент: `getDependency`-замыкание аллоцировалось на каждый compile при уже существующей shared-фабрике рядом.)
- **Заявленные метрики выгоды пересчитай:** «−N строк», «−M аллокаций», «закрывает K issues» — `git diff --stat`/скан, не вера. Расхождение метрики при сохранённой структурной ценности — MINOR-находка с честной цифрой; расхождение, рушащее мотивацию решения, — SHOULD-FIX с эскалацией автору. (Прецедент: план B обещал −150 строк, реально −11 — ценность пересформулирована как структурная.)
- **Sub-ms перф-утверждения** («не медленнее», «+2 сравнения — шум») на hot-path — ТОЛЬКО same-session A/B (swap+rebuild+back-to-back, бенчи пакета); кросс-сессионные цифры не принимаются. RFC обещал perf-замер → он должен быть выполнен или явно передан пользователю как хвост.
- **«Можно лучше → надо», симметрия:** найденная оптимизация портируется во ВСЕ применимые места diff, не только в предъявленное; «здесь уже быстро → пропустим» — не аргумент.
- **Анти-цель:** не предлагать микрооптимизации cold-path ценой читаемости и не «оптимизировать» без замера то, что RFC осознанно оставил простым.

## Шаг 5 — Упрощаемость (обязательный)

- **Дубли, внесённые реализацией:** скопипащенные замыкания/комментарии/хелперы — особенно когда в том же файле уже есть shared-паттерн для соседа (несогласованность = будущий drift). (Прецедент: `getValidator` продублирован в двух wire-функциях при shared `compileFactory` строкой выше.)
- **Мёртвое:** экспорты/реэкспорты/параметры/поля без консьюмеров (`grep`-ом наличие, node-сканом отсутствие); леса времянки (пробные хелперы, debug-логи, `(temp)`-маркеры).
- **Лишняя церемония:** класс на один call-site, builder без вариативности, интерфейс-прослойка с единственным имплементором, тройное повторение списка полей — кандидаты на схлопывание. Проверь и ОБРАТНОЕ: не пере-схлопнула ли реализация то, что RFC осознанно оставил разделённым.
- **Удаление оправдывается usage-сканом** (вызовы, `git log -S`), не памятью и не комментарием.
- Разграничение с /simplify: здесь упрощение сверяется с каноном RFC — «проще» не должно ломать пункт документа; каждое предложение-упрощение указывает пункт RFC, которому оно не противоречит.

## Шаг 6 — Doc-sync и хвосты

- Doc-коммит реализации сам аудитится: скан по СТАРЫМ именам/формулировкам во всём репо — включая ДРУГИЕ пакеты (тайпинги/комментарии в core-types!), тесты-комментарии, wiki, `IMPLEMENTATION_NOTES`/`ARCHITECTURE`/`CLAUDE.md` всех уровней. «Нулевой остаток» доказывается детерминированным сканом, не выборочным grep.
- Line-цитаты в комментариях/доках сдвигаются соседними мержами — проверяй адреса на HEAD; полу-обновлённый комментарий (одна строка поправлена, соседняя ссылается на старое имя) — типовая находка.
- Самопротиворечие доков: новый контракт в одном месте против старой формулировки в другом (тот же файл считается!).
- Changeset-дисциплина: bump по типу (fix=patch), один пакет на файл, private→changeset на core; cross-package правка src публичного пакета (даже комментарий в тайпингах) требует своего changeset.

## Шаг 7 — Гейты и вердикт

- Точечные прогоны затронутого пакета: type-check (`tsc`, IDE-диагностике не верить — бывает stale), functional c coverage-порогом, property, lint/prettier изменённых файлов, changeset-линт. Полный `pnpm build` НЕ гонять — финальная валидация у пользователя.
- **Отчёт:** (1) таблица канона со статусами и доказательствами; (2) findings **BLOCKER / SHOULD-FIX / MINOR**, каждая с категорией `correctness | rfc-deviation | optimality | simplification | doc-drift`, `file:line` и однострочным фиксом; (3) вердикт: реализация принимается / принимается с правками / возвращается — отдельно для кода и для соответствия канону; (4) если находки меняют сам RFC (новая грань, опровергнутая посылка) — предложи ревизию документа, не молчаливый дрейф.
- При `--fix`: correctness-фиксы — только с regression-тестом на каждый; после правок — повторные гейты; сводка «что применено / что оставлено автору».

## Жёсткие правила

- Несущие вердикты (баг подтверждён / класс закрыт / метрика сходится) — только собственным исполнением; агентское резюме — гипотеза.
- Дефолтный конфиг проверяется раньше максимального (opt-in-слои маскируют дыры).
- **При accept/reject-дивергенции двух слоёв — определи, КТО неправ, round-trip'ом принятой стороны, ПРЕЖДЕ чем фиксировать находку и её направление.** `gate=REJECT / backstop=ACCEPT` имеет ДВА взаимоисключающих прочтения: (a) gate ложно реджектит ВАЛИДНЫЙ роут — принятая сторона round-trip'ит (`buildPath`→`match` возвращает параметры) → баг в gate; (b) backstop молча принимает МЁРТВЫЙ роут (`buildPath` эмитит URL, который его же `match` отвергает) → баг в backstop, иной класс и severity. «Дивергенция подтверждена» без round-trip принятой стороны — недоказанное направление (частый источник ложной severity). (Прецедент tokenizer-review 07-10: `~:id` gate=REJECT/backstop=ACCEPT сперва прочитан как «gate ложно реджектит валидное», но принятая сторона оказалась DEAD → это backstop-баг класса «путь без ведущего /», другое направление и другой владелец фикса.)
- Не навязывай вкусовщину: отклонение от RFC без противоречия и без ухудшения — вердикт «эквивалентно», не находка.
- Правки реализации = обычный код-флоу репо (changeset, тесты, конвенции); правки RFC-документа — с ревизией в журнале. Push/commit — только по явной просьбе или заведённому в сессии порядку.
