# path-matcher

Internal package providing the segment-trie URL matcher, parameter encoding/decoding, constraint validation, and path building. Not published to npm -- consumed by `route-tree` which wraps it via `createMatcher`.

## Exports

Public surface is intentionally narrow (#740): `index.ts` exports only these plus types. Encoding helpers, `createSegmentNode`, and constraint validation are **internal** — used inside `SegmentMatcher`/`registration`, and in tests imported directly from `src/*` (not the package index).

| Export | Description |
|--------|-------------|
| `SegmentMatcher` | Core class: `registerTree()`, `match()`, `buildPath()`, `hasRoute()` |
| `buildParamMeta(path)` | Extracts URL/query/splat params + constraints |

### Types

| Type | Description |
|------|-------------|
| `SegmentMatcherOptions` / `ResolvedMatcherOptions` | Matcher configuration (caseSensitive, strictTrailingSlash, etc.) |
| `MatchResult` | Match output: `{ segments, params, meta }` |
| `CompiledRoute` | Internal compiled route with build slots, constraints, caches |
| `MatcherInputNode` | Tree node input for `registerTree()` |
| `SegmentNode` | Trie node: static children, param child, splat child |
| `ParamMeta` | Extracted param metadata (urlParams, queryParams, spatParams, constraintPatterns) |
| `BuildParamSlot` / `BuildPathOptions` | Path building types |
| `ConstraintPattern` | `{ pattern: RegExp, constraint: string }` |
| `URLParamsEncodingType` | `"default"` / `"uri"` / `"uriComponent"` / `"none"` |

## Gotchas

- **Zero dependencies (runtime AND dev)** -- `path-matcher` imports nothing from `search-params` anywhere: not in `src/**`, not in `tests/**`. The test helper `tests/helpers/createTestMatcher.ts` ships its own minimal inline `parse`/`build` that mirrors `search-params` no-strategies defaults. This keeps the package fully self-contained
- **`parseQueryString`/`buildQueryString` are REQUIRED** in `SegmentMatcherOptions` -- no default. Constructing a matcher without them is a type error. Tests use `createTestMatcher()` from `tests/helpers/createTestMatcher.ts` which wires an inline minimal parser; production uses `route-tree/createMatcher()` which wires `search-params.parse`/`build`
- **Inline test parser can drift from `search-params`** -- if you change query-string semantics in `search-params` (boolean/null handling, encoding rules), update `tests/helpers/createTestMatcher.ts` in lockstep. There is no automatic drift check — tests will keep passing against stale semantics
- **Single-pass scanner** -- `SegmentMatcher.match()` uses a single-pass `#scanPath` that replaces 4 separate scans (hash, unicode, query, double-slash) for performance
- **Static cache** -- Routes without URL params are cached after first `registerTree()` for O(1) matching
- **Constraint validation is two-phase** -- During `match()`, constraints filter out non-matching routes; during `buildPath()`, constraints throw on invalid param values. Both validate the **decoded** value: `match()` runs `#validateConstraints` *after* `#decodeParams`, so a constraint describes the value the consumer receives, not the raw URL segment — `/users/%35` satisfies `<\d+>` (decodes to `5`), and a raw form matching the regex but decoding to a violating value is rejected (#857)
- **Splat encoding** -- Splat params (`*path`) encode each segment individually (preserving `/` separators) via the single `encodeParam(value, encoding)` in `encoding.ts` (splat-only, 2-arg since #1397 dropped the dead `isSplat` flag); `compileBuildParts` reuses it instead of inlining, so the `encodeParam` unit/property suites assert the exact code prod runs (#860)
- **`buildParamMeta` query detection is mask-based** -- before locating the query `?`, both the optional-param marker (`:param?`) **and** `?` inside a `<...>` constraint (e.g. the lazy quantifier in `:id<\d?>`) are neutralized on a length-preserving mask, so neither is mistaken for the query separator and the match index still maps onto the original path (#738/#741)
- **Single param-name grammar** -- `PARAM_NAME_PATTERN` (`[^/?<]+`, exported from `buildParamMeta.ts`) is the char CLASS a name may contain; the per-segment name BOUNDARY (where the name ends — including the #1324 trailing-marker rule) is owned by the `parseSegment` tokenizer, consumed by **all four** path-grammar layers: `buildParamMeta` (L1), `compileBuildParts` (L2 build — `splitPathSegments` + `parseSegment` per segment, re-adding the `/` separators `splitPathSegments` strips to rebuild `buildStaticParts` via `makeBuildParamSlot`), `registration`'s `extractParamName` (L3 trie), and route-tree's gate via `findSegmentGrammarError` (L4) — so meta, build, match, and the gate cannot disagree on a name (#738/#1324). L2 was the last layer migrated off a parallel `paramRgx` (Реш.1 of the post-tokenizer-simplification RFC): behavior-preserving — `buildPath` is byte-identical on every accepted route (old-vs-new diff + `buildPath` A/B vs master, 0-diff), and the inverse-pair round-trip property still locks build ≡ match. `PARAM_NAME_PATTERN` remains the char class the query mask (`OPTIONAL_PARAM_MARKER_RGX`) and the #1050 fused-marker backstop derive from
- **Single constraint grammar (#804)** -- `CONSTRAINT_BODY_PATTERN` (`[^>]*`, exported from `constraint-grammar.ts`) is the one definition of the `<...>` delimiter body; all three constraint regexes (`OPTIONAL_PARAM_MARKER_RGX`, `CONSTRAINT_RGX` in `buildParamMeta`; `CONSTRAINT_PATTERN_RGX` in `registration`) derive from it (Реш.1 removed the fourth — `compileBuildParts`'s `paramRgx` — when L2 moved to `parseSegment`), reconciling a former `+`/`*` desync (match side was `<[^>]+>`, strip/build `<[^>]*>` — they disagreed on the empty `<>`). `isConstraintBalanced` is the one balance predicate, consumed by route-tree's gate (replacing its former local `hasBalancedConstraints`). The constraint-axis counterpart of the param-name single source (#738)
- **Name-less marker rejected at registration (#858)** -- `PARAM_NAME_PATTERN` requires ≥1 char, so a bare marker with no name (`:`, `*`, or one carrying only a modifier — `:?`, `:<\d+>`) is not a valid param. `registerTree()` throws `[SegmentMatcher.registerTree] Empty parameter name …` in `registerNode`'s per-segment `parseSegment` pre-pass (Реш.2 — one place, was every child-creation site); `extractParamName` keeps a defensive `throwEmptyParamName` for the now-unreachable case. Without the guard a name-less marker compiled to a phantom empty-named slot: `match("/files/x")` captured `{ "": "x" }` while `buildPath` emitted a literal `/files/*` and `buildParamMeta` reported no param — a three-way match/build/meta desync of the #736/#738 class
- **Fused mid-segment marker rejected at registration (#1050)** -- a `:`/`*` marker glued to a static prefix *within* a segment (`/a:b`, `/users/x:id`, `/a*b`) is extracted as a param by build/`buildParamMeta` (their marker regex is unanchored) but the trie compiles a non-marker-led segment as a static literal — so `buildPath` emits an unmatchable URL while `match` rejects it. `registerNode`'s per-segment `parseSegment` pre-pass throws `[SegmentMatcher.registerTree] Fused parameter marker …` (Реш.2 replaced the former `FUSED_MARKER_RGX` char-scan in `processSegment`'s static branch — `parseSegment`'s `fused-marker` code, the SAME the gate reads, now decides it), the sibling of the #858 name-less rejection. A marker-*led* segment whose name itself contains `:`/`*` (`/:a:b` → the greedy param `a:b`) is unaffected — only a marker *after* a static prefix drifts. route-tree's validation gate catches it first with a route-contextual error
- **Trailing parameter marker rejected at registration (#1324)** -- a param or splat name ending in a bare marker (`/:y*`, `/:y:`, `/*y:`): the name-class `[^/?<]+` greedily swallows the trailing `:`/`*` into the name (`y*`), so bare core registered a dead route — `buildPath` then threw `Missing required param 'y*'` — while route-tree's gate rejected it (name-less), a gate↔backstop divergence. `registerNode`'s per-segment `parseSegment` pre-pass (Реш.2) throws `[SegmentMatcher.registerTree] Trailing parameter marker …` — `parseSegment` ends the name BEFORE a trailing marker, the SAME boundary the gate reads (`extractParamName` in the trie still tokenizes the name, but the malformed form is already rejected upstream). A MID-name marker (`/:a:b` → param `a:b`) is unaffected. The sibling of #858/#1050 on the trailing-marker axis; this closes the LAST gate↔backstop divergence — formerly the one excluded parity case, now route-tree GBP6
- **Unbalanced/empty constraint rejected at registration (#804)** -- `registerNode` throws `[SegmentMatcher.registerTree] Unbalanced constraint delimiter …` / `Empty constraint …` before trie insertion for a path whose `<...>` delimiters are unbalanced (`/x/:id<\d+`, `/x/:id<`, a stray `>`) or empty (`/x/:id<>`, which compiles to a never-matching `^()$`). Unbalanced is caught by `isConstraintBalanced` — the ONE whole-path check kept, since a stray `>` is invisible to the per-segment tokenizer; empty `<>` is now caught by `registerNode`'s `parseSegment` pre-pass (`empty-constraint` code, Реш.2 replaced the former `includes("<>")` literal check). Bare core built these silently and `buildPath` then emitted a garbage URL (`/x/1<\d+`); this is the backstop covering the `createRouter`-first flow for every consumer, the sibling of the name-less (#858) and fused (#1050) rejections. route-tree's validation gate catches it first with a route-contextual error
- **Optional splat `*name?` rejected at registration (#1149)** -- `*path?` desynced build/meta (a splat: multi-segment, `/`-preserving encoder) from the trie (the optional fork `segment.endsWith("?")` compiled a plain single-segment param via `ensureParamChild`), so `buildPath({ path: "a/b" })` emitted `/files/a/b` which `match` rejects — a dead deep-link. `registerNode`'s per-segment `parseSegment` pre-pass throws `[SegmentMatcher.registerTree] Optional splat … is not supported` (Реш.2 removed the former `startsWith("*")` check in `insertIntoTrieFrom`'s optional fork — `parseSegment`'s `optional-splat` code decides it), the sibling of the name-less (#858) / fused (#1050) rejections. A required splat `*name` (incl. `*path?query`, where `?` starts the query and is stripped before the trie) is unaffected. route-tree's validation gate catches it first with a route-contextual error. Product decision (reject, not support): the shape only ever matched 0–1 segments (pre-1.0 → minor)
- **Optional param directly before a dynamic successor (#1263/#1264, "SUPPORT-NARROWER")** -- an optional `:opt?` immediately followed by a param or splat is a `paramChild` **fork** (`ForkMeta` on the slot; `markOptionalFork` in `registration/trieNodes.ts` for the optional fork — the *non-optional* constrained-param + splat-sibling case is #1266, handled by `markConstrainedParamFork`, see the end of this entry). `match` disambiguates the omit form two ways. **A1 opt→splat** (`/:v<c>?/*rest`, e.g. versioned-API / preview-mode): *try-take-if-valid* — take the segment as the optional only if its **decoded** value satisfies the constraint (`#tryTakeFork` inline-decodes before `#decodeParams`, the #857 decode-edge; malformed-percent → skip → splat captures → `#decodeParams` rejects → UNMATCH, never throws), else the splat captures it. **A2 opt→required-param** (`/:a?/:b`): *segment count* — the omit form is one segment shorter, so on the LAST segment (`segmentEnd >= length`) the optional is omitted and the segment binds under the successor's name (`fork.skipName`), not the optional's (the wrong-name bug); the optional's own constraint is validated post-traverse (the #1148 `Object.hasOwn` guard skips it when absent). **Limitation (#1286):** this covers only a TERMINAL successor — a non-terminal one (`/:a?/:b/c`, `/:a?/:b/:c`) is NOT disambiguated (`segmentEnd >= length` holds only when the omit makes the successor the last segment), so the omit form binds under the OPTIONAL's name (`/x/c` → `{a:"x"}`, wanted `{b:"x"}`). Pinned as a caveat-lock; the general fix (variable-depth remaining-segment counting) is disproportionate. The fork branches are **inlined** in `#traverseFrom` (a helper call regressed single-param ~5%, spike-measured — hence its `cognitive-complexity` disable). **B: unconstrained opt→splat (`/:v?/*rest`) is REJECTED** at registration (`throwUnconstrainedOptionalSplat`) — no constraint = no validity signal, so every multi-segment value would silently reshape; add a constraint or split into two routes. `opt+opt` (`/:a?/:b?`) is out of scope (present-first, documented — INVARIANTS #8). `buildPath` is non-injective on opt→splat (two param sets → one URL), so the inverse-pair property asserts params-equality for opt+required but only path-fixpoint (#18) for opt→splat. **#1266 (required generalization, no `optional` anywhere):** a constrained **required** param sharing its trie level with a splat sibling from ANOTHER route (`/*rest` + `/:v<c>/*rest`) gets the SAME A1 try-take-if-valid. `markConstrainedParamFork` marks `fork.constraint` **unconditionally** for every constrained required param; `match` acts on it only when a splat sibling is ALSO present (`node.splatChild`), which makes it **registration-order independent** (the splat may be registered before or after, by a different route). Without it the `:v` paramChild greedily committed, its constraint failed only post-traverse (no backtrack), and the catch-all was unreachable — its `buildPath` a dead deep-link (INVARIANTS #8 rigidity, the #1264 class with no optional). **Follow-up adversarial review (#1283/#1284/#1285/#1287, all in this batch):** **#1283** — a constraint-SATISFYING take that dead-ends on the LAST segment (take-node has no terminal route, only a would-be-empty splat child) now ALSO skips to the splat (dedicated check since subsumed by the #1288 validated sub-traverse — behaviour unchanged) (`/v1` on `/:v<c>?/*rest` or `/*rest`+`/:v<c>/*rest` was UNMATCH — the fork reproduced the very dead-deep-link class it closed); a take-node WITH a route stays present-first. The property's `arbNonVersionSplat` was blind to single-segment version values → single-segment-omit property added. **#1284** — a slot serving routes with DIFFERENT constraints now uses the **disjunction** of all of them (`markConstrainedParamFork` composes `(?:src1)|(?:src2)` of the anchored sources, one `.test`; `pc.fork` re-created via spread since `ForkMeta.constraint` is readonly), not the first-registered — which silently killed a same-name sibling under a splat, order-dependent. **#1285** — `markConstrainedParamFork` short-circuits on `!compiled.hasConstraints` before its `extractParamName` call, and `hasNonAsciiSegment` uses a `charCodeAt` index loop (registerTree perf, ~58% of the SSR clone tax). **#1287** — ≥2 optional params directly before a splat are REJECTED (`hasMultipleOptionalsBeforeSplat`, now in `parseSegment.ts` on the RAW path via `splitPathSegments` — shared verbatim with route-tree's validation gate so the two can't drift): one slot carries only one opt→splat fork, so the omit-outer/take-inner form (`/ab/x`) silently reshaped into the splat
- **Param+splat junction resolves by validated sub-traverse (#1288)** -- at a node carrying BOTH a `paramChild` and a `splatChild`, `match` no longer decides by the current segment alone: the param branch is tried on a scratch object (recursive `#traverseFrom`) and commits only when it structurally completes AND the reached route's constraints hold on the decoded scratch values (`#branchConstraintsHold` — #857 decoded semantics, #1148 absent-param skip, undecodable → not taken, #737); otherwise the splat sibling captures. The uniform INVARIANTS #8 rule: **"param wins if its branch can complete"**. Subsumes the #1283 last-segment dead-end branch (removed) and lifts the former unconstrained greedy carve-out — `/user/:id/profile` + `/user/*rest` now resolves `/user/x/settings` to the catch-all (was UNMATCH), as does a deep constraint dead-end below the junction (`/*rest` + `/:v<v\d+>/:id<\d+>` on `/v1/abc`). The decoded-constraint fast-reject (`#tryTakeFork`) is kept as an allocation-free short-circuit before the scratch. The same `#branchConstraintsHold` guards `#matchSplat`'s specific-child candidate (the #24 blind spot: `/files/*any` + child `/:id<\d+>` on `/files/xx` structurally reached the child, its constraint then killed the whole match — now it falls back to the wildcard). Nested junctions are polynomial (spike: a 32-level chain ≈ 85 µs, no exponent; a bottom failure falls back LOCALLY to that level's splat — no cascades, because a splat always completes); non-junction param hops never enter the block, so the common single-param path is untouched (spike: family take pays ~+8-11% for the scratch, fast-reject/skip ≈ 0, everything else in noise).
- **Static segments do not backtrack** -- once a segment matches a static child, traversal commits to it; if the rest fails it does NOT retry a param sibling (`/users/new/posts` with `/users/new`+`/users/:id/posts` → `undefined`). Intentional/greedy (INVARIANTS Matching #16, #740)
- **Empty required param rejected at build** -- `buildPath` throws for `""` on a required param (it would collapse the segment and match the parent); optional params unaffected (#740)
- **Performance-critical encoding** -- `encodeURIComponentExcludingSubDelims` uses pre-check regex to skip encoding for all-safe strings (29-57x faster for alphanumeric). A **lone (unpaired) surrogate** matches the regex → slow path → `encodeURIComponent(match)` throws `URIError` (the `u` flag only coalesces *paired* surrogates). `totalize` wraps the slow path (and `uri`/`uriComponent` in `ENCODING_METHODS`) to sanitize it to U+FFFD via a lone-surrogate regex (a lib-agnostic manual `toWellFormed` — consumer tsconfigs may predate es2024), keeping `buildPath` total (#1315) — the fast path never sees a surrogate, so it pays nothing. Twin of search-params' `safeEncode` (zero-dep package → inlined, not imported), lossy like `-0` (INVARIANTS Encoding #1)
- **Param-name conflict detection (#736)** -- a trie position is keyed by *position*, not by name; the captured value is written under the name recorded on that position. `registerTree()` therefore throws `[SegmentMatcher.registerTree] Parameter name conflict …` when **two different routes** put differently-named params/splats on the **same** position (`/user/:id` + `/user/:slug/profile`) — otherwise first-registration would silently win and the second route would capture under the wrong key. The guard is strictly cross-route: a single route's own consecutive optionals (`/a/:b?/:c?/d`, `/a/:b?/:c/d`) legitimately reuse one position via the optional-omit branch, tracked by the per-route `ownNodes` set threaded through `registration/trie.ts` so they never trip the guard
- **Fused constraint suffix rejected at registration (#1150)** -- static text glued to a constraint's closing `>` within a segment (`/:year<\d+>-archive`, `/:id<\d+>.html`) desyncs build (strips `<…>` then re-extracts the name greedily, fusing the suffix — name `year-archive`) from meta (ends the name at `<` — name `year`), compiling to a silent dead route. `registerNode`'s per-segment `parseSegment` pre-pass throws `Text fused to a constraint …` (Реш.2 removed the `hasFusedConstraintSuffix` char-scan; `parseSegment`'s `fused-constraint-suffix` code — a `>` not followed by `/`, `?`, or end-of-input, the SAME token the gate reads — decides it, so the two can't drift). The mirror of #1050 on the OTHER side of the param; route-tree's gate backstops with a route-contextual error
- **Constraint in a static segment rejected at registration (#1311)** -- a `<...>` constraint filling a STATIC segment (no `:`/`*` marker) — `/foo<bar>`, `/a<b>` — is stripped by the marker-agnostic `CONSTRAINT_PATTERN_RGX` to `/foo` / `/a`, silently reshaping the route. `registerNode`'s per-segment `parseSegment` pre-pass throws `Constraint '<...>' in a static segment …` (Реш.2 removed the `hasConstraintInStaticSegment` char-scan; `parseSegment`'s `constraint-in-static` code — the SAME the gate reads — decides it). Because that scan checks `<` in a static BEFORE checking emptiness, an empty `<>` in a static (`/foo<>`) now reports `constraint-in-static`, not `empty-constraint` — a deliberate reject-*reason* shift (Реш.2 F-A; accept/reject unchanged). A constraint on a PARAM (`/:id<\d+>`) is unaffected. route-tree's gate backstops with a route-contextual error. The sibling of #1050/#1150 on the static-segment axis
- **Duplicate param name in one route rejected at registration (#1151)** -- the same name twice in one route (`/:id/:id`), a param+splat clash (`/:x/*x`), or a parent's param reused by a child bind two trie positions under ONE name — match's later capture silently overwrites the earlier and `rewritePathOnMatch` rewrites the user's URL (`/1/2` → `/2/2`). `compileAndRegisterRoute` throws `Duplicate parameter name …` when `buildParamNamesSet.size ≠ buildParamNames.length`. The #736 guard only fires on DIFFERENTLY-named params; this is the same-name axis (route-tree's gate mirrors the same-path case)
- **Duplicate effective path rejected at registration (#1153)** -- two routes compiling to the same trie terminal (flat `/a/b` vs nested `a`→`b`, `/x` vs `/x/`, two `/`) silently shadowed each other (the later's deep link resolved to the earlier). `writeStrongRoute` distinguishes STRONG (full-insertion) terminal ownership from WEAK (optional-omit `??=`) via a `routeIsStrong` boolean on `SegmentNode` (registration-only, never read on the match hot path) and throws `Duplicate route path` on a second STRONG write by a different route; a weak owner is legitimately displaced and a same-route revisit is idempotent (`node.route === compiled`). Matcher-level ONLY — the collision materializes in the compiled trie, invisible to route-tree's per-path gate. A same-named splat is inherently terminal, so its shared-position case is a dup here
- **Non-ASCII static segment rejected at registration (#1154)** -- a raw non-ASCII static (`/café`, `/меню`) registers but never matches (match rejects input ≥ U+0080 in `#scanPath` AND compares static trie keys raw, never percent-decoded), so `buildPath` emits `/café` which `match` rejects — a dead route. `processSegment`'s static branch throws `Non-ASCII static segment …` via `hasNonAsciiSegment` with the percent-encode workaround (`/caf%C3%A9`). A non-ASCII PARAM name or constraint body (`:id<[а-я]+>`, matched against the DECODED value) is unaffected — only static text is compared raw. route-tree's gate backstops with a route-contextual error
- **Malformed query-param declarations rejected at registration (#1242 §5.1/§5.3)** -- a query-param name containing `<`/`>` (a constraint leaked in via a reverse-order modifier typo — `/a/:b?<\d+>` parses the `?` as the query start, so `<\d+>` becomes the query name) or one colliding with a path-param name (`/a/:tab?tab`, where `buildPath` emits the value twice — `/a/x?tab=x`) is rejected. `validateQueryParamDeclarations` scans `buildParamMeta.urlParams` + the declared query names; route-tree's gate mirrors it. A `=` in a declaration (`?name=value`, §5.2) is NOT rejected — see next
- **`?name=value` in a route definition is NOT a query-default feature (#1242 §5.2)** -- the `=value` becomes part of the query-param NAME (`buildParamMeta("/search?q=term").queryParams === ["q=term"]`), so the param is literally named `q=term` and never round-trips cleanly. Bare core tolerates it (the existing "valid query params" tests accept it), so it is left accepted rather than made breaking — but declare a plain `?name`; a per-param default value is not supported here
- **Index route under an optional-param (ANY position) or splat parent rejected at registration (#1242 §5.4, #1294)** -- an index (`path: "/"`) under a parent whose path carries an OPTIONAL param in **any** position (`/a/:b?`, `/a/:b?/c`) or ends in a SPLAT (`/files/*rest`) is unreachable/inconsistent (under an optional it binds only the take form — `/a/:b?/c` + idx: `/a/x/c/` → index but `/a/c/` → parent, because `walkTrie` lands `slashChildRoute` on the full-take terminal only and does not fan out omit forms; under a splat `slashChildRoute` sits on the splat node, which `#matchSplat`'s fast path never reads). `insertSlashChildIntoTrie` scans all parent segments (`throwSlashChildUnderDynamicParent`); #1242 checked only the LAST segment and missed mid-path optionals (#1294). A REQUIRED-param parent (`/users/:id`, `/a/:b/c`) has a single form and its slash-child is coherent — **allowed** (an existing tested feature). Matcher-level only — a trie-terminal-shape concern the per-path route-tree gate can't see (like #1153)
- **The matcher is INVALID after a `registerTree` throw (#1242 §5.6)** -- `registerTree` registers routes one-by-one and `routesByName.set` runs before trie insertion, so if route N is malformed and throws, routes 1..N-1 are already half-registered (they still `match`, and `buildPath` of the rejected route emits garbage). Unreachable via `createRouter` (it throws wholesale and discards the matcher); relevant only to a DIRECT `SegmentMatcher` consumer, which must **discard the matcher after a `registerTree` throw**, not reuse it (product decision: document, not make registration atomic — the case is unreachable through the public API)
- **A `__proto__` query key survives the merge as an own property (#1293)** -- `#mergeQueryParams` folds the parsed query into the params accumulator via `Object.defineProperty` (a small `assignQueryParam` helper), not a plain `params[key] = …`. For the literal key `"__proto__"` a plain assign invokes the inherited setter and silently drops the param — annulling search-params' #855 own-key hardening one layer up (a string value vanishes, an array value swaps the local prototype). `defineProperty` writes a genuine own entry; no prototype pollution either way (`makeState` spreads own keys into a fresh object). Mirrors search-params' "Prototype-name keys are own properties" gotcha, one pipeline layer up
