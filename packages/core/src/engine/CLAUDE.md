# engine (`core/src/engine`)

The router's **routing engine** — a self-contained, zero-external-dependency subsystem of `@real-router/core` (the merged engine, engine-merge #1510). The former `route-tree` facade lives at this directory's root; `path-matcher` (segment-trie URL matching) and `search-params` (query-string parse/build) are folded in as internal **layers** under `path-matcher/` and `search-params/`. Not a package — it has no `package.json` and is bundled into core at build time; `@real-router/validation-plugin` reaches `validateRoute` through core's `@real-router/core/validation` subpath (segment lookup / existence via the matcher), never reaching in here directly (#1301, enforced by a plugin-level guard test).

> **Folded into core (engine-merge iteration 2).** This was `packages/engine` (a bare internal workspace package, never published to npm); iteration 2 of the engine-merge RFC absorbed it into `core/src/engine` (this directory) — the engine is the router's core, not a utility, so it lives in `src/`, not `src/utils/`. Its tests moved to `packages/core/tests/engine/`; its layer-boundary + white-box tiers ported into `packages/core/eslint.config.mjs`. (The standalone engine's RFC §5.5 facade-reachability ratchet was **not** kept — a non-gated meta-guard that became redundant once the engine is internal core code, tested like the rest of core; the 3 white-box tiers + the 100% coverage gate remain the guards.) Layer boundaries (path-matcher must not import search-params — the DI seam) and the 3 white-box test tiers (facade / path-matcher-layer / search-params-layer) are enforced by core's `eslint.config.mjs`. The gotchas below describe the route-tree facade layer; each folded-in layer keeps its own semantics (see git history of the pre-merge `path-matcher` / `search-params` packages, and of `packages/engine`).

The rest of this file documents the **route-tree facade layer** (the src root — `createRouteTree`, `createMatcher`, `validateRoute`, tree ops).

## Exports

| Export                                | Description                                                                                                                    |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `createRouteTree(name, path, routes)` | Builds immutable `RouteTree` from route definitions (always frozen — the `options?`/`skipFreeze` builder was removed in #1302) |
| `getSegmentsByName(tree, routeName)`  | O(1)-per-level lookup of route nodes by dot-notation name                                                                      |

| `DEFAULT_QUERY_PARAMS`                | Default query string options (re-exported from `search-params`)                                                                |
| `validateRoute(route, method, ...)`   | Batch route validation with cross-batch duplicate detection                                                                    |

## Gotchas

- **Immutable, always** -- `createRouteTree` always `Object.freeze`s the tree (no opt-out; the former `skipFreeze` `TreeBuildOptions` — and the standalone `createRouteTreeBuilder` — were removed in #1302 as core-unreachable API). `addRoute` rebuilds the whole tree via `createRouteTree`.
- **`paramMeta` is frozen too (#747)** -- `processNode` freezes the nested `paramMeta` object and its `urlParams`/`queryParams`/`spatParams` arrays, not just the node. (`paramMeta` carries no `constraintPatterns` Map — the path grammar has no regex constraints.)
- **Dot-notation is internal** -- User-provided route names must NOT contain dots; dots are only used in computed `fullName` (e.g., `"users.profile"`)
- **`createMatcher` hides DI** -- It wires `search-params` parse/build into `SegmentMatcher` so consumers never import `search-params` directly
- **`validateRoute` is recursive** -- Validates children arrays; detects duplicates both in the existing tree AND within the current batch
- **`route-batch.ts` carries a `getTypeDescription` twin, hardened in lockstep (#903/#1052)** -- route-tree has no `type-guards` dependency, so `getTypeDescription` is duplicated here for `validateRoute`'s error messages. The `constructor`/`.name` read is wrapped in `try/catch` in both copies, so an adversarial throwing accessor — a `constructor`/`.name` getter that throws, an **inherited** throwing getter reached by the custom-prototype check's `getTypeDescription(route)` (`:109`, _before_ the own-key getter scan, which uses `Object.keys` and misses inherited getters), or a Proxy `[[Get]]` — yields the clean `must be a plain object` `TypeError`, not the getter's exception. Keep the two copies in sync
- **`validateRoutePath` rejects an optional modifier (`:x?` / `*x?`)** -- `describeRemovedForm` (over the shared `parseSegment` tokenizer, on the query-stripped `pathPattern`) flags a `:x?` / `*x?` optional modifier and throws the route-contextual **optional-removed** recipe — it names the offending segment AND computes the two replacement sibling paths from the actual path (drop the segment → without-form; drop the trailing `?` → required form). Optionality is expressed with sibling routes, so a single check covers every optional shape — an optional splat (`*x?`), an optional before a splat, and two optionals before a splat all reduce to it. path-matcher backstops it at `registerTree` with a shorter, path-free recipe (`throwOptionalRemoved`)
- **`validateRoutePath` rejects a `<`/`>` in a path segment** -- `describeRemovedForm` flags any `<`/`>` in a path segment, in ANY position (after a param `:id<\d+>`, fused with a suffix `:year<\d+>-archive`, filling a static segment `/foo<bar>`, a stray delimiter `/a>b`, or in a param name `:id>`), and throws the **constraint-removed** recipe (`<`/`>` are reserved in path segments; match the segment as a plain string and validate the value in a guard). A single check covers every position — the tokenizer's name scan breaks on `<`/`>`, so a stray `>` in a param name is caught too. path-matcher backstops it at `registerTree` with a shorter recipe (`throwConstraintRemoved`)
- **`validateRoutePath` rejects fused mid-segment markers (#1050)** -- `findSegmentGrammarError` (the shared `parseSegment` tokenizer, #1324 Phase 3) flags a `:`/`*` after a static prefix within a segment (`/a:b`, `/users/x:id`, `/a*b`): build/meta extract it as a param while the trie compiles the segment as a literal, so `buildPath` emits a URL its own `match` rejects. Runs after the name-less check (#863) on the query-stripped `pathPattern`; a marker that _starts_ its segment — including a marker-led name containing `:`/`*` (`/:a:b` → param `a:b`) — is not flagged. A char-scan, not a strip. path-matcher backstops it at `registerTree` (the sibling of #858/#863)
- **`validateRoutePath` rejects a duplicate param name (#1151)** -- `validateUniqueParamNames` flags a name repeated within one route's own path (`/:id/:id`, a param+splat clash `/:x/*x`) — `buildParamMeta.urlParams` lists every path-binding name (params AND splats) in order, keeping duplicates, so a single pass catches both. The trie binds the duplicates at different positions under one name, so match's later capture overwrites the earlier and `rewritePathOnMatch` rewrites the user's URL. path-matcher's `registerTree` backstop additionally catches CROSS-level dups (a parent's param reused by a child), which this per-path gate cannot see
- **`validateRoutePath` rejects a non-ASCII static segment (#1154)** -- `hasNonAsciiStatic` flags a code point ≥ U+0080 in a STATIC segment (`/café`, `/меню`): match rejects non-ASCII input and compares static keys raw, so the route registers but never matches. A marker-aware char-scan — a non-ASCII PARAM name (`:café`) is skipped, so only static text is flagged. The percent-encoded form (`/caf%C3%A9`) already works. path-matcher backstops at `registerTree`
- **`validateRoutePath` rejects a malformed query-param declaration (#1242 §5.1/§5.3)** -- `validateQueryParamDeclarations` (over `buildParamMeta`'s `urlParams` + `queryParams`) flags a query-param name containing `<`/`>` (`/a?filter<x` — never round-trips) or one colliding with a path-param name (`/a/:tab?tab`, where `buildPath` emits the value twice). A `=` in a declaration (`?name=value`, §5.2) is deliberately NOT flagged — bare core tolerates it. (A reverse-order form like `/a/:b?<\d+>` is caught earlier as an **optional-removed** form — the `?` does not parse as the query start when a `<` tail follows.) path-matcher backstops at `registerTree`. The `#1242` §5.4 slash-child reject (an index under a SPLAT parent) is matcher-level only (a trie-terminal-shape concern this per-path gate can't see, like #1153)
- **Gate ↔ backstop parity is property-locked (#1320)** -- every reject check above has a path-matcher `registerTree` backstop twin. The per-segment name/marker grammar — name-less #858, fused-marker #1050, trailing-marker #1324, and the two rejection codes (**optional-removed**, **constraint-removed**) — is single-sourced in path-matcher's `parseSegment` tokenizer: the gate consumes it via `findSegmentGrammarError` / `describeRemovedForm`, the backstop via `extractParamName` + `registerNode`'s grammar pass (`throwSegmentGrammarError`), so those cannot drift by construction. The gate's removed-form recipe is richer (route name + computed sibling paths); the backstop's is a shorter path-free recipe — but the **verdict** (which segment, which error code) is the same tokenizer, so a `describeRemovedForm` reason is emitted only when the removed form is ALSO the first grammar error (it returns `undefined` on any earlier non-removed error, keeping the gate's reason in lockstep with the backstop's first-error verdict). For the checks the two layers still implement SEPARATELY (non-ASCII #1154, dup-param #1151), `tests/engine/property/gate-backstop-parity.properties.ts` proves they agree (both reject a malformed shape / both accept a valid one) across the generated input space, so a future edit that makes one layer stricter than the other fails the property. The #1324 closed divergence still holds: a marker fused to the END of a param (`/:y*`, `/:y:`, `/*y:`) routes through the same tokenizer on both layers, so both reject it. See INVARIANTS.md "Gate ↔ Backstop Parity"
- **Functional tests exercise the PUBLIC API only (white-box guardrail)** -- `tests/engine/functional/**/*.test.ts` must import from the `src/engine` barrel (`createRouteTree`, `createMatcher`, `getSegmentsByName`, `routeTreeToDefinitions`, `validateRoute`, + public types), NEVER a deeper relative `src/engine/*` path — enforced by a `no-restricted-imports` block in core's `eslint.config.mjs` (§5 facade tier; allowlist EMPTY). The whitebox audit proved `validateRoutePath` (the one truly-internal gate a test reached directly) is FULLY reachable through the public `validateRoute`, which delegates to it verbatim (same `TypeError` messages) — including the `~`-under-parameterized-parent branch, reached with a REAL `createRouteTree("p", "/:id", [])` param root (its root node carries `paramMeta.urlParams`), not the `__mocks__` builder. A test of a genuinely internal PURE function belongs in `tests/engine/property/` (exempt — the gate↔backstop parity property legitimately imports `validateRoutePath` from `src/engine`); a genuinely publicly-unreachable branch gets a documented KEEP-narrow `ignores:` entry in core's `eslint.config.mjs`. `__mocks__/route-tree-mock.ts` (hand-built `RouteTree`) survives only for `validation-route-batch.test.ts`'s duplicate-parent fixtures
- **Internal layers (not external deps)** -- the `path-matcher/` (segment trie) and `search-params/` (query-string serialization) layers are co-located subfolders wired via the DI seam (`createMatcher` bakes in the search-params parse/build), NOT workspace dependencies — the whole engine is one zero-external-dependency subsystem of core
