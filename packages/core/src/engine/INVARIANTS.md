# Invariants

> Property-based invariants verified via [fast-check](https://fast-check.dev/). Implementations moved to `packages/core/tests/engine/property/` when the engine folded into `core/src/engine` (engine-merge iteration 2).

## Full Roundtrip

| #   | Invariant                    | Description                                                                                                                                                                                                              |
| --- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | URL param roundtrip          | `match(buildPath("users.profile", {id}))` restores both the route name and the original `id` value. This is the core correctness property: if navigation builds a path and can't match it back, the router is broken.    |
| 2   | Query-only roundtrip         | For a route with only query params, `buildPath` then `match` preserves all query param values. Verifies the `search-params` integration via the `parseQueryString`/`buildQueryString` dependency injection.              |
| 3   | Array query params roundtrip | For all 4 array formats (`none`, `brackets`, `index`, `comma`), array values roundtrip correctly through the integration layer. Each format has distinct encoding behavior that must compose correctly with the matcher. |
| 4   | URL/query param isolation    | After matching a path on a route with both URL params and query params, each param appears in exactly one source. URL params are typed `"url"` and query params are typed `"query"` in the result meta.                  |

## createRouteTree Normalization

| #   | Invariant                             | Description                                                                                                                                                                                                                                                                         |
| --- | ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Tree build idempotency                | Over randomly-generated nested trees (`arbRouteForest`), `routeTreeToDefinitions(createRouteTree(routes))` reproduces the input **exactly** (names, paths incl. `~`/params/query/splat, nesting, child order) and a second roundtrip is a fixed point. Essential for `replace()` in the core router.                                                                                         |
| 2   | Route preservation                    | Over randomly-generated nested trees, `createRouteTree` drops nothing at **any** level: `tree.children.size === routes.length` AND the total node count equals the total definition count. No routes are silently lost during construction.                                                                                                                                      |
| 3   | Absolute path normalization           | After tree building, no node's `path` property anywhere in the tree starts with `~` (verified over `arbRouteForest`). The tilde prefix is consumed during normalization and replaced by the `absolute: true` flag.                                                                                                                        |
| 4   | Absolute path roundtrip               | For randomized trees with absolute path children, `~` is stripped from stored paths AND `routeTreeToDefinitions` → `createRouteTree` roundtrip preserves the `absolute` flag and path for every node.                                                                               |

## Computed Caches

| #   | Invariant                       | Description                                                                                                                                                                                                                                                                                                                            |
| --- | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Tree immutability               | Every node in the tree — plus its `children` Map, `paramTypeMap`, `nonAbsoluteChildren` array, nested `paramMeta` object, and its `urlParams`/`queryParams`/`spatParams` arrays — passes `Object.isFrozen` (#747). Verified over randomly-generated trees (`arbRouteForest`) spanning plain/URL/query/splat params and absolute flags. **Exception (Map shell-freeze):** both `paramMeta.constraintPatterns` and the `children` Map are `Map`s, and `Object.freeze` locks only the Map **shell** (no reassignment, no new own-properties) — NOT its entries: `.set`/`.delete`/`.clear` still pass. So `Object.isFrozen(children) === true` does **not** imply entry-immutability the way the truly-frozen `nonAbsoluteChildren` array (whose `.push` throws) does; entry-immutability rests solely on the `ReadonlyMap` type. `constraintPatterns` is deliberately left unwrapped, whereas `children` **is** wrapped (a cheap shell-freeze that kills mutant 44 per the mutation-audit of 2026-06-24 — keep the wrap) — but the same caveat applies to both. Corollary (#1009): paramless (leaf) nodes reuse the shared frozen `EMPTY_CHILDREN_MAP` sentinel across every leaf of every tree in the process, so a cast or plain-JS consumer calling `.set` on one leaf's `children` contaminates the sentinel and the mutation surfaces in an unrelated tree; a correctly-typed `ReadonlyMap` consumer reaching the tree via `getTree()` cannot trigger this without a cast — the same accepted posture as `constraintPatterns`.                                                               |
| 2   | Non-absolute children filtering | For every node (over `arbRouteForest`), `nonAbsoluteChildren` equals the children from the `children` Map where `absolute === false` **as an ordered sequence** (definition order) — verified by ordered equality, not bare membership, so a reordering is caught. Absolute children appear in `children` but not in `nonAbsoluteChildren`.                                                                                                        |
| 3   | paramTypeMap classification     | For every node, each entry in `paramTypeMap` correctly classifies the parameter source: URL path parameters (`:param`) **and splat params** (`*param`) are typed `"url"` (splat also appears in `paramMeta.spatParams`), and query string parameters (`?param`) are typed `"query"`. No extra entries exist beyond the declared URL/splat/query params. Validated by building trees from randomized URL+query+splat param combinations. |
| 4   | fullName correctness            | For every node (over `arbRouteForest`), `fullName` equals the dot-joined chain of `name` fields from the root to that node, independently reconstructed by walking the `parent` chain. Validates `computeFullName` in `computeCaches` over arbitrary structures. |

## getSegmentsByName

> Generalized from the old fixed DEEP_TREE fixture to random nested trees (`arbRouteForest`). The oracle is the tree structure itself: the chain returned by `getSegmentsByName` (a Map-walk down from the root) must equal an independent walk up the `parent` backrefs — a model-based cross-check run over every node of thousands of shapes.

| #   | Invariant       | Description |
| --- | --------------- | ----------- |
| 1   | Chain fidelity  | For every node, `getSegmentsByName(tree, node.fullName)` returns the exact root→node chain: the segment list equals the independent `parent`-walk chain (root excluded) and its last element **is** that node. A single model-based cross-check that subsumes correctness (right node), the prefix property (each `fullName` + `.` prefixes the next), and the full-name chain. |
| 2   | Length          | The number of segments returned equals the number of dot-separated parts in `node.fullName`. No intermediate segments are missing from the chain. |
| 3   | Null on unknown | `getSegmentsByName(tree, name)` returns `null` for names not in the tree — both a real name extended with a non-existent child segment (partial-walk-then-miss) and a wholly-unknown top-level name (first-lookup miss). |

| --- | --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Extraction            | Query params declared in a route path (`?param1&param2`) are correctly extracted into `paramMeta.queryParams`. All declared param names must be present and in order.                           |

## nodeToDefinition

| #   | Invariant                         | Description                                                                                                                                                                                                                                                                         |
| --- | --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | nodeToDefinition absolute restoration | `nodeToDefinition(node).path` starts with `~` if and only if `node.absolute` is `true`. For absolute nodes, removing the `~` prefix yields `node.path`. For non-absolute nodes, the output path equals `node.path` unchanged. Verified over every node of randomly-generated trees (`arbRouteForest`). |

## Route Name Validation

| #   | Invariant                 | Description                                                                                                                                                                                                       |
| --- | ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Valid name acceptance     | Names matching the pattern `[a-zA-Z_][a-zA-Z0-9_-]*` pass `validateRoute` without throwing. The ROUTE_NAME_PATTERN regex accepts the full valid character set.                                                    |
| 2   | Invalid name rejection    | Names that are empty, whitespace-only, contain dots, start with a digit, or start with a hyphen cause `validateRoute` to throw `TypeError`. Covers all string-format rejection branches in `validateRouteName`.   |
| 3   | System route bypass       | Names prefixed with `@@` bypass the pattern check entirely — they can contain dots, slashes, and other characters that would normally be rejected. Used for internal system routes like `@@router/UNKNOWN_ROUTE`. |
| 4   | Non-string name rejection | Non-string values (`number`, `null`, `undefined`, `boolean`) cause `validateRoute` to throw `TypeError` at the `typeof` check, before any pattern validation runs.                                                |

## Route Path Validation

| #   | Invariant                           | Description                                                                                                                                                                                                 |
| --- | ----------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Valid path acceptance               | Empty strings, absolute paths (`/path`), tilde paths (`~path`), query paths (`?param`), and relative segments all pass `validateRoutePath` without throwing.                                                |
| 2   | Non-string path rejection           | Non-string values (`number`, `null`, `undefined`, `boolean`, `array`) cause `validateRoutePath` to throw `TypeError` at the type check.                                                                     |
| 3   | Whitespace rejection                | Paths containing whitespace characters (space, tab, newline) cause `validateRoutePath` to throw `TypeError`. Whitespace is never valid in route paths.                                                      |
| 4   | Double-slash rejection              | Paths containing consecutive slashes (`//`) cause `validateRoutePath` to throw `TypeError`. Double slashes indicate a path construction error.                                                              |
| 5   | Absolute under parameterized parent | Tilde-prefixed paths (`~path`) under a parent node that has URL parameters in its `paramTypeMap` cause `validateRoutePath` to throw `TypeError`. Absolute paths cannot be used under parameterized parents. |
| 6   | Unbalanced constraint rejection     | Paths with a stray constraint delimiter — a `<` with no closing `>` (`/:id<\d+`, `/:id<`) or a `>` with no opening `<` — cause `validateRoutePath` to throw `TypeError`. A linear scan tracks open/close (a `<` opens a constraint, the first following `>` closes it, a `<` inside the body is allowed), so a `>` outside a constraint or a `<` left unclosed is unbalanced and would desync match vs build downstream (`buildPath` throws `Missing required param`). Balanced constraints (`/:id<\d+>`) and a `<` inside a constraint body (`/:id<[a<b]>`) are accepted (#749). |
| 7   | Name-less param marker rejection    | A `:` or `*` marker with no parameter name — `/x/:`, `/x/*`, `/x/:?` (only a modifier), `/x/:<\d+>` (only a constraint), `/x/:/n` (boundary) — causes `validateRoutePath` to throw `TypeError`. The check derives from path-matcher's single `PARAM_NAME_PATTERN` grammar (`[:*]` then a negative lookahead for the name class) and scans only the URL-path portion (`buildParamMeta(path).pathPattern`), so a `:`/`*` inside a query declaration (`/x?:`) is **not** flagged. path-matcher rejects name-less markers at `registerTree` (#858); this gate rejects them earlier with a route-contextual error (#863). A bare `/*` is **not** a catch-all — the catch-all is the named `/*rest`. |
| 8   | Fused mid-segment marker rejection  | A `:`/`*` marker fused to a static prefix **within** a segment (`/a:b`, `/users/x:id`, `/a*b`) causes `validateRoutePath` to throw `TypeError`. build/`buildParamMeta` extract the marker as a param (their marker regex is unanchored) while the trie honors a marker only at segment start and compiles the segment as a static literal — so `buildPath` would emit a URL its own `match` rejects. This route-tree gate rejects it early with a route-contextual error; path-matcher backstops at `registerTree`. The sibling of the name-less rejection (row 7) on the marker-anchoring axis (#1050). |
| 9   | Fused constraint suffix rejection   | Static text (or a second marker) fused to a constraint's closing `>` **within** a segment (`/:year<\d+>-archive`, `/:id<\d+>.html`) causes `validateRoutePath` to throw. build strips `<…>` then re-extracts the param name greedily, fusing the suffix (name `year-archive`), while meta ends the name at `<` (name `year`) — build name ≠ meta name, so the route is a silent dead route (`buildPath` throws `Missing required param`, `match` keys the constraint on a phantom name). A linear scan (run after the balance check, so every `>` is a constraint closer) flags a `>` not followed by `/`, `?`, or end-of-input. The mirror of the fused mid-segment marker (row 8) on the OTHER side of the param; path-matcher backstops at `registerTree` (#1150). |
| 10  | Optional splat rejection            | An optional splat `*name?` (`/files/*path?`) causes `validateRoutePath` to throw. build treats it as a multi-segment splat while the trie's optional fork compiles a single-segment plain param, so `buildPath` emits a URL its own `match` rejects. Scanned on the query-stripped `pathPattern`, so a required splat followed by a query (`*path?download`) is **not** flagged. Product decision: reject (the shape only ever matched 0–1 segments). Sibling of the fused rejections; path-matcher backstops at `registerTree` (#1149). |
| 11  | Unconstrained optional-before-splat rejection | An **unconstrained** optional param immediately before a splat (`/:v?/*rest`) causes `validateRoutePath` to throw. Without a constraint there is no validity signal to disambiguate "take the optional" from "let the splat capture", so every multi-segment value has two readings and `match` would silently reshape half the input space. A char-scan flags an optional `?` whose preceding char is NOT `>` (unconstrained — a constrained optional ends `<…>?`) and which is immediately followed by `/*`. A **constrained** optional→splat (`/:v<c>?/*rest`) IS supported (path-matcher's try-take-if-valid); an opt→param (`/:a?/:b`) or opt→static is unaffected. path-matcher backstops at `registerTree` (#1264). |
| 12  | Duplicate param name rejection      | A param name repeated within one route's own path (`/:id/:id`, a param+splat clash `/:x/*x`) causes `validateRoutePath` to throw. `buildParamMeta.urlParams` lists every path-binding name (params AND splats) in order and keeps duplicates, so a single pass catches both. The trie would bind the duplicates at different positions under ONE name, so `match`'s later capture silently overwrites the earlier and `rewritePathOnMatch` rewrites the user's URL (`/1/2` → `/2/2`). The #736 conflict guard only fires on *differently*-named params; this is the same-name axis. path-matcher's `registerTree` backstop additionally catches CROSS-level dups (a parent's param reused by a child), which this per-path gate cannot see (#1151). |
| 13  | Non-ASCII static segment rejection  | A raw non-ASCII code point (≥ U+0080) in a **static** segment (`/café`, `/меню`) causes `validateRoutePath` to throw. `match` rejects non-ASCII input and compares static trie keys raw (never percent-decoded), so the route would register but never match (`buildPath` emits `/café`, which `match` rejects). A marker- and constraint-aware char-scan: a non-ASCII param **name** (`:café`) or constraint body (`:id<[а-я]+>`, matched against the *decoded* value) is skipped — only static text is flagged. The percent-encoded form (`/caf%C3%A9`) works. path-matcher backstops at `registerTree` (#1154). |
| 14  | Query-param declaration validation  | A query-param name containing `<`/`>` (a constraint leaked in via a reverse-order modifier typo — `/a/:b?<\d+>` parses the `?` as the query start, so `<\d+>` becomes the query name) or one that collides with a path-param name (`/a/:tab?tab`, where `buildPath` emits the value twice, `/a/x?tab=x`) causes `validateRoutePath` to throw. `validateQueryParamDeclarations` scans `buildParamMeta`'s `urlParams` + `queryParams`. A `=` in a declaration (`?name=value`) is deliberately NOT flagged — bare core tolerates it (#1242 §5.2). path-matcher backstops at `registerTree` (#1242 §5.1/§5.3). |
| 15  | Constraint in a static segment rejection | A `<...>` constraint filling a **static** segment (no `:`/`*` marker) — `/foo<bar>`, `/a<b>` — causes `validateRoutePath` to throw. path-matcher's marker-agnostic `CONSTRAINT_PATTERN_RGX` silently strips it (`/foo<bar>` → `/foo`), reshaping the route with no signal. the gate's `findSegmentGrammarError` reports `constraint-in-static` (via the shared, constraint-aware `parseSegment` tokenizer) after the fused-suffix check — row 9 (#1150) catches only a constraint fused with TRAILING text; one cleanly ENDING a static segment slips through (the residual of #1242 §5.5). A constraint on a param (`/:id<\d+>`) is unaffected. path-matcher backstops at `registerTree` (#1311). |
| 16  | Two-optionals-before-splat rejection | ≥2 optional params directly before a splat (`/:a<c>?/:b<c>?/*rest`) causes `validateRoutePath` to throw: a single trie slot carries one optional→splat fork, so the outer optional's mark overwrites the inner's and the omit-outer/take-inner form silently reshapes into the splat. `hasMultipleOptionalsBeforeSplat` is SINGLE-SOURCED in path-matcher (`parseSegment.ts`, RAW path via `splitPathSegments`) and imported by both the gate and the `registerTree` backstop, so — like `isConstraintBalanced` — they can't drift. The #1264 check (row 11) runs first, so this fires for the two-optional (constrained) shape it doesn't catch (#1287). |

## Route Duplicate Detection

> `validateRoute`'s cross-route guarantees (the reason `seenNames` / `seenPathsByParent` exist) — distinct from the single-route format checks above.

| #   | Invariant                          | Description                                                                                                                                                                                                                                  |
| --- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | Batch name duplicate               | Validating a batch where any route name is repeated (even buried mid-list) throws `[…] Duplicate route "<name>" in batch`. Names are unique within a batch.                                                                                  |
| 2   | Batch path duplicate (same parent) | Two routes that share a `path` under the **same** parent level throw `[…] Path "<path>" is already defined`, even when their names differ.                                                                                                   |
| 3   | Path uniqueness is parent-scoped   | The same `path` under **different** parents is **allowed** (no false positive). Duplicate-path detection is keyed by parent, not global. Mutating the keying to ignore the parent breaks this invariant — it is the parent-scoping guard.    |
| 4   | Tree name duplicate                | Validating a route whose name already exists in the supplied existing tree throws `[…] Route "<name>" already exists` (the tree-relative check, used by `addRoute`).                                                                          |
| 5   | All-distinct acceptance            | A batch with all-unique names and all-unique paths passes validation without throwing — the duplicate checks produce no false positives on legitimate route sets.                                                                            |

## Gate ↔ Backstop Parity

> The route-tree `validateRoutePath` **gate** (a whole-path pre-pass) and the path-matcher `registerTree` **backstop** (per-segment, during trie insertion) independently reject the same malformed single-path shapes. Tier 1 of #1320 single-sourced the whole-path scans the two layers could literally share (`isConstraintBalanced`; `hasMultipleOptionalsBeforeSplat` was added by #1287); #1324/#1332 went further, subsuming the per-segment grammar — including the former `hasConstraintInStaticSegment` / `hasFusedConstraintSuffix` char-scans — into the shared `parseSegment` tokenizer, so those cannot drift by construction. The scans still kept **separate** below (non-ASCII #1154, unconstrained-opt→splat #1264, dup-param #1151) merge poorly with the perf-sensitive (#1285) / trie-interleaved backstop. This parity contract locks them against drift — proven across the generated input space, not merged in code. If a future edit makes one layer stricter than the other, a generated shape lands in the gap and the property fails.

| #    | Invariant                                     | Description                                                                                                                                                              |
| ---- | --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| GBP1 | Fused mid-segment marker parity               | A marker fused to a static prefix (`/a:b`, `/foo*bar`) is rejected by BOTH the gate (VP8) and the backstop (`parseSegment`'s `fused-marker` code — single-sourced since #1324/#1332, was `FUSED_MARKER_RGX`); a clean marker-led segment (`/:id`, `/*rest`) is accepted by both (#1050).       |
| GBP2 | Non-ASCII static parity                       | A non-ASCII code point in a static segment (`/café`) is rejected by both (VP13 ↔ `hasNonAsciiSegment`); a pure-ASCII static segment by neither (#1154).                                                            |
| GBP3 | Optional splat parity                         | An optional splat (`/*path?`) is rejected by both (VP10 ↔ backstop optional fork); a required splat (`/*rest`) by neither (#1149).                                                                                 |
| GBP4 | Unconstrained optional-before-splat parity    | An unconstrained optional directly before a splat (`/:v?/*rest`) is rejected by both (VP11 ↔ backstop); a required param before a splat (`/:id/*rest`) by neither (#1264).                                          |
| GBP5 | Duplicate param name parity                   | A same-name param repeated in one path (`/:id/:id`) is rejected by both (VP12 ↔ backstop cross-position dup); two distinct names (`/:a/:b`) by neither (#1151).                                                     |
| GBP6 | Trailing parameter marker parity              | A marker fused to the END of a param/splat name (`/:y*`, `/:y:`, `/*y:`) is rejected by BOTH the gate (`findSegmentGrammarError` → trailing-marker) and the backstop (`extractParamName` → `parseSegment`); a clean marker-led name (`/:id`, `/*rest`) by neither (#1324).                                          |
| GBP7 | Two-optionals-before-splat parity             | Two optionals directly before a splat (`/:a<c>?/:b<c>?/*rest`) is rejected by both via the SHARED `hasMultipleOptionalsBeforeSplat` predicate (VP16 ↔ the SAME function in both layers, single-sourced like `isConstraintBalanced`); a single constrained optional→splat (`/:v<c>?/*rest`) by neither (#1287).                                                                          |

> **Excluded — documented single-layer behaviour, NOT parity bugs:** gate-only string-format checks (`//` double-slash VP4, whitespace VP3, non-string VP2) — the backstop consumes an already-segmented path. (The trailing-marker `/:y*` case, formerly excluded, is CLOSED as of #1324 — the backstop now shares the gate's `parseSegment` name boundary via `extractParamName` — and is **GBP6** above.)
