# Invariants

> Property-based invariants verified via [fast-check](https://fast-check.dev/). Implementations moved to `packages/core/tests/engine/property/` when the engine folded into `core/src/engine` (engine-merge iteration 2).

## Full Roundtrip

| #   | Invariant                    | Description                                                                                                                                                                                                              |
| --- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | URL param roundtrip          | `match(buildPath("users.profile", {id}))` restores both the route name and the original `id` value. This is the core correctness property: if navigation builds a path and can't match it back, the router is broken.    |
| 2   | Query-only roundtrip         | For a route with only query params, `buildPath` then `match` preserves all query param values. Verifies the `search-params` integration via the `parseQueryString`/`buildQueryString` dependency injection.              |
| 3   | Array query params roundtrip | For all 4 array formats (`none`, `brackets`, `index`, `comma`), array values roundtrip correctly through the integration layer. Each format has distinct encoding behavior that must compose correctly with the matcher. |
| 4   | URL/query param isolation    | After matching a path on a route with both URL params and query params, each param appears in exactly one source. URL params are typed `"url"` and query params are typed `"query"` in the result meta.                  |

## createRouteTree Normalization

| #   | Invariant                   | Description                                                                                                                                                                                                                                                                                          |
| --- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Tree build idempotency      | Over randomly-generated nested trees (`arbRouteForest`), `routeTreeToDefinitions(createRouteTree(routes))` reproduces the input **exactly** (names, paths incl. `~`/params/query/splat, nesting, child order) and a second roundtrip is a fixed point. Essential for `replace()` in the core router. |
| 2   | Route preservation          | Over randomly-generated nested trees, `createRouteTree` drops nothing at **any** level: `tree.children.size === routes.length` AND the total node count equals the total definition count. No routes are silently lost during construction.                                                          |
| 3   | Absolute path normalization | After tree building, no node's `path` property anywhere in the tree starts with `~` (verified over `arbRouteForest`). The tilde prefix is consumed during normalization and replaced by the `absolute: true` flag.                                                                                   |
| 4   | Absolute path roundtrip     | For randomized trees with absolute path children, `~` is stripped from stored paths AND `routeTreeToDefinitions` → `createRouteTree` roundtrip preserves the `absolute` flag and path for every node.                                                                                                |
| 5   | Leading-slash normalization | `createNode` prepends a missing leading `/` after the `~`-strip (`#1407`), so the trie never sees a bare relative segment. All four spellings of an absolute single segment — relative `s`, `/s`, `~s`, `~/s` — collapse to the stored path `/s` and round-trip through the matcher (`buildPath` then `match`). A query-only path (`?q`) is exempt (no path segment to slash). `arbRouteForest` only generates leading-`/`/`~/` forms, so invariant #1 (exact idempotency) is unaffected — this covers the otherwise-untested slash-less space (`tree.properties.ts` describe 5). |

## Computed Caches

| #   | Invariant                       | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --- | ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Tree immutability               | Every node in the tree — plus its `children` Map, `paramTypeMap`, `nonAbsoluteChildren` array, nested `paramMeta` object, and its `urlParams`/`queryParams`/`spatParams` arrays — passes `Object.isFrozen` (#747). Verified over randomly-generated trees (`arbRouteForest`) spanning plain/URL/query/splat params and absolute flags. **Exception (Map shell-freeze):** the `children` Map is a `Map`, and `Object.freeze` locks only the Map **shell** (no reassignment, no new own-properties) — NOT its entries: `.set`/`.delete`/`.clear` still pass. So `Object.isFrozen(children) === true` does **not** imply entry-immutability the way the truly-frozen `nonAbsoluteChildren` array (whose `.push` throws) does; entry-immutability rests solely on the `ReadonlyMap` type. `children` **is** wrapped (a cheap shell-freeze that kills mutant 44 per the mutation-audit of 2026-06-24 — keep the wrap). Corollary (#1009): paramless (leaf) nodes reuse the shared frozen `EMPTY_CHILDREN_MAP` sentinel across every leaf of every tree in the process, so a cast or plain-JS consumer calling `.set` on one leaf's `children` contaminates the sentinel and the mutation surfaces in an unrelated tree; a correctly-typed `ReadonlyMap` consumer reaching the tree via `getTree()` cannot trigger this without a cast. (`paramMeta` carries no `constraintPatterns` Map — the path grammar has no regex constraints.) |
| 2   | Non-absolute children filtering | For every node (over `arbRouteForest`), `nonAbsoluteChildren` equals the children from the `children` Map where `absolute === false` **as an ordered sequence** (definition order) — verified by ordered equality, not bare membership, so a reordering is caught. Absolute children appear in `children` but not in `nonAbsoluteChildren`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| 3   | paramTypeMap classification     | For every node, each entry in `paramTypeMap` correctly classifies the parameter source: URL path parameters (`:param`) **and splat params** (`*param`) are typed `"url"` (splat also appears in `paramMeta.spatParams`), and query string parameters (`?param`) are typed `"query"`. No extra entries exist beyond the declared URL/splat/query params. Validated by building trees from randomized URL+query+splat param combinations.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| 4   | fullName correctness            | For every node (over `arbRouteForest`), `fullName` equals the dot-joined chain of `name` fields from the root to that node, independently reconstructed by walking the `parent` chain. Validates `computeFullName` in `computeCaches` over arbitrary structures.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |

## getSegmentsByName

> Generalized from the old fixed DEEP_TREE fixture to random nested trees (`arbRouteForest`). The oracle is the tree structure itself: the chain returned by `getSegmentsByName` (a Map-walk down from the root) must equal an independent walk up the `parent` backrefs — a model-based cross-check run over every node of thousands of shapes.

| #   | Invariant       | Description                                                                                                                                                                                                                                                                                                                                                                     |
| --- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Chain fidelity  | For every node, `getSegmentsByName(tree, node.fullName)` returns the exact root→node chain: the segment list equals the independent `parent`-walk chain (root excluded) and its last element **is** that node. A single model-based cross-check that subsumes correctness (right node), the prefix property (each `fullName` + `.` prefixes the next), and the full-name chain. |
| 2   | Length          | The number of segments returned equals the number of dot-separated parts in `node.fullName`. No intermediate segments are missing from the chain.                                                                                                                                                                                                                               |
| 3   | Null on unknown | `getSegmentsByName(tree, name)` returns `null` for names not in the tree — both a real name extended with a non-existent child segment (partial-walk-then-miss) and a wholly-unknown top-level name (first-lookup miss).                                                                                                                                                        |

## Route Name Validation

| #   | Invariant                 | Description                                                                                                                                                                                                       |
| --- | ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Valid name acceptance     | Names matching the pattern `[a-zA-Z_][a-zA-Z0-9_-]*` pass `validateRoute` without throwing. The ROUTE_NAME_PATTERN regex accepts the full valid character set.                                                    |
| 2   | Invalid name rejection    | Names that are empty, whitespace-only, contain dots, start with a digit, or start with a hyphen cause `validateRoute` to throw `TypeError`. Covers all string-format rejection branches in `validateRouteName`.   |
| 3   | System route bypass       | Names prefixed with `@@` bypass the pattern check entirely — they can contain dots, slashes, and other characters that would normally be rejected. Used for internal system routes like `@@router/UNKNOWN_ROUTE`. |
| 4   | Non-string name rejection | Non-string values (`number`, `null`, `undefined`, `boolean`) cause `validateRoute` to throw `TypeError` at the `typeof` check, before any pattern validation runs.                                                |

## Route Path Validation

| #   | Invariant                                 | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| --- | ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Valid path acceptance                     | Empty strings, absolute paths (`/path`), tilde paths (`~path`), query paths (`?param`), and relative segments all pass `validateRoutePath` without throwing.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| 2   | Non-string path rejection                 | Non-string values (`number`, `null`, `undefined`, `boolean`, `array`) cause `validateRoutePath` to throw `TypeError` at the type check.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| 3   | Whitespace rejection                      | Paths containing whitespace characters (space, tab, newline) cause `validateRoutePath` to throw `TypeError`. Whitespace is never valid in route paths.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| 4   | Double-slash rejection                    | Paths containing consecutive slashes (`//`) cause `validateRoutePath` to throw `TypeError`. Double slashes indicate a path construction error.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| 5   | Absolute under parameterized parent       | Tilde-prefixed paths (`~path`) under a parent node that has URL parameters in its `paramTypeMap` cause `validateRoutePath` to throw `TypeError`. Absolute paths cannot be used under parameterized parents.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| 6   | Regex constraint rejected — stray `<`/`>` | A `<` (`/:id<\d+>`, `/:id<`) or a `>` (`/a>b`) anywhere in a segment causes `validateRoutePath` to throw the **constraint-removed** recipe (`<`/`>` are reserved in path segments; validate the value in a guard). `<`/`>` have no meaning in the grammar — a `<` with no `>` is not "unbalanced", it is simply reserved. Single-sourced via `parseSegment`; path-matcher backstops at `registerTree`.                                                                                                                                                                                                                       |
| 7   | Name-less param marker rejection          | A `:` or `*` marker with no parameter name — `/x/:`, `/x/*`, `/x/:?` (only a modifier), `/x/:/n` (boundary) — causes `validateRoutePath` to throw `TypeError`. The check derives from path-matcher's single `parseSegment` grammar (marker then an empty name) and scans only the URL-path portion, so a `:`/`*` inside a query declaration (`/x?:`) is **not** flagged. path-matcher rejects name-less markers at `registerTree` (#858); this gate rejects them earlier with a route-contextual error (#863). The name scan stops at the first `<`/`>` too, so `/x/:<…>` is a name-less marker (empty name) checked before the constraint-removed recipe. A bare `/*` is **not** a catch-all — the catch-all is the named `/*rest`.                |
| 8   | Fused mid-segment marker rejection        | A `:`/`*` marker fused to a static prefix **within** a segment (`/a:b`, `/users/x:id`, `/a*b`) causes `validateRoutePath` to throw `TypeError`. build/`buildParamMeta` extract the marker as a param (their marker regex is unanchored) while the trie honors a marker only at segment start and compiles the segment as a static literal — so `buildPath` would emit a URL its own `match` rejects. This route-tree gate rejects it early with a route-contextual error; path-matcher backstops at `registerTree`. The sibling of the name-less rejection (row 7) on the marker-anchoring axis (#1050).                                                                                                                                            |
| 9   | Regex constraint rejected — fused suffix  | A `<`/`>` carrying a fused suffix (`/:year<\d+>-archive`, `/:id<\d+>.html`) causes `validateRoutePath` to throw the **constraint-removed** recipe. The `<`/`>` are reserved, so the whole segment is rejected regardless of any suffix. path-matcher backstops at `registerTree`.                                                                                                                                                                                                                                                                                                                            |
| 10  | Optional modifier rejected — splat        | An optional splat `*name?` (`/files/*path?`) causes `validateRoutePath` to throw the **optional-removed** recipe (declare two sibling routes). A trailing `?` on a `:param`/`*splat` is not part of the grammar. Scanned on the query-stripped `pathPattern`, so a required splat followed by a query (`*path?download`) is **not** flagged — the `?` there is the query separator, not an optional marker. path-matcher backstops at `registerTree`.                                                                                                                                                                                                                                                   |
| 11  | Optional modifier rejected — before splat | An optional param before a splat (`/:v?/*rest`) causes `validateRoutePath` to throw the **optional-removed** recipe. The `:v?` optional is not part of the grammar, so there is no take-vs-splat ambiguity to disambiguate. path-matcher backstops at `registerTree`.                                                                                                                                                                                                                                                                                                                                                                                          |
| 12  | Duplicate param name rejection            | A param name repeated within one route's own path (`/:id/:id`, a param+splat clash `/:x/*x`) causes `validateRoutePath` to throw. `buildParamMeta.urlParams` lists every path-binding name (params AND splats) in order and keeps duplicates, so a single pass catches both. The trie would bind the duplicates at different positions under ONE name, so `match`'s later capture silently overwrites the earlier and `rewritePathOnMatch` rewrites the user's URL (`/1/2` → `/2/2`). The #736 conflict guard only fires on _differently_-named params; this is the same-name axis. path-matcher's `registerTree` backstop additionally catches CROSS-level dups (a parent's param reused by a child), which this per-path gate cannot see (#1151). |
| 13  | Non-ASCII static segment rejection        | A raw non-ASCII code point (≥ U+0080) in a **static** segment (`/café`, `/меню`) causes `validateRoutePath` to throw. `match` rejects non-ASCII input and compares static trie keys raw (never percent-decoded), so the route would register but never match (`buildPath` emits `/café`, which `match` rejects). A marker-aware char-scan: a non-ASCII param **name** (`:café`) is skipped — only static text is flagged. The percent-encoded form (`/caf%C3%A9`) works. path-matcher backstops at `registerTree` (#1154).                                                                                                                                                                                                                          |
| 14  | Query-param declaration validation        | A query-param name containing `<`/`>` (`/a?filter<x` — never round-trips) or one that collides with a path-param name (`/a/:tab?tab`, where `buildPath` emits the value twice, `/a/x?tab=x`) causes `validateRoutePath` to throw. `validateQueryParamDeclarations` scans `buildParamMeta`'s `urlParams` + `queryParams`. A `=` in a declaration (`?name=value`) is deliberately NOT flagged — bare core tolerates it (#1242 §5.2). A reverse-order form like `/a/:b?<\d+>` is caught earlier as an **optional-removed** form (the `?` does not parse as the query start when a `<` tail follows). path-matcher backstops at `registerTree` (#1242 §5.1/§5.3).                                                                  |
| 15  | Regex constraint rejected — static segment | A `<...>` filling a **static** segment (`/foo<bar>`, `/a<b>`) causes `validateRoutePath` to throw the **constraint-removed** recipe. `<`/`>` are reserved anywhere in a segment, so a `<...>` that cleanly ends a static segment is rejected like any other. path-matcher backstops at `registerTree`.                                                                                                                                                                                                                                                                                                                                             |
| 16  | Optional modifier rejected — two before splat | Two (or more) optional params before a splat (`/:a?/:b?/*rest`) causes `validateRoutePath` to throw the **optional-removed** recipe on the first optional. Each `?` optional is not part of the grammar. path-matcher backstops at `registerTree`.                                                                                                                                                                                                                                                                                                                                                           |

## Route Duplicate Detection

> `validateRoute`'s cross-route guarantees (the reason `seenNames` / `seenPathsByParent` exist) — distinct from the single-route format checks above.

| #   | Invariant                          | Description                                                                                                                                                                                                                               |
| --- | ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Batch name duplicate               | Validating a batch where any route name is repeated (even buried mid-list) throws `[…] Duplicate route "<name>" in batch`. Names are unique within a batch.                                                                               |
| 2   | Batch path duplicate (same parent) | Two routes that share a `path` under the **same** parent level throw `[…] Path "<path>" is already defined`, even when their names differ.                                                                                                |
| 3   | Path uniqueness is parent-scoped   | The same `path` under **different** parents is **allowed** (no false positive). Duplicate-path detection is keyed by parent, not global. Mutating the keying to ignore the parent breaks this invariant — it is the parent-scoping guard. |
| 4   | Tree name duplicate                | Validating a route whose name already exists in the supplied existing tree throws `[…] Route "<name>" already exists` (the tree-relative check, used by `addRoute`).                                                                      |
| 5   | All-distinct acceptance            | A batch with all-unique names and all-unique paths passes validation without throwing — the duplicate checks produce no false positives on legitimate route sets.                                                                         |

## Gate ↔ Backstop Parity

> The route-tree `validateRoutePath` **gate** (a whole-path pre-pass) and the path-matcher `registerTree` **backstop** (per-segment, during trie insertion) independently reject the same malformed single-path shapes. The per-segment grammar — name/marker forms AND the two rejection codes (**optional-removed**, **constraint-removed**) — is subsumed into the shared `parseSegment` tokenizer, so those cannot drift by construction: the gate reads it via `findSegmentGrammarError` / `describeRemovedForm`, the backstop via `extractParamName` + `registerNode`'s grammar pass. The checks still kept **separate** below (non-ASCII #1154, dup-param #1151) merge poorly with the perf-sensitive (#1285) / trie-interleaved backstop. This parity contract locks them against drift — proven across the generated input space, not merged in code. If a future edit makes one layer stricter than the other, a generated shape lands in the gap and the property fails.

| #    | Invariant                            | Description                                                                                                                                                                                                                                                                                                               |
| ---- | ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| GBP1 | Fused mid-segment marker parity      | A marker fused to a static prefix (`/a:b`, `/foo*bar`) is rejected by BOTH the gate (VP8) and the backstop (`parseSegment`'s `fused-marker` code — single-sourced since #1324/#1332, was `FUSED_MARKER_RGX`); a clean marker-led segment (`/:id`, `/*rest`) is accepted by both (#1050).                                  |
| GBP2 | Non-ASCII static parity              | A non-ASCII code point in a static segment (`/café`) is rejected by both (VP13 ↔ `hasNonAsciiSegment`); a pure-ASCII static segment by neither (#1154).                                                                                                                                                                   |
| GBP3 | Optional modifier parity              | An optional form (`/:v?`, `/:v?/x`, `/*v?`) is rejected by BOTH via the shared `parseSegment` **optional-removed** code (VP10/VP11/VP16 ↔ backstop `throwOptionalRemoved`); the same shapes without the `?` by neither.                                                                                           |
| GBP4 | Regex constraint parity               | A `<...>` or stray `<`/`>` (`/:v<\d+>`, `/a<b>`, `/a>b`) is rejected by BOTH via the shared **constraint-removed** code (VP6/VP9/VP15 ↔ backstop `throwConstraintRemoved`); the plain forms (`/:v`, `/a`) by neither.                                                                                   |
| GBP5 | Duplicate param name parity          | A same-name param repeated in one path (`/:id/:id`) is rejected by both (VP12 ↔ backstop cross-position dup); two distinct names (`/:a/:b`) by neither (#1151).                                                                                                                                                           |
| GBP6 | Trailing parameter marker parity     | A marker fused to the END of a param/splat name (`/:y*`, `/:y:`, `/*y:`) is rejected by BOTH the gate (`findSegmentGrammarError` → trailing-marker) and the backstop (`extractParamName` → `parseSegment`); a clean marker-led name (`/:id`, `/*rest`) by neither (#1324).                                                |
| GBP7 | Name-less optional-modifier parity   | A marker-less segment with a trailing `?` (`/faq?`) is rejected by BOTH as name-less — the backstop via its `endsWith("?")` fork → `parseSegment`, the gate via `findSegmentGrammarError` — both consuming the SAME name-less verdict (the former F1 drift); the same static without the `?` (`/faq`) by neither (#1241). |

> **Excluded — documented single-layer behaviour, NOT parity bugs:** gate-only string-format checks (`//` double-slash VP4, whitespace VP3, non-string VP2) — the backstop consumes an already-segmented path. (The trailing-marker `/:y*` case, formerly excluded, is CLOSED as of #1324 — the backstop now shares the gate's `parseSegment` name boundary via `extractParamName` — and is **GBP6** above.)

### Matching

| #   | Invariant                             | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| --- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 8   | Param wins if its branch can complete | At a param+splat junction (a node with BOTH a param child and a splat sibling) the param branch wins **iff its sub-traverse structurally completes** (#1288): the branch is tried on a scratch object and commits only when it reaches a route; otherwise the splat sibling captures. `/user/:id/profile` + `/user/*rest` resolves `/user/x/settings` to the catch-all (`{rest:"x/settings"}`) — the param branch dead-ends. Cited by `SegmentMatcher.#traverseFrom` / `#matchSplat`. |
| 24  | Splat backtracking                    | When a splat node has a child route, a remainder that matches the child resolves to the more-specific child; a remainder that does not falls back to the wildcard capture (`/n/*rest` + `/n/*rest/edit`: `/n/edit` → the child, `/n/x/y` → `{rest:"x/y"}`). Cited by `#matchSplat` / `splat-backtrack.stress.ts`.                                                                                                                                                          |
| 25  | Query overrides same-named path param | A query key equal to a path-param name **overwrites** the captured path value: `match("/u/5?id=9").params` → `{id:"9"}` (query merged last into the same object, `#mergeQueryParams`). Intentional/documented (#843): `buildPath` never emits a path param as a query key, so the build→match roundtrip is unaffected. **Dies in wave 2** (the flat-bag `state.params` merge is replaced by a split `params`/`search` model). Cited by `SegmentMatcher.#mergeQueryParams`.                                                                                         |

### Roundtrip Extensions

| #   | Invariant                                  | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --- | ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2   | Encoding-aware value corruption (uri/none) | Under `urlParamsEncoding: "uri"` or `"none"` a non-splat param value carrying a reserved char corrupts the round-trip **silently**: a `?` splits (`{id:"x?tab=1"}` → builds `/users/x?tab=1` → matches back `{id:"x", tab:"1"}`), a `#` truncates (`{id:"x#sec"}` → `{id:"x"}`), a non-ASCII value under `none` builds an unmatchable URL. More dangerous than the `/`-case (which merely won't match) — the round-trip returns a _different but plausible_ value. Mitigation: `default`/`uriComponent` (which percent-encode `?`/`#`/non-ASCII), or a splat for multi-segment values. |

> **Retired invariant numbers (not reused):** Matching #9/#10/#23/#26 (constraint satisfaction / rejection / filtering / decode-consistency), #15 (constraint-aware query detection), #29/#30 (constraint-balance and two-optionals-before-splat registration rejections), the optional-aliasing **build-side caveat** on Matching #2, and the optional-specific Matching #3 / #17 / Roundtrip Extensions #1. The grammar has no regex constraints or optional params — see the optional/constraint-rejection checks (Route Path Validation rows 6/9/10/11/15/16) above.
