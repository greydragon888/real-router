# SSR / SSG Comparison ‚Äî @real-router/vue vs vue-router

> Authoritative source for Real-Router capabilities: [`README.md`](./README.md) (synthesised from the four child READMEs in [`ssr/`](./ssr), [`ssr-streaming/`](./ssr-streaming), [`ssg/`](./ssg), [`ssr-mixed/`](./ssr-mixed)).
> Competitor data sourced from official docs accessed via Context7 on **2026-05-06**:
> - `vue-router` v4 ‚Äî official router for Vue 3: https://router.vuejs.org/, https://router.vuejs.org/guide/migration/, https://router.vuejs.org/guide/essentials/history-mode.html
> - `vue-router/experimental` data loaders (still **experimental** as of 2026-05-06): https://router.vuejs.org/data-loaders/basic/, https://router.vuejs.org/data-loaders/colada/, https://router.vuejs.org/data-loaders/defining-loaders.html
> - `unplugin-vue-router` (typed routes / file-based routing, separate package): https://uvr.esm.is/

This comparison is restricted to **standalone** routers for Vue 3. Nuxt and Quasar are deliberately excluded ‚Äî they bundle the router with their own server runtime, build pipeline, and deployment model, putting them in the meta-framework category. `vue-router` is the official Vue.js routing library and the only standalone option in the Vue ecosystem; this comparison therefore has a single competitor.

## TL;DR

`vue-router` 4 owns **routing** and only routing ‚Äî `createMemoryHistory()` + `router.push(req.url)` + `router.isReady()` is the entire SSR contract on the router side, with data fetching, state hydration, error ‚Üí HTTP mapping, and SSG enumeration left to the application. The `vue-router/experimental` data-loader API (`defineBasicLoader`, `defineColadaLoader` with Pinia Colada) is the closest equivalent to a Real-Router-style loader, but it is **explicitly experimental** as of 2026-05-06 and not yet recommended for production. **Real-Router on Vue 3 ships per-route loaders via `ssr-data-plugin`, per-route SSR mode opt-out (`{ ssr?, loader? }` with `SsrMode` triple + function form), typed loader-error ‚Üí HTTP envelope (`LoaderRedirect` / `LoaderNotFound` / `LoaderTimeout` ‚Üí 301/302/404/504), `cloneRouter` per-request DI, build-time path enumeration with strict typed-error abort, and a documented per-pipeline HTTP envelope** ‚Äî all built on the same Vue 3 limitations (blocking `<Suspense>` SSR, atomic hydration, no out-of-order placeholders) that constrain `vue-router` users equally. **`vue-router` wins on ecosystem maturity, typed routes via `unplugin-vue-router` file-based routing, the experimental Pinia Colada SWR cache, and the breadth of navigation guards / hooks**; **Real-Router wins on ready-for-production loader semantics, typed errors, per-pipeline HTTP envelope, build-time stale-id detection, and the public plugin protocol**.

## Where Real-Router wins

- **Per-route loaders that are production-ready, not experimental.** `ssr-data-plugin` is dogfooded in three shipped examples and writes data into `state.context.data` consistently across SSR, streaming, and SSG. `vue-router`'s closest equivalent (`defineBasicLoader` / `defineColadaLoader`) is **explicitly under `vue-router/experimental`** and not yet recommended for production ‚Äî verified at https://router.vuejs.org/data-loaders/defining-loaders.html (retrieved 2026-05-06). Until the experimental API stabilises, vue-router users assemble SSR data with `serverPrefetch()` + Pinia store or hand-rolled prefetch.
- **Typed loader-error ‚Üí HTTP envelope.** [`README.md` "Typed HTTP error model"](./README.md#capabilities) ‚Äî `LoaderRedirect` / `LoaderNotFound` / `LoaderTimeout` / `CANNOT_ACTIVATE` map deterministically to 301/302/404/504/302 in the `entry-server.ts` dispatch table. The streaming pipeline catches `LoaderNotFound` **before** stream construction (`Transfer-Encoding: chunked` is absent on 404, `cleanup()` always runs ‚Äî no router leak). `vue-router` provides `router.onError` and `isNavigationFailure` for typing failures, but the application owns the HTTP envelope.
- **SSG that fails the build on stale data.** Empirical finding 6 in [`README.md`](./README.md): Vue's SSG opts into the strict typed-error path ‚Äî unknown ids throw `LoaderNotFound` and `ssg-build.ts` aborts the build with a non-zero exit, guarding against stale `entries.ts` ids silently emitting "user not found" pages with HTTP 200. `vue-router` has no SSG primitive; the application's loop chooses the contract.
- **Per-pipeline HTTP envelope is documented, not accidental.** The `Cache-Control` / `ETag` / `AbortController` table in [`README.md`](./README.md#capabilities) codifies three deliberate HTTP contracts (strong sha256 ETag for `renderToString`, none for `renderToWebStream` because in-flight buffering defeats streaming, weak file-mtime ETag for SSG). `vue-router` does not own HTTP concerns.
- **Per-request isolation via `cloneRouter`.** The DI seam carries `currentUser` (auth/role guards) and `abortSignal` (cancellation); `entry-server/_auth.ts` and `entry-client.ts` mirror cookie parsing through shared `_known-users.ts` so post-hydration `canActivate(getDep("currentUser"))` outcomes match the SSR pass. `vue-router` runs as a singleton per-app instance; isolation in SSR comes from creating a fresh `createSSRApp` + `createMemoryHistory` per request, with no router-level DI seam.
- **`canActivate` + DI guards.** Auth/role gating reads DI via `getDep(...)` on a per-route resolver. `vue-router`'s `beforeEach` / `beforeEnter` is comparable in expressiveness, but state has to come through the application's own scope (Pinia, request locals).
- **Public plugin protocol.** `claimContextNamespace` + `usePlugin` + `addInterceptor` is the same surface every shipped plugin uses (`ssr-data-plugin`, `lifecycle-plugin`, `validation-plugin`, `search-schema-plugin`, `browser-plugin`, ‚Ä¶). `vue-router` exposes navigation guards but not a comparable public extension protocol for plugin authors.
- **Vue 3.5 lazy hydration as a documented adapter feature.** `ssr/` demonstrates `defineAsyncComponent({ hydrate: hydrateOnVisible() })` end-to-end ‚Äî server fully renders, client defers JS hydration until the strategy fires, the deferred component's chunk is **not preloaded** by the initial HTML (verified by an e2e test). `vue-router` is neutral about this ‚Äî the same Vue 3.5 capability works either way, but Real-Router's example proves the integration.
- **Selective SSR per route (#597).** `ssrDataPluginFactory` accepts `{ ssr?, loader? }` per route with `SsrMode = "full" | "data-only" | "client-only"`, the `true`/`false` boolean aliases, and a function form `(state) => SsrMode` resolving per-navigation. The `ssr-mixed/` example demonstrates all four shapes on one server with `getSsrDataMode(state)` branching `entry-server.ts` between `await renderToString(createSSRApp(...))` and shell HTML. `vue-router` does not expose per-route SSR-pipeline branching either in the stable surface or in the experimental loader API ‚Äî opting routes out of SSR is application-level work (e.g. application code branches on path before invoking `renderToString`).

## Where vue-router wins

- **Maturity, ecosystem, and the official Vue brand.** `vue-router` is the official Vue.js router with years of stable releases, broad third-party plugin coverage, and large-scale production use. Real-Router on Vue is dogfooded in three examples but is a much younger library with a smaller ecosystem.
- **Typed routes via `unplugin-vue-router`.** File-based routing with full type inference ‚Äî `useRoute('/users/[id]')` narrows `to.params.id` directly. Real-Router has equivalent type machinery in `core` / `route-tree`, but the SSR README does not exhibit a comparable file-based-routing experience.
- **Pinia Colada SWR cache (experimental).** `defineColadaLoader` with `staleTime` / `key` / `refresh()` / `reload()` provides built-in dedup, caching, and revalidation when the user opts into the experimental API. Real-Router's SSR README does not claim a built-in SWR layer; `lifecycle-plugin.onNavigate` + TanStack Query is the documented composition.
- **Rich global navigation hooks.** `router.beforeEach` / `router.beforeResolve` / `router.afterEach` / `router.onError` / `isNavigationFailure` is a wide surface for cross-cutting concerns (analytics, auth, scroll restoration, page titles). Real-Router's lifecycle is intercept-via-plugin which is more powerful but less plug-and-play.
- **`<RouterView>` slot props and named views.** `vue-router` ships named views (`<router-view name="header" />`) and slot props for transitions out of the box; Real-Router covers similar concerns through plugins (`@real-router/lifecycle-plugin`) and adapter components, but the `vue-router` API is denser.
- **Documentation and migration guides.** Official docs include detailed migration playbooks (v3 ‚Üí v4), SSR cookbooks, and a large body of community recipes. Real-Router's docs are comprehensive but younger.

## Where they tie

- **Same Vue 3 framework-side floor.** Both routers inherit Vue 3 stable's blocking `<Suspense>` SSR semantics, atomic hydration via `app.mount()`, no out-of-order placeholders, and Vue 3.5's per-async-component lazy hydration. Neither router can compensate for these at its own layer.
- **`createMemoryHistory` / per-request app instance is the SSR contract.** Both routers depend on `createSSRApp(...)` + a memory-mode router on the server; both depend on `createSSRApp(...).mount()` to claim the SSR DOM on the client.
- **`vue/server-renderer` is the only render surface.** Streaming via `renderToWebStream`, classical via `renderToString` ‚Äî both routers delegate to the framework.
- **`<Teleport>` `:disabled="!mounted"` + `onMounted` gate.** Empirical finding 2 in [`README.md`](./README.md) is a Vue framework concern, not a router concern; `vue-router` users hit the same hydration-mismatch warning without the gate.
- **AbortSignal in loaders.** Real-Router via `cloneRouter(base, { abortSignal })` + `getDep("abortSignal")`; vue-router's experimental loaders via `(to, { signal }) => ‚Ä¶` ‚Äî semantically equivalent.
- **Server runtime portability.** Both routers run on Node, Bun, Edge given a Web Fetch adapter; neither requires a specific server.
- **CSR‚ÜîSSR code reuse.** Both keep the same route definitions across CSR and SSR; differences live in the entry layer and the history-mode switch.
- **No ISR.** Neither router implements on-demand revalidation; both delegate to the deployment platform's HTTP cache.
