# Vue SSR Examples

> Server-rendering with Real-Router and Vue 3 across four delivery models — classical SSR, chunked-streaming SSR with blocking Suspense, static site generation, and hybrid per-route mode.

Four standalone Vite apps. Each one is a real, e2e-tested dogfooding consumer of `@real-router/*` for a specific server-rendering shape; this document is the synthesis of what the router can and cannot do on Vue 3, sourced exclusively from the four child READMEs.

## Capabilities

What Real-Router actually delivers on Vue 3 across these four shapes, grouped by concern. Each item is sourced from one or more child READMEs.

**Per-request isolation.** Runtime examples (`ssr/`, `ssr-streaming/`, `ssr-mixed/`) use `createRequestScope(req, baseRouter, deps?)` from `@real-router/core/utils` — the four-step boilerplate replacement that produces an isolated router clone, attaches an `AbortSignal` to the request's lifetime (Node `IncomingMessage#"close"` event), injects `abortSignal` into the deps map, and exposes `dispose()` for `finally`. `ssg/` calls `cloneRouter(base)` directly — at build time there's no per-request lifecycle to wire. All scopes are disposed in `finally` so a typed-error short-circuit, a streaming pump exit, or a client disconnect never leaks state. The DI seam carries `currentUser` (auth/role guards) alongside the auto-injected `abortSignal` (cancellation); `entry-server/_auth.ts` and `entry-client.ts` mirror cookie parsing through shared `_known-users.ts` so post-hydration `canActivate(getDep("currentUser"))` outcomes match the SSR pass.

**Single data-loading plugin shape, with strict SSG.** `@real-router/ssr-data-plugin` writes plain JSON-shaped data into `state.context.data` for all four pipelines and ships a typed-errors subpath (`@real-router/ssr-data-plugin/errors`) with `LoaderNotFound`/`LoaderRedirect`/`LoaderTimeout` + `withTimeout`. **Note:** Vue SSG opts into the strict typed-error path — unlike Solid/Svelte SSG which return `user: undefined` for unknown ids, Vue's `users.profile` and `users.profile.posts` loaders throw `LoaderNotFound` and `ssg-build.ts` aborts the build with a non-zero exit code, guarding against stale `entries.ts` ids silently emitting "user not found" pages with HTTP 200.

**State serialization + hydration, Vue-shape.** `serializeRouterState()` is XSS-safe; `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad on `RouterInternals.hydrationState` and calls `router.start(state.path)` once on the client; `ssr-data-plugin`'s start interceptor reads the scratchpad and reuses the server-resolved `state.context.data` directly, skipping the loader call (#596). Vue uses `createSSRApp(...).mount()` to claim existing DOM (the same call that produces an SSR-aware app instance — there is no separate `hydrate()` function as in Svelte/Solid). SSG dual-mode mount: `if (rootElement.firstElementChild)` → `createSSRApp` (hydrate); else → `createApp` (fresh render) — same `entry-client.ts` works for SSG-built pages and Vite dev fallbacks.

**Two render APIs, one router contract.** The router is identical across pipelines; the renderer is the variable:

| Renderer | Output | `<Suspense>` behaviour | Used in |
|---|---|---|---|
| `renderToString` | full HTML string (awaits all `<Suspense>` boundaries) | blocking — content after boundary waits for async children | `ssr/`, `ssg/` |
| `renderToWebStream` | `ReadableStream<Uint8Array>` with chunked HTTP | blocking — chunks emit as render tree resolves, but Suspense still serial | `ssr-streaming/` |

Vue's streaming model is **chunked transfer of the rendered output as the render tree resolves** — real `Transfer-Encoding: chunked` and improved TTFB vs `renderToString` — but `<Suspense>` boundaries remain blocking (no out-of-order placeholders).

**Typed HTTP error model.** Loader errors map to HTTP responses via `entry-server.ts` dispatch: `CANNOT_ACTIVATE` → 302, `LOADER_REDIRECT` → 301/302, `LOADER_NOT_FOUND` → 404 text/plain, `LOADER_TIMEOUT` → 504 text/plain. Streaming pipeline catches `LoaderNotFound` **before** stream construction — `Transfer-Encoding: chunked` is absent on 404, `scope.dispose()` always runs (no router leak). Build-time SSG: typed errors abort loudly; verified by an e2e test that imports the compiled `entry-server.js` and asserts `render('/users/9999')` rejects with `LoaderNotFound`.

**Render-time HTTP status (`<HttpStatusCode :code>`).** Complementary to the loader-driven model: when the status is decided by the *rendered component* (e.g. a glob `*` route → `NotFound.vue`) rather than by a loader exception, mount `<HttpStatusCode>` from `@real-router/vue/ssr`. The component reads the optional `inject(HTTP_STATUS_KEY, null)` sink and writes `props.code` in `setup()`; `entry-server.ts` reads `sink.code ?? 200` after `renderToString` to set the response status. Renders nothing (setup returns `() => null`); silent no-op on the client when no provider is mounted. **Vue-specific: `entry-client.ts` mounts the same `<HttpStatusProvider>` symmetrically** with a throwaway sink — Vue emits `<!--[-->` / `<!--]-->` fragment markers around the provider's slot, and an asymmetric mount logs `Hydration completed but contains mismatches`. Same architectural fix Solid uses; React/Preact don't need it. 3 dedicated dogfood e2e tests verify the contract. Streaming/SSG examples don't dogfood this — `ssg/` builds 404 at build-time via `404.html`; `ssr-streaming/` short-circuits unknown ids via `LoaderNotFound`.

**Vue 3.5 lazy hydration as a first-class adapter feature.** `ssr/` demonstrates `defineAsyncComponent({ loader, hydrate: hydrateOnVisible() })` for `HeavyAnalytics.vue`: server fully renders the HTML (crawlers and JS-disabled clients see everything), but the client defers JS hydration until the strategy fires. Until then, `onMounted` does NOT run, event handlers are not attached, and the component's separate JS chunk is not loaded — pure savings on time-to-interactive for content the user has not seen yet. Strategies: `hydrateOnVisible()`, `hydrateOnIdle()`, `hydrateOnInteraction()`, `hydrateOnMediaQuery()`, custom function. Maps to Angular's `@defer (on viewport) + withIncrementalHydration()`.

**Vue 3.5 `useId()` for SSR-stable form IDs.** `ssr/`'s `SearchForm.vue` uses `useId()` for `<label :for="...">` ↔ `<input :id="...">` pairing. Returns a stable per-component-instance ID; SSR and client produce identical values, no hydration mismatch. Maps to Solid's `createUniqueId()` and React's `useId()`.

**Custom directive lifecycle on Real-Router routes.** `v-track-view` (in `ssr/`) demonstrates the full directive contract: `mounted(el, binding)` (SSR-skipped — DOM-only APIs safe without `typeof window === "undefined"` guard), `updated(el, binding)` (fires on reactive binding change, Vue diffs and calls with the new value), `unmounted(el)` (cleanup). The streaming example additionally exercises directive update lifecycle through `manualOverride` flipping a tracked id.

**`<Teleport>` streaming-safe portal pattern.** `ssr-streaming/`'s `ProductSpecsModal.vue` uses `<Teleport :disabled="!mounted">` paired with `onMounted(() => mounted.value = true)`. Server-side, `mounted = false` → `<Teleport>` disabled → content renders inline at the declared site (or nothing if closed). Client-side, `onMounted` fires after hydration, `mounted` flips, and any open content moves to `#modal-target`. The host `<div id="modal-target">` lives in `index.html` (sibling of `#root`) so it ships in every server response, decoupling the portal target from the Vue tree.

**Suspense Error Boundary.** `<ReviewsErrorBoundary>` uses `onErrorCaptured` returning `false` to stop propagation — mirrors React's `componentDidCatch`. Test fixture id `4`: server resolves `Promise.resolve([])` (no SSR error), client `fetchReviews("4")` returns `Promise.reject()` → boundary catches the post-hydration throw → sibling `<RelatedItems>` keeps rendering.

**Per-route SSR mode (#597).** `@real-router/ssr-data-plugin` accepts a per-route `{ ssr?, loader? }` config that branches the server pipeline by `SsrMode`: `"full"` (run loader + render), `"data-only"` (run loader, ship JSON only, shell HTML), `"client-only"` (skip loader, ship shell, application fetches client-side). `ssr: false` aliases `"client-only"`; `ssr: true` aliases `"full"`. Function form `(state) => SsrMode` resolves per-navigation. The plugin publishes the resolved mode to `state.context.ssrDataMode`; `entry-server.ts` reads it via `getSsrDataMode(state)` to branch between `await renderToString(createSSRApp(...))` from `vue/server-renderer` (full) and a shell `<div data-ssr-shell data-ssr-mode="…">` placeholder. The `ssr-mixed/` example demonstrates all four config shapes on one server. Composes with #596: hydration scratchpad reuse applies when mode allows the loader; `client-only` skips the loader unconditionally regardless of scratchpad contents (mode wins).

**Per-namespace CSR revalidation (#605).** `@real-router/ssr-data-plugin` exports `invalidate(router, namespace)` — a peek-then-clear-after-write primitive that marks a context namespace stale. Dogfooded in `ssr-mixed/`'s Home page: `invalidate(router, "data")` + `router.navigate(name, params, search, { reload: true })` re-runs the loader on the same route. The plugin's `stabilizeState` was updated so `{ reload: true }` is not short-circuited as a same-state no-op. Loaders observe the navigation's `AbortSignal` via `SsrLoaderContext.signal` — rapid double-clicks abort the first nav, its `signal.addEventListener("abort", …)` fires, and the cancel-safe write semantics keep the stale flag set so the second nav's leave handler can consume it. Verified by 3 dedicated e2e tests in `ssr-mixed/`.

**SSR-boundary primitives (#604).** `@real-router/vue/ssr` exports `<ClientOnly>` (renders fallback on the server, children after hydration) and `<ServerOnly>` (renders children on the server, fallback after hydration) — paired primitives for declarative branch-by-environment markup. Dogfooded in `ssr/`'s Home page; verified by 2 e2e tests asserting JS-disabled HTML shows the server branch and post-hydration DOM swaps both branches without `hydrat`/`mismatch` console warnings. Cross-adapter (React/Preact/Vue/Solid/Svelte/Angular).

**`defer()` API is available but not dogfooded here.** `@real-router/vue/ssr` ships `useDeferred`/`<Await>`/`<Streamed>` and `@real-router/ssr-data-plugin` ships `defer({ critical, deferred })` (#610) with the `__rrDefer__("key", json)` inline-script settle wire format emitted by `injectDeferredScripts` (from `@real-router/ssr-data-plugin/server`). The Vue `ssr-streaming/` example deliberately uses Vue-native `async setup()` instead — Vue's `<Suspense>` is chunked-blocking in SSR, so layering `__rrDefer__` script chunks on top of Vue's body chunks would trip the hydration walker. The cross-adapter `defer()` contract remains available for application code that wants to bypass `<Suspense>` (e.g. an SPA layer over a non-streaming Vue SSR).

**Build-time path enumeration.** `getStaticPaths(router)` walks the route tree and returns **leaf** paths only. `ssg-build.ts` derives intermediate parent URLs from leaves (e.g. `/users/:id` from `/users/:id/posts`) and adds the `/users` list page manually. Result: 8 pre-rendered URLs. `appType: "mpa"` in `vite.config.ts` is load-bearing — disables Vite's SPA fallback so each pre-rendered file ships at its own URL.

**Per-route HTTP semantics, per pipeline.** Three of four examples drive `Cache-Control` from a single `cache-policies.ts` lookup (`ssr-mixed/` is a mode-shape demo without cache wiring). The HTTP envelope differs by pipeline:

| Pipeline | `Cache-Control` | `ETag` | `AbortController` |
|---|---|---|---|
| `ssr/` (renderToString) | per-route, `must-revalidate` | strong sha256 (16 base64url chars) | `createRequestScope(req, baseRouter, { currentUser })` auto-injects `abortSignal` into deps; `slow` loader reads it via `getDep("abortSignal")` and composes with `withTimeout(..., { upstreamSignal })` (#598) — composed signal cancels upstream `fetch` on deadline or client disconnect |
| `ssr-streaming/` (renderToWebStream) | per-route, **no `must-revalidate`** (CDN edge handles freshness) | **none, intentional** (buffering defeats streaming) | `createRequestScope` exposes `scope.signal`; pump checks `signal.aborted` between `reader.read()` calls, `reader.cancel()` + `reader.releaseLock()` in `finally` |
| `ssg/` (build-time) | per-route via `ssgServe()` Vite plugin (intercepts `writeHead`) | weak from file mtime (Vite preview default) | not applicable |
| `ssr-mixed/` (per-route mode) | not wired (mode-shape demo) | not wired | `createRequestScope(req, baseRouter)` auto-injects `abortSignal`; `home` loader reads it via `SsrLoaderContext.signal` for #605 cancel-safety |

## Limitations

Each item is categorised: **(framework-side)** — Vue 3 / Vue runtime constraint not under Real-Router's control; **(design choice)** — deliberate Real-Router (or plugin) omission with documented rationale, often closeable by user-side composition; **(example design choice)** — per-example scope narrowing or simplification, not a library/adapter constraint.

- **(framework-side) Vue 3 stable does NOT implement out-of-order Suspense placeholders in SSR.** The `ssr-streaming/` README's comparison table is explicit: React 19 emits `<!--$?-->` markers + `<template>` chunks, Solid emits inline `<script>` patches, Vue does not. Application code that streams `<Suspense>` boundaries gets chunked transfer of the rendered output but no "fallback now, real content later" wire pattern. See "Why no `ssr-rsc/`?" below.
- **(framework-side) Vue 3 stable does NOT implement selective hydration.** `app.mount()` claims the DOM atomically; there is no per-island hydration as in React 19 (per Suspense boundary), Solid (`_$HY` patches), or Angular (`@defer` + `withIncrementalHydration()`). Vue 3.5 lazy hydration via `defineAsyncComponent({ hydrate: ... })` is the closest Vue equivalent, but it is opt-in per async component, not automatic per Suspense boundary.
- **(framework-side) `<Suspense>` semantics are blocking in SSR.** Content after a `<Suspense>` boundary waits for async children. The "Critical content precedes deferred sections" positional invariant is enforced (Scenario 11), and `reviews-fallback`/`related-fallback` markers never appear in the DOM at all (Scenario 14) — fallbacks only matter for the client transition, not the streamed bytes.
- **(framework-side) `<Teleport>` SSR + streaming + hydration requires a `:disabled` + `onMounted` gate.** Without `<Teleport :disabled="!mounted">`, Vue's SSR Teleport emits placeholder markers that the streaming pipeline + hydration walker don't always match precisely — triggering `Hydration completed but contains mismatches`. Pattern is the documented escape hatch for any teleported content.
- **(framework-side) No ETag for streamed responses.** Inherent: a streaming pipeline never holds the body in memory as a single buffer, so adding ETag would mean buffering the whole stream first — which defeats streaming. Production setups rely on CDN-level shared caching (`s-maxage`) plus the CDN's own buffered ETag layer applied at the edge.
- **(framework-side) `withTimeout()` cancellation is cooperative.** When the deadline elapses, the loader's `{ signal }` aborts *before* the race rejects with `LoaderTimeout`; loaders that thread `signal` into their I/O actually cancel the underlying work. Loaders that don't propagate the signal still run to completion in the background. The `slow` loader in `ssr/` composes the deadline with `options.upstreamSignal` (client-disconnect from per-request DI).
- **(framework-side) `404`/`504` response bodies are plain text, not the SSR-rendered NotFound page.** Rendering a rich 404 would require a second `render()` pass — kept simple in the demo.
- **(example design choice) `ssr-streaming/` does not dogfood `defer()` + `<Await>` + `<Streamed>`** even though the adapter ships them (`@real-router/vue/ssr` exports `useDeferred`/`<Await>`/`<Streamed>`; `@real-router/ssr-data-plugin` exports `defer()` and `injectDeferredScripts`). Vue 3's `<Suspense>` is chunked-blocking in SSR — the renderer awaits every `async setup()` *before* emitting that chunk — and layering `__rrDefer__("key", json)` settle scripts on top of Vue's body chunks trips the hydration walker (`Hydration completed but contains mismatches`). The streaming example uses Vue-native `async setup()` with per-component fetchers as the canonical path; the `defer()` API is available for application code that bypasses `<Suspense>` (e.g. SPA layered over non-streaming SSR). See `packages/ssr-data-plugin/CLAUDE.md` "Adapters that intentionally don't dogfood `defer()`" for the full rationale.
- **(design choice) No router-specific streaming API beyond `defer()` + `<Await>` + `<Streamed>`.** Streaming behaviour comes from `renderToWebStream` + `<Suspense>` + `async setup()`; the router's role in `ssr-streaming/` is identical to `renderToString` SSR. The cross-adapter `defer()` API is available but not dogfooded in `ssr-streaming/` (see above).
- **(design choice) `ssr-data-plugin` (boot-path) and `lifecycle-plugin onNavigate` (CSR navigation) split the lifecycle by phase, on purpose.** `ssr-data-plugin` intercepts `router.start()` — runs once during SSR; on the client, `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad and the plugin reuses the server-resolved `state.context.data` directly (#596) instead of calling the loader again. CSR navigations via `<Link>` do NOT re-run its loader; that is `lifecycle-plugin.onNavigate`'s job, and the canonical CSR+SSR stack composes both plugins. Single-responsibility per plugin keeps SSG-only build scripts and pure-CSR SPAs from carrying interception code they don't use. One-off escape hatches that don't change the architecture: `router.navigate(name, params, search, { reload: true })`, or a layered CSR fetcher (TanStack Query, SWR).
- **(example design choice) `ssr-streaming/` deliberately omits cookie-based auth gates.** Vue adapter supports `canActivate` + DI; the streaming example just doesn't use it (`routes.ts` declares `home` / `products.list` / `products.detail` only — no `canActivate`). Sibling `ssr/` covers auth end-to-end. Pure scope narrowing for `<Suspense>` focus.
- **(framework-side) `ssr-streaming/` `Cache-Control` omits `must-revalidate`.** Streamed bodies cannot be ETag'd in-flight (buffering for hashing defeats streaming), so freshness validation is delegated to the CDN edge layer (which buffers + ETags at its own boundary). The `cache-policies.ts` file documents this rationale inline. Inherent streaming trade-off, not Vue-specific. Self-hosted setups without an edge cache should add `must-revalidate` themselves.

### Why no `ssr-rsc/`?

Vue does not implement React Server Components. The `ssr-streaming/README.md` comparison table characterises Vue 3's network model as "sequential, top-down" `<Suspense>` with "blocking — content after the boundary waits for async children" semantics and atomic hydration via `app.mount()`. There is no Flight-style server-component-tree wire format and no per-component server/client boundary marker. The `ssr-streaming/` example documents Vue's actual blocking-Suspense + chunked-streaming model "honestly, without promising APIs that don't exist". If you need RSC, use the React adapter — see [`../../react/ssr-examples/ssr-rsc/`](../../react/ssr-examples/ssr-rsc).

## Vue-unique aspects

What's specific to the Vue 3 adapter, expressed as Real-Router integration points rather than Vue theory:

- **Vue 3.5 lazy hydration is the closest Vue equivalent to React/Solid selective hydration.** It runs at the component level (`defineAsyncComponent({ hydrate: ... })`) rather than the boundary level — opt-in per async component, with five strategies (`hydrateOnVisible`/`hydrateOnIdle`/`hydrateOnInteraction`/`hydrateOnMediaQuery`/custom). Real-Router does nothing different — the lazy-hydrated component is a regular page descendant — but the `ssr/` example proves the integration: SSR ships full markup, the JS chunk is not loaded until visible, `onMounted` and event handlers wait for the strategy to fire.
- **Chunked streaming AS RENDERED, not chunked streaming WITH OOO.** Vue's position is unique in this monorepo: real `Transfer-Encoding: chunked` (Scenario 12 verifies body wall-clock ≥1100 ms, `Content-Length` absent), but `<Suspense>` boundaries remain serial. Adapter-shopping readers expecting React 19 / Solid behaviour need to know this — the bytes ship progressively, but the semantic ordering follows source.
- **`<Teleport>` is the canonical Vue portal pattern, with a streaming-specific footgun.** Real-Router does not own teleport behaviour, but the `ssr-streaming/` example documents `:disabled="!mounted"` + `onMounted` gate as the load-bearing fix for hydration consistency. Same family of pitfalls as React's `createPortal` `mounted` gate in `react/ssr-examples/ssr-streaming/`.

## Empirical findings

Findings that the four READMEs flag as non-obvious, empirically-verified, or footgun-class — distilled here so adapter-shopping readers don't have to re-discover them.

1. **Vue chunked streaming is real at the HTTP level but blocks at the Suspense boundary — verified by three Vue-specific surrogate scenarios.** `ssr-streaming/` Scenario 12 captures `Transfer-Encoding: chunked` set + `Content-Length` absent + body delivery wall-clock ≥1100 ms (proves genuine streaming, not buffered response). Scenario 11 enforces the positional invariant: resolved critical HTML appears in the response BEFORE the `<Suspense>`-rendered sections. Scenario 14 asserts no fallback flicker — `reviews-fallback`/`related-fallback` markers never appear in the DOM. Together these pin down "real chunked transfer + blocking Suspense" — the actual Vue 3 contract, distinct from React 19's "fallback now, real content later".

2. **`<Teleport>` without `:disabled` + `onMounted` gate triggers `Hydration completed but contains mismatches`.** SSR Teleport emits placeholder markers that the streaming pipeline + hydration walker don't always match precisely. The canonical streaming-safe pattern is documented as the recommended escape hatch in `ssr-streaming/`.

3. **Vue 3.5 lazy hydration economics: until the strategy fires, the JS chunk is not loaded.** The deferred component's separate JS chunk does NOT appear in the initial HTML's preload set, `onMounted` does not run, and event handlers are not attached. Verified by an e2e test asserting "code-split chunk exists on disk + is NOT preloaded by initial HTML". This is what makes lazy hydration a real time-to-interactive saving rather than a deferred-execution illusion.

4. **`ssr-data-plugin` skips the loader on the post-hydration `start()` (#596).** `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad on `RouterInternals.hydrationState` before `router.start(state.path)`; the plugin's start interceptor reads it and reuses the server-resolved `state.context.data` directly. Result: zero loader-driven calls on first paint after hydration. Verified by `post-hydration loader skip (#596)` Playwright tests in each child example's `e2e/` folder.

5. **`AbortController` mid-stream release is fast.** `ssr-streaming/` Scenario 20: client `timeout: 150ms`, server releases the stream-pump within 800 ms (well under the 1200 ms full-stream span for `/products/:id`). The pump's `finally` always calls `reader.cancel()` + `reader.releaseLock()` regardless of how the loop exits — required so `ReadableStream` locks don't block GC. Without this wiring, the server would keep pulling chunks until completion even after the client gives up.

6. **Vue SSG opts into strict typed errors, diverging from Solid/Svelte SSG.** `LoaderNotFound` thrown for unknown ids aborts the build with a non-zero exit code, guarding against stale `entries.ts` ids silently emitting "user not found" pages with HTTP 200. Solid and Svelte SSG return `user: undefined` and let the page render the undefined branch — both contracts are valid, but the Vue choice is the strict one.

7. **`invalidate(router, namespace)` + `{ reload: true }` survives rapid double-clicks (#605).** Naive `invalidate + navigate` is unsafe: a late-resolving loader from nav A can race nav B and overwrite fresh data with stale. The plugin's peek-then-clear-after-write semantics keeps the stale flag set while the loader is in flight; if the navigation aborts, the late loader sees `signal.aborted` and skips the write — the flag stays set for the next nav to consume. Verified in `ssr-mixed/` by the `aborts` counter ticking up on rapid double-clicks while `fetchedAt` still lands on a fresh timestamp.

8. **`@real-router/vue/ssr` is a separate package entry (#609), not a root export.** All SSR-only components (`<ClientOnly>`, `<ServerOnly>`, `<Await>`, `<Streamed>`, `<HttpStatusCode>`, `<HttpStatusProvider>`) and composables (`useDeferred`, `createHttpStatusSink`) ship under the `/ssr` subpath — symmetric with React/Preact/Solid/Svelte/Angular. CSR-only consumers don't pay for the SSR surface; SSR consumers import from one place.
