# SSR / SSG Comparison — @real-router/preact vs preact-iso

> Authoritative source for Real-Router capabilities: [`README.md`](./README.md) (synthesised from the four child READMEs in [`ssr/`](./ssr), [`ssr-streaming/`](./ssr-streaming), [`ssg/`](./ssg), [`ssr-mixed/`](./ssr-mixed)).
> Competitor data sourced from official docs accessed via Context7 and direct repo on **2026-05-06**:
>
> - `preact-iso` repo: https://github.com/preactjs/preact-iso
> - Preact v10 guide: https://preactjs.com/guide/v10/preact-iso/
> - `preact-render-to-string` releases: https://github.com/preactjs/preact-render-to-string/releases (referenced for shared underlying capability set)

This comparison is restricted to **standalone** routing libraries for Preact. There are no Preact meta-frameworks comparable to Next/Nuxt/SvelteKit at the routing layer — `preact-iso` is the de-facto standard standalone router for Preact 10, sitting alongside `preact-router` (older) and the general-purpose `wouter`. `preact-iso` is selected because it is the official Preact project's own router and the only one that ships a first-class `prerender` SSG entry point.

## TL;DR

`preact-iso` is the smaller, denser library — a single import surface (`Router` + `LocationProvider` + `lazy` + `ErrorBoundary` + `prerender` + `hydrate`) that gives you SSG and post-hydration routing in roughly four lines of setup. **It does not ship per-route loaders, per-route error boundaries, typed loader-error → HTTP code mapping, an explicit streaming SSR pipeline, or per-request isolation primitives.** Real-Router on Preact 10 ships four dogfooded delivery shapes (`renderToString` for SSG, the **Preact-unique** `renderToStringAsync` for single-shot async SSR, `renderToReadableStream` for chunked streaming with `<preact-island>` swap-in, and a hybrid per-route mode example), wires `ssr-data-plugin` loaders into all of them, and codifies a typed-error → HTTP envelope contract. Preact 10 framework-side limits (no `use(promise)`, async function components silently skip in streaming, no RSC) constrain both routers identically. **Real-Router wins on data-loading lifecycle, typed errors, per-pipeline HTTP envelope, build-time path enumeration, and per-request DI; `preact-iso` wins on byte budget, setup velocity, and a one-call `prerender` whose `links[]` return value is a crawler-style enumeration shortcut.**

## Where Real-Router wins

- **Per-route loaders with a typed error → HTTP envelope.** `LoaderRedirect` / `LoaderNotFound` / `LoaderTimeout` / `CANNOT_ACTIVATE` map to 301/302/404/504/302 in the `ssr/` `entry-server.tsx` dispatch table. The streaming example catches `LoaderNotFound` _before_ constructing the stream so `cleanup()` stays reachable — see [`README.md`](./README.md) "Typed HTTP error model" — without it, a generic throw would skip `router.dispose()` and leak the per-request router. `preact-iso` ships no loader concept and therefore no typed-error mapping.
- **Three rendering modes wired through one router contract.** [`README.md` "Three Preact rendering modes, one router contract"](./README.md#capabilities) — sync `renderToString` for SSG, the **Preact-unique** `renderToStringAsync` for single-shot async SSR with `Content-Length`, and `renderToReadableStream` for chunked streaming. `preact-iso`'s `prerender()` covers only the async-single-shot path; users wanting streaming or sync-renderable SSG bypass `preact-iso` and call `preact-render-to-string` directly.
- **Build-time path enumeration with leaf-only contract.** `getStaticPaths(router, entries)` walks the route tree and returns leaves only; parent URLs are derived in user code. The 8-URL output for the SSG demo is `1 home + 1 list + 3 profiles + 3 posts`. `preact-iso`'s prerender returns `links[]` discovered while rendering, which is a crawler-style enumeration — useful but order-sensitive and dependent on what the home page links to.
- **SSG that fails the build on stale data.** Empirical finding 5 in [`README.md`](./README.md): an unexpected `LoaderNotFound` at build time (e.g. stale id in `entries.ts`) aborts the build with a non-zero exit. Without typed errors, the build silently emits a "user not found" page and serves it as 200 with empty content.
- **Per-pipeline HTTP envelope is documented, not accidental.** The `Cache-Control`/`ETag`/`AbortController` table in [`README.md`](./README.md#capabilities) codifies three deliberate HTTP contracts (strong sha256 ETag for the async-single-shot path; no asserted ETag for streamed bodies; weak file-mtime ETag for SSG). `preact-iso` does not own HTTP concerns.
- **Per-request isolation via `cloneRouter`.** The DI seam carries `currentUser`, `abortSignal`, etc.; `entry-client.tsx` mirrors the same DI wiring over `document.cookie` so post-hydration guards see the same values as SSR. `preact-iso` has no router-level isolation primitive.
- **Plugin protocol is public.** `ssr-data-plugin` is one of many — `lifecycle-plugin`, `validation-plugin`, `search-schema-plugin`, `browser-plugin`, etc. — using `claimContextNamespace` + `usePlugin` + `addInterceptor`. `preact-iso` has callbacks (`onRouteChange`, `onLoadStart`, `onLoadEnd`) but no plugin protocol.
- **Auth/role gates with DI.** `canActivate` + `getDep("currentUser")` is a documented contract on every route; `cloneRouter(base, { currentUser, abortSignal })` is the injection seam. Reproducing this in `preact-iso` is application-level.
- **Selective SSR per route (#597).** `ssrDataPluginFactory` accepts `{ ssr?, loader? }` per route with `SsrMode = "full" | "data-only" | "client-only"`, the `true`/`false` boolean aliases, and a function form `(state) => SsrMode` resolving per-navigation. The `ssr-mixed/` example demonstrates all four shapes on one server with `getSsrDataMode(state)` branching `entry-server.tsx` between `await renderToStringAsync(<App/>)` and shell HTML. `preact-iso` has no loader concept and therefore no per-route SSR-pipeline branching at all — opting routes out of SSR is application-level work.

## Where preact-iso wins

- **Smallest setup, smallest API surface.** A `prerender(<App />)` call plus `hydrate(<App />, root)` covers SSG + hydration; the routing API is `Router` + `Route` + `LocationProvider` + `lazy` + `ErrorBoundary`. The README synthesis of Real-Router on Preact 10 documents three examples and a plugin layer for the same overall job — that is by-design (it covers more), but for a small SSG site, `preact-iso` is denser.
- **Crawler-style route discovery for SSG.** `prerender()` returns `{ html, links }` — `links[]` is internal navigation discovered during the render, which can be recursively walked to enumerate the site without an explicit `entries.ts`. Real-Router prefers an explicit list (deterministic, testable) but the author of a small marketing site may prefer the crawler.
- **Type-safe param props with one generic.** `RoutePropsForPath<'/user/:id'>` narrows component props directly — documented per `preactjs.com/guide/v10/preact-iso/`. Real-Router's adapter has equivalent type machinery in `core` / `route-tree`, but the SSR README does not exhibit it for SSR specifically; readers shopping for "type-safe params" will see `preact-iso`'s example first.
- **Built-in async-aware route transitions.** `Router` preserves the current view while a `lazy()` import for the next route resolves — that is the library's banner feature. Real-Router does not own a comparable transition-suspending primitive at the router layer (it composes with `<Suspense>` / `lazy()` on the Preact side).
- **No plugin layer to assemble.** `preact-iso` is one import. Real-Router's SSR story bundles `@real-router/preact` + `@real-router/ssr-data-plugin` + `@real-router/browser-plugin` (+ optionally `lifecycle-plugin`); fewer moving parts is a real DX win for the SSG-only case.
- **Smaller bundle.** `preact-iso` is one of the smallest standalone routers in the Preact ecosystem — verifiable on bundlephobia. Real-Router's footprint is identical across adapters (the README notes Preact 10's own size advantage as the lever), but `preact-iso` is denser per byte.
- **Optional Navigation API entry point.** `preact-iso/router/navigation-api` is shipped as a separate entry. Real-Router's `@real-router/navigation-plugin` provides the same capability and more (incl. server runtime considerations), but `preact-iso`'s in-tree variant is one import less.

## Where they tie

- **Same Preact 10 framework-side floor.** Both routers inherit Preact 10's lack of `use(promise)`, the silent skip of async function components in `renderToReadableStream`, the `<preact-island>` wire signature for `lazy()` boundaries, and the absence of RSC. Neither router can compensate for these at its own layer.
- **`renderToStringAsync` capability.** `preact-iso`'s `prerender()` and Real-Router's `ssr/` example both use it. The capability comes from `preact-render-to-string@>=6.6.x`, not from either router.
- **`useId()` SSR/CSR parity.** Available since `preact-render-to-string@6.6.5` (Dec 2025), available to both routers identically.
- **Server runtime portability.** Both routers produce strings or `ReadableStream`s; both run on Node, Bun, Edge given a Web Fetch adapter (Real-Router uses Express in the examples).
- **CSR↔SSR code reuse.** Both routers keep the same route definitions across CSR and SSR; differences live in the entry layer, not the route tree.
- **Hydration via Preact's `hydrate`.** Both wrap or compose with `hydrate(vnode, parent)`; both depend on `<preact-render-to-string>` producing markup the client can re-walk.
- **No ISR.** Neither router implements on-demand revalidation; both delegate to the deployment platform's HTTP cache.
- **Two-versions-of-Preact pitfall.** Empirical finding 4 in [`README.md`](./README.md) — `resolve.dedupe: ["preact", "preact/hooks", "preact/jsx-runtime"]` is required for any Vite monorepo example using `@real-router/preact`. The same dedup is required for `preact-iso` users hitting the same monorepo trap; this is not a router-side concern.
