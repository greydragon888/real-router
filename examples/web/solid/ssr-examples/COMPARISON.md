# SSR / SSG Comparison — @real-router/solid vs @solidjs/router and @tanstack/solid-router

> Authoritative source for Real-Router capabilities: [`README.md`](./README.md) (synthesised from the four child READMEs in [`ssr/`](./ssr), [`ssr-streaming/`](./ssr-streaming), [`ssg/`](./ssg), [`ssr-mixed/`](./ssr-mixed)).
> Competitor data sourced from official docs accessed via Context7 on **2026-05-06**:
> - `@solidjs/router` — official Solid router: https://docs.solidjs.com/solid-router/, https://docs.solidjs.com/solid-router/rendering-modes/ssr, https://docs.solidjs.com/solid-router/data-fetching/how-to/preload-data, https://docs.solidjs.com/solid-router/reference/data-apis/query, https://docs.solidjs.com/solid-router/concepts/actions
> - `@tanstack/solid-router` — Solid variant of TanStack Router: https://tanstack.com/router/latest (Solid framework section), https://github.com/tanstack/router/blob/main/packages/solid-router-ssr-query/README.md, https://github.com/tanstack/router/blob/main/docs/start/framework/solid/guide/selective-ssr.md

This comparison is restricted to **standalone** routers for Solid. SolidStart (Solid's meta-framework, which bundles `@solidjs/router` with a server runtime, build orchestration, and file-based routing) and TanStack Start are deliberately excluded. `@solidjs/router` is the official Solid project's own router (used by SolidStart but available standalone); `@tanstack/solid-router` is TanStack's Solid variant (used by TanStack Start but available standalone). Both are in scope here — readers picking between standalone routers for a hand-rolled Solid SSR setup get a side-by-side comparison.

## TL;DR

`@solidjs/router` is the framework's own router with the deepest possible Solid integration — `query()` + `createAsync()` + `action()` form a coherent data layer with automatic deduplication, automatic revalidation after mutations, and `preload`-on-hover/click. `@tanstack/solid-router` brings the strongest type safety (typed params, `validateSearch`, type-safe loader returns) and selective-SSR (`ssr: true | false | 'data-only'`), at the cost of being newer than its React sibling. **Real-Router on Solid ships four dogfooded delivery shapes** (`renderToString` for SSG, `renderToStringAsync` for single-buffered SSR, `renderToStream` for true OOO streaming verified by TCP frame timing, and a hybrid per-route mode example) wired through one router contract, plus typed loader-error → HTTP envelope, build-time path enumeration with optional fail-fast, `cloneRouter` per-request DI, the cross-adapter plugin protocol that ports to React/Vue/Svelte/Preact/Angular, **and per-route SSR mode opt-out (#597) that closes a previous TanStack-only gap**. **`@solidjs/router` wins on idiomatic Solid data flow** (`query`/`createAsync`/`action` is denser and more reactive than `ssr-data-plugin` + `state.context.data`); **`@tanstack/solid-router` wins on type safety and SWR caching**; **Real-Router wins on typed loader-error → HTTP, view-agnostic plugin protocol, a documented per-pipeline HTTP envelope, and per-route SSR mode (function-form `(state) => SsrMode` is unique among the three)**. Solid-side framework features (`<Suspense>` + `createResource` + `<ErrorBoundary>` with reset, `_$HY` selective hydration, `use:` directives) are framework-side and work for all three routers identically.

## Where Real-Router wins

- **Three rendering modes wired through one router contract.** [`README.md` "Three Solid SSR rendering modes, one router contract"](./README.md#capabilities) — sync `renderToString` for SSG, `renderToStringAsync` for buffered async SSR with a graceful no-JS fallback, `renderToStream` for true OOO streaming. Both competitors are render-mode-neutral, but Real-Router is the only one that **dogfoods all three side-by-side as e2e-tested examples**, including TCP-frame-timing verification of true HTTP streaming (Scenario 17).
- **Typed loader-error → HTTP envelope.** `LoaderRedirect` / `LoaderNotFound` / `LoaderTimeout` / `CANNOT_ACTIVATE` map deterministically to 301/302/404/504/302 in the `entry-server` dispatch table; the streaming pipeline catches `LoaderNotFound` **before** stream construction so `cleanup()` always runs. `@solidjs/router`'s `redirect()` and thrown `Response` cover the redirect path; richer error → status mapping is application-level. `@tanstack/solid-router` is the same.
- **Cross-adapter consistency.** The same `ssr-data-plugin` loader shape works on React, Vue, Solid, Svelte, Preact, and Angular — a team migrating between adapters keeps the same loader contract and typed-error subpath. `@solidjs/router`'s `query`/`createAsync` is Solid-only by design; `@tanstack/solid-router` is Solid-shape but with a different surface than the React variant.
- **Build-time path enumeration with leaf-only contract and optional fail-fast.** `getStaticPaths(router, entries)` returns leaves; `ssg-build.ts` derives parent URLs in user code; the fail-fast guard is wired but dormant under the example's tolerant loaders — opting into the strict `LoaderNotFound` path activates it without library changes. Neither competitor ships SSG enumeration.
- **Per-pipeline HTTP envelope is documented, not accidental.** The `Cache-Control` / `ETag` / `AbortController` table in [`README.md`](./README.md#capabilities) codifies three deliberate HTTP contracts (strong sha256 ETag for `renderToStringAsync`, none for `renderToStream` because in-flight buffering defeats streaming, none for SSG by example-design divergence). Neither competitor owns HTTP concerns.
- **Per-request `AbortController` threaded into loader DI + composed via `withTimeout`.** `cloneRouter(base, { abortSignal })` + `getDep("abortSignal")` + `withTimeout(..., { upstreamSignal })` (#598) gives the `slow` loader in `ssr/` a single composed `AbortSignal` that fires on deadline or client disconnect; threaded into `fetch(..., { signal })` it cancels at the network layer. `@solidjs/router`'s `query()` does not document an explicit `AbortSignal` parameter in the fetcher signature; `@tanstack/solid-router` exposes `abortController.signal` in the loader signature but lacks a built-in deadline + upstream composer.
- **`canActivate` + DI guards with mirrored client cookie parsing.** Auth/role gating reads DI via `getDep("currentUser")`; `entry-server` and `entry-client` mirror cookie parsing through shared `_known-users.ts` so post-hydration `canActivate` outcomes match the SSR pass. Replicating this with either competitor is application-level.
- **Idempotent disposal in streaming.** `ssr-streaming/` documents the `disposed: boolean` guard so `onCompleteAll` and the server-layer `finally { cleanup() }` cannot double-dispose. Cited in the README as a real-world streaming gotcha.
- **Public plugin protocol.** `claimContextNamespace` + `usePlugin` + `addInterceptor` is the same surface every shipped plugin uses. `@solidjs/router` and `@tanstack/solid-router` expose hooks but no comparable extension protocol.
- **Selective SSR per route — closing a previous TanStack-only gap (#597).** `ssrDataPluginFactory` accepts `{ ssr?, loader? }` per route with `SsrMode = "full" | "data-only" | "client-only"`, the `true`/`false` boolean aliases, and a function form `(state) => SsrMode` resolving per-navigation. The `ssr-mixed/` example demonstrates all four shapes on one server with `getSsrDataMode(state)` branching `entry-server.tsx`. Per-navigation function form is symmetric with TanStack Solid Router's functional `ssr` form; `@solidjs/router` has no equivalent. **Real-Router-distinct**: the function form receives the resolved post-routing `state`, and the resolved mode is published to `state.context.ssrDataMode` so application code can read it from any component via the public `getSsrDataMode(state)` helper — TanStack's `ssr` is route-config-only, not exposed as state.

## Where @solidjs/router wins

- **Idiomatic Solid data flow.** `query(fn, name)` + `createAsync(() => getQuery(arg))` is the canonical Solid data pattern — `createAsync` is a fine-grained reactive `Accessor<T>` that re-runs only when its inputs change; `query()` deduplicates concurrent calls by `[name, ...args]` automatically. `ssr-data-plugin`'s `state.context.data` is correct but less reactive — readers shopping for Solid-native ergonomics will see `@solidjs/router`'s example first.
- **Automatic revalidation after mutations.** Calling `action(async fd => …)` automatically revalidates queries on the same page after success; `json(data, { revalidate: "posts" })` lets the action target specific keys. Real-Router's equivalent is `router.navigate(name, params, undefined, { reload: true })` plus `lifecycle-plugin.onNavigate` — composable but more verbose.
- **`preload` on hover/click intent.** The `preload` prop on `<Route>` triggers the cache fetcher when the user hovers/clicks a link, before the route is mounted. Real-Router has `@real-router/preload-plugin` separately, but `@solidjs/router`'s primitive is one prop.
- **Smaller and simpler.** `@solidjs/router` is the official lightweight router; the SSR contract is `<Router url={isServer ? req.url : ""} />` plus the data API. No plugin layer to assemble for a simple SSR site.
- **Documented in `docs.solidjs.com`.** Official Solid docs cover SSR setup, `query`/`createAsync`/`action`, `preload`, `redirect`, `json`, `revalidate` end-to-end with examples retrieved 2026-05-06.
- **Action-result plumbing via `revalidate` directive.** Mutations declaratively revalidate queries by key — a denser pattern than wiring up cache-busting manually.

## Where @tanstack/solid-router wins

- **Type-safe params, search, and loader returns.** `Route.useParams()` / `Route.useSearch()` infer types from the route definition; `validateSearch: zodSchema` validates and types search params. Real-Router has core route-tree types but no `validateSearch`-shaped surface in the SSR README; `@solidjs/router` requires explicit generics on `useParams` / `useSearchParams`.
- **Built-in SWR caching.** `staleTime` + `gcTime` per route, comparable to TanStack Query. Real-Router's SSR README does not claim a built-in SWR layer; `@solidjs/router`'s `query()` deduplicates but stable docs do not commit to TTL semantics.
- **`@tanstack/solid-router-ssr-query` for Solid Query integration.** A documented one-import setup (`setupRouterSsrQueryIntegration({ router, queryClient })`) for SSR + Solid Query out of the box. Real-Router would compose `lifecycle-plugin.onNavigate` + Solid Query manually.
- **`errorComponent` / `notFoundComponent` per route.** Router-side error UI primitives. Real-Router defers to Solid's `<ErrorBoundary>` for runtime errors and to typed loader errors for HTTP envelope mapping.
- **Inheritance from the React sibling's design.** Type-safe loaders, `beforeLoad`, `context`-passing, and the broader TanStack DX language carry over to the Solid variant.

## Where they tie

- **Same Solid framework floor.** All three depend on `solid-js/web`'s `renderToString` / `renderToStringAsync` / `renderToStream`; all three use Solid `<Suspense>` + `createResource` + `<ErrorBoundary>`; all three benefit from `_$HY` selective hydration on streaming. None of the routers can change framework-side limits at their layer.
- **`generateHydrationScript()` is required.** Solid's `_$HY` runtime needs the hydration script in `<head>` for any SSR mode. All three routers require it on the consumer side.
- **`use:` directives are SSR-safe by construction.** Solid runtime skips action invocations on the server. The babel-preset quirk (directive symbol must be imported in the consuming module) applies to all three.
- **Server runtime portability.** All three run on Node, Bun, Edge given a Web Fetch adapter; none requires a specific server.
- **CSR↔SSR code reuse.** All three keep the same route definitions across CSR and SSR; differences live in the `<Router url>` / SSR-entry layer.
- **No ISR.** Neither standalone router implements on-demand revalidation; all three delegate to the deployment platform.
- **Per-route SSR mode opt-out (Real-Router #597 vs TanStack `ssr: true | false | 'data-only'`).** Both Real-Router and `@tanstack/solid-router` expose per-route SSR opt-out with a functional form. Surface differs (`{ ssr?, loader? }` plugin config vs `ssr` route prop), but the capability is equivalent at the user-facing level. `@solidjs/router` has no equivalent.
- **vite-plugin-solid quirks affect everyone.** The `<RouteView.NotFound>` + multi-`Match` hydration-key divergence and the `<SuspenseList revealOrder>` mismatch are vite-plugin-solid 2.11.x bugs. All three routers run on top of vite-plugin-solid; none can fix the upstream issue.
