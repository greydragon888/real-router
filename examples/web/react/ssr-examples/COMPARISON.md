# SSR / SSG Comparison — @real-router/react vs react-router (data mode) and @tanstack/react-router

> Authoritative source for Real-Router capabilities: [`README.md`](./README.md) (the five child READMEs in `ssr/`, `ssr-streaming/`, `ssg/`, `ssr-rsc/`, `ssr-mixed/` synthesise into that file).
> Competitor data sourced from official docs accessed via Context7 on **2026-05-06**:
>
> - `react-router` v7 — library/data mode (not framework mode): https://reactrouter.com/start/data/custom, https://reactrouter.com/api/data-routers/createStaticRouter, https://reactrouter.com/start/modes
> - `@tanstack/react-router` (standalone, not TanStack Start): https://tanstack.com/router/latest/docs/framework/react/how-to/setup-ssr, https://tanstack.com/router/latest/docs/framework/react/guide/data-loading, https://tanstack.com/router/latest/docs/framework/react/guide/deferred-data-loading

This comparison is restricted to **standalone** routing libraries. TanStack Start, Remix/React Router framework mode, Next.js, and similar meta-frameworks are deliberately excluded — they bundle routing with their own server runtime, build pipeline, and deployment model and are not apples-to-apples with a router that you assemble into your own server. Where TanStack Start or React Router framework mode supplies a feature that the standalone library does not, the matrix reflects the standalone surface only and notes the gap.

## TL;DR

Real-Router ships the smallest, most explicit SSR surface of the three: five orthogonal delivery shapes (classical SSR, streaming SSR, SSG, RSC, hybrid per-route mode) wired through the same `cloneRouter` → `start(url)` → `serializeRouterState` → `hydrateRouter` lifecycle, with no router-owned streaming or RSC abstraction — those reduce to React 19 primitives and a bundler-agnostic ReactNode payload. **TanStack Router has the strongest typed-data DX** (typed params, typed search via `validateSearch`, typed loader returns, `staleTime`/`gcTime` SWR, automatic streaming) at the cost of a larger surface and a Vite plugin requirement. **React Router data mode** is the only one of the three that exposes a non-framework SSR API (`createStaticHandler` + `createStaticRouter`) explicitly; loaders are runtime-aware via `request`/`request.signal`, but data-mode SSG is DIY (the `prerender` config and full type inference belong to framework mode, which is excluded here). **Real-Router wins on per-pipeline isolation, RSC bundler-agnosticism, and typed loader-error → HTTP envelope mapping**; it loses on built-in SWR caching and on type-safe search params validation, neither of which the README claims.

## Where Real-Router wins

- **Bundler-agnostic RSC plugin.** `@real-router/rsc-server-plugin` writes a `ReactNode` to `state.context.rsc`; the application's `entry.rsc.tsx` decides whether to ship it as raw Flight (`/__rsc?route=…`) or pipe it through SSR + `injectRSCPayload` for the initial HTML load. The plugin itself does not depend on `@vitejs/plugin-rsc` — it is the single integration seam, and the Express server is reduced to a thin Web↔Node bridge. Neither competitor offers RSC outside its meta-framework cousin (TanStack Start / React Router framework mode).
- **Typed loader-error → HTTP envelope, mirrored across pipelines.** `LoaderRedirect`/`LoaderNotFound`/`LoaderTimeout`/`CANNOT_ACTIVATE` map deterministically to 301/302/404/504/302; the same shapes are exported from both `@real-router/ssr-data-plugin/errors` and `@real-router/rsc-server-plugin/errors` so application catch-paths read identically regardless of pipeline. React Router data mode and TanStack Router both expose `redirect()` / `Response`-throw, but neither defines the timeout / cannot-activate codes nor mirrors them across two pipelines.
- **Per-pipeline HTTP envelope is documented, not accidental.** `cache-policies.ts` + the table in the README codify the deliberate divergence: classical SSR ships strong sha256 ETag + `must-revalidate`; streaming and RSC ship neither (in-flight hashing defeats streaming) and document the reason inline. The four examples encode four production HTTP contracts you can copy.
- **Per-request isolation via `cloneRouter`.** Concurrent-load e2e in three of the four examples (10 / 9 / 10 concurrent loads) prove the model. `react-router` data mode achieves isolation through `query(request)` returning a fresh context, but does not expose a router-level `clone(deps)` injection seam for `currentUser` / `abortSignal` / `db`. TanStack Router solves this via `context` per route; the API is comparable but does not document the same isolation guarantees in the standalone surface.
- **SSG that fails the build on stale data.** `ssg/` aborts the build with a non-zero exit on an unexpected `LoaderNotFound`, instead of silently emitting a "user not found" page for a stale `entries.ts` id. Standalone `react-router` and `@tanstack/react-router` leave SSG to the user, so this contract is whatever you write.
- **Build-time path enumeration with explicit "leaves only" contract.** `getStaticPaths(router, entries)` is documented to return leaves; parent URLs are derived in user code. The alternative (recursive enumeration) was rejected as order-sensitive — that decision is in the README and in `IMPLEMENTATION_NOTES.md`.
- **Plugins are first-class — server actions plumb through router state.** `rscActionPluginFactory(getResult)` writes Server-Action results into `state.context.rscAction`; any Server Component (e.g. `<NotificationBanner>`) can read it without prop-drilling. Cross-component action result via a `rsc:server-action-response` `CustomEvent` is documented as the explicit choice over a global mutable ref. Neither competitor provides a comparable extension surface in standalone mode.
- **Selective SSR per route in standalone mode (#597).** `ssrDataPluginFactory` accepts `{ ssr?, loader? }` per route with `SsrMode = "full" | "data-only" | "client-only"`, the `true`/`false` boolean aliases, and a function form `(state) => SsrMode` resolving per-navigation. The `ssr-mixed/` example demonstrates all four shapes on one server, with `getSsrDataMode(state)` branching `entry-server.tsx` between `renderToString(<App/>)` and shell HTML. **TanStack Router's `ssr: true | false | 'data-only'` is a Start-only feature** (not the standalone library); React Router data mode does not have it at all. Real-Router closes this gap as a plain plugin config — no Start-equivalent meta-framework required.

## Where competitors win

- **TanStack Router has the strongest typed-data DX.** Typed params, typed search via `validateSearch`, typed loader returns inferred from the route, and built-in SWR caching (`staleTime` / `gcTime`) come out of the box. `@real-router/react` has type-safe path params, but search-param validation lives in a separate plugin (`@real-router/search-schema-plugin`) and SWR caching is not a core or SSR concern.
- **TanStack Router auto-streams without an `<Await>` wrapper.** Any unawaited promise in a loader becomes a Suspense boundary downstream; the user does not write `<Await>`. Real-Router's `ssr-streaming/` writes `<Suspense>` + `use(promise)` by hand — that is the explicit "Library Philosophy" position, but TanStack's API is shorter for the common case.
- **React Router has a richer error-boundary API.** `errorElement` per route + `isRouteErrorResponse` + `useRouteError` is a documented, typed contract; loaders can throw `Response` or `redirect()` directly. Real-Router defers to React's `<ErrorBoundary>` and a Suspense Error Boundary for streamed-promise rejections — equivalent in capability, less in router-side ergonomics.
- **Larger ecosystems and tooling.** `react-router` has the longest history and the broadest integration surface (auth, devtools, examples, third-party hosts). TanStack Router has TanStack Query / Form / Table integration. Real-Router's ecosystem is plugin-centric and smaller.
- **Built-in revalidation hooks.** `react-router` exposes `useRevalidator()` / `router.revalidate()`; TanStack Router exposes `router.invalidate()`. Real-Router uses `router.navigate(name, params, { reload: true })` or composes `lifecycle-plugin.onNavigate` for CSR refetch — equivalent functionality, but the competitors' APIs are first-class.

## Where they tie

- **Render fundamentals via React 19 primitives.** All three delegate to `renderToString` / `renderToReadableStream` / `<Suspense>` for the actual rendering work. Streaming wire signature (out-of-order Suspense placeholders, byte-ordered chunk boundaries) is a React 19 property, not a router property.
- **No ISR / on-demand revalidation in standalone mode.** None of the three implements ISR at the router level; all three delegate to the deployment platform's HTTP cache.
- **Loader cancellation on rapid navigation.** All three thread an `AbortSignal` into loader work; cancellation of the underlying I/O depends on the application calling `fetch(url, { signal })` or equivalent. **Real-Router differentiates with composed deadline + upstream:** `withTimeout(..., { upstreamSignal })` (#598) composes the per-request `cloneRouter(base, { abortSignal })` with an internal deadline via `AbortSignal.any`, so a single `{ signal }` cancels on whichever fires first.
- **CSR↔SSR code reuse.** All three keep the same route definitions across CSR and SSR; differences live in the entry layer, not the route tree.
- **Server runtime portability.** The library / data-mode / standalone surface of all three runs on Node, Bun, and Edge given a Web Fetch adapter; lock-in arises only when you adopt the meta-framework cousin (Start / framework mode).
