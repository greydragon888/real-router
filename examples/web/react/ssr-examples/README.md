# React SSR Examples

> Server-rendering with Real-Router and React 19 across five delivery models — classical SSR, HTTP-streaming SSR, static site generation, React Server Components, and hybrid per-route mode.

Five standalone Vite apps. Each one is a real, e2e-tested dogfooding consumer of `@real-router/*` for a specific server-rendering shape; this document is the synthesis of what the router can and cannot do, sourced exclusively from the five child READMEs.

| Subdir | Demonstrates |
|---|---|
| [`ssr/`](./ssr) | Classical per-request `cloneRouter` → `usePlugin(ssrDataPluginFactory)` → `start(url)` → `renderToString` → `serializeRouterState` → `hydrateRouter`. Auth + role gates via `canActivate` + DI; typed loader errors (`LoaderRedirect`/`LoaderNotFound`/`LoaderTimeout` + `withTimeout`) → 301/302/404/504; per-route meta with absolute canonical + og; production HTTP: sha256 strong **ETag** (16 base64url chars), per-route `Cache-Control`, `AbortController` threaded into loader DI via `cloneRouter(base, { currentUser, abortSignal })`. `useId()` proves SSR/CSR id parity. |
| [`ssr-streaming/`](./ssr-streaming) | React 19 native streaming: `renderToReadableStream` + `<Suspense>` + `use(promise)` + selective hydration. `ssr-data-plugin` provides critical data resolved before the shell. Suspense Error Boundary contains rejected promises at the boundary granularity. `createPortal` modal with a `mounted` gate. Production HTTP: per-route `Cache-Control`, `AbortController`, **no ETag** (intentional — see Limitations). Notably ships **zero router-specific streaming API**. |
| [`ssg/`](./ssg) | Build-time pre-rendering: `getStaticPaths(router, entries)` → `cloneRouter` + `start` + `renderToString` per URL → write `dist/{url}/index.html`. Nested SSG (`users.profile.posts` → 8 paths total), empty-state path, absolute canonical + og, filesystem-layout assertion, `404.html`, `sitemap.xml`. `LoaderNotFound` at build time fails the build loudly. Preview pipeline ships per-route `Cache-Control` (custom `ssgServe()` Vite middleware) + weak `ETag` from file mtime. |
| [`ssr-rsc/`](./ssr-rsc) | React Server Components with `@real-router/rsc-server-plugin` (ReactNode payload variant) + `@vitejs/plugin-rsc`. Two-endpoint architecture: `GET /:path` (HTML + injected Flight) and `GET /__rsc?route=…` (raw Flight) flow through one `entry.rsc.tsx` fetch handler. Server/Client component boundary; React 19 Server Actions (`'use server'`, `useActionState`, progressive enhancement); cross-component action result via `rscActionPluginFactory` writing to `state.context.rscAction`. Race-safe client navigation: per-fetch `AbortController` cancels stale in-flight `/__rsc` requests. |
| [`ssr-mixed/`](./ssr-mixed) | Per-route SSR mode (#597) — four routes × four config shapes on one `entry-server.tsx`: `home` short-form (full), `admin.dashboard` `{ ssr: false }` (client-only), `users.profile` `{ ssr: "data-only", loader }` (data-only), `docs.detail` `{ ssr: (state) => …, loader }` (function-form, branches on `?format=`). `entry-server.tsx` reads `getSsrDataMode(state)` and either `renderToString(<App/>)` (full) or emits `<div data-ssr-shell data-ssr-mode="…">` shell. 4 e2e scenarios verify each mode's HTTP response shape. Composes with #596: hydration scratchpad reuse for `full`/`data-only`; `client-only` skips loader unconditionally. |

## Capabilities

What Real-Router actually delivers across these four shapes, grouped by concern. Each item is sourced from one or more child READMEs.

**Per-request isolation.** `cloneRouter(base, deps?)` produces an isolated router per request; the four examples all dispose it in `finally` so a typed-error short-circuit, a streaming pump exit, or a client disconnect never leaks state. Verified by concurrent-load e2e in each runtime example — `ssr/` runs 10 concurrent `/users/:id` requests, `ssr-streaming/` runs 9 concurrent loads (Scenario 9), and `ssr-rsc/` Scenario 5 runs 10 concurrent `/users/{0..9}`.

**Two data-loading plugins, one shape.** `@real-router/ssr-data-plugin` writes plain JSON-shaped data into `state.context.data` for `ssr/`/`ssr-streaming/`/`ssg/`. `@real-router/rsc-server-plugin` writes ReactNode payloads into `state.context.rsc` for `ssr-rsc/`. Both are start-interceptor plugins; both ship a typed-errors subpath (`/errors`) with the same `LoaderNotFound`/`LoaderRedirect`/`LoaderTimeout` shapes — explicitly mirrored in the RSC plugin so application catch-paths are identical regardless of pipeline shape.

**State serialization + hydration.** `serializeRouterState()` is XSS-safe; the resulting blob lands in `window.__SSR_STATE__`. `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad on `RouterInternals.hydrationState` and calls `router.start(state.path)` once on the client; `ssr-data-plugin`'s start interceptor reads the scratchpad and reuses the server-resolved `state.context.data` directly, skipping the loader call (#596). The hydration component tree sees the same `state.context.data` / `state.context.rsc` the server rendered. SSG auto-detects hydrate vs `createRoot` via `rootElement.firstElementChild` so the same `entry-client` works for SSG-built pages and Vite dev fallbacks.

**Typed HTTP error model.** Loader errors have explicit codes that map to HTTP responses (`ssr/src/entry-server.tsx` dispatch table at lines 130-174 → `server/index.ts` HTTP envelope): `CANNOT_ACTIVATE` → 302, `LOADER_REDIRECT` → 301/302, `LOADER_NOT_FOUND` → 404 text/plain, `LOADER_TIMEOUT` → 504 text/plain. The same shape applies on both endpoints of the RSC dual-endpoint architecture (`/users/9999` and `/__rsc?route=/users/9999` both yield 404 text/plain — Scenarios 16a/16b). At build time in SSG, an unexpected `LoaderNotFound` aborts the build with a non-zero exit instead of silently emitting a "user not found" page for a stale `entries.ts` id.

**Build-time path enumeration.** `getStaticPaths(router, entries)` walks the route tree and returns **leaf** paths only. Application code derives intermediate parent URLs from the leaves (e.g. `/users/:id` from `/users/:id/posts` by suffix-stripping) and adds purely-parent pages manually. The 8-URL output set in `ssg/` is the empirical demonstration.

**Per-route HTTP semantics, per pipeline.** All four examples drive `Cache-Control` from a single `cache-policies.ts` lookup. The HTTP envelope differs by pipeline:

| Pipeline | `Cache-Control` | `ETag` | `AbortController` |
|---|---|---|---|
| `ssr/` (renderToString) | per-route, `must-revalidate` | strong sha256 (16 base64url chars) | threaded into loader DI via `cloneRouter(base, { currentUser, abortSignal })` |
| `ssr-streaming/` (renderToReadableStream) | per-route, **no `must-revalidate`** | **none, intentional** | aborts stream-pump on `req.on("close")`; loader DI not threaded |
| `ssg/` (build-time) | per-route via `ssgServe()` Vite middleware | weak, derived from file mtime by Vite preview | not applicable (pre-rendered) |
| `ssr-rsc/` (Flight + injected HTML) | per-route on **both** endpoints, **no `must-revalidate`** | **none, intentional** | per-request controller fires on `req.on("close")`; signal in `request.signal` but not in loader DI by default |

**Auth + role gates.** `canActivate` guards read DI via `getDep(...)`; `cloneRouter` is the injection seam (`cloneRouter(base, { currentUser, abortSignal })`). The post-hydration client must mirror server-side DI logic over `document.cookie` or the client guard diverges — handled in `entry-client.tsx`. `/dashboard` requires `currentUser`; `/admin` adds `role === "admin"`.

**Per-route SSR mode (#597).** `@real-router/ssr-data-plugin` accepts a per-route `{ ssr?, loader? }` config that branches the server pipeline by `SsrMode`: `"full"` (run loader + render), `"data-only"` (run loader, ship JSON only, shell HTML), `"client-only"` (skip loader, ship shell, application fetches client-side). `ssr: false` aliases `"client-only"`; `ssr: true` aliases `"full"`. Function form `(state) => SsrMode` resolves per-navigation. The plugin publishes the resolved mode to `state.context.ssrDataMode`; `entry-server.tsx` reads it via `getSsrDataMode(state)` to branch between `renderToString(<App/>)` (full) and a shell `<div data-ssr-shell data-ssr-mode="…">` placeholder. The `ssr-mixed/` example demonstrates all four config shapes — short form + boolean alias + string-form + function-form — on one server. Composes with #596: hydration scratchpad reuse applies when mode allows the loader; `client-only` skips the loader unconditionally regardless of scratchpad contents (mode wins).

**RSC integration without RSC-specific routing.** `rsc-server-plugin` writes a complete `ReactNode` per route into `state.context.rsc`. The router still uses the same `start(url)` lifecycle, the same loader error contract, the same `cloneRouter` isolation. The client-side single-listener pattern (`router.subscribe → fetch /__rsc`) carries `<Link>` clicks, programmatic `router.navigate({ reload: true })`, and `popstate` (history) through one code path.

**Server Actions, end-to-end.** Real-Router does not own Server Actions — they are React 19 + `@vitejs/plugin-rsc` primitives — but the example proves the integration: `decodeAction(formData)` for the no-JS path (progressive enhancement via `$ACTION_REF_*` form fields), `loadServerAction(id)` + `decodeReply(body)` for the with-JS path, plus `createTemporaryReferenceSet()` so non-serialisable references survive the round trip. The cross-component result pattern is router-flavoured: `rscActionPluginFactory(getResult)` publishes the action result to `state.context.rscAction`, which any Server Component (e.g. `<NotificationBanner>` in any layout) can read without prop-drilling from the form.

**SEO/head plumbing.** Per-route `meta` resolver returns a `PageMeta` block — `{ title, description, canonical, ogTitle, ogDescription }` (5 fields; `og:url` is rendered from `canonical`, not stored separately). The server renders the block as 6 tags and splices the result into a `<!--ssr-meta-->` placeholder. Canonical URLs are absolute (`SITE_ORIGIN`-prefixed, default `https://example.com`) — relative canonicals are rejected by crawlers. SSG additionally emits `sitemap.xml` from the same URL set, with a filesystem-layout assertion that catches drift between sitemap, on-disk HTML, and `entries.ts`.

## Limitations

Each item is categorised: **(framework-side)** — React/streaming/RSC constraint not under Real-Router's control; **(design choice)** — deliberate Real-Router (or plugin) omission with documented rationale, often closeable by user-side composition; **(example design choice)** — per-example scope narrowing or simplification, not a library/adapter constraint.

- **(design choice) `ssr-data-plugin` (boot-path) and `lifecycle-plugin onNavigate` (CSR navigation) split the lifecycle by phase, on purpose.** `ssr-data-plugin` intercepts `router.start()` — runs once during SSR; on the client, `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad and the plugin reuses the server-resolved `state.context.data` directly (#596) instead of calling the loader again. CSR navigations via `<Link>` and `router.navigate()` do NOT re-run its loader; that is `lifecycle-plugin.onNavigate`'s job, and the canonical CSR+SSR stack composes both plugins. Single-responsibility per plugin keeps SSG-only build scripts from carrying navigation-interception code, and pure-CSR SPAs from carrying start-path interception code. The RSC example sidesteps both with a third application-level pattern — re-fetching `/__rsc?route=…` on every navigation through a single `router.subscribe` listener (no plugin needed for that path). One-off escape hatches that don't change the architecture: `router.navigate(name, params, { reload: true })`, or a layered CSR fetcher (TanStack Query, SWR).
- **(design choice) No `<Await>` / `defer()` wrapper API.** `ssr-streaming/` ships `<Suspense>` + `use(promise)` directly; the `ssr-streaming/README.md` "Library Philosophy" section is explicit that pure delegation to React 19 native primitives is the position. Ergonomic parity with React Router v7's `<Await>` + `defer()` without the abstraction.
- **(design choice) No router-specific streaming API.** Streaming behaviour comes from `renderToReadableStream` + `<Suspense>` + `use(promise)`; the router's role in `ssr-streaming/` is identical to non-streaming SSR.
- **(framework-side) No ETag for streamed responses.** Both `ssr-streaming/` and `ssr-rsc/` document this as inherent: hashing the body requires buffering, which defeats streaming (TTFB jumps to "after the slowest deferred section resolves"). The expectation is that production CDNs add a buffered ETag layer at the edge while the origin still benefits from chunked transfer.
- **(framework-side) SSR cannot render `createPortal`.** The target DOM node does not exist in a string render. `ssr-streaming/` uses a `mounted` state gate (`useState(false)` + `useEffect(() => setMounted(true), [])`) so the SSR pass ships only the trigger button and hydration completes without mismatch. Application-level pattern, not a router concern, but documented because it is the React-19 + streaming-SSR canonical shape.
- **(framework-side) `getStaticPaths` returns leaf paths only.** Application code in `ssg-build.ts` derives intermediate parent URLs (e.g. `/users/:id` from `/users/:id/posts`) and adds purely-parent pages (e.g. `/users` for `UsersList`) manually. Documented as a deliberate API shape — the alternative (returning every path) would make the function order-sensitive and ambiguous for routes that have both a leaf component and children.
- **(framework-side) `withTimeout()` cancellation is cooperative.** When the deadline elapses, the loader's `{ signal }` aborts *before* the race rejects with `LoaderTimeout`; loaders that thread `signal` into their I/O — e.g. `fetch(url, { signal })` — actually cancel the underlying work (proven end-to-end by the `withTimeout (#598) network cancellation` test in `ssr/`). Loaders that don't propagate the signal still run to completion in the background. The `slow` loader in `ssr/` composes the deadline with `options.upstreamSignal` (client-disconnect from per-request DI).
- **(example design choice) RSC example does not thread `abortSignal` into loader DI.** `cloneRouter`'s second argument is an arbitrary `Record<string, unknown>` deps map (`packages/core/src/api/cloneRouter.ts`); the runtime `ssr/` example passes `{ currentUser, abortSignal }`, the RSC example passes only `{ db: database }` for focus. Adding `abortSignal` to the RSC example is one Edit — no library change needed. The signal is observable at request level via `request.signal` either way.
- **(framework-side) Streaming pipelines omit `must-revalidate` from `Cache-Control`.** Both `ssr-streaming/cache-policies.ts` and `ssr-rsc/cache-policies.ts` ship policies without `must-revalidate` (verified rule-by-rule); the inline footnote in each file explains the related reason — streamed bodies cannot be ETag'd in-flight (buffering for hashing defeats streaming), so freshness validation is delegated to the CDN edge layer (which buffers + ETags at its own boundary). Inherent streaming trade-off. Self-hosted setups without an edge cache should add `must-revalidate` themselves.

## RSC specifics

What's unique to `ssr-rsc/`, expressed in terms of Real-Router integration rather than RSC theory:

- **`@real-router/rsc-server-plugin` is bundler-agnostic.** The plugin emits a `ReactNode` to `state.context.rsc`; `entry.rsc.tsx` decides whether to ship that payload as raw Flight (`/__rsc?route=…`, `Content-Type: text/x-component`) or pipe it through SSR + `injectRSCPayload` for the initial HTML load. The plugin itself does not know about `@vitejs/plugin-rsc` — it only knows about `state.context`.
- **Two endpoints, one fetch handler.** `entry.rsc.tsx`'s `default { fetch }` owns request routing. Express is reduced to a thin Web↔Node bridge (`expressToFetchRequest`), reusable between dev and prod servers. `serverHandler: false` on `@vitejs/plugin-rsc` is what disables the bundler's own request handler — this is the integration seam.
- **Single source of truth for client navigation.** `App.tsx`'s `router.subscribe` listener owns every `/__rsc` re-fetch. `<Link>` clicks, `router.navigate({ reload: true })`, and `popstate` (history) all flow through one place. Each fetch carries its own `AbortController`; when navigation B starts, navigation A is aborted so a slow A response cannot land in the DOM after B has rendered (race-safety verified by Scenario 11).
- **Server Actions, with router-aware result plumbing.** Real-Router does not implement Server Actions, but `rscActionPluginFactory(getResult)` extends them with a cross-component result mechanism: action results land in `state.context.rscAction`, any Server Component anywhere in the layout can read it (e.g. `<NotificationBanner>` rendered as a sibling of the page Server Component). The transport-to-tree-update bridge is a `rsc:server-action-response` `CustomEvent` — explicitly chosen over a global mutable `setNode` reference for decoupling.
- **Loader-error parity.** `@real-router/rsc-server-plugin/errors` is the deliberate mirror of `@real-router/ssr-data-plugin/errors`; application catch-paths look identical regardless of which pipeline a route uses.

## Empirical findings

Findings that the four READMEs flag as non-obvious, empirically-verified, or footgun-class — distilled here so adapter-shopping readers don't have to re-discover them.

1. **Selective hydration in React 19 has a visible wire signature, and it can be asserted in e2e.** `ssr-streaming/` Scenario 17 looks for `<!--$?-->` Suspense placeholder markers alongside `<template id="B:0">` resolved-content tags in the streamed bytes; Scenario 18 does byte-offset analysis to prove `reviews-section` (600 ms server delay) appears before `related-section` (1200 ms) — out-of-order completion in HTTP byte order. In this monorepo only Solid's `renderToStream` produces an equivalent wire signature; Vue 3 stable, Preact 10 (without `lazy()`), and Svelte 5 do not.

2. **Per-render memoised promises, not module-level caches, are the correct streaming idiom.** `ssr-streaming/`'s `Reviews` component uses `useMemo(() => fetchReviews(productId), [productId])` so a fresh promise is constructed per request. The naive alternative — `React.lazy`'s singleton cache — would make streaming visible only on the first request and then deliver from cache, defeating the demo. This is documented as the canonical shape, not a workaround.

3. **`getStaticPaths` returns leaves only — by design.** `ssg/` derives intermediate parent paths in user code (suffix-strip `/posts` → `/users/:id`; add `/users` for the list page manually). The 8-path output for the demo is `1 home + 1 list + 3 profiles + 3 posts`, not "every URL the tree could produce". Easy to mis-spec by assuming the function is recursive.

4. **`createPortal` + SSR + hydration requires a `mounted` gate.** `ProductSpecsModal` defers portal rendering until `useEffect` flips `mounted = true`, otherwise hydration walker mismatches. SSR ships only the trigger button; the portal target node lives in `index.html` as a sibling of `#root`. Verified by Scenarios 15-16 in `ssr-streaming/`.

5. **Race-safe client navigation requires per-fetch `AbortController` in RSC.** Without it, rapid A→B `<Link>` clicks let A's older `/__rsc` response land in the DOM after B has rendered. `App.tsx:53-103` (the navigation `useEffect`) aborts the previous controller before issuing the next fetch. Verified by Scenario 11 in `ssr-rsc/`.

6. **HTTP envelope diverges by pipeline shape, not by accident.** The four examples ship deliberately different ETag / `must-revalidate` / `AbortController-in-DI` matrices because of inherent constraints (streaming can't be ETagged in-flight) and intentional contracts (RSC delegates revalidation to the CDN edge). Cross-pipeline copy-paste of `cache-policies.ts` is fine; cross-pipeline copy-paste of `server/index.ts` is not.
