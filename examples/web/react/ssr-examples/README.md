# React SSR Examples

> Server-rendering with Real-Router and React 19 across five delivery models — classical SSR, HTTP-streaming SSR, static site generation, React Server Components, and hybrid per-route mode.

Five standalone Vite apps. Each one is a real, e2e-tested dogfooding consumer of `@real-router/*` for a specific server-rendering shape; this document is the synthesis of what the router can and cannot do, sourced exclusively from the five child READMEs.

| Subdir | Demonstrates |
|---|---|
| [`ssr/`](./ssr) | Classical per-request `createRequestScope(req, baseRouter, { currentUser })` (from `@real-router/core/utils` — clones base, attaches `AbortSignal` to `req.on("close")`, injects `abortSignal` into the deps map, returns `dispose()`) → `usePlugin(ssrDataPluginFactory)` → `start(url)` → `renderToString` → `serializeRouterState` → `hydrateRouter`. Auth + role gates via `canActivate` + DI; typed loader errors (`LoaderRedirect`/`LoaderNotFound`/`LoaderTimeout` + `withTimeout`) → 301/302/404/504; **render-time HTTP status via `<HttpStatusCode code={404}/>` + `<HttpStatusProvider>` + `createHttpStatusSink()` from `@real-router/react/ssr`** — `NotFound` page declares its own status, `entry-server` reads `sink.code ?? 200` after `renderToString` (replaces the prior `state.name === UNKNOWN_ROUTE ? 404 : 200` server-side check); **SSR-boundary primitives `<ClientOnly>` + `<ServerOnly>` from `@real-router/react/ssr` (#604) dogfooded on the Home page** (e2e/ssr-boundaries.spec.ts asserts JS-disabled HTML shows the server branch and post-hydration DOM swaps both without `console.error`); per-route meta with absolute canonical + og; production HTTP: sha256 strong **ETag** (16 base64url chars), per-route `Cache-Control`, `AbortController` auto-injected into loader DI via `createRequestScope` so `getDep("abortSignal")` works inside loaders. `useId()` proves SSR/CSR id parity. |
| [`ssr-streaming/`](./ssr-streaming) | React 19 native streaming: `renderToReadableStream` + `<Suspense>` + `use(promise)` + selective hydration. `ssr-data-plugin` provides critical data via `defer({ critical, deferred })` — the formal `@real-router/ssr-data-plugin` API (not an application-level convention); deferred promises stream in via inline `<script>__rrDefer__("key", json)</script>` settle scripts emitted by `injectDeferredScripts` (`@real-router/ssr-data-plugin/server`). Consumers read via `<Await name>` + `<Streamed>` + `useDeferred()` from `@real-router/react/ssr`. Production HTTP: per-route `Cache-Control`, `AbortController` (via `createRequestScope`), **no ETag** (intentional — see Limitations). |
| [`ssg/`](./ssg) | Build-time pre-rendering: `getStaticPaths(router, entries)` → `cloneRouter` + `start` + `renderToString` per URL → write `dist/{url}/index.html` (`createRequestScope` is not used at build time — no per-request lifecycle to wire). Nested SSG (`users.profile.posts` leaves + derived `users.profile` parents + manually added `/users` → 8 paths total), empty-state path, absolute canonical + og, filesystem-layout assertion, `404.html`, `sitemap.xml`. `LoaderNotFound` at build time fails the build loudly. Preview pipeline ships per-route `Cache-Control` (custom `ssgServe()` Vite middleware in `vite.config.ts`) + weak `ETag` from file mtime. |
| [`ssr-rsc/`](./ssr-rsc) | React Server Components with `@real-router/rsc-server-plugin` (ReactNode payload variant) + `@vitejs/plugin-rsc`. Two-endpoint architecture: `GET /:path` (HTML + injected Flight) and `GET /__rsc?route=…` (raw Flight) flow through one `entry.rsc.tsx` fetch handler. Web `Request`-shaped scope via `createRequestScope(request, baseRouter, { db: database })` (reads `request.signal` directly, no Node `req.on("close")` needed). Race-safe client navigation: per-fetch `AbortController` cancels stale in-flight `/__rsc` requests. |

Composes with #596: hydration scratchpad reuse for `full`/`data-only`; `client-only` skips loader unconditionally. |

## Capabilities

What Real-Router actually delivers across these five shapes, grouped by concern. Each item is sourced from one or more child READMEs.

**Per-request isolation.** Runtime examples (`ssr/`, `ssr-streaming/`, `ssr-rsc/`, `ssr-mixed/`) use `createRequestScope(req|request, baseRouter, deps?)` from `@real-router/core/utils` — the four-step boilerplate replacement that produces an isolated router clone, attaches an `AbortSignal` to the request's lifetime (Node `IncomingMessage#"close"` event or Web `Request.signal`), injects `abortSignal` into the deps map, and exposes `dispose()` for `finally`/`Symbol.asyncDispose`. `ssg/` calls `cloneRouter(base)` directly — at build time there's no per-request lifecycle to wire. All scopes are disposed in `finally` so a typed-error short-circuit, a streaming pump exit, or a client disconnect never leaks state. Verified by concurrent-load e2e in each runtime example — `ssr/` runs 10 concurrent `/users/:id` requests, `ssr-streaming/` runs 9 concurrent loads (Scenario 9), and `ssr-rsc/` Scenario 5 runs 10 concurrent `/users/{0..9}`.

**Two data-loading plugins, one shape.** `@real-router/ssr-data-plugin` writes plain JSON-shaped data into `state.context.data` for `ssr/`/`ssr-streaming/`/`ssg/`/`ssr-mixed/`. `@real-router/rsc-server-plugin` writes ReactNode payloads into `state.context.rsc` for `ssr-rsc/`. Both are start-interceptor plugins; both ship a typed-errors subpath (`/errors`) with the same `LoaderNotFound`/`LoaderRedirect`/`LoaderTimeout` shapes — explicitly mirrored in the RSC plugin so application catch-paths are identical regardless of pipeline shape.

**State serialization + hydration.** `serializeRouterState()` is XSS-safe; the resulting blob lands in `window.__SSR_STATE__`. `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad on `RouterInternals.hydrationState` and calls `router.start(state.path)` once on the client; `ssr-data-plugin`'s start interceptor reads the scratchpad and reuses the server-resolved `state.context.data` directly, skipping the loader call (#596). The hydration component tree sees the same `state.context.data` / `state.context.rsc` the server rendered. SSG auto-detects hydrate vs `createRoot` via `rootElement.firstElementChild` so the same `entry-client` works for SSG-built pages and Vite dev fallbacks.

**Typed HTTP error model.** Loader errors have explicit codes that map to HTTP responses (`ssr/src/entry-server.tsx` `catch (error)` dispatch block → `server/index.ts` HTTP envelope): `CANNOT_ACTIVATE` → 302, `LOADER_REDIRECT` → 301/302, `LOADER_NOT_FOUND` → 404 text/plain, `LOADER_TIMEOUT` → 504 text/plain. The same shape applies on both endpoints of the RSC dual-endpoint architecture (`/users/9999` and `/__rsc?route=/users/9999` both yield 404 text/plain — Scenarios 16a/16b). At build time in SSG, an unexpected `LoaderNotFound` aborts the build with a non-zero exit instead of silently emitting a "user not found" page for a stale `entries.ts` id.

**Render-time HTTP status (`<HttpStatusCode code={N}/>`).** Complementary to the loader-driven model: when the status is decided by the *rendered component* (e.g. a glob `*` route → NotFound page) rather than by a loader exception, mount `<HttpStatusCode>` from `@real-router/react/ssr`. The component reads the optional `<HttpStatusProvider sink={…}>` context and writes `code` to the per-request sink; `entry-server.tsx` reads `sink.code ?? 200` after `renderToString` to set the response status. Returns `null` (no DOM); silent no-op on the client (no provider mounted by default → `useContext` returns null). Demonstrated in `ssr/`'s `NotFound.tsx` and verified by 3 dedicated e2e tests (render-time path, no phantom 404 leak across requests, hydration silence). The streaming, SSG, and RSC examples intentionally do NOT dogfood `<HttpStatusCode>` — `ssg/` builds 404 at build-time via `404.html`; `ssr-streaming/` and `ssr-rsc/` short-circuit unknown routes via `LoaderNotFound` before any rendering. The component covers a different orthogonal concern (render-time decision), and overloading the streaming/RSC demos with it would muddy them.

**Build-time path enumeration.** `getStaticPaths(router, entries)` walks the route tree and returns **leaf** paths only. Application code derives intermediate parent URLs from the leaves (e.g. `/users/:id` from `/users/:id/posts` by suffix-stripping) and adds purely-parent pages manually. The 8-URL output set in `ssg/` is the empirical demonstration.

**Per-route HTTP semantics, per pipeline.** Four of five examples drive `Cache-Control` from a single `cache-policies.ts` lookup (`ssr-mixed/` is a mode-shape demo without cache wiring). The HTTP envelope differs by pipeline:

| Pipeline | `Cache-Control` | `ETag` | `AbortController` |
|---|---|---|---|
| `ssr/` (renderToString) | per-route, `must-revalidate` | strong sha256 (16 base64url chars) | `createRequestScope(req, baseRouter, { currentUser })` auto-injects `abortSignal` into deps; `slow` loader reads it via `getDep("abortSignal")` and composes with `withTimeout({ upstreamSignal })` |
| `ssr-streaming/` (renderToReadableStream) | per-route, **no `must-revalidate`** | **none, intentional** | `createRequestScope(req, baseRouter)` auto-injects `abortSignal`; aborts stream-pump on `req.on("close")`; current loaders don't read `getDep("abortSignal")` |
| `ssg/` (build-time) | per-route via `ssgServe()` Vite middleware (inline `CACHE_RULES` in `vite.config.ts`) | weak, derived from file mtime by Vite preview | not applicable (pre-rendered) |
| `ssr-rsc/` (Flight + injected HTML) | per-route on **both** endpoints, **no `must-revalidate`** | **none, intentional** | `createRequestScope(request, baseRouter, { db })` reads `request.signal` directly (Web `Request`-shaped); `abortSignal` is auto-injected into deps but current loaders don't read it |
| `ssr-mixed/` (per-route mode) | not wired (mode-shape demo) | not wired | `createRequestScope(req, baseRouter)` auto-injects `abortSignal`; `home` loader reads it via `SsrLoaderContext.signal` for #605 cancel-safety |

**Auth + role gates.** `canActivate` guards read DI via `getDep(...)`; `createRequestScope(req, baseRouter, { currentUser })` is the injection seam — `currentUser` lands in the deps map and `abortSignal` is added alongside it automatically. The post-hydration client must mirror server-side DI logic over `document.cookie` or the client guard diverges — handled in `entry-client.tsx`. `/dashboard` requires `currentUser`; `/admin` adds `role === "admin"`.

**Per-route SSR mode (#597).** `@real-router/ssr-data-plugin` accepts a per-route `{ ssr?, loader? }` config that branches the server pipeline by `SsrMode`: `"full"` (run loader + render), `"data-only"` (run loader, ship JSON only, shell HTML), `"client-only"` (skip loader, ship shell, application fetches client-side). `ssr: false` aliases `"client-only"`; `ssr: true` aliases `"full"`. Function form `(state) => SsrMode` resolves per-navigation. The plugin publishes the resolved mode to `state.context.ssrDataMode`; `entry-server.tsx` reads it via `getSsrDataMode(state)` to branch between `renderToString(<App/>)` (full) and a shell `<div data-ssr-shell data-ssr-mode="…">` placeholder. The `ssr-mixed/` example demonstrates all four config shapes — short form + boolean alias + string-form + function-form — on one server. Composes with #596: hydration scratchpad reuse applies when mode allows the loader; `client-only` skips the loader unconditionally regardless of scratchpad contents (mode wins).

**RSC integration without RSC-specific routing.** `rsc-server-plugin` writes a complete `ReactNode` per route into `state.context.rsc`. The router still uses the same `start(url)` lifecycle, the same loader error contract, the same per-request scope (`createRequestScope(request, …)` over the Web `Request` shape — `request.signal` covers client-disconnect, no Node listener needed). The client-side single-listener pattern (`router.subscribe → fetch /__rsc`) carries `<Link>` clicks, programmatic `router.navigate({ reload: true })`, and `popstate` (history) through one code path.

**Server Actions, end-to-end.** Real-Router does not own Server Actions — they are React 19 + `@vitejs/plugin-rsc` primitives — but the example proves the integration: `decodeAction(formData)` for the no-JS path (progressive enhancement via `$ACTION_REF_*` form fields), `loadServerAction(id)` + `decodeReply(body)` for the with-JS path, plus `createTemporaryReferenceSet()` so non-serialisable references survive the round trip. The cross-component result pattern is router-flavoured: `rscActionPluginFactory(getResult)` publishes the action result to `state.context.rscAction`, which any Server Component (e.g. `<NotificationBanner>` in any layout) can read without prop-drilling from the form.

**SEO/head plumbing.** Per-route `meta` resolver returns a `PageMeta` block — `{ title, description, canonical, ogTitle, ogDescription }` (5 fields; `og:url` is rendered from `canonical`, not stored separately). The server renders the block as 6 tags and splices the result into a `<!--ssr-meta-->` placeholder. Canonical URLs are absolute (`SITE_ORIGIN`-prefixed, default `https://example.com`) — relative canonicals are rejected by crawlers. SSG additionally emits `sitemap.xml` from the same URL set, with a filesystem-layout assertion that catches drift between sitemap, on-disk HTML, and `entries.ts`.

## Limitations

Each item is categorised: **(framework-side)** — React/streaming/RSC constraint not under Real-Router's control; **(design choice)** — deliberate Real-Router (or plugin) omission with documented rationale, often closeable by user-side composition; **(example design choice)** — per-example scope narrowing or simplification, not a library/adapter constraint.

- **(design choice) `ssr-data-plugin` (boot-path) and `lifecycle-plugin onNavigate` (CSR navigation) split the lifecycle by phase, on purpose.** `ssr-data-plugin` intercepts `router.start()` — runs once during SSR; on the client, `hydrateRouter(router, ssrState)` deposits the parsed state into a one-shot scratchpad and the plugin reuses the server-resolved `state.context.data` directly (#596) instead of calling the loader again. CSR navigations via `<Link>` and `router.navigate()` do NOT re-run its loader; that is `lifecycle-plugin.onNavigate`'s job, and the canonical CSR+SSR stack composes both plugins. Single-responsibility per plugin keeps SSG-only build scripts from carrying navigation-interception code, and pure-CSR SPAs from carrying start-path interception code. The RSC example sidesteps both with a third application-level pattern — re-fetching `/__rsc?route=…` on every navigation through a single `router.subscribe` listener (no plugin needed for that path). One-off escape hatches that don't change the architecture: `router.navigate(name, params, search, { reload: true })`, or a layered CSR fetcher (TanStack Query, SWR).
- **(design choice) No router-specific streaming API beyond `defer()` + `<Await>` + `<Streamed>`.** Streaming behaviour comes from React 19's `renderToReadableStream` + `<Suspense>` + `use(promise)`; the router's contribution is the cross-adapter `defer({ critical, deferred })` loader-return contract and the `__rrDefer__("key", json)` NDJSON wire format (so consumers can pick between `<Await>`/`<Streamed>` adapter components and the inline `use(useDeferred("key"))` form — both reduce to the same `<Suspense>` + `use(promise)` underneath). Application code is never expected to invent its own wire shape — that's the point of formalising `defer()`. See `ssr-streaming/README.md` "Library Philosophy" for the contract.
- **(framework-side) No ETag for streamed responses.** Both `ssr-streaming/` and `ssr-rsc/` document this as inherent: hashing the body requires buffering, which defeats streaming (TTFB jumps to "after the slowest deferred section resolves"). The expectation is that production CDNs add a buffered ETag layer at the edge while the origin still benefits from chunked transfer.
- **(framework-side) SSR cannot render `createPortal`.** The target DOM node does not exist in a string render. `ssr-streaming/` uses a `mounted` state gate (`useState(false)` + `useEffect(() => setMounted(true), [])`) so the SSR pass ships only the trigger button and hydration completes without mismatch. Application-level pattern, not a router concern, but documented because it is the React-19 + streaming-SSR canonical shape.
- **(framework-side) `getStaticPaths` returns leaf paths only.** Application code in `ssg-build.ts` derives intermediate parent URLs (e.g. `/users/:id` from `/users/:id/posts`) and adds purely-parent pages (e.g. `/users` for `UsersList`) manually. Documented as a deliberate API shape — the alternative (returning every path) would make the function order-sensitive and ambiguous for routes that have both a leaf component and children.
- **(framework-side) `withTimeout()` cancellation is cooperative.** When the deadline elapses, the loader's `{ signal }` aborts *before* the race rejects with `LoaderTimeout`; loaders that thread `signal` into their I/O — e.g. `fetch(url, { signal })` — actually cancel the underlying work (proven end-to-end by the `withTimeout (#598) network cancellation` test in `ssr/`). Loaders that don't propagate the signal still run to completion in the background. The `slow` loader in `ssr/` composes the deadline with `options.upstreamSignal` (client-disconnect from per-request DI; `createRequestScope` auto-injects `abortSignal` into the deps map).
- **(example design choice) RSC example's loaders don't observe `abortSignal`.** `createRequestScope` auto-injects `abortSignal` into the deps map for every runtime example, including `ssr-rsc/`; the RSC example's loaders (`users.profile` reads `getDep("db")` only) simply don't call `getDep("abortSignal")` today. Wiring it in is a one-line read — no library change needed. The signal is also observable at request level via `request.signal` either way.
- **(framework-side) Streaming pipelines omit `must-revalidate` from `Cache-Control`.** Both `ssr-streaming/src/router/cache-policies.ts` and `ssr-rsc/src/router/cache-policies.ts` ship policies without `must-revalidate` (verified rule-by-rule); the inline footnote in each file explains the related reason — streamed bodies cannot be ETag'd in-flight (buffering for hashing defeats streaming), so freshness validation is delegated to the CDN edge layer (which buffers + ETags at its own boundary). Inherent streaming trade-off. Self-hosted setups without an edge cache should add `must-revalidate` themselves.

## RSC specifics

What's unique to `ssr-rsc/`, expressed in terms of Real-Router integration rather than RSC theory:

- **`@real-router/rsc-server-plugin` is bundler-agnostic.** The plugin emits a `ReactNode` to `state.context.rsc`; `entry.rsc.tsx` decides whether to ship that payload as raw Flight (`/__rsc?route=…`, `Content-Type: text/x-component`) or pipe it through SSR + `injectRSCPayload` for the initial HTML load. The plugin itself does not know about `@vitejs/plugin-rsc` — it only knows about `state.context`.
- **Two endpoints, one fetch handler.** `entry.rsc.tsx`'s `default { fetch }` owns request routing. Express is reduced to a thin Web↔Node bridge (`expressToFetchRequest`), reusable between dev and prod servers. `serverHandler: false` on `@vitejs/plugin-rsc` is what disables the bundler's own request handler — this is the integration seam.
- **Single source of truth for client navigation.** `App.tsx`'s `router.subscribe` listener owns every `/__rsc` re-fetch. `<Link>` clicks, `router.navigate({ reload: true })`, and `popstate` (history) all flow through one place. Each fetch carries its own `AbortController`; when navigation B starts, navigation A is aborted so a slow A response cannot land in the DOM after B has rendered (race-safety verified by Scenario 11).
- **Server Actions, with router-aware result plumbing.** Real-Router does not implement Server Actions, but `rscActionPluginFactory(getResult)` extends them with a cross-component result mechanism: action results land in `state.context.rscAction`, any Server Component anywhere in the layout can read it (e.g. `<NotificationBanner>` rendered as a sibling of the page Server Component). The transport-to-tree-update bridge is a `rsc:server-action-response` `CustomEvent` — explicitly chosen over a global mutable `setNode` reference for decoupling.
- **Loader-error parity.** `@real-router/rsc-server-plugin/errors` is the deliberate mirror of `@real-router/ssr-data-plugin/errors`; application catch-paths look identical regardless of which pipeline a route uses.

## Empirical findings

Findings that the five READMEs flag as non-obvious, empirically-verified, or footgun-class — distilled here so adapter-shopping readers don't have to re-discover them.

1. **Selective hydration in React 19 has a visible wire signature, and it can be asserted in e2e.** `ssr-streaming/` Scenario 17 looks for `<!--$?-->` Suspense placeholder markers alongside `<template id="B:0">` resolved-content tags in the streamed bytes; Scenario 18 does byte-offset analysis to prove `reviews-section` (600 ms server delay) appears before `related-section` (1200 ms) — out-of-order completion in HTTP byte order. In this monorepo only Solid's `renderToStream` produces an equivalent wire signature; Vue 3 stable, Preact 10 (without `lazy()`), and Svelte 5 do not.

2. **Per-request promises must be constructed inside the loader factory, not module-level.** `ssr-streaming/`'s loader returns `defer({ critical: { product }, deferred: { reviews: fetchReviews(id), related: fetchRelated(id) } })` — `fetchReviews(id)` is called fresh on every request, so each request's `state.context.ssrDataDeferred.reviews` is its own promise. A module-level singleton (e.g. caching the promise outside the loader factory) would resolve once on the first request, then deliver from cache for all subsequent requests — invisible streaming on every request after the first. The `<Await name="reviews">` consumer reads `state.context.ssrDataDeferred[name]` and feeds it to `use(promise)`; the live promise has to be there per request for `<Suspense>` to suspend.

3. **`getStaticPaths` returns leaves only — by design.** `ssg/` derives intermediate parent paths in user code (`/users/:id/posts` is the leaf; suffix-strip `/posts` → `/users/:id`; add `/users` for the list page manually). The 8-path output for the demo is `1 home (/) + 3 leaf posts + 3 derived profiles + 1 manual /users`, not "every URL the tree could produce". Easy to mis-spec by assuming the function is recursive.

4. **`createPortal` + SSR + hydration requires a `mounted` gate.** `ProductSpecsModal` defers portal rendering until `useEffect` flips `mounted = true`, otherwise hydration walker mismatches. SSR ships only the trigger button; the portal target node lives in `index.html` as a sibling of `#root`. Verified by Scenarios 15-16 in `ssr-streaming/`.

5. **Race-safe client navigation requires per-fetch `AbortController` in RSC.** Without it, rapid A→B `<Link>` clicks let A's older `/__rsc` response land in the DOM after B has rendered. `ssr-rsc/src/App.tsx`'s navigation `useEffect` (the `router.subscribe` listener) aborts the previous controller before issuing the next fetch. Verified by Scenario 11 in `ssr-rsc/`. The same race-safety pattern surfaces inside `@real-router/ssr-data-plugin` as #605 (cancel-safe loader contract) — `ssr-mixed/` exercises it via rapid double-click on `invalidate(router, "data")` (3 dedicated e2e tests).

6. **HTTP envelope diverges by pipeline shape, not by accident.** The four pipelines that wire HTTP (`ssr/`, `ssr-streaming/`, `ssg/`, `ssr-rsc/`) ship deliberately different ETag / `must-revalidate` / `AbortController-in-DI` matrices because of inherent constraints (streaming can't be ETagged in-flight) and intentional contracts (RSC delegates revalidation to the CDN edge). Cross-pipeline copy-paste of `cache-policies.ts` is fine; cross-pipeline copy-paste of `server/index.ts` is not. `ssr-mixed/` deliberately omits HTTP wiring — it's a mode-shape demo.

7. **`createRequestScope` replaces the four-step boilerplate; build-time SSG sticks with `cloneRouter`.** Every runtime example now reduces to `const scope = createRequestScope(req, baseRouter, deps); scope.router.usePlugin(...); try { ... } finally { await scope.dispose(); }` — `AbortController + req.on("close") + cloneRouter + deps-injection` collapse into one call. `ssg-build.ts` keeps `cloneRouter(baseRouter)` because at build time there's no request to listen on. New SSR adapters should reach for `createRequestScope` first; reach back for `cloneRouter` only when there's no request-lifetime signal available (e.g., a build script, or running in a Worker after `request.signal` is already wired upstream).
