# SSR / SSG Comparison — @real-router/angular vs @angular/router (+ @angular/ssr)

> Authoritative source for Real-Router capabilities: [`README.md`](./README.md) (synthesised from the four child READMEs in [`ssr/`](./ssr), [`ssr-streaming/`](./ssr-streaming), [`ssg/`](./ssg), [`ssr-mixed/`](./ssr-mixed)).
> Competitor data sourced from official Angular docs accessed via Context7 on **2026-05-06**:
> - `@angular/router` API: https://angular.dev/api/router, https://angular.dev/api/router/ResolveFn, https://angular.dev/api/router/CanActivateFn, https://angular.dev/api/router/Routes
> - `@angular/ssr`: https://angular.dev/guide/ssr, https://angular.dev/api/ssr
> - Hydration: https://angular.dev/guide/hydration, https://angular.dev/guide/hydration#incremental-hydration
> - Deferrable views: https://angular.dev/guide/templates/defer
> - `provideClientHydration`: https://angular.dev/api/platform-browser/provideClientHydration, `provideZonelessChangeDetection`: https://angular.dev/api/core/provideZonelessChangeDetection

This comparison is restricted to **standalone** routers. The natural Angular counterpart is `@angular/router` paired with `@angular/ssr` — both are first-party Angular packages but do not constitute a meta-framework: they ship as libraries that the user assembles into an `@angular/build:application` build target, just as `@real-router/angular` does. Analog (Angular meta-framework) is excluded from this comparison.

## TL;DR

`@angular/router` is the framework's own router with the deepest possible Angular integration — `ResolveFn`/`CanActivateFn` data resolvers and guards, `RenderMode.Prerender` working out of the box, type-aware `ActivatedRouteSnapshot`/`paramMap`, official `TestBed`/`RouterTestingHarness`, and a decade of ecosystem integrations. **Real-Router on Angular reuses the same `@angular/ssr` render pipeline** (`AngularNodeAppEngine.handle(req)`, `provideClientHydration(withIncrementalHydration(), withEventReplay())`, `@defer` blocks) and **adds**: a typed loader-error → HTTP envelope (`LoaderRedirect`/`LoaderNotFound`/`LoaderTimeout` → 301/302/404/504), a build-time path-enumeration contract (`getStaticPaths` leaf-only), an in-process SSG pipeline that produces byte-identical SSG/runtime output, two complementary HTTP-status strategies (declarative `ServerRoute` config + middleware-mapped typed errors), and the cross-adapter consistency that the same `ssr-data-plugin` shape works on React/Vue/Solid/Svelte/Preact too. **Where `@angular/router` wins**: native Angular DX (rxjs-aware resolvers, full guard taxonomy `CanActivate`/`CanDeactivate`/`CanLoad`/`CanMatch`/`CanActivateChild`), `RenderMode.Prerender` working without an in-process server workaround, `TitleStrategy`, scroll restoration, the depth of the official ecosystem. **Where Real-Router wins**: typed loader-error → HTTP envelope, view-agnostic plugin protocol, build-time SSG enumeration with optional fail-fast contract, two HTTP-status strategies side-by-side. Angular 21 framework limits (no progressive HTTP flush, body in 1 TCP frame, per-`@defer` selective hydration only) constrain both routers identically.

## Where Real-Router wins

- **Typed loader-error → HTTP envelope, with two complementary strategies side-by-side.** [`README.md` "Two complementary HTTP-status strategies"](./README.md#capabilities) — declarative `ServerRoute` config pins status/headers at routing time (rich SSR body still rendered for `path: "gone"` → 410 with `Sunset`/`Deprecation`/`Link` headers); typed loader errors caught by Express middleware map data-driven failures (`LoaderRedirect`/`LoaderNotFound`/`LoaderTimeout`) to 301/302/404/504. Both are verified side-by-side in one e2e test. `@angular/router` exposes `ServerRoute` `status`/`headers` and `RedirectCommand` for the redirect path, but does not provide the typed-error contract for arbitrary HTTP statuses — the application has to roll its own error → status mapper.
- **Cross-adapter consistency.** The same `ssr-data-plugin` loader shape works on React, Vue, Solid, Svelte, Preact, and Angular — a team migrating between adapters or running multiple frontends keeps the same loader contract, the same typed-error subpath, the same `getStaticPaths` enumeration. `@angular/router`'s `ResolveFn` is Angular-only by design.
- **Build-time path enumeration with leaf-only contract.** `getStaticPaths(router, entries)` walks the route tree and returns leaves only; `ssg-build.ts` derives parent URLs in user code (8 pre-rendered URLs in the demo: 1 home + 1 list + 3 profiles + 3 posts). Optional fail-fast guard (`process.exit(1)` on any throw) is wired but dormant under tolerant loaders by example design — opting into the strict `LoaderNotFound` path activates it without library changes.
- **In-process SSG pipeline that is byte-identical to runtime SSR.** `scripts/ssg-build.ts` boots the compiled `server.mjs` on a build-only port and `fetch`-es each URL through the live `AngularNodeAppEngine`. Same render pipeline as production, no `renderApplication` direct (which produces NG0201 with `provideRealRouterFactory`). `@angular/ssr`'s `RenderMode.Prerender` works directly but goes through a different code path than runtime SSR — divergence is theoretically possible in long-lived deployments; Real-Router's choice removes that surface area.
- **Public plugin protocol with conditional server/client registration.** `claimContextNamespace` + `usePlugin` + `addInterceptor` is the public extension surface; the function form of `provideRealRouterFactory` `plugins` lets the same factory branch on `REQUEST` nullness so server registers `ssrDataPluginFactory` only, client registers `browserPluginFactory` + `ssrDataPluginFactory`. `@angular/router` has guards/resolvers/`RouteReuseStrategy` as router extension points but no plugin protocol layered above.
- **Per-pipeline HTTP envelope is documented, not accidental.** The `Cache-Control` / `ETag` / `AbortController` table in [`README.md`](./README.md#capabilities) — strong sha256 ETag for buffered SSR (response is buffered via `arrayBuffer()` for hashing), narrow envelope for `ssr-streaming/` (intentionally), sirv defaults for SSG preview. `@angular/router` does not own HTTP concerns; the application chooses.
- **Per-request `AbortController` threaded into loader DI + composed via `withTimeout`.** `cloneRouter(base, { abortSignal })` + `getDep("abortSignal")` + `withTimeout(..., { upstreamSignal })` (#598) gives the `slow` loader in `ssr/` a single composed `AbortSignal` that fires on deadline or client disconnect; threaded into `fetch(..., { signal })` it cancels at the network layer. `ResolveFn` does not receive an `AbortSignal` in its signature (Angular 21).
- **Data-driven per-route SSR mode (#597).** `ssrDataPluginFactory` accepts `{ ssr?, loader? }` per route with `SsrMode = "full" | "data-only" | "client-only"`, the `true`/`false` boolean aliases, and a function form `(state) => SsrMode` resolving per-navigation against `state.params` / `state.path` / `state.name`. `@angular/router`'s closest equivalent is `ServerRoute` `renderMode`, which is **path-based only** — `RenderMode.Client` for `/marketing` is decided by URL pattern at routing time, not by request-specific data. Real-Router's `docs.detail` route in `ssr-mixed/` decides between `"full"` and `"client-only"` based on `state.params.format === "pdf"` (a query parameter resolved by routing); Angular cannot express this with `ServerRoute` alone. The Angular integration uses an Express middleware before `AngularNodeAppEngine.handle(req)` for branching — a different shape than React/Vue/Solid/Svelte/Preact `ssr-mixed` examples (which branch inside `entry-server`), explained as a deliberate architectural choice in the child README.

## Where @angular/router wins

- **`RenderMode.Prerender` works out of the box.** `app.routes.server.ts` declares `{ path: '/posts/:id', renderMode: RenderMode.Prerender, getPrerenderParams: …}` and the build runs the prerender pass automatically. Real-Router's example documents that `RenderMode.Prerender` is **not supported** with `provideRealRouterFactory` (REQUEST is null in the prerender pipeline → every URL falls back to `"/"`), and works around it with the in-process SSR pipeline. The workaround produces byte-identical output but adds the `outputMode: "server"` + `ssr.entry` overhead even for SSG-only deployments.
- **Native data resolvers with rxjs.** `ResolveFn<T>: (route, state) => MaybeAsync<T>` accepts `T` / `Promise<T>` / `Observable<T>` and integrates with the entire Angular reactive ecosystem — `HttpClient`, `toSignal()`, `takeUntilDestroyed()`. Real-Router's `ssr-data-plugin` loaders are plain async functions, idiomatic but not rxjs-native.
- **Full guard taxonomy.** `CanActivate` / `CanActivateChild` / `CanDeactivate` / `CanLoad` / `CanMatch` each address a different lifecycle moment with native typing and `RedirectCommand` support. Real-Router has `canActivate` (the most common one) and the `cloneRouter` DI seam; the equivalents of `CanDeactivate` (unsaved-changes prompts) and `CanMatch` (conditional route matching) are application-level.
- **Tighter Angular ecosystem integration.** `RouterModule.forRoot()` / `provideRouter()` is the official surface; `TestBed` + `RouterTestingHarness` cover testing patterns; `TitleStrategy` covers per-route titles; `RouterLink` / `RouterLinkActive` / `RouterOutlet` are the documented templating primitives; scroll restoration is a `withInMemoryScrolling()` feature on `provideRouter`. Real-Router on Angular ships `provideRealRouterFactory` + `<route-view>` + `<a realLink>` + `injectRoute` — equivalent in capability, smaller in third-party tooling.
- **`provideRouter` features compose deeply.** `withInMemoryScrolling`, `withViewTransitions`, `withDebugTracing`, `withDisabledInitialNavigation`, `withRouterConfig`, `withComponentInputBinding`, `withHashLocation`, `withPreloading` — each is a one-import opt-in. Real-Router covers similar ground through dedicated plugins (`@real-router/scroll-restoration`, `@real-router/preload-plugin`, etc.) but the API surface is per-plugin rather than per-feature flag.
- **Lazy-loaded children.** `loadChildren: () => import('./admin/admin.module').then(m => m.AdminModule)` is the documented one-liner for code-splitting an entire route subtree. Real-Router has `@defer` + `loadComponent`-style patterns in the components, but `@angular/router`'s lazy-children primitive is denser.
- **Maturity and community.** `@angular/router` is the framework-author's own router with a decade of releases, broad coverage of edge cases (named outlets, secondary routes, query-param-based navigation, `RouteReuseStrategy` for tab-style apps), and large-scale production use.

## Where they tie

- **Same Angular 21 framework floor.** No progressive HTTP flush — body delivered in 1 TCP frame regardless of router; per-`@defer` block selective hydration via `withIncrementalHydration()` available on both; `withEventReplay()` captures clicks during slow chunk hydration on both. Empirical findings 1, 2, 7, 8 in [`README.md`](./README.md) are framework-side facts that apply to both routers.
- **Same `@angular/ssr` render pipeline.** `AngularNodeAppEngine.handle(req)` is the public surface; `provideServerRendering(withRoutes(serverRoutes), withAppShell(AppComponent))` is the server bootstrap shape; both routers are consumers of the same engine.
- **Mandatory ceremonial wiring is identical.** `provideRouter([NgRouterStub])` peer (Real-Router) vs `provideRouter(routes)` (Angular's own) — both are calls to `provideRouter`. `withAppShell(AppComponent)` is required for both. `security.allowedHosts: ["localhost"]` is required for both. `server-runner.mjs` Node wrapper or its equivalent for `isMainModule` fragility applies to both. `outputMode: "server"` + `ssr.entry` is required for both.
- **Server runtime portability.** Both run on Node + Express via `AngularNodeAppEngine`; both are theoretically portable to Hono/Bun once those expose compatible request/response shapes.
- **CSR↔SSR code reuse.** Both keep the same route definitions across CSR and SSR; differences live in the `provideServerRendering(...)` server-bootstrap layer.
- **No ISR.** Neither standalone router implements on-demand revalidation; both delegate to the deployment platform.
- **No Real-Router-side advantage on Angular 21 lazy hydration mechanics.** `@defer` triggers, `@placeholder`/`@loading`/`@error` blocks, `withIncrementalHydration()`, `withEventReplay()` — all framework-side; both routers integrate with them by being neutral about template-level concerns.
